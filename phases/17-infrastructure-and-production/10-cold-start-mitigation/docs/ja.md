# サーバーレス LLM のコールドスタート軽減

> 20 GB モデルイメージが 5-10 分（7B）から 20 分以上（70B）かかり、コールドからサービング開始まで。本当のサーバーレス世界では、それはウォームアップではなく、停止です。軽減は 5 つのレイヤーで動作します：事前シードされたノードイメージ（AWS 上の Bottlerocket、デュアルボリュームアーキテクチャ）、モデルストリーミング（NVIDIA Run:ai Model Streamer、vLLM でネイティブ）、GPU メモリスナップショット（Modal チェックポイント、再起動が最大 10 倍高速化）、ウォームプール（`min_workers=1`）、段階的ロード（ServerlessLLM の NVMe→DRAM→HBM パイプライン、10-200 倍のレイテンシ削減）、および入力トークン（KB）ではなく KV キャッシュ（GB）を移動するライブマイグレーション。Modal は 2-4 秒のコールドスタートを床として発行し、Baseten は 5-10 秒のデフォルト、事前ウォーミングでサブ秒。このレッスンは、5 つのレイヤーを測定、予算化、スタックする方法を教えます。

**タイプ:** Learn
**言語:** Python (stdlib、おもちゃのコールドスタートパスシミュレータ)
**前提条件:** Phase 17 · 02（推論プラットフォームエコノミクス）、Phase 17 · 03（GPU オートスケーリング）
**所要時間:** 約 60 分

## 学習目標

- コールドスタート軽減の 5 つのレイヤーを列挙し、各レイヤーで 1 つのツール またはパターンを名前で述べてください。
- 70B モデルの合計コールドスタート時間を（ノード提供）+（重みダウンロード）+（HBM への重み読み込み）+（エンジン初期化）の合計として計算してください。
- ライブマイグレーションが KV キャッシュ（GB）ではなく入力トークン（KB）を転送する理由と、ペナルティ（再計算）を説明してください。
- ウォームプールトレードオフ（アイドル GPU に支払うか、コールドスタート末尾を受け入れるか）と、`min_workers > 0` が必須になる SLA しきい値を名前で述べてください。

## 問題

サーバーレス LLM エンドポイントは夜間にゼロにスケールします。午前 8 時にトラフィックスパイク。最初のリクエストは以下を待ちます：

1. Karpenter が GPU ノードをプロビジョニングします：45-60 秒。
2. コンテナが重み付きの 30 GB イメージをプルします：120-300 秒。
3. エンジンが HBM に重みをロードします：モデルサイズとストレージ速度に応じて 45-120 秒。
4. vLLM または TRT-LLM が CUDA グラフ、KV キャッシュプール、トークナイザーを初期化します：10-30 秒。

合計：220-510 秒（大約 3-8 分）トークンが 1 つ戻ってくるまで。SLA は 2 秒です。ウォームプール（`min_workers=1`）を出荷し、問題は消えたように見えます — ただし、1 つのアイドル GPU に 24x7 を支払います。サービスに 5 つの製品がある場合、各々1 つのウォーム レプリカで、それは 5 × 24 × 30 = 3,600 GPU 時間/月、単一のユーザーが呼んだかどうかに関わらず。

コールドスタート軽減は、サーバーレスエコノミクスを維持しながら、常時稼働のレイテンシに近づく方法です。

## コンセプト

### レイヤー 1 — 事前シードされたノードイメージ（Bottlerocket）

AWS では、Bottlerocket のデュアルボリュームアーキテクチャが OS からデータを分離します。コンテナイメージが事前にプルされた状態でデータボリュームをスナップショットします；`EC2NodeClass` でスナップショット ID を参照してください。新しいノードは重みがローカル NVMe 上に既にある状態でブートします — ステップ 2 とステップ 3 の一部が消えます。Karpenter でネイティブに動作します。典型的な節約：大規模モデルの場合、コールドスタートあたり 2-4 分。

GCP での同等のもの：事前ベークされたコンテナレイヤーを備えたカスタム VM イメージ。Azure では：同じパターンでの管理ディスクスナップショット。

### レイヤー 2 — モデルストリーミング（Run:ai Model Streamer）

完全なファイルをロードしてから最初のリクエストに応答する代わりに、重みを GPU メモリレイヤーごとにストリーミングしてロードし、最初のトランスフォーマーブロックが常駐するとすぐに処理を開始します。NVIDIA Run:ai Model Streamer は vLLM 2026 にネイティブに出荷されます。S3、GCS、ローカル NVMe と連動します。I/O 計算セットアップと重複させることで、大規模モデルの重みロード時間をおおよそ半減します。

### レイヤー 3 — GPU メモリスナップショット（Modal）

Modal は GPU 状態（重み、CUDA グラフ、KV キャッシュ領域）のチェックポイントを取得します。最初のロード後。後続の再起動は HBM に直接逆シリアル化します — 再初期化より 10 倍高速化。これは「ウォーム GPU を 2 秒でブート」に最も近いものです。トレードオフ：スナップショットは GPU トポロジごとなので、Karpenter があなたを別の SKU に移行させた場合、再チェックポイントします。

### レイヤー 4 — ウォームプール（min_workers=1）

最もシンプルな軽減：1 つのレプリカを常に準備しておきます。コストは 1 つの GPU の時給 24x7。算数は小規模モデルでは苛酷です（30 秒のコールドスタートを回避するために $0.85-$1.50/hr を支払う）そして大規模なものには親切です（5 分のコールドスタートを回避するために $4/hr を支払う）。ウォームプールが必須になる SLA しきい値：通常 70B+ モデルでの TTFT P99 < 60 秒。

### レイヤー 5 — 段階的ロード（ServerlessLLM）

ServerlessLLM はストレージを階層として扱います：NVMe（高速だが大きい）、DRAM（中程度だが段階的）、HBM（小さいが即座）。重みは DRAM に事前ロードされ、HBM にオンデマンドロードされます。ペーパーは、ナイーブディスク-to-HBM と比較して、コールドロードのレイテンシを 10-200 倍削減したと報告しています。本番採用は初期ですが、vLLM との統合が存在します。

### レイヤー 6 — ライブマイグレーション（ボーナスパターン）

ノードが使用不可になる場合（スポット削除、ノードドレーン）、従来のパターンは別のレプリカをコールドスタートしてリクエストキューをドレーンすることです。ライブマイグレーションは入力トークン（キロバイト）を、モデルロード済みでコールドスタート可能な宛先に移動し、宛先の KV キャッシュを再計算します。再計算は GB の KV キャッシュをネットワーク経由で転送するよりも安いです。分散デプロイメントに適用可能。

### ウォームプール数学

P99 TTFT SLA が 2 秒のサービスの場合、質問は「ウォームプール はい/いいえ」ではなく「何個のウォームレプリカ、どのパスにそれらを取得するか」です。

- 高価値インタラクティブパス（ライブチャット、音声エージェント）：`min_workers=1-2`。
- バックグラウンドバッチパス（夜間分類）：スケールツーゼロ許容、5-10 分のコールドスタート許容。
- プレミアムティア：テナントあたりの `min_workers` 専用容量付き。

### 最適化前に測定

新しいノード上の 70B モデルのコールドスタート解剖（説明目的）：

| フェーズ | 時間 | 軽減 |
|-------|------|-----------|
| ノード提供 | 50 秒 | Bottlerocket + 事前シードイメージ、ウォームプール |
| イメージプル | 180 秒 | 事前シードデータボリューム（削除） |
| HBM への重み | 75 秒 | モデルストリーマー（半減）；GPU スナップショット（削除） |
| エンジン初期化 | 20 秒 | 永続的 CUDA グラフキャッシュ |
| 最初のフォワード | 3 秒 | 最小固有レイテンシ |
| **合計コールド** | **328 秒** | |
| **軽減付き合計** | **~15 秒** | 22 倍削減 |

### 覚えるべき数字

- Modal コールドスタート：2-4 秒（GPU スナップショット付き）。
- Baseten デフォルトコールドスタート：5-10 秒；事前ウォーミング付きサブ秒。
- 生の 70B コールドスタート：3-8 分。
- Run:ai Model Streamer：約 2 倍重みロード高速化。
- ServerlessLLM 段階的ロード：10-200 倍レイテンシ削減（ペーパー数字）。

## 使用方法

`code/main.py` は、各軽減ありなしのコールドスタートパスをモデルしています。合計コールドスタート時間、ウォームプール コスト、およびウォームプール がそれ自体に支払う break-even リクエストレートを報告します。

## 出荷方法

このレッスンは `outputs/skill-cold-start-planner.md` を生成します。SLA、モデルサイズ、トラフィック形状が与えられた場合、どの軽減をスタックするかを選択します。

## 演習

1. `code/main.py` を実行してください。ウォームレプリカが SLA での追加リクエストドロップに対する冷却スタート税の支払いより安くなる break-even リクエストレートを計算してください。
2. 13B モデルを P99 TTFT SLA 3 秒でデプロイします。それを達成する最小軽減スタック（最も少ないレイヤー）を選択してください。
3. Bottlerocket 事前シード化はイメージプルを削除しますが、重みは依然としてスナップショットから HBM にロードされます。スナップショット支援 NVMe が 7 GB/s で読み取る場合、70B モデルの壁時計を計算してください。
4. サーバーレスプロバイダーは GPU スナップショット（Modal）を提供し、チームは「スナップショットが PII をリークする」ため拒否します。両側を議論してください — 現実的なリスクは何か、軽減は何ですか（一時的なスナップショット、暗号化、ネームスペース分離）？
5. 段階的ウォームプール ポリシーを設計してください：有料ユーザー、試用ユーザー、バッチワークロード向けの何個のウォームレプリカ？数学を示してください。

## 主要用語

| 用語 | 人々の言い方 | 実際の意味 |
|------|----------------|------------------------|
| コールドスタート | "大きな一時停止" | 新しいレプリカでリクエストから最初のトークンまでの時間 |
| ウォームプール | "常時最小値" | 最少 1 つのレプリカを準備しておく `min_workers >= 1` |
| 事前シードイメージ | "ベークされた AMI" | コンテナ重みが事前常駐するノードイメージ |
| Bottlerocket | "AWS ノード OS" | デュアルボリュームスナップショットサポート付き AWS コンテナ最適化 OS |
| モデルストリーマー | "ストリーミングロード" | 重み I/O を計算セットアップと重複させる |
| GPU スナップショット | "HBM へのチェックポイント" | 事後ロード GPU 状態をシリアル化；再起動時に逆シリアル化 |
| 段階的ロード | "NVMe + DRAM + HBM" | ストレージティアの階層；オンデマンドロード |
| ライブマイグレーション | "トークンを移動" | 入力（KB）を転送、宛先で KV を再計算 |
| `min_workers` | "ウォームレプリカ" | サーバーレス最小キープアライブカウント |
| スケールツーゼロ | "フル サーバーレス" | アイドル時コスト無し；完全コールドスタート税を受け入れる |
