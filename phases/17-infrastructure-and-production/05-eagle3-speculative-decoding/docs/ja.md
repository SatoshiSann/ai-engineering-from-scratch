# 本番環境での EAGLE-3 推測デコード

> 推測デコードは高速ドラフトモデルとターゲットモデルをペア。ドラフトが K トークンを提案。ターゲットが 1 つのフォワードで検証。受理トークンは無料。2026 年、EAGLE-3 は本番ノート — ドラフト頭がトークンではなくターゲットモデルの隠れ状態で訓練、受理率アルファを一般的チャットで 0.6-0.8 帯に押す。正しい質問は「ドラフト はどれだけ高速か」ではなく「私のトラフィックのアルファは何か？」。アルファが 0.55 以下に落ちたら、高同時度で推測デコードは各 削除ドラフトがターゲットフォワードパスを秒で費用がかかるため net-negative。このレッスンはアルファを最初に計測し、フラグを 2 番目に反転する方法を教える。

**タイプ:** 学習
**言語:** Python（stdlib、簡単な受理率シミュレーター）
**前提条件:** フェーズ 17 · 04（vLLM サービング インターナル）、フェーズ 10 · 18（マルチトークン予測）
**所要時間:** 約 60 分

## 学習目標

- 推測デコードの 3 つの世代に名前を付け、EAGLE-3 が EAGLE-2 とクラシックドラフトモデルから何が変わるかを説明。
- 受理率アルファを定義、アルファと K（ドラフト長）からの期待スピードアップを計算、目標同時度の損益分岐点アルファを特定。
- vLLM 2026 でなぜ推測デコードはオプトイン（デフォルトでない）か説明し、アルファを計測せずに有効にするが本番アンチパターンである理由。
- 計測計画を書く：どのベンチマーク、どのプロンプト分布、どの同時度ポイント、どのメトリクスをゲート。

## 問題

デコードはメモリ境界。H100 で Llama 3.3 70B FP8 を実行、各デコードトークン は約 140 GB/s の重みを読み、1 トークンを出力。GPU コンピュートはデコード中ほぼアイドル — ボトルネックは HBM 帯域幅、matmul スループット ではなく。

推測デコードはギャップを活用。安いドラフトモデルで K 候補トークンを生成、ターゲットモデルにすべて K を 1 つのフォワードで検証するよう依頼。各検証トークンは実質無料（ターゲットが無いほうにしなければならなかった K バッチフォワードに償却）。

古典的なドラフトモデルアプローチは同じ家族の小さいモデルを使用（Llama 3.3 70B をドラフト用 Llama 3.2 1B）。機能しますが受理率は中程度 — より小さいモデル分布はターゲットから発散。EAGLE、次に EAGLE-2、次に EAGLE-3 は軽いドラフト頭をターゲットモデルの内部状態で直接訓練、ドラフト分布がターゲットははるかに密接に追跡。アルファが 0.4 から 0.6-0.8 に行く理由。

キャッチ：EAGLE-3 は vLLM 2026 でオプトイン。`speculative_config` は明示的に設定する必要。フラグなし、加速なし。実際のトラフィックでアルファを計測することなく有効にするチームはしばしば、テイル遅延がより低くなるではなくより良くなるを見ます。

## コンセプト

### 推測デコードが実際にもたらすもの

推測デコードなし、トークン当たりのコストは 1 つのターゲットフォワード。推測デコード at ドラフト長 K と受理アルファで、期待トークン/ターゲットフォワードは `1 + K * alpha`。スピードアップは `(1 + K * alpha) / (1 + epsilon)` ここで epsilon はドラフト・プラス・検証オーバーヘッド。K=5、alpha=0.7：`(1 + 5*0.7) / (1 + 0.1) = 4.5 / 1.1 = 4.1x`。実数はクラスタ周り 2-3x（アルファはトラフィックで稀にその高い本番と epsilon はバッチサイズで成長）。

### アルファが唯一重要なメトリック である理由

削除トークンは消えない — ターゲットフォワード（再ロール）を最初の削除トークン用に強制。アルファが 0.4 に落ちるワークロード、ドラフトオーバーヘッド + 検証 + 再ロール を支払う。高同時度（例 256 同時）、デコード バッチは既に大きく、「ターゲットのみ」と「ターゲット・プラス・検証」間のメモリ帯域幅ギャップはシュリンク。ほとんどの 2026 ハードウェアで alpha 0.55 以下で推測デコードは net-negative。

アルファはワークロード別。ShareGPT スタイル一般チャットで、ShareGPT で訓練した EAGLE-3 は 0.6-0.8。ドメイン特定トラフィック（コード、医療、法）、一般データで訓練したドラフト頭は 0.4-0.6。ドメイン特定ドラフト頭を訓練はアルファを復元 — ターゲットファインチューニングに比べて軽い、速い訓練ジョブ。

### EAGLE 世代一目

- **古典的ドラフトモデル**：同じ家族の小さいモデル。アルファ 0.3-0.5。インフラシンプル — 2 つのモデルロード、ドラフトは目標フォワードごと K フォワード実行。
- **EAGLE-1（2024）**：ターゲット隠れ状態（最後のレイヤー）で訓練した単一ドラフト頭。アルファ 0.5-0.6。ターゲットの小さいパラム オーバーヘッド。
- **EAGLE-2（2025）**：適応ドラフト長と木ベースドラフト（1 つのターゲットパスで複数ブランチ検証）。アルファ 0.6-0.7。複雑なドラフト スケジューラー。
- **EAGLE-3（2025-2026）**：複数ターゲットレイヤー（最後のみではなく）で訓練したドラフト頭、より良い調整。アルファ 0.6-0.8 一般チャット。

### 2026 年本番レシピ

1. ターゲットモデル無地をシップ。ターゲット同時度での TTFT、ITL、スループット を測定。
2. vLLM `speculative_config` 経由で EAGLE-3 ドラフト を有効にする。ベンチマークを再実行。
3. 受理率アルファ をログ。vLLM V1 は `spec_decode_metrics.accepted_tokens_per_request` として報告。要求ドラフト長で除算してアルファを得る。
4. 本番トラフィック分布で alpha < 0.55 な ら、推測デコード を無効 or ドメイン特定 EAGLE-3 ドラフトを訓練。
5. 本番同時度で再実行。P99 ITL がより悪くならなかったことを確認。

### 本番落とし穴：P99 テイル

平均 ITL は推測デコードで落ちる。P99 はチューニングなしで悪化の可能性。削除ドラフト は 2 パスシーケンス（ドラフト + 検証失敗 + 再ロール）をトリガー。フルバッチ下、2 つのパスはシリアライズ。P99 ITL、P50 を監視しない。

### EAGLE-3 が既にデプロイされるところ

Google は 2025 年に AI Overviews で推測デコードをデプロイ（同じ品質、高速応答）。vLLM V1 は `speculative_config` をドキュメント化されたインターフェイスとしてシップ。V1 の N-gram GPU 推測デコードは チャンク化プリフィル 互換変種。SGLang は プリフィクス 重いワークロード用推奨ドラフト パスとして EAGLE-3 をサポート。

### 1 行での損益分岐点算数

期待スピードアップ：`S(alpha, K) = (1 + K*alpha) / (1 + verify_overhead)`。`S = 1` で解く：`alpha_breakeven = verify_overhead / K`。典型的な verify_overhead 約 0.15、K=5：`alpha_breakeven = 0.03`。しかしそれは生のデコード算数。高同時度で検証オーバーヘッド上昇、デコード バッチ既にシーケンス間メモリ読み償却、有効 alpha_breakeven は実行で 約 0.45-0.55 に登ります。

### 推測デコードを使用しない場合

- レイテンシが重要ない バッチ-1 オフライン生成。無地ターゲット使用。
- 非常に短い出力（50 トークン以下）。ドラフト オーバーヘッド と検証コスト優位。
- ドメイン特定なし、ドメイン訓練ドラフト頭。アルファ 低すぎる。
- vLLM v0.18.0 + ドラフト モデル推測デコード + `--enable-chunked-prefill`。この組み合わせはコンパイル しない。ドキュメント例外は V1 の N-gram GPU spec decode。

## 使用

`code/main.py` はアルファ値とドラフト長 K の範囲を通じてデコード ループをシミュレート（推測デコード あり/なし）。損益分岐点アルファ、測定スピードアップ、テイル動作を印刷。いくつかの（アルファ、K）組み合わせで実行して、推測デコードが停止し支払うところを見る。

## 配布

このレッスンは `outputs/skill-eagle3-rollout.md` を生成。ターゲットモデル、トラフィック分布説明、同時度目標を考えると、段階化 EAGLE-3 ロールアウト計画を生成 — ベースライン ベンチマーク、有効設定、計測アルファ、アルファ >= 0.55 でゲート、P99 ITL 監視。

## 演習

1. `code/main.py` を実行。K=5、2x スピードアップに何のアルファが必要か？3x スピードアップ？それは verify_overhead にどれだけ敏感か？
2. 本番トラフィック分割 70% 一般チャット、30% コード。一般チャットはShareGPT で訓練した EAGLE-3 で alpha 0.7、コード は alpha 0.4。ブレンド アルファは？推測デコードは net-positive か？
3. vLLM `speculative_config` ドキュメント読む。3 つのモード（ドラフト モデル、EAGLE、N-gram）に名前付け、チャンク化プリフィル どれが互換。
4. 有効にした後、平均 ITL が 25% に落ちたが P99 ITL が 15% 上昇を見る。診断と軽減を提案。
5. Llama 3.3 70B の EAGLE-3 ドラフト頭のメモリコストを計算。Llama 3.2 1B とクラシック ドラフトでの実行と比較。

## 重要用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|-------------|----------|
| 推測デコード | 「ドラフト・プラス・検証」 | 安いモデルで K トークン提案、1 つのターゲットフォワードで全K 検証 |
| 受理率アルファ | 「spec accept rate」 | ターゲットが受理するドラフトトークンの分数。唯一重要なメトリック |
| ドラフト長 K | 「spec k」 | ドラフトがターゲットフォワード ごと提案するトークン。典型的 4-8 |
| 検証オーバーヘッド エプシロン | 「spec overhead」 | 検証・再ロール vs 無地ターゲット フォワードへの追加コスト。バッチで成長 |
| EAGLE-3 | 「latest EAGLE」 | 2025-2026 変種。複数ターゲット レイヤーでドラフト頭を訓練。アルファ 0.6-0.8 一般チャット |
| `speculative_config` | 「vLLM spec config」 | vLLM V1 の明示的 opt-in。デフォルト無し は加速なし |
| N-gram spec decode | 「N-gram draft」 | GPU 側ドラフト、プロンプトの N-gram ルックアップ使用。チャンク化プリフィル 互換 |
| 損益分岐点アルファ | 「no-op alpha」 | 推測デコードが ゼロスピードアップを与える アルファ。本番同時度で監視。
| 削除ドラフト 2 パス | 「reroll cost」 | ドラフト削除時 2 つのターゲットフォワード。P99 テイル駆動 |
