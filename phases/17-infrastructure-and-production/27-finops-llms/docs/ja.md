# LLM 向け FinOps — ユニット経済学とマルチテナント帰属

> 従来の FinOps は LLM スペンドで崩壊。コストはリソースアップタイム不動産ではなくトークントランザクション。タグはマップされない — API 呼び出しはアセットではなくトランザクション。エンジニアリング決定（プロンプト設計、コンテキスト ウィンドウ、出力長）は財務決定。2026年の手順は初日にインストルメント化する 3 つの帰属次元を持つ：ユーザーごと（`user_id`）シート価格と拡張用、タスクごと（`task_id` + `route`）製品サーフェスコストと優先順位付け用、テナントごと（`tenant_id`）ユニット経済学とリニューアル用。4 つのトークン層 — プロンプト、ツール、メモリ、レスポンス — 1 つのバケットがスペンドを隠す。マルチテナント製品の施行ラダー：テナントごとレート制限（予想ピークの 2-3 倍、クリア 429 + リトライ後）、日次スペンド上限（契約上限の 1.5-3 倍、レート締付け + アラート発動）、z スコア > 4 でのキルスイッチ（自動ポーズ + オンコール ページング）。帰属パターン：タグ&集計、テレメトリジョイナー（トレース ID → 請求、最高精度）、サンプリング&外挿、モデルベース割当、イベントソース化、リアルタイムストリーミング。ユニット メトリック：解決済みクエリごとのコスト、生成アーティファクトごとのコスト — $/M トークンではない。事後的タグ付けは常にミス、リクエスト作成時にインストルメント化。

**タイプ:** Learn
**言語:** Python（stdlib、キルスイッチ付きオモチャコスト帰属シミュレータ）
**前提条件:** Phase 17 · 13 (Observability)、Phase 17 · 14 (Caching)
**所要時間:** ~60分

## 学習目標

- 従来の FinOps（タグ + 階級）が LLM スペンドで崩壊する理由を説明し、3 つの新しい帰属次元の名前を述べる。
- 4 つのトークン層（プロンプト、ツール、メモリ、レスポンス）を列挙し、単一バケット請求がなぜコストを隠すかを説明。
- マルチテナント製品の施行ラダー（レート → スペンド上限 → キルスイッチ）を設計。
- $/M トークンではなくユニット メトリック（解決済みクエリあたりコスト / アーティファクト）を選択。

## 問題

請求書が $40,000。以下は分からない：
- どのテナントがそれを使った。
- どの製品機能がそれを駆動。
- 個別ユーザーが悪用的か。
- プロンプト ブロート、ツール呼び出し、メモリ増幅が原因か。

タグ&集計はプロバイダー側で雲リソース（EC2、S3）に機能する。タグが明細項目に伝播。LLM API 呼び出しは自動タグされない — ユーザー/タスク/テナントを呼び出しサイトにスタンプして伝播する必要。事後的帰属は常にエッジケースをミス。

## コンセプト

### 3 つの帰属次元

**ユーザーごと** (`user_id`)：誰がコストする。シート価格、拡張対話、パワーユーザーを識別するドライブ。

**タスクごと** (`task_id` + `route`)：どの製品サーフェスがコストするか。機能優先順位付け、高コスト機能削除決定をドライブ。

**テナントごと** (`tenant_id`)：どの顧客が利益か。ユニット経済学、リニューアル価格、階級閾値をドライブ。

呼び出しサイトで初日にすべて 3 つをインストルメント化。事後的は常により悪い。

### 4 つのトークン層

| 層 | 例 | 典型 % 合計 |
|-------|---------|---------------------|
| プロンプト | システム + ユーザー入力 | 40-60% |
| ツール | ツール呼び出し結果をフィードバック | 20-40%（エージェント ワークロード） |
| メモリ | 前の対話 / 取得ドキュメント | 10-30% |
| レスポンス | モデル出力 | 10-30% |

4 つすべてをバケットすると最適化は見えない。帰属スキーマで分割。

### 施行ラダー

1. **レート制限** テナントごと。予想ピークの 2-3 倍。429 と `Retry-After` で返す。テナントが摩擦を見、驚き請求なし。

2. **日次スペンド上限** テナントごと。契約上限の 1.5-3 倍。トリガー：レート制限を締付け + 顧客成功にアラート。

3. **キルスイッチ** テナント ベースラインに対する z スコア > 4 でのスペンド。テナントを自動ポーズ、オンコール ページング、オプス + CS にエスカレート。

### 帰属パターン

- **タグ&集計**：メタデータ ヘッダーをスタンプ、後で集計。シンプル、粗。
- **テレメトリ ジョイナー**：トレース ID で請求にトレース結合。最高精度。成熟チームが行う。
- **サンプリング&外挿**：5-10% をサンプリング、乗算。粗スペンド用の費用効率的、テール をミス。
- **モデルベース割当**：回帰でコスト駆動を推測。タグなし従来データ用。
- **イベントソース化**：イベント ストリーム内のコスト（Kafka / Kinesis）。リアルタイム。
- **リアルタイム ストリーミング**：ダッシュボード更新サブ秒。

### ユニット メトリックは Cost per X

$/M トークンはベンダー話。製品メトリック：

- 解決済みサポート チケットごとのコスト。
- 生成記事ごとのコスト。
- 成功したエージェント タスクごとのコスト。
- ユーザーセッション分当たりのコスト。

コストを製品成果にタイ。さもなくば最適化はアンカーなし。

### コスト帰属トレース形状

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

すべての呼び出しで発信。データ レイクに保存。次元ごとに集計。Phase 17 · 13 観測スタックがここにある。

### 複合セービング スタック

スタック：キャッシュ + バッチ + ルート + ゲートウェイ。4 つすべてで：
- L2 キャッシュ（Phase 17 · 14）：~10 倍安い入力。
- バッチ（Phase 17 · 15）：50% 割引。
- 安いモデルへのルート（Phase 17 · 16）：60% コスト削減。
- ゲートウェイ効率（Phase 17 · 19）：冗長性 + リトライ。

ベストケース スタック：~ナイーブ ベースラインの 5-10%。ほとんどのチームは 2-3 のレバー、数チームがすべて 4 つをスタック。

### 覚えるべき数字

- 帰属次元：ユーザーごと、タスクごと、テナントごと。
- 4 つのトークン層：プロンプト、ツール、メモリ、レスポンス。
- キルスイッチ：z スコア > 4。
- ユニット メトリック：解決済みクエリ当たりコスト、$/M トークンではない。
- スタック最適化：~ベースラインの 5-10% 可能。

## これを使用

`code/main.py` はマルチテナント LLM サービスで 3 階級施行ラダーをシミュレート。虐待テナントを注入してキルスイッチ発火をデモ。

## 配送

このレッスンは `outputs/skill-finops-plan.md` を生成。製品とスケール指定で、帰属スキーマと施行ラダーを設計。

## 演習

1. `code/main.py` を実行。キルスイッチはどの z スコアで発火？閾値をどう選ぶ？
2. テナント毎、タスク毎コスト ダッシュボードを設計。まず構築する 5 つのビューは何？
3. 最大テナントがユニット経済学負。3 つの介入を顧客インパクト順で提案。
4. サポート製品の解決済みチケット当たりコストを計算：3M トークン/チケット、~800 チケット/日、GPT-5 キャッシュ率。
5. 事後的タグ付けが機能できるかを議論。いつが許容可能か？

## キー用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|----------------|--------|
| ユーザーごと帰属 | 「ユーザーレベル コスト」 | すべての呼び出しにスタンプされた `user_id` |
| タスクごと帰属 | 「機能コスト」 | 製品サーフェスを識別する `task_id` + `route` |
| テナントごと帰属 | 「顧客コスト」 | `tenant_id`、ユニット経済学をドライブ |
| 4 つのトークン層 | 「コスト層」 | プロンプト + ツール + メモリ + レスポンス |
| レート制限 | 「429 ガード」 | ゲートウェイで施行されるテナント毎上限 |
| 日次スペンド上限 | 「日次上限」 | アラート付きテナントスコープ予算 |
| キルスイッチ | 「自動ポーズ」 | z スコア > 4 でのスペンドが自動中止をトリガー |
| 解決済み当たりコスト | 「製品ユニット メトリック」 | トークンではなく製品成果にタイされたコスト |
| テレメトリ ジョイナー | 「トレース-請求」 | 最高精度帰属パターン |
| スタック最適化 | 「キャッシュ+バッチ+ルート+ゲートウェイ」 | ~5-10% ベースラインへの複合セービング |
