# マルチリージョン LLM サービングと KV キャッシュ局所性

> ラウンドロビン ロードバランシングはキャッシュされた LLM 推論に対して能動的に有害です。その接頭辞を保有するノードに着陸しないリクエストは完全なプリフィル コストを支払います — 長いプロンプトの P50 で約 800 ミリ秒対キャッシュ ヒットで約 80 ミリ秒。2026 年の本番パターンは、KV キャッシュイベントを消費し、接頭辞ハッシュ マッチでルーティングするキャッシュ認識ルータ（Rust 内の vLLM Router、llm-d router）です。最近の研究（GORGO）は、クロスリージョン ネットワーク レイテンシをルーティング目標の明示的な項として作成します。商用「クロスリージョン推論」オファリング（Bedrock クロスリージョン推論、GKE マルチクラスタ ゲートウェイ）は推論を不透明として扱います — 可用性を処理し、TTFT ではありません。JPMorgan と Mayo Clinic は 2024 年 11 月に us-east-1 フェイルオーバーを約 22 分で実行しました。DR 現実：LLM DR 障害の 32% はチームが重みはバックアップしたが tokenizer ファイルまたは量子化コンフィグを忘れたためです。

**タイプ:** Learn
**言語:** Python (stdlib、おもちゃの接頭辞キャッシュ認識ルータシミュレータ)
**前提条件:** Phase 17 · 04（vLLM Serving）、Phase 17 · 06（SGLang RadixAttention）
**所要時間:** 約 60 分

## 学習目標

- ラウンドロビン ロードバランシングがキャッシュされた推論を破壊する理由を説明し、TTFT ペナルティを定量化してください。
- キャッシュ認識ルータを図にしてください：入力（KV キャッシュイベント）、アルゴリズム（接頭辞ハッシュ マッチ）、タイブレーカー（GPU 使用率）。
- 32% DR 障害ドライバー（欠落 tokenizer ファイル/量子化コンフィグ）を名前で述べ、3 ファイル DR チェックリストを述べてください。
- 商用クロスリージョン オファリング（Bedrock CRI、GKE Multi-Cluster Gateway）をキャッシュ対応ルータと区別してください。

## 問題

サービスは us-east-1、us-west-2、eu-west-1 で実行されます。ALB を前に置き、ラウンドロビン。本番環境の接頭辞キャッシュ ヒット率は 8% に低下します。TTFT P50 は 3 倍になります。vLLM ログはすべてのリクエストが完全なプリフィル コストを支払っていることを示します。

ラウンドロビン はステートレスサービスに最適です。LLM 推論はデザインでステートフルです — KV キャッシュはモデルが見たすべてをエンコードします。ブラインドでルーティングは間違ったキャッシュへのルーティングです。

別に、チームには DR プランがあります。モデル重みを S3 クロスリージョンにバックアップします。リージョン停止が発生；フェイルオーバーを試みる；レプリカが開始を拒否します。tokenizer.json、量子化コンフィグ、RoPE スケーリング コンフィグを忘れたため、異なるバケットにありました。

マルチリージョン LLM サービングはキャッシュ問題、ルーティング問題、DR 衛生問題です — ロードバランサー問題ではありません。

## コンセプト

### キャッシュ認識ルーティング

リクエストはプロンプトで到着します。ルータは接頭辞をハッシュします（例えば、最初の 512 トークン）；各レプリカに「この接頭辞はキャッシュされていますか？」と尋ねます。レプリカは KV キャッシュイベントをパブサブ チャネルで発行して、ブロックを割り当て、削除します。ルータはマッチするレプリカを選択し、誰もマッチしない場合、GPU 使用率ベースのタイブレーカーにフォールスルーします。

**vLLM Router**（Rust、2026 本番スタック）：`kv.cache.block_added` イベントをサブスクライブし、接頭辞ハッシュ → レプリカ インデックスを保守し、O(1) ルックアップでルーティング。マッチなしのときは最小キュー深度にフォールスルー。

**llm-d router**：同じパターン、Kubernetes ネイティブ。ControlPlane API 経由でイベントを発行します。

**SGLang RadixAttention**（Phase 17 · 06）はレプリカ内同等物です。クロスレプリカ ルーティングは厳密に上流です。

### 数字

TTFT P50 2K トークン プロンプト、Llama 3.3 70B FP8、H100 上：
- キャッシュ ヒット（同じレプリカ、接頭辞常駐）：約 80 ミリ秒。
- キャッシュ ミス（コールド プリフィル）：約 800 ミリ秒。

10 倍ギャップ。レプリカ全体で接頭辞キャッシュの 60-80% にヒットするルータを持つ場合、N レプリカ容量のシングルレプリカ パフォーマンスを近似します。10% にヒットする場合、ナイーブ スケーリングを近似します。

### クロスリージョンは新しい制約を持っています — ネットワーク レイテンシ

リージョン間 RTT：
- us-east-1 ↔ us-west-2：約 65 ミリ秒。
- us-east-1 ↔ eu-west-1：約 75 ミリ秒。
- us-east-1 ↔ ap-southeast-1：約 220 ミリ秒。

ルーティングが us-east-1 からのリクエストを ap-southeast-1 の ホット接頭辞に受け取る場合、保存されたプリフィル（800 → 80 ミリ秒）は 440 ミリ秒のラウンドトリップ に劇的に矮小化されます。GORGO（2026 研究）はこれを明示的に作成します — `prefill_time + network_latency` を共同に最小化、プリフィルのみではなく。多くの場合、答えはほぼマルチ MB 接頭辞でプリフィルが支配的な場合を除き、リージョナル以外のルーティングを保つことです。

### 商用「クロスリージョン推論」はここで役に立たない

AWS Bedrock クロスリージョン推論は容量圧力中にリクエストを自動的に他のリージョンにルーティングします。TTFT ではなく可用性を最適化し、推論を不透明として扱います。GKE Multi-Cluster Gateway は同じです — サービスレベル フェイルオーバー、KV キャッシュの意識なし。

これらを使用する場合でも、アプリレイヤー キャッシュ認識ルータが必要です。「us-east-1 は火事です」ケースを処理します。キャッシュ認識ルーティングは TTFT ケースを処理します。

### DR 衛生 — 32% 欠落ファイル問題

広く引用された 2026 統計：LLM DR 障害の 32% はチームが重みをバックアップしたが以下を忘れたため発生します：

- `tokenizer.json` または `tokenizer.model`
- 量子化コンフィグ（`quantize_config.json`、AWQ スケール、GPTQ ゼロポイント）
- モデル固有コンフィグ（RoPE スケーリング、アテンション マスク、チャット テンプレート）
- エンジン コンフィグ（`vllm_config.yaml`、サンプリング デフォルト、LoRA アダプター マニフェスト）

修正は最小 3 ファイル DR マニフェスト：

1. HF モデル レポ下のすべてのファイル（重み + コンフィグ + tokenizer）。
2. エンジン固有のサービング コンフィグ。
3. デプロイメント マニフェスト（K8s YAML、Dockerfile、依存ロック）。

さらに：四半期ごとに DR ドリルを実行します。JPMorgan us-east-1 ドリルは、プレイブックが習得されたため、2024 年 11 月にのみ 22 分の回復に当たりました。

### データレジデンス は直交

EU 顧客 PHI は EU を離れることはできません。キャッシュ認識ルータがパリ オリジン リクエストを接頭辞マッチで us-east-1 に送信する場合、TTFT 利得に関わらず GDPR に違反しました。ル ータをキャッシュの最適化前にレジデンス境界でパーティション化してください。

### 覚えるべき数字

- キャッシュ ヒット対ミス TTFT ギャップ：約 10 倍（2K プロンプトで 80 ミリ秒対 800 ミリ秒）。
- リージョン間 RTT US-EU：約 75 ミリ秒。
- DR 障害：32% は tokenizer/quant コンフィグを欠落。
- JPMorgan us-east-1 フェイルオーバー 2024 年 11 月：22 分（30 分 SLA）。

## 使用方法

`code/main.py` は、3 つのルーティング戦略（ラウンドロビン、キャッシュ認識リージョナル、キャッシュ認識グローバル）をマルチリージョン ワークロードでシミュレートします。キャッシュ ヒット率、TTFT P50/P99、クロスリージョン 請求書を報告します。

## 出荷方法

このレッスンは `outputs/skill-multi-region-router.md` を生成します。リージョン、レジデンス制約、SLA が与えられた場合、ルーティング計画を設計します。

## 演習

1. `code/main.py` を実行してください。75 ミリ秒 RTT が与えられた場合、ローカルのみのルーティングより クロスリージョン ルーティングが勝つプロンプト長はどこですか？
2. キャッシュ ヒット率は 70% から 12% に低下します。3 つの可能な原因を診断し、各々を確認する観測可能なものを述べてください。
3. vLLM で 5 つの LoRA アダプターで提供される 70B AWQ 量子化モデル向けの DR マニフェストを設計してください。あらゆるファイルと コンフィグをリストしてください。
4. 厳密な TTFT SLO を持つ fintech に対して Bedrock クロスリージョン推論が「十分」かどうかを議論してください。具体的な動作を引用してください。
5. パリオリジン リクエストが us-east-1 の接頭辞にマッチします。ルーティングしますか？ポリシーを書いてください。

## 主要用語

| 用語 | 人々の言い方 | 実際の意味 |
|------|----------------|------------------------|
| キャッシュ認識ルーティング | "スマート LB" | KV キャッシュ保有レプリカへの接頭辞ハッシュ マッチでルーティング |
| KV キャッシュイベント | "キャッシュ パブサブ" | レプリカはブロック追加/削除を発行；ルータはインデックス化 |
| 接頭辞ハッシュ | "キャッシュ キー" | ルータ ルックアップとして使用される最初 N トークンのハッシュ |
| GORGO | "クロスリージョン ルーティング 研究" | arXiv 2602.11688；ネットワーク レイテンシを明示的な項として |
| クロスリージョン推論 | "Bedrock CRI" | AWS 製品；可用性フェイルオーバー、TTFT 認識ではない |
| DR マニフェスト | "バックアップ リスト" | 復元に必要なあらゆるファイル — 重みのみではない |
| データレジデンス | "GDPR 境界" | ユーザー データが見ることができるリージョンに関する法的制約 |
| RTT | "ラウンドトリップ時間" | ネットワーク レイテンシ；75 ミリ秒 US-EU、220 ミリ秒 US-APAC |
| LLM 認識 LB | "キャッシュヒット LB" | 製品カテゴリとしてのキャッシュ認識ルータ |
