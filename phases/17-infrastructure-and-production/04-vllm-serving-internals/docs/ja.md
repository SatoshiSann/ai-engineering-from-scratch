# vLLM サービング インターナル：PagedAttention、連続バッチング、チャンク化プリフィル

> vLLM の 2026 年における優位性は単一のトリックでなく 3 つの複合デフォルトに基づいています。PagedAttention は常時オン。連続バッチングは各デコード反復間に新しいリクエストをアクティブバッチに注入。チャンク化プリフィルは長いプロンプトをスライスし、デコードトークンが飢える ことがない。3 つすべてをオンにし、Llama 3.3 70B FP8 on H100 SXM5 が 128 同時で 2,200-2,400 tok/s をプッシュ — vLLM 独自のデフォルトより約 25% 高く、素朴な PyTorch ループより 3-4 倍。このレッスンはスケジューラーと注意カーネルを図解できるレベルで読み、`code/main.py` の簡単な連続バッチャーで終わります（vLLM が行うようにプリフィルとデコードをスケジュール）。

## 学習目標

- PagedAttention を KV キャッシュアロケータとして説明：ブロック、ブロックテーブル、本番負荷で断片化が 4% 未満に留まる理由。
- 反復レベルで連続バッチングを図解：完了したシーケンスがバッチを抜け、ドレインせずに新しいものが参加する方法。
- チャンク化プリフィルを 1 文で説明し、それが保護するレイテンシメトリック（ヒント：TTFT テイル、平均スループットではない）に名前を付け。
- 2026 年 vLLM v0.18.0 のゴッチャに名前を付け（すべての最適化を一度にオンにするチーム）。

## 問題

素朴な PyTorch サーブループは 1 度に 1 リクエスト：トークン化、プリフィル、EOS までデコード、返す。1 ユーザーでは機能。100 ユーザーでは忍耐強い人のキュー。明らかな修正 — 静的バッチング — すべてのリクエストをウィンドウで最長プロンプトまでパッド、すべてのデコードを最長期待出力までパッド、最も遅いシーケンスでバッチ全体を失速させます。使用しないパッディングに支払い、高速リクエストは遅いものを待ちます。

vLLM は 3 つの問題を一度に解決。PagedAttention は KV キャッシュ断片化が古典的な連続割り当てのように HBM の 60-80% を食うのを防止。連続バッチングはリクエストが各デコード反復間にバッチに参加・抜け可能 — バッチは常に実作業で満杯。チャンク化プリフィルは 32k トークンプロンプトを約 512 トークンスライスに分割してデコード と相互使用 — 長いプロンプトが GPU のすべてのデコードトークンをフリーズしない。

2026 年本番デフォルトはすべて 3 つ。各々が何をするかを理解する必要、障害モードはすべてスケジューラーにあり、モデルではなく。

## コンセプト

### KV キャッシュアロケータとしての PagedAttention の仮想メモリ システム

KV キャッシュは `num_layers × 2 × num_heads × head_dim × seq_len × bytes_per_element` をシーケンスごと。Llama 3.3 70B at 8192 トークンで、それは BF16 でおよそ 1.25 GB/シーケンス。すべてのリクエストに 8192 スロットをプリ予約しますが平均リクエストは 1500 トークンのみ使用なら、予約した HBM の約 82% を浪費。古典的なバッチングこの廃棄を支払う。

PagedAttention は OS 仮想メモリから考えを借ります。KV キャッシュはシーケンスごとに連続ではない。固定サイズブロック（デフォルト 16 トークン）で割り当て。各シーケンスは論理トークン位置を物理ブロック ID にマップするブロックテーブルを持つ。シーケンスが割り当てブロックを超えて成長すると、1 つ以上のブロックが追加。完了すると、ブロックプールに戻る。

断片化は 60-80%（古典的）から 4% 未満（PagedAttention）に落ちる。PagedAttention をフラグで有効にしない — それは vLLM が配布する唯一のアロケータ。ノブは `--gpu-memory-utilization`（デフォルト 0.9）、重みと活性化のロード後に KV ブロック用に予約する HBM の量を vLLM に伝える。

### 反復レベルでの連続バッチング

古い「動的バッチング」はウィンドウ（例 10 ms）をバッチ満杯まで待機し、すべてのシーケンスが完了するまでプリフィル + デコード + デコード + デコード を実行。高速シーケンスは早期に出て、GPU が遅いものを完了する間、アイドル。

連続バッチングは各デコードステップ間で動作。実行中シーケンスのセットを `RUNNING` リストと呼ぶ。各反復：

1. `RUNNING` の EOS か max_tokens に到達したシーケンスは削除。
2. スケジューラーは待機キューを見る。空いている KV ブロックがあれば、新しいシーケンス（プリフィルまたは再開）を許可。
3. フォワードパスは現在 `RUNNING` の何でも実行、各シーケンスごと 1 つの新しいトークン を出力。

バッチサイズは固定数までパッドされない。出力の異なる位置のシーケンスは 1 つの融合フォワードを共有。2026 年 vLLM でこれは `V1 scheduler` と呼ばれる。重要不変量：スケジューラーはリクエスト当たり 1 回ではなく、各デコード反復ごとに 1 回実行。

### チャンク化プリフィル は TTFT テイルを保護

プリフィルは計算境界。Llama 3.3 70B 上の 32k トークンプロンプトは 1 つの H100 で約 800 ms の純粋なプリフィル。プリフィルが実行されている間、バッチ内のすべての他のシーケンスのデコードトークンは待機。サーブループでは、1 つの長いプロンプトの最初トークンレイテンシ（TTFT）は数十ユーザーの他のものの トークン間レイテンシ（ITL）ブリップになる。

チャンク化プリフィルはプリフィルを固定サイズチャンク（デフォルト 512 トークン）に分割し、各チャンク を単位としてスケジュール。チャンク間でスケジューラーはデコード シーケンスを 1 トークン進める可能性があります。小さい絶対プリフィル遅延（チャンクごと数 ms）をより低いデコード時間ジッター用にトレード。発表ベンチマークで混合負荷下の P99 ITL は約 50 ms から約 15 ms に落ちます。

### 3 つのデフォルトは相互作用

3 つの特徴すべて相互仮定。PagedAttention は細かいサイズの KV リソースをスケジューラーに与え、トレード可能。連続バッチングは新しいシーケンスのプリフィルがグローバルリシャッフルを強制しないようにその細かいリソースが必要。チャンク化プリフィルは同じ `RUNNING` リストでスケジューラーが行う判断 — 別のシステムではなく 1 つのスケジューラーポリシーです。

すべてのフラグを知る必要はない。スケジューラーが最適化するものを知る必要：KV ブロックバジェット下の goodput、チャンク化プリフィル スライシング対象。

### 2026 年 v0.18.0 ゴッチャ

vLLM v0.18.0 ではドラフトモデル推測デコード（`--speculative-model`）と `--enable-chunked-prefill` を組み合わせることができません。ドキュメント例外は V1 スケジューラー内の N-gram GPU 推測デコード。すべてのフラグをリリースノートなし で読まずにオンにするチーム、ソフト回帰ではなく起動時にランタイム エラー を得ます。推測ゲインがチャンク化プリフィル有効化のための価値があった場合、選択を再検討 — 2026 年の正しい答えはしばしば、チャンク化プリフィル コンパイルしないドラフトモデル + チャンク化プリフィルではなく EAGLE-3 。

### 覚えるべき数値

- Llama 3.3 70B FP8、H100 SXM5、128 同時、すべて 3 つオン：2,200-2,400 tok/s。
- 同じモデル、デフォルト vLLM（チャンク化プリフィル なし）：約 1,800 tok/s。
- 同じモデル、素朴な PyTorch フォワードループ：約 600 tok/s。
- 本番負荷で PagedAttention 下の KV 断片化廃棄：< 4%。
- 混合負荷下の P99 ITL：チャンク化プリフィル で約 15 ms、なしで約 50 ms。

### スケジューラーは何に見えるか

```
while True:
    finished = [s for s in RUNNING if s.is_done()]
    for s in finished: release_blocks(s); RUNNING.remove(s)

    while WAITING and have_free_blocks_for(WAITING[0]):
        s = WAITING.pop(0)
        allocate_initial_blocks(s)
        RUNNING.append(s)

    # schedule prefill chunks + decode in one batch
    batch = []
    for s in RUNNING:
        if s.in_prefill:
            batch.append(next_prefill_chunk(s))   # e.g. 512 tokens
        else:
            batch.append(decode_one_token(s))     # 1 token

    run_forward(batch)                            # one fused GPU call
```

`code/main.py` はまさしくこのループ、stdlib Python で偽トークン数と偽フォワード遅延。実行すると、チャンク化プリフィル が長いプリフィル 中のデコード シーケンスをどのように生きたままにするか示す。

## 使用

`code/main.py` はトグル可能な特徴を持つ vLLM スタイルスケジューラーをシミュレート。実行して見る：

- `NAIVE` モード：1 度に 1 リクエスト、バッチングなし。
- `STATIC` モード：パッド・ウェイト、古典的バッチング。
- `CONTINUOUS` モード：反復レベル許可と解放。
- `CONTINUOUS + CHUNKED` モード：デコード と相互使用されるプリフィル スライス。

出力はトータルスループット（仮想秒当たりトークン）、TTFT 平均、P99 ITL を示す。`CONTINUOUS + CHUNKED` 行は混合トラフィックで優位に。

## 重要用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|-------------|----------|
| PagedAttention | 「KV トリック」 | KV キャッシュ用固定サイズブロックアロケータ。断片化 < 4% |
| ブロックテーブル | 「ページテーブル」 | 論理トークン位置を物理 KV ブロックにマップするシーケンスごと |
| 連続バッチング | 「動的バッチング、ただし正しい」 | 許可/解放判定を各デコード反復に行う |
| チャンク化プリフィル | 「プリフィル分割」 | 長プリフィルを 512 トークン スライスに分割、デコード と相互使用 |
| TTFT | 「初トークン時間」 | プリフィル + キュー + ネットワーク。長プロンプトで主に プリフィル |
| ITL | 「トークン間レイテンシ」 | 連続デコードトークン間の時間。主に バッチサイズ |
| Goodput | 「SLO スループット」 | トークン/秒（すべての SLO をまだ満たすリクエスト） |
| V1 スケジューラー | 「新しいスケジューラー」 | vLLM 2026 スケジューラー。N-gram spec decode は チャンク化プリフィル 互換パス |
| `--gpu-memory-utilization` | 「メモリノブ」 | 重みと活性化後の KV ブロック用に予約する HBM の分数 |
