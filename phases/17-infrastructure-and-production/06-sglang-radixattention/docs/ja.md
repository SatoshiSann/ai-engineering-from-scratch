# SGLang と RadixAttention プリフィクス 重いワークロード用

> SGLang は KV キャッシュを再利用可能な 1 級リソースとして radix 木に保存。vLLM がリクエストを FCFS（先着順）でスケジュール、SGLang のキャッシュ対応スケジューラーはより長い共有プリフィクスを持つリクエストを優先 — 実質的に radix 深さ優先トラバーサル、ホットブランチが HBM に居座る。Llama 3.1 8B on ShareGPT-like 1K プロンプト、SGLang はじゃ 約 16,200 tok/s to vLLM の約 12,500、約 29% エッジ。プリフィクス重い RAG ワークロード でアドバンテージ 6.4x に到達。音声クローニング形状ワークロード でキャッシュ ヒット率 86%。2026 年全体で 400,000+ GPU に xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS で デプロイ。ゴッチャは 6.4x 数字がプリフィクス順序が矛盾するいなくなること — 順序はエンジニアのレバー。

**タイプ:** 学習
**言語:** Python（stdlib、簡単な radix ツリーキャッシュ + キャッシュ対応スケジューラー）
**前提条件:** フェーズ 17 · 04（vLLM サービング インターナル）、フェーズ 14（Agentic RAG）
**所要時間:** 約 75 分

## 学習目標

- RadixAttention を図解：プリフィクスが radix 木に保存されるやり方、同じブランチに根ざしているシーケンス間で KV ブロックが共有される方法。
- キャッシュ対応スケジューリングと FCFS がプリフィクス重トラフィックに誤りの理由を説明。
- プリフィクス キャッシュ ヒット率とプロンプト長分布を考えると、ワークロードの期待スピードアップを計算。
- 6.4x 数字が実際（vs 失われた上向き）にするプロンプト順序ディシプリンに名前を付け。

## 問題

古典的サービングはリクエストのプロンプトを不透明に見なす。5,000 個の RAG リクエストがすべて 2,000 トークン システム プロンプト + 同じ取得プリアンブル で開始しても、vLLM は そのプリフィクスを 5,000 回プリフィル。GPU は何度も何度も同じ作業をする。

観察：エージェント と RAG ワークロードのプロンプトはほぼ常にしばしば長いプリフィクスを共有。システムプロンプト、ツール スキーマ、少数ショット例、取得ヘッダー、会話履歴 — すべてリクエスト間で繰り返す。そのプリフィクスの KV キャッシュを 1 度保存し再利用したら、再びプリフィル しない。

RadixAttention はまさしくそれをする。トークンは radix 木でインデックス。各ノードは トークンシーケンスのパスで所有 KV ブロック。新しいリクエストは木をウォーク：マッチするトークンのノードは KV ブロックを再利用。プリフィル コストは完全プロンプトではなく「新しい」サフィックスに比例になる。

チャレンジはスケジューリング。2 つのリクエストが 2,000 トークンプリフィクスを共有、3 つが同じプリフィクスの 200 トークンのみを共有なら、2 つの長い共有リクエストを一緒に配信したい — ロングプリフィクスが次のロング プリフィクス リクエストが 衝突する前に HBM にとどまるように。FCFS は反対 — 最初に到着した人を配信、長いプリフィク スが次のロング プリフィクス リクエストが衝突する前に HBM にとどまるように。

## コンセプト

### KV インデックスとしての Radix 木

Radix 木（圧縮トライ）はトークンシーケンスを保存。各ノードはトークン範囲を所有してその範囲の計算 KV ブロック。子どもは 1 つ以上のトークン シーケンスを拡張。

```
root
 |- "You are a helpful assistant..."  (2,000 tokens, 124 KV blocks)
      |- "Context: <doc A>..."        (500 tokens, 31 blocks)
           |- "Question: Alice..."    (80 tokens, 5 blocks)
           |- "Question: Bob..."      (95 tokens, 6 blocks)
      |- "Context: <doc B>..."        (520 tokens, 33 blocks)
```

新しいリクエスト がシステムプロンプト + "Context: <doc A>" + "Question: Carol" で到着。スケジューラーはウォーク：システムプリフィクス マッチ（124 ブロック再利用）、doc-A ブランチ マッチ（31 ブロック再利用）、その後 "Question: Carol" （4 ブロック）のみ新しいブロックアロケーション。プリフィル コスト：4 ブロック新しいトークン。木なし：160 ブロック。約 40x プリフィル セービング。

### キャッシュ対応スケジューリング

Radix ツリー支援再利用はキャッシュがかき混ぜられたら無意味。2 つの重要ポリシー：

1. **深さ優先発送**。キューから次のリクエストを選ぶとき、現在実行中セットと同じブランチに根ざしたリクエストを好む。ホットブランチを固定。
2. **ブロック レベル ではなくブランチレベル LRU**。個々のブロックではなく全ブランチ をエビクト（最短使用リーフから開始）、キャッシュシェイプが radix シェイプに一致。

FCFS は両方に反する。2,000 トークンを共有するリクエストが 50 を共有するリクエスト後ろに座る、その後 2,000 トークンブランチ が 50 トークン 1 つを許可するためにエビクト。

### ベンチマーク 数字 — を記憶すべき

- Llama 3.1 8B、H100、ShareGPT 1K プロンプト：SGLang 約 16,200 tok/s vs vLLM 約 12,500（約 29% エッジ）。
- プリフィクス重 RAG（同じシステム + 同じ doc、変動質問）：SGLang で最大 6.4x。
- 音声クローニング ワークロード：86.4% プリフィクス キャッシュ ヒット率。
- SGLang 顧客全体で本番ヒット率：50-99% プロンプト ディシプリンに応じ。
- 2026 年全体で 400,000+ GPU に デプロイ。

### 順序ゴッチャ

6.4x 数字は一貫したプロンプト テンプレート順序に依存。クライアント がプロンプトを構成したら `[system, tools, context, history, question]` 一部リクエスト、その他は `[system, context, tools, history, question]` ですと、木は共有プリフィクスを見つけることができない。人間に共有プリフィクスのように見えるものは radix 木に 2 つの異なるシーケンス。

エンジニア レバー：プロンプト テンプレート はキャッシュキー。順序を修正。すべての不変（システム、ツール、スキーマ）を最初に。取得コンテキスト次。ユーザー質問は最後に。プリフィクスに動的コンテンツを相互使用しない。

実際の研究ケース：プリフィクスからの動的コンテンツの移動は 1 つのデプロイ を 1 つの変更で 7% から 74% キャッシュ ヒット率に取った。

### RadixAttention はどこで勝ち負けするか

勝ち：
- RAG（同じ取得プリアンブル、変動質問）。
- エージェント（同じツール スキーマ、変動クエリ）。
- ロングシステムプロンプト でのチャット。
- 反復プリアンブル付き音声 / ビジョンワークロード。

負け（vLLM レベルスループット へ返す）：
- 一意なプロンプトでの単一ショット生成（コード完了、システムプロンプト なしのオープンエンドチャット）。
- 動的プロンプト、そこで各リクエストが プリフィクスに一意コンテンツを相互使用。

### これはスケジューラー問題である理由、カーネル問題だけではなく

KV 再利用をカーネル トリックとして実装できる。SGLang の洞察は再利用が ホットブランチをメモリに保つスケジューラー場合のみ支払う。 ナイーブ「利用可能なら再利用」ポリシーは混合負荷下キャッシュがかき混ぜられる。Radix-tree-indexed スケジューラーがカーネルトリックをしたら 本番エッジにターンします。

### vLLM との相互作用

2 つのシステムはしっかり競争相手ではない。2026 年 vLLM は プリフィクスキャッシング（`--enable-prefix-caching`）と キャッシュ対応ルーター（Rust の vLLM ルーター）を追加。ギャップは閉じましたが完全には消えない — SGLang の全スタックは radix ファースト。vLLM はそれをグラフト。プリフィクス再利用に支配されたワークロード用、SGLang はデフォルトに留まる。強いプリフィクスパターンなしで一般目的サービング用、vLLM は同等またはより優れています。

## 使用

`code/main.py` はおもちゃ radix ツリー KV キャッシュ + 2 つのポリシーを持つスケジューラーを実装：FCFS とキャッシュ対応。同じワークロード を両方に通し、プリフィクス キャッシュ ヒット率とスループット デルタ を報告。その後「スクランブル順序」ワークロード を実行して 6.4x を見せる崩壊。

## 配布

このレッスンは `outputs/skill-radix-scheduler-advisor.md` を生成。ワークロード説明（プロンプト テンプレート形状、取得パターン、同時テナント数）を考えると、プロンプト 順序処方と SGLang 採用に go/no-go を生成。

## 演習

1. `code/main.py` を実行。同じワークロード 上で FCFS とキャッシュ対応を比較。デルタはどこから来る — プリフィル セービング、デコード セービング、キュー遅延？
2. ワークロード を修正、プロンプト をランダム に `[system, tools, context]` を並べ替え。再実行。ヒット率 に何が起こる？ なぜ？
3. Llama 3.1 8B 上 で 2,000 トークンシステムプロンプト を 1 つの radix ブランチとして居座るの HBM コスト を計算。プリフィクス再利用なし 16 シーケンスバッチ のコスト と比較。
4. SGLang RadixAttention ペーパーを読む。木形 LRU エビクトがプリフィクス重負荷 でブロック形 LRU を上回る理由 3 文で 説明。
5. 顧客が 8% キャッシュ ヒット率のみを報告。3 つの可能な原因に名前付け、各用の診断を実行。

## 重要用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|-------------|----------|
| RadixAttention | 「SGLang 物」 | KV キャッシュ radix 木でインデックス、共有プリフィクス再利用ブロック |
| Radix 木 | 「圧縮トライ」 | 木ここで各ノード トークン範囲とその KV ブロック を所有 |
| キャッシュ対応スケジューラー | 「ホットブランチファースト」 | 居座るブランチを共有するリクエスト を好むスケジューラー |
| プリフィクス キャッシュ ヒット率 | 「プロンプトの無料部分」 | 再利用 KV ブロックから配信されるプロンプトトークンの分数 |
| FCFS | 「先着順」 | デフォルトスケジューリングがプリフィクス ローカリティを壊す |
| ブランチレベル LRU | 「リーフをエビクト」 | ブロック形ではなくradix 形にマッチするエビクト ポリシー |
| プロンプト テンプレート順序 | 「キャッシュキー」 | プロンプトのコンポーネント順序は木が共有できるもの を決定 |
| システムプロンプト ピンニング | 「居座るプリフィクス」 | エビクション スラッシュを避けるため不変システム部分を居座り |
