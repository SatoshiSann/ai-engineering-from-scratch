# セキュリティ — シークレット、APIキーローテーション、監査ログ、ガードレール

> シークレットの分散を集中管理化されたボルト（HashiCorp Vault、AWS Secrets Manager、Azure Key Vault）で排除。認証情報をコンフィグファイル、VCS内のenvファイル、スプレッドシートに保存しない。静的キーより IAM ロールを使用。CI/CDでは OIDC を使用。AIゲートウェイパターンは2026年の解決策：アプリ → ゲートウェイ → モデルプロバイダ。ゲートウェイが実行時にボルトから認証情報を取得。ボルトでローテーションすると、すべてのアプリが数分内に新しいキーを取得 — 再デプロイなし、Slack での「誰が新しいキーを持っているか」という質問なし。ローテーションポリシー ≤90日。TruffleHog / GitGuardian / Gitleaks でコミットごとにスキャン。ゼロトラスト：MFA、SSO、RBAC/ABAC、短命トークン、デバイス態勢。PII スクラビングはエンティティ認識を使用して PHI/PII をマスクしてから転送。一貫性のあるトークン化（Mesh アプローチ）は機密値を安定したプレースホルダーにマップするため、LLM がコード/関係セマンティクスを保持。ネットワーク流出：LLM サービスは専用 VPC/VNet サブネット内で `api.openai.com`、`api.anthropic.com` などのみをホワイトリスト。他の全アウトバウンドをブロック。2026年のインシデント駆動要因：Vercel サプライチェーン攻撃。侵害された CI/CD 認証情報から環境変数を流出し、数千の顧客デプロイメント全体に及ぶ。

**タイプ:** Learn
**言語:** Python（stdlib、オモチャの PII スクラバー + 監査ログライター）
**前提条件:** Phase 17 · 19 (AI Gateways)、Phase 17 · 13 (Observability)
**所要時間:** ~60分

## 学習目標

- 4 つのシークレット管理のアンチパターン（VCS内のコンフィグファイル、ハードコードされたenv、スプレッドシート、静的キー）を列挙し、それぞれの代替案を名前で述べる。
- AIゲートウェイがボルトから取得するパターンを2026年の本番環境標準として説明する。
- 一貫性のあるトークン化（同じ値 → 同じプレースホルダー）を使用した PII スクラバーを実装し、セマンティクスが保持されるようにする。
- 2026年の Vercel サプライチェーンインシデントを名前で述べ、CI/CD 認証情報の衛生について何を学んだかを説明する。

## 問題

インターンが `.env` を API キーと一緒にコミット。すぐに削除。キーはすでに git ヒストリーに存在 — GitGuardian スキャンが検出、ローテーションプロセスは「チームに Slack で連絡、40 個のコンフィグファイルを更新、すべてのサービスを再デプロイ」。8 時間後、半分のサービスがライブで、半分が デプロイ ウィンドウを待っている。

別の問題として、ユーザープロンプトに「My SSN is 123-45-6789」が含まれている。プロンプトが OpenAI に送信。BAA があるが、内部ポリシーでは転送前に PII をマスクすることが必須。やっていない。

さらに別の問題として、EKS クラスタの LLM ポッドはインターネット上の任意のホストに到達できる。誰かが攻撃者制御のドメインへの DNS ルックアップ経由でデータを流出。何もそれをブロックしていない。

LLM サービスのセキュリティは、この 3 つのベクトル全てに対処する必要がある。ボルトバックの認証情報。PII スクラビング。ネットワーク流出フィルタリング。監査ログ。

## コンセプト

### 集中管理ボルト + IAM ロールプル

**Vault**: HashiCorp Vault、AWS Secrets Manager、Azure Key Vault、GCP Secret Manager。唯一の真実の源。

**IAM ロール**: アプリ/ゲートウェイは静的キーではなく IAM アイデンティティで認証。ボルトはトークンのライフタイムのシークレットを返す。

**AIゲートウェイパターン**: ゲートウェイはリクエスト時にボルトから `OPENAI_API_KEY` を取得。ボルト内でローテーション、次のリクエストが新しいキーを取得。再デプロイなし。

### ローテーションポリシー ≤ 90 日

すべての API キー、ボルト ルート トークン、CI/CD 認証情報。可能な場所は自動ローテーション。手動ローテーションはログ記録と追跡。

### シークレット スキャン

- **TruffleHog** — コミット上の正規表現 + エントロピー。
- **GitGuardian** — 商用、高精度。
- **Gitleaks** — OSS、CI 内実行。

すべてのコミットでスキャン。新しいシークレット検出時は PR をブロック。

### ゼロトラスト態勢

- すべてのアカウントで MFA 必須。
- SAML/OIDC 経由の SSO。
- 細かいアクセス制御のための RBAC（ロールベース）または ABAC（属性ベース）。
- 短命トークン（日数ではなく時間）。
- デバイス態勢 — ディスク暗号化を備えた企業デバイスのみ。

### PII / PHI スクラビング

プロンプトがインフラを離れる前に：

1. エンティティ認識（spaCy NER、Presidio、商用）。
2. マッチしたエンティティをマスク：`"My SSN is 123-45-6789"` → `"My SSN is [SSN_TOKEN_A3F]"`。
3. 一貫性のあるトークン化（Mesh アプローチ）：同じ値が同じプレースホルダーにマップするため、LLM が関係を保持。
4. オプションで LLM レスポンスの逆マップ。

静的な正規表現フィルタは基本的なパターンをキャッチ、NER はより多くをキャッチ。両方を使用。

### 入力 + 出力ガードレール

入力：既知のジェイルブレイク、禁止トピックをブロック、ユーザーごとにレート制限。

出力：漏洩したシークレット（API キーパターン、拒否コンテキストのメールパターン）の正規表現スクラブ、ポリシー違反のクラシファイア。

### ネットワーク流出ホワイトリスト

LLM サービスを専用サブネット内に配置：
- ホワイトリスト：`api.openai.com`、`api.anthropic.com`、ベクトル DB エンドポイント、ボルト エンドポイント。
- その他すべて：ドロップ。
- DNS をホワイトリストのみのリゾルバー経由（DNS トンネルリング流出を回避）。

### 監査ログ

すべての LLM 呼び出しの不変ログ：
- タイムスタンプ。
- ユーザー / テナント。
- プロンプト ハッシュ（プライバシーのため生プロンプトではない）。
- モデル + バージョン。
- トークン数。
- コスト。
- レスポンス ハッシュ。
- すべてのガードレール トリップ。

規制要件に従って保持（SOC 2 1 年、HIPAA 6 年）。

### 2026年の Vercel インシデント

サプライチェーン攻撃：侵害された CI/CD 認証情報は環境変数を数千の顧客デプロイメント全体に流出。教訓：CI/CD 認証情報は本番環境相当。ボルトに保存。狭くスコープ。積極的にローテーション。

### 覚えるべき数字

- ローテーションポリシー：≤ 90 日。
- すべてのコミットでスキャン：TruffleHog / GitGuardian / Gitleaks。
- Vercel 2026：CI/CD creds 侵害 → 数千の顧客 env vars 漏洩。
- 監査ログ保持：SOC 2 = 1 年、HIPAA = 6 年。

## 演習

1. `code/main.py` を実行。同じ SSN を参照する 2 つのプロンプトを送信。両方が同じプレースホルダーを取得することを確認。
2. vLLM-on-EKS デプロイメント向けのネットワーク流出ポリシーを設計。OpenAI + Anthropic + Weaviate を呼び出す。
3. git ヒストリーでキーを検出（2 年前）。正しい対応は何か — キーをローテーション、ヒストリーをスクラブ、またはその両方？正当化する。
4. 監査ログが 1 日 10 GB に成長。保持層を設計（ホット 30 日、ウォーム 12 か月、コールド 6 年）。
5. 逆トークン化（実際の値を LLM レスポンスに置換）がプレースホルダーを見えたままにする複雑性に値するかどうかを議論。

## キー用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|----------------|--------|
| Vault | 「シークレット ストア」 | 集中管理された認証情報管理サービス |
| IAM ロール | 「アイデンティティベース認証」 | アプリが想定するロール、短命 creds を返す |
| CI/CD 向け OIDC | 「クラウド発行トークン」 | CI に静的キーなし — OIDC 経由のアイデンティティ |
| TruffleHog / GitGuardian / Gitleaks | 「シークレット スキャナー」 | コミット時シークレット検出 |
| RBAC / ABAC | 「アクセス制御」 | ロールベース vs 属性ベース |
| PII スクラビング | 「データ マスキング」 | 機密エンティティを削除またはトークン化 |
| 一貫性のあるトークン化 | 「安定したプレースホルダー」 | 同じ値 → 毎回同じトークン |
| Mesh アプローチ | 「Mesh トークン化」 | セマンティック保存トークン化パターン |
| 流出ホワイトリスト | 「アウトバウンド許可リスト」 | 許可されたドメインのみ到達可能 |
| 監査ログ | 「不変履歴」 | コンプライアンスのための追記のみレコード |
