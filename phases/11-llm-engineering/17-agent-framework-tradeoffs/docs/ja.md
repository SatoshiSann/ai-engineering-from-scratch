# エージェントフレームワークのトレードオフ — LangGraph 対 CrewAI 対 AutoGen 対 Agno

> すべてのフレームワークは同じデモ(研究エージェント報告書を構築)を売却し、同じバグを隠す(ステートスキーマはオーケストレーションレイヤーと戦う)。問題の形にマッチするフレームワーク抽象化を選びます；他はすべて2回書くグルです。

**タイプ:** 学習
**言語:** Python
**前提条件:** Phase 11 · 09 (Function Calling)、Phase 11 · 16 (LangGraph)
**所要時間:** 約45分

## 問題

複数のLLM呼び出しが必要なタスク。研究ワークフロー(計画、検索、要約、引用)。コードレビューパイプライン(差分解析、批判、パッチ、検証)。複数ターンアシスタント(フライト予約、メール書き、経費報告)。フレームワークを選ぶ。

3日後、フレームワークの抽象化が漏洩。CrewAIはロール与えるが「研究」が構造化計画を「書き」に渡す必要があるときに戦う。AutoGenはエージェント間チャットを与えるが、第1クラスステートなし、チェックポイントは会話ログの漬物。LangGraphはステートグラフを与えるが、出発前のすべての遷移命名に強制。Agnoは単一エージェント抽象化を与えるが、3つの並行労働に試みるとき悲鳴。

修正「最高のフレームワークを選ぶ」ではない。フレームワークのコアアブストラクションをあなたの問題の形にマッチするまで。このレッスンはその地図を描く。

## コンセプト

4つのフレームワークが2026年風景を支配。彼らのコア抽象化は同じではない。

| フレームワーク | コア抽象化 | 最良の適合 | 最悪の適合 |
|-----------|------------------|----------|-----------|
| **LangGraph** | `StateGraph` — 型ステート、ノード、条件エッジ、チェックポインター。 | 明示ステートと人間ループ割り込みを持つ; 本番エージェント時間旅行デバッグが必要。 | トポロジー未知で役割駆動ブレーンストーム。 |
| **CrewAI** | `Crew` — ロール(目標、背景、物語)、タスク、プロセス(直列または階層)。 | ロール演技またはペルソナ駆動ワークフロー短い直列/階層計画。 | ステートを超える複雑なブランチ；複雑なブランチ。 |
| **AutoGen** | `ConversableAgent`ペア — 2つ以上エージェント回転まで退場状態。 | マルチエージェント*ダイアログ*(先生学生、提案者批評家、俳優評論者)、チャットから出現。 | 既知DAG決定論的ワークフロー；リスタート全体耐久ステート。 |
| **Agno** | `Agent` — 単一LLM + ツール + メモリ、チームに合成可能。 | 高速単一エージェントと軽量チーム；強多様性と組み込みストレージ。 | 深く明示分岐グラフ、カスタムリデューサー。 |

### 抽象化は何で実際かを意味

フレームワークのコア抽象化はアーキテクチャをピッチするときホワイトボードで描くもの。

- **LangGraph** → グラフを描く。ノードはステップ、エッジは遷移、ステートオブジェクトすべてのポイント型。心的モデルはステートマシン。
- **CrewAI** → 組織図描く。各ロールはジョブ説明、マネージャールーティング タスク。心的モデルはスペシャリストの小チーム。
- **AutoGen** → Slack DM描く。2つのエージェント メッセージ；3番目は必要ならジョイン。心的モデルはチャット。
- **Agno** → 単一ボックスツール描く。ツール隣. チームのはボックス隣. 心的モデルは「バッテリー含まエージェント」。

### ステート問題

ステートは本番フレームワーク選択が大部分壊れる場所。

- **LangGraph。** 型ステート(TypeDictまたはPydanticモデル)、フィールドリデューサー、第1クラスチェックポインター(SQLite/Postgres/Redis)。再開、割り込み、時間旅行は無料。*(Phase 11 · 16を見.)* 
- **CrewAI。** ステートはコンテキスト場経由タスク間文字列、または構造`output_pydantic`経由。出BOXの耐久チームストアなし；再起動生き残る必要ならこれをボルト。
- **AutoGen。** ステートはチャット履歴と任意のユーザー定義`context`。会話成績簿は永続化；任意のワークフロー ステートはアダプターを書かなければ。
- **Agno。** 組み込みストレージドライバー(SQLite、Postgres、Mongo、Redis、DynamoDB) `Agent`に`storage=`経由— 会話セッションとユーザーメモリは自動的に持続。完全グラフチェックポインターではなく；会話スコープセッション店。

### ブランチ質問

すべての非自明なエージェントはブランチ。誰が判断するかが重要。

- **LangGraph** — 開発者は条件エッジ経由判断。ルーティングはPython関数、名命ブランチ。ブランチはコンパイルされたグラフで第1クラス；チェックポインターはどのブランチが取られたかを記録。
- **CrewAI** — マネージャーは階層的モード判断；直列モード判断は構築時。ルーティングはマネージャープロンプトで暗黙；第1クラス「もし」なし。
- **AutoGen** — エージェント はチャット判断。ブランチはチャットから出現。`GroupChatManager`は次のスピーカーを選びます；手書き`speaker_selection_method`可能だがデフォルトはLLM駆動。
- **Agno** — エージェントはどのツールを次に呼ぶか判断。チームはコーディネーター/ルーター/協力者モード；その先のブランチは開発者責任。

### 可視性問題

- **LangGraph** — OpenTelemetry via LangSmith または任意OTel 輸出。すべてのノード遷移はトレーススパン；チェックポイントはリプレイ可能トレースとして倍。LangSmithは第1方選択肢；Langfuse/Phoenix もアダプターあり。
- **CrewAI** — 2025年後期から第1クラスOpenTelemetry；Langfuse、Phoenix、Opik、AgentOps統合。
- **AutoGen** — OpenTelemetry `autogen-core`経由統合；AgentOps と Opikコネクター。トレーシング粒度はノード、エージェントメッセージあたり。
- **Agno** — 組み込み`monitoring=True`フラグと OpenTelemetry 輸出；Langfuse用タイト統合セッション トレース。

### コストと遅延

すべての4つのフレームワークは呼び出しあたりオーバーヘッド追加(フレームワークロジック、検証、連列化)。増加オーバーヘッドの約順：Agno ≈ LangGraph < CrewAI ≈ AutoGen。差分はフレームワークがどのくらい余分LLMルーティングをするかで支配。CrewAIの階層マネージャーはトークンを次に誰が行くかを決定; AutoGenの`GroupChatManager`同様。LangGraphはモデル呼び出しのみ`llm.invoke`を書くところをトークンを使う。Agnoの単一エージェント路径は薄い。

実行あたりコストが重要のとき、明示ルーティングを優先(LangGraphエッジ、AutoGen`speaker_selection_method`)をLLM選択ルーティング。

### 相互運用性

- **LangGraph** ↔ **LangChain**ツール、検索、LLM。第1クラスMCPアダプター(MCPサーバーとしてツール輸入)。
- **CrewAI** ↔ ツール継承`BaseTool`；LangChainツール、LlamaIndex ツール、MCPツール、すべてアダプト。Crew-to-crewデリゲーション`allow_delegation=True`経由。
- **AutoGen** → `FunctionTool`が任意Pythonラップ呼び出し可能；MCPアダプター利用可能。AG2エコシステムへのタイト結合エージェント間パターン。
- **Agno** → `@tool`デコレータまたはBaseTool サブクラス；MCPアダプター；ツールをエージェント、チーム全体で共有可。

## スキル

> あなたは1文で、なぜ与えられたフレームワークが与えられたエージェント問題に正しいのか説明できます。

構築前チェックリスト：

1. **形を描く。** これはグラフ(型ステート、命名遷移)か？役割演技(スペシャリスト引き渡す)か？チャット(エージェント回転)か？単一エージェントツール付きか？
2. **ブランチを判断。** 開発者判断ルーティング → LangGraph。マネージャーエージェント判断 → CrewAI階層。チャット出現 → AutoGen。ツール呼び出し判断 → Agno。
3. **ステート予算をチェック。** チェックポイント再開必要か？時間旅行？中実行人間割り込み？はいなら、LangGraphはデフォルト；Agnoセッションカバー会話スコープステート。
4. **コスト予算をチェック。** LLM選択ルーティングはターン毎余分トークン。エージェント毎日数千回実行なら、明示ルーティング優先。
5. **フレームワーク オーバーヘッド予算。** すべてのフレームワークは別依存。タスク2つのLLM呼び出しとツールなら、30行プレーンPython書く；フレームワークなしが安い。

グラフ、組織図、チャット、またはエージェントボックスを描く前にフレームワークに到達拒否。それの状態モデルを戦う強制するを選ぶ拒否あなたが実際に必要なもの。

## 判断行列

| 問題形 | 優先フレームワーク | 理由 |
|---------------|---------------------|-----|
| DAG型ステート、人間承認、長実行のワークフロー | LangGraph | 第1クラスステート、チェックポインター、割り込み、時間旅行。 |
| 研究/書く異なるロール付きパイプライン | CrewAI(直列)またはLangGraphサブグラフ | ロール-per-task cheap CrewAI ; CrewAIでLangGraph複雑ブランチが拡大。 |
| 提案者批評家または先生学生ダイアログ | AutoGen | 2つエージェントチャットはネイティブ形。 |
| 単一エージェントツール、セッション、メモリ | Agno | 最も薄いセットアップ、組み込みストレージ、メモリ。 |
| 数千平列フアウト削減器付き | LangGraph + `Send` | 平列ディスパッチAPIのみ第1クラス。 |
| クイックプロトタイプ、フレームワークなし | プレーンPython + プロバイダSDK | フレームワークなしが最速。 |

## 演習

1. **簡単。** 同じタスク — 「Anthropicの本部を研究、200単語簡潔を書く、ソース引用」 — をLangGraph(4ノード：計画、検索、書く、引用)でとCrewAI(3ロール：研究者、書き手、編者)で実装。ランごとのトークンコストとコード行を報告。
2. **中程度。** 同じタスクをAutoGen(研究者 ↔ 書き手チャット、編集者がグループチャットにジョイン)とAgno(単一エージェント`search_tools`と`write_tools`、セッション店)でビルド。4実装を(a)ランあたりコスト、(b)クラッシュ後再開能力、(c)書く前人間承認を注射能力でランク。
3. **難しい。** デシジョンツリースクリプト`pick_framework.py`を短い問題説明(JSON: `{has_typed_state, has_roles, has_dialogue, has_parallel_fanout, needs_resume}`)を取ってフレームワーク推奨を返す、1文正当化付きビルド。6自分で設計するケースで検証。
