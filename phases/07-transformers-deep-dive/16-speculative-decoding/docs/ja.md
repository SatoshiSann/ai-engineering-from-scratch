# 推測デコーディング — ドラフト、確認、繰り返し

> 自己回帰デコーディングはシリアル。各トークンは前のトークンを待つ。推測デコーディングはチェーンを破る: 安価なモデルがNトークンをドラフトし、高価なモデルがすべてのNを1つの順伝播で確認する。ドラフトが正しい場合、Nの生成に対して1つの大きな順伝播を支払う。

**タイプ:** ビルド
**言語:** Python
**前提条件:** Phase 7 · 07 (GPT Causal LM)、Phase 7 · 12 (KV Cache & Flash Attention)
**所要時間:** ~60分

## 問題

H100で70B LLMをサンプリングして1トークンを取得するのに約30ms。3Bドラフトモデルは約3ms。3Bドラフトを5トークン先にさせてから、70Bを*一度*実行してすべての5を確認する場合、合計は `5×3 + 30 = 45 ms` で最大5個の受け入れられたトークン — 対 `5×30 = 150 ms` ストレートラインの生成。これが完全な推測デコーディングのピッチ: 少量の追加GPUメモリ(ドラフトモデル)を交換して、2-4倍低いデコードレイテンシ。

トリックは分布を保存する必要がある。推測サンプリング、Leviathan et al. (2023)によって導入され、Chenらによって並行して、出力シーケンスが大モデルが単独で生成したであろう**同一分布**であることを保証する。品質トレードオフなし。より速いだけ。

4つの家族のドラフト-検証ペアが2026年の推論を支配する:

1. **バニラ推測 (Leviathan 2023)。** 個別のドラフトモデル(例: Llama 3 1B) + 検証ツール(例: Llama 3 70B)。
2. **Medusa (Cai 2024)。** 検証ツール上の複数のデコーディングヘッドが並列で位置 `t+1..t+k` を予測。個別のドラフトモデルなし。
3. **EAGLEファミリー (Li 2024、2025)。** 検証ツールの隠された状態を再利用する軽量ドラフト; バニラより高い受け入れ率; 典型的に3-4倍。
4. **先読みデコーディング (Fu 2024)。** ヤコビ反復; ドラフトモデルはまったく必要ない。自己推測。ニッチだが依存性がない。

2026年のすべての本番推論スタックは推測デコーディングをデフォルトで出荷する。vLLM、TensorRT-LLM、SGLang、およびllama.cppはすべてバニラ + EAGLE-2の少なくとも一つをサポート。

## コンセプト

### コアアルゴリズム

検証ツール `M_q` およびより安価なドラフト `M_p` を与える:

1. `x_1..x_k` を既にデコードされた接頭辞とする。
2. **ドラフト**: `M_p` を使用して自動回帰的に `d_{k+1}, d_{k+2}, ..., d_{k+N}` を提案し、ドラフト確率 `p_1..p_N` を取得。
3. **並列で確認**: `M_q` を `x_1..x_k, d_{k+1}, ..., d_{k+N}` で1回実行し、位置 `k+1..k+N+1` の検証ツール確率 `q_1..q_{N+1}` を取得。
4. **左から右へ各ドラフトトークンを受け入れ/拒否**: 各 `i` に対して、確率 `min(1, q_i(d_i) / p_i(d_i))` で受け入れる。
5. 位置 `j` での最初の拒否: 「残差」分布 `(q_j - p_j)_+` 正規化から `t_j` をサンプル。`j` 後のすべてのドラフトは破棄。
6. すべての `N` を受け入れた場合: `q_{N+1}` から1つの追加トークン `t_{N+1}` をサンプル(無料ボーナストークン)。

残差分布トリックは、出力が正確に `M_q` がスクラッチからサンプリングされたときと同じように分布されているという数学的な洞察である。

### スピードアップを決定するもの

`α` = ドラフトトークンあたりの予期される受け入れ率とする。`c` = ドラフト対検証ツールコスト比。ステップあたり:

- ナイーブな生成は1つの大モデル呼び出しをトークンあたり行う。
- 推測は `(1 - α^{N+1}) / (1 - α) ≈ 1/(1-α)` トークンあたり1つの大モデル呼び出しを行う、`α` が高い場合。

典型的なルール `α = 0.75` および `N = 5` で: 3倍少ない大モデル呼び出し。ドラフトコストは5倍安い。総ウォールクロックは約2.5倍低下。

**α の根拠:**

- どのくらい良くドラフトが検証ツールを近似するか。同じファミリー/同じ訓練データは α を著しく上げる。
- デコーディング戦略。貪欲ドラフト対貪欲検証ツール: 高 α。温度サンプリング: マッチさせるのが難しい; 受け入れは低下。
- タスク型。コードおよび構造化出力はより多く受け入れる(予測可能); フリーフォーム創造的執筆はより少なく受け入れる。

### Medusa — ドラフトモデルなしのドラフト

Medusaはドラフトモデルを検証ツール上の追加出力ヘッドと置き換える。位置 `t` で:

```
共有トランク → 隠された h_t
    ├── head_0: t+1でのトークンを予測 (標準LMヘッド)
    ├── head_1: t+2でのトークンを予測
    ├── head_2: t+3でのトークンを予測
    ├── head_3: t+4でのトークンを予測
```

各ヘッドは独自のロジットを出力。推論時に各ヘッドからサンプリングして候補シーケンスを取得してから、すべての候補継続を一度に検討するツリーアテンション方式を使用して1つの順伝播で確認。

長所: 2番目のモデルなし。短所: 訓練可能なパラメータを追加; 教師なし微調整段階が必要(~1B トークン); 受け入れ率は良いドラフトを持つバニラ推測より少し低い。

### EAGLE — 隠された状態の再利用により優れたドラフト

EAGLE-1/2/3 (Li et al., 2024-2025) は、検証ツールの最後のレイヤー隠された状態を摂取する小さなトランスフォーマー(通常1レイヤー)を作成する。ドラフトは検証ツールの機能表現を見るので、その予測は検証ツール出力分布と強く相関する。受け入れ率は約0.6(バニラ)から0.85+に上昇。

EAGLE-3 (2025) は候補継続上のツリー検索を追加。vLLMおよびSGLangはEAGLE-2/3をLlama 3/4およびQwen 3のデフォルト仕様経路として配布。

### KVキャッシュダンス

確認は `N` ドラフトトークンを検証ツールの1つの順伝播に供給。これは検証ツールのKVキャッシュを `N` エントリまで拡張。いくつかのドラフトが拒否される場合、受け入れられた接頭辞長にキャッシュをロールバックする必要がある。

本番実装(vLLMの `--speculative-model`、TensorRT-LLMのLookaheadDecoder)はスクラッチKVバッファで処理。最初に書き込み、受け入れ時にコミット。概念的には難しくないが、煩雑である。

## ビルドする

`code/main.py` を見よ。コア推測サンプリングアルゴリズム(拒否ステップ+残差分布)を実装する:

- 決定論的-softmaxをハンドコード分布(アテンション数学を分析的に確認できるよう)上で実装する「大モデル」。
- 大モデルの摂動である「ドラフトモデル」。
- 直接サンプリングと同じ周辺分布を生成する受け入れ/拒否ループ。

### ステップ 1: 拒否ステップ

```python
def accept_or_reject(q_prob, p_prob, draft_token, u):
    ratio = q_prob / p_prob if p_prob > 0 else float("inf")
    return u < min(1.0, ratio)
```

`u` はユニフォームランダム数。`q_prob` はドラフトトークンの検証ツール確率。`p_prob` はドラフトモデル確率。Leviathanの定理は、このベルヌーイ決定、続く残差でのサンプリングが検証ツール分布を正確に保存することである。

### ステップ 2: 残差分布

```python
def residual_dist(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    return [r / s for r in raw]
```

要素単位で `p` を `q` から減算、負の値をゼロにクランプ、再正規化。拒否時にこれからサンプル。

### ステップ 3: 1つの推測ステップ

```python
def spec_step(prefix, q_model, p_model, N, rng):
    drafts = []
    p_probs = []
    ctx = list(prefix)
    for _ in range(N):
        p_dist = p_model(ctx)
        d = sample(p_dist, rng)
        drafts.append(d)
        p_probs.append(p_dist[d])
        ctx.append(d)

    q_dists = [q_model(prefix + drafts[:i]) for i in range(N + 1)]

    for i, d in enumerate(drafts):
        u = rng.random()
        q_prob = q_dists[i][d]
        p_prob = p_probs[i]
        if u < min(1.0, q_prob / p_prob if p_prob > 0 else float("inf")):
            prefix = prefix + [d]
        else:
            res = residual_dist(q_dists[i], p_model(prefix))
            prefix = prefix + [sample(res, rng)]
            return prefix
    prefix = prefix + [sample(q_dists[N], rng)]
    return prefix
```

5つの受け入れ → 1つのボーナス → 1つの検証ツール処理で6トークン生成。

### ステップ 4: 受け入れ率を測定

異なるドラフト品質レベルで10,000の推測ステップを実行。受け入れ率対ドラフトと検証ツール分布間のKLダイバージェンスをプロット。単調な関係を見るべき。

### ステップ 5: 分布等価性を確認

経験的: 推測ループによって生成されたトークンのヒストグラムは、検証ツールから直接サンプリングによって生成されたヒストグラムと一致するべき。これはLeviathanの定理が実際である。カイ二乗検定はサンプリング誤差内で確認。

## 使用

本番:

```bash
# EAGLEを使用したvLLM
vllm serve meta-llama/Llama-3.1-70B-Instruct \
    --speculative-model /models/llama-3.1-eagle-70b \
    --speculative-draft-tensor-parallel-size 1 \
    --num-speculative-tokens 5

# バニラドラフトモデルを使用したvLLM
vllm serve meta-llama/Llama-3.1-70B-Instruct \
    --speculative-model meta-llama/Llama-3.2-1B-Instruct \
    --num-speculative-tokens 5
```

TensorRT-LLMは2026年中盤の最も速いMedusaパス。`faster-whisper` は小さなドラフトでWhisper-largeのための推測デコーディングをラップ。

**ドラフト選択:**

| 戦略 | いつ選ぶか | スピードアップ |
|----------|--------------|---------|
| バニラドラフト (1B/3B Llamaファミリー) | 高速プロトタイプ、訓練なし | 1.8-2.3倍 |
| Medusaヘッド | 検証ツールを微調整できる | 2-3倍 |
| EAGLE-2 / 3 | 本番、最高速度 | 3-4倍 |
| 先読み | ドラフトなし、訓練なし、追加パラメータなし | 1.3-1.6倍 |

**推測デコーディング NOT する場合:**

- 1-5トークンの単一シーケンス生成。オーバーヘッドが支配。
- ワイルドで創造的/高温度サンプリング(α低下)。
- メモリ制約された展開(ドラフトモデルはVRAM追加)。

## 演習

1. **簡単。** `code/main.py` を実行。50,000トークンでカイ二乗p > 0.05内で推測トークン分布が検証ツール直接サンプル分布と一致することを確認。
2. **中程度。** `α = 0.5, 0.7, 0.85` の関数としてスピードアップ(大モデル順伝播あたりトークン)をプロット。各 α の最適 `N` を識別。(ヒント: 検証呼び出しあたりの予期されるトークン = `(1 - α^{N+1}) / (1 - α)`。)
3. **難しい。** 小さなMedusaを実装: Lesson 14のキャプストーンGPTを取る、位置t+2、t+3、t+4を予測する3つの追加LMヘッドを追加。tinyshakespeareで結合マルチヘッド損失で訓練。バニラドラフトを同じモデルを切り詰めることで作られた対受け入れ率を比較。
4. **難しい。** ロールバックを実装: 10トークン接頭辞KVキャッシュから開始、5ドラフトトークンを供給、位置3での拒否をシミュレート。キャッシュの読み取りが次の反復で「接頭辞+最初の2受け入れられたドラフト」と正しく一致することを確認。

## 重要な用語

| 用語 | 人々が言う | 実際の意味 |
|------|-----------------|-----------------------|
| ドラフトモデル | 「安いやつ」 | 候補トークンを提案する小さいモデル; 通常検証ツールより10-50倍安い。 |
| 検証ツール | 「大きいやつ」 | 分布を保存するターゲットモデル; 推測ステップあたり1回実行。 |
| 受け入れ率 (α) | 「ドラフトがどれだけ正しいか」 | トークンあたりの確率、検証ツールがドラフトを受け入れる。典型的に0.7-0.9。 |
| 残差分布 | 「拒否フォールバック」 | `(q - p)_+` 正規化; 拒否でこれからサンプリング。 |
| ボーナストークン | 「フリーなやつ」 | すべてのN ドラフト受け入れ時、検証ツールの次ステップ分布から1つ多く。 |
| Medusa | 「ドラフトレス推測」 | 検証ツール上の複数LMヘッド、並列で位置t+1..t+kを予測。 |
| EAGLE | 「隠された状態ドラフト」 | 検証ツール最後のレイヤー隠された状態で条件付けられた小さいトランスフォーマードラフト。 |
| 先読みデコーディング | 「ヤコビ反復」 | 固定点反復を使用した自己推測; ドラフトモデルなし。 |
| ツリーアテンション | 「多くの候補を一度に確認」 | いくつかのドラフト継続を同時に検討する分岐確認。 |
| KVロールバック | 「拒否されたドラフト元に戻す」 | スクラッチKVバッファ; 受け入れでコミット、拒否で破棄。 |
