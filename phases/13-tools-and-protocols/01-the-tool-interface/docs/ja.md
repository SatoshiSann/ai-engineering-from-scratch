# ツールインターフェース — エージェントが構造化I/Oを必要とする理由

> 言語モデルはトークンを生成します。プログラムはアクションを実行します。この2つの間のギャップを埋めるのがツールインターフェースです。モデルがアクションをリクエストし、ホストがそれを実行するという契約です。2026年のスタック — OpenAI、Anthropic、Geminiの関数呼び出し、MCPの`tools/call`、A2Aのタスクパーツ — はすべて、同じ4ステップループの異なるエンコーディングです。このレッスンではそのループに名前を付け、それを実行するための最小限の仕組みを示します。

**タイプ:** 学習
**言語:** Python（stdlib、LLMなし）
**前提条件:** フェーズ11（LLM補完API）
**所要時間:** 約45分

## 学習目標

- LLMがテキストのみを生成できる場合、それ自体では実世界に対するアクションを実行できない理由を説明する。
- 4ステップツール呼び出しループ（描写→決定→実行→観察）を描き、各ステップを所有する者を特定する。
- ツール記述を3つの部分として記述する：名前、JSONスキーマ入力、および決定論的な実行関数。
- 純粋なツールと副作用のあるツールを区別し、その分割が安全性にとって重要な理由を述べる。

## 問題

LLMは次のトークンに対する確率分布を放出します。これが唯一の出力面です。チャットモデルに「ベンガルールの現在の天気は？」と聞くと、もっともらしい文を書くことができますが、天気APIにダイヤルすることはできません。その文は偶然正しいかもしれませんが、3日前の古い情報かもしれません。

このギャップを埋めることが、ツールインターフェースの目的です。ホストプログラム — エージェントランタイム、Claude Desktop、ChatGPT、Cursor、またはカスタムスクリプト — は、呼び出し可能なツールのリストをモデルにアドバタイズします。モデルがアクションが必要だと判断すると、ツールとその引数に名前を付けた構造化ペイロードを放出します。ホストはそのペイロードをパースしてツールを実際に実行し、結果をフィードバックします。モデルが呼び出しを放出しなくなるまで、ループが続きます。

このコントラクトの最初のバージョンは、2023年6月のOpenAIの「functions」パラメータとしてリリースされました。Anthropicはその後、Claude 2.1で`tool_use`ブロックで続きました。Geminiは数ヶ月後に`functionDeclarations`を追加しました。すべてのプロバイダーは現在、同じ形状を公開しています：入力としてのJSONスキーマ型ツールリスト、出力としてのJSONペイロードツール呼び出し。モデルコンテキストプロトコル（2024年11月）は、1つのツールレジストリがあらゆるモデルに対応するようにコントラクトを汎用化しました。A2A（2026年4月、v1.0）は、エージェント間の委譲に同じプリミティブを階層化しました。

4ステップループは、これらすべての下にある不変量です。フェーズ13の他のすべてはその精密化です。

## コンセプト

### ステップ1：描写

ホストは各ツールを3つのフィールドで宣言します。

- **名前。** 安定した、機械可読な識別子。「weather thing」ではなく`get_weather`。
- **説明。** 1段落の自然言語概要。「ユーザーが特定の都市について現在の状態を尋ねるときに使用します。歴史的データには使用しないでください。」
- **入力スキーマ。** ツールの引数を記述するJSONスキーマオブジェクト（ドラフト2020-12）。

モデルはリストを受け取ります。最新のプロバイダーは、これらの宣言をプロバイダー固有のテンプレートを使用してシステムプロンプトにシリアライズするため、呼び出し者として構造化形式だけを扱います。

### ステップ2：決定

ユーザーのメッセージと利用可能なツールが与えられた場合、モデルは3つの動作の1つを選択します。

1. **テキストで直接回答する。** ツール呼び出しなし。
2. **1つ以上のツールを呼び出す。** 構造化呼び出しオブジェクトを放出する。`parallel_tool_calls: true`の下で（OpenAIおよびGeminiではデフォルト、Anthropicではオプトイン）、モデルは1つのターン内に複数の呼び出しを放出できます。
3. **拒否。** 厳密モード構造化出力は、呼び出しの代わりに型付き`refusal`ブロックを生成できます。

ツール呼び出しペイロードには3つの安定したフィールドがあります：呼び出し`id`、ツール`name`、およびJSON`arguments`オブジェクト。IDは、ホストが後の結果を特定の呼び出しと相関させることができるために存在し、これは平行呼び出しが順序外で戻ってくるときに重要です。

### ステップ3：実行

ホストは呼び出しを受け取り、引数を宣言されたスキーマに対して検証し、実行関数を実行します。無効な引数は、モデルがフィールドを幻想化したか、間違った型を使用したことを意味します — これは弱いモデルで非常に一般的な障害モードです。本番ホストは、無効な引数に対して3つのうちの1つを行います：失敗し、エラーをモデルに表示する、制約付きパーサーでJSONを修復する、またはプロンプトに検証エラーを含めてモデルを再試行します。

実行関数自体は通常のコードです。Python、TypeScript、シェルコマンド、データベースクエリ。それは結果を生成し、通常は文字列ですが、任意のJSON値またはMCPの構造化コンテンツブロック（テキスト、画像、またはリソース参照）です。結果はシリアライズ可能である必要があります。

### ステップ4：観察

ホストはツール結果を会話に追加し（一致する`id`を持つ`tool`ロールメッセージとして）、モデルを再起動します。モデルはテキストをフィアンスし、コンテキスト内にツール出力があり、最終的な回答を生成するか、より多くの呼び出しをリクエストできます。これは、モデルが呼び出しを放出しなくなるか、ホストが反復回数の安全制限に達するまで続きます。

### 信頼分割

安全性に重要なツールには2つの特徴があります。

- **純粋。** 読み取り専用、決定論的、副作用なし。`get_weather`、`search_docs`、`get_current_time`。推測的に呼び出すのは安全です。
- **重要。** 状態を変更し、お金を費やし、ユーザーデータに触れます。`send_email`、`delete_file`、`execute_trade`。ゲートが必要です。

Metaの2026年「Rule of Two」(エージェントセキュリティ用)では、単一のターンは最大2つを組み合わせることができます：信頼できない入力、機密データ、重要なアクション。ツールインターフェースは、呼び出しを拒否したり、ユーザー確認を要求したり、スコープをエスカレートしたりすることで、その規則を適用する場所です。完全なセキュリティチャプターはフェーズ13・15を参照し、エージェントレベルのパーミッションポリシーはフェーズ14・09を参照してください。

### ループが存在する場所

| コンテキスト | 誰が描写するか | 誰が決定するか | 誰が実行するか |
|---------|---------------|-------------|--------------|
| シングルターン関数呼び出し（OpenAI/Anthropic/Gemini） | アプリ開発者 | LLM | アプリ開発者 |
| MCP | MCPサーバー | MCPクライアント経由LLM | MCPサーバー |
| A2A | エージェントカード発行者 | 呼び出しエージェント | 呼び出されたエージェント |
| Webブラウザ（関数呼び出しエージェント） | ブラウザ拡張機能 / WebMCP | LLM | ブラウザランタイム |

どこでも、同じ4つのステップです。列名が変わります。構造は変わりません。

### なぜモデルに単にJSONを放出するよう促さないのか？

「モデルにJSONで返信するように求める」は関数呼び出し前のパターンでした。これは最前線のモデルでは約5～15％の時間に失敗し、より小さいモデルでははるかに多くの時間に失敗します。障害モードには、欠落している括弧、末尾のコンマ、幻想化されたフィールド、および間違った型が含まれます。その後、JSONの修復パス、再試行、または制約付きデコーダーが必要になります。

ネイティブ関数呼び出しは3つの理由でより優れています。まず、プロバイダーは正確な呼び出し形状でモデルをエンドツーエンドで訓練するため、有効なJSON率は厳密モードで98～99％に上昇します。次に、呼び出しペイロードは独自のプロトコルスロットに配置され、自由テキスト内ではないため、ツール呼び出しがユーザーに見える返信にリークされることはありません。第3に、プロバイダーは制約付きデコーディング（OpenAIの厳密モード、Anthropicの`tool_use`、Geminiの`responseSchema`）でスキーマ準拠を適用します。出力は検証が保証されます。

フェーズ13・02は3つのプロバイダーAPIを並べて説明します。フェーズ13・04は構造化出力に深く掘り込みます。

### サーキットブレーカー

ループは、モデルが呼び出しの放出を停止するか、ホストが最大ターン数に達したときに終了します。本番ホストはこれを5～20ターンの間に設定します。それ以上は、ほぼ確実にモデルが出られないループに入っています。Claude Codeはデフォルト20に設定されています。OpenAI Assistantsは10に。Cursorのエージェントモードは25に。

代替案 — 無制限ループ — は6ヶ月ごとに「エージェントが夜間にAPI呼び出しで$400を費やした」戦後として表示されます。バウンドなしで出荷しないでください。

フェーズ14・12はエラー回復と自己修復について詳しく説明しています。フェーズ17は本番レート制限をカバーしています。

### フェーズ13がここから進む場所

- レッスン02～05は、プロバイダーレベルのツール呼び出し表面を磨きます。
- レッスン06～14は、ループをMCPに汎用化します。
- レッスン15～18は、ループを敵対的サーバー、敵対的ユーザー、および認証されていないリモート認証面から守ります。
- レッスン19～22は、エージェント間コラボレーション、可観測性、ルーティング、およびパッケージング向けのパターンを拡張します。
- レッスン23は、すべてのプリミティブを使用して完全なエコシステムを出荷します。

残りのすべてのレッスンは、この4ステップループの精密化です。それを不変量として心に留めておいてください。

## 使用方法

`code/main.py`はLLMなしで4ステップループを実行します。偽の「決定」関数はユーザーメッセージにパターンマッチングすることでモデルをシミュレートします。実行関数、スキーマ検証、および観察ステップハーネスは本物です。完全な要求/応答の振付け付き印刷可能な中間状態を見るために実行してから、偽の決定を後のレッスンで実際のプロバイダーに置き換えます。

見るべきもの：

- ツールレジストリはツールごとに3つのフィールドを保持します：名前、説明、スキーマ、および実行関数参照。
- バリデータは、stdlibのみで記述された最小限のJSONスキーマサブセット（タイプ、必須、列挙、最小/最大）です。フェーズ13・04はより完全なものを配布します。
- ループは反復回数を5で制限します。本番エージェントには、まさにこの種のサーキットブレーカーが必要です。

## 出荷

このレッスンは`outputs/skill-tool-interface-reviewer.md`を生成します。ドラフトツール定義（名前+説明+スキーマ+実行関数の概要）が与えられた場合、スキルはループフィットネス監査を行います：名前は機械的に安定しているか、説明は完全な使用概要か、スキーマはJSONスキーマ2020-12を正しく使用しているか、純粋対重要な分類は明示的か。

## 演習

1. `code/main.py`に`get_stock_price(ticker)`という4番目のツールを追加します。その説明を「ユーザーがティッカーで現在の株価を尋ねるときに使用します。履歴価格や市場サマリーには使用しないでください。」と記述します。ハーネスを実行し、偽の決定がティッカーに関するクエリを新しいツールにルーティングすることを確認します。

2. スキーマ検証を破棄します。必須フィールドが欠落している`arguments`オブジェクトを持つ呼び出しを渡し、ホストが実行前にそれを拒否することを確認します。次に、追加の未知フィールドを持つ呼び出しを渡します。決定：ホストは拒否すべきか、それとも無視すべきか？安全性議論で選択を正当化してください。

3. ハーネス内の各ツールを純粋または重要として分類します。必要なレジストリエントリに`consequential: true`フラグを追加し、重要なツールが選択されるたびに「ユーザーに確認するだろう」ラインを出力するようにループを変更します。これは、すべての本番ホストが必要とする確認ゲートの形状です。

4. 上記の提供者列テーブルを埋めた紙に4ステップループを描きます。お気に入りのクライアント（Claude Desktop、Cursor、ChatGPT、またはカスタムスタック）のために。フェーズ13・06のMCP固有の変種と相互参照してください。

5. OpenAIの関数呼び出しガイドを上から下まで読んでください。ここで提示されている4ステップループにはあるが、リクエストには存在しないフィールドを特定します。それが何を追加し、なぜそれが必須ではなく便利なのかを説明します。

## キーターム

| 用語 | 人々が言うこと | それが実際に意味するもの |
|------|----------------|------------------------|
| ツール | 「モデルが呼び出せるもの」 | 名前+JSONスキーマ型入力+実行関数の3つ組 |
| 関数呼び出し | 「ネイティブツール使用」 | 散文ではなく構造化ツール呼び出しを放出するためのプロバイダーレベルAPIサポート |
| ツール呼び出し | 「モデルの行動要求」 | モデルが放出した`id`、`name`、`arguments`を含むJSONペイロード |
| ツール結果 | 「ツールが返したもの」 | 一致するIDを持つ`tool`ロールメッセージでラップされた実行関数の出力 |
| 並列ツール呼び出し | 「一度に多くの呼び出し」 | 1つのモデルターン内の複数呼び出しオブジェクト、独立で、IDで順序付け可能 |
| 厳密モード | 「保証されたJSON」 | 宣言されたスキーマに対してモデルの出力を検証するように強制する制約付きデコーディング |
| 純粋ツール | 「読み取り専用ツール」 | 副作用なし。再実行するのは安全 |
| 重要ツール | 「アクションツール」 | 外部状態を変更します。ゲート、監査、またはユーザー確認が必要 |
| 4ステップループ | 「ツール呼び出しサイクル」 | 描写→決定→実行→観察 |
| ホスト | 「エージェントランタイム」 | ツールレジストリを保持し、モデルを呼び出し、実行関数を実行するプログラム |
