# MCP トランスポート — stdio と Streamable HTTP と SSE からの移行

> stdio はローカルでのみ機能し、他の場所では機能しません。Streamable HTTP（2025-03-26）がリモート標準です。古い HTTP+SSE トランスポートは非推奨であり、2026年中盤に削除される予定です。正しいトランスポートを選択することで移行が不要になり、セッションの継続性と DNS リバインディング保護を備えたリモートでホスト可能な MCP サーバーが実現します。

**タイプ:** Learn
**言語:** Python（stdlib、Streamable HTTP エンドポイントスケルトン）
**前提条件:** Phase 13 · 07、08（MCP サーバーとクライアント）
**所要時間:** 約45分

## 学習目標

- デプロイ形態（ローカル vs リモート、シングルプロセス vs フリート）に基づいて stdio と Streamable HTTP を選択する。
- Streamable HTTP シングルエンドポイントパターンを実装する：リクエスト用の POST、セッションストリーム用の GET。
- `Origin` 検証とセッション ID のセマンティクスを強制して DNS リバインディングを防止する。
- 2026年中盤の削除期限前に、レガシー HTTP+SSE サーバーを Streamable HTTP に移行する。

## 問題

最初の MCP リモートトランスポート（2024-11）は HTTP+SSE でした：2つのエンドポイント、1つはクライアントの POST 用、もう1つはサーバーからクライアントへのストリーム用の Server-Sent-Events チャネル。機能していました。また問題もありました：セッションごとに2つのエンドポイント、一部の CDN の前のキャッシュが壊れ、長時間存続する SSE 接続への依存があり、一部の WAF が積極的に終了させます。

2025-03-26 の仕様はこれを Streamable HTTP に置き換えました：1つのエンドポイント、クライアントリクエスト用の POST、セッションストリーム確立用の GET、両者が `Mcp-Session-Id` ヘッダーを共有します。その後構築または移行されたすべてのサーバーは Streamable HTTP を使用しています。古い SSE モードは非推奨になっています — Atlassian Rovo が 2026年6月30日に削除、Keboola が 2026年4月1日、ほとんどの残存するエンタープライズサーバーが 2026年末までに削除します。

そして stdio はローカルサーバーにとってなお重要です。Claude Desktop、VS Code、すべての IDE 型クライアントは stdio 経由でサーバーを生成します。正しいメンタルモデルは：stdio は「このマシン」用、Streamable HTTP は「ネットワーク経由」用。交差なし。

## コンセプト

### stdio

- 子プロセストランスポート。クライアントがサーバーを生成し、stdin/stdout 経由で通信します。
- 1 行に 1 つの JSON オブジェクト。改行区切り。
- セッション ID なし；プロセス ID がセッションです。
- 認証不要（子プロセスは親の信頼境界を継承）。
- リモートサーバーには決して使用しないでください — SSH または socat でトンネルする必要がありますが、その場合は Streamable HTTP を使用してください。

### Streamable HTTP

単一のエンドポイント `/mcp`（または任意のパス）。3つの HTTP メソッドをサポート：

- **POST /mcp.** クライアントが JSON-RPC メッセージを送信します。サーバーは単一の JSON レスポンスまたは 1 つ以上のレスポンスの SSE ストリームで返信します（バッチレスポンスと関連する通知に便利）。
- **GET /mcp.** クライアントが長時間存続する SSE チャネルを開きます。サーバーはサーバーからクライアントへのリクエスト（サンプリング、通知、引き出し）に使用します。
- **DELETE /mcp.** クライアントがセッションを明示的に終了します。

セッションは、サーバーが最初のレスポンスで設定する `Mcp-Session-Id` ヘッダーとクライアントが後続のすべてのリクエストでエコーバックするもので識別されます。セッション ID は暗号的にランダムである必要があります（128ビット以上）；クライアントが選択した ID は安全上の理由で拒否されます。

### シングルエンドポイント vs 2つ

古い仕様からの 2 エンドポイントモードは 2026年でもまだ呼び出し可能です — 仕様では「レガシー互換」と宣言しています。しかし、すべての新しいサーバーはシングルエンドポイントであるべきです。公式 SDK はシングルエンドポイントを発行します；レガシーモードは移行されていないリモートと通信する場合のみ使用してください。

### `Origin` 検証と DNS リバインディング

ブラウザは MCP クライアントではありませんが（今のところ）、攻撃者は `localhost:1234/mcp` に POST するようブラウザを説得するウェブページを作成できます — ユーザーのローカル MCP サーバーがリッスンしている場所。サーバーが `Origin` をチェックしない場合、ブラウザの同一オリジンポリシーはそれを保存しません。なぜなら `Origin: http://evil.com` は有効なクロスオリジンだからです。

2025-11-25 仕様では、サーバーが `Origin` がアローリストにない場合、リクエストを拒否する必要があります。アローリストは通常、MCP クライアントホスト（`https://claude.ai`、`vscode-webview://*`）とローカルバリアントを含みます。

### セッション ID のライフサイクル

1. クライアントが `Mcp-Session-Id` なしで最初のリクエストを送信します。
2. サーバーがランダム ID を割り当て、レスポンスヘッダーで `Mcp-Session-Id` を設定します。
3. クライアントがストリーム用のすべての後続リクエストと `GET /mcp` でそのヘッダーをエコーします。
4. セッションはサーバーが取り消すことができます；クライアントは後続のリクエストで 404 を見ており、再初期化する必要があります。
5. クライアントがセッションを明示的に DELETE してクリーンシャットダウンできます。

### キープアライブと再接続

SSE 接続はドロップします。クライアントは同じ `Mcp-Session-Id` で再 GET することで再確立します。サーバーは障害中に逃したイベントをキューに入れ（妥当なウィンドウまで）、クライアントがエコーバックする `last-event-id` ヘッダー経由で再生する必要があります。

Phase 13 · 13 は、タスクについて説明しており、長時間実行される作業が完全なセッション再接続でも存続するようにします。

### バックワード互換性プローブ

古い新しいサーバーの両方をサポートしたいクライアント：

1. `/mcp` に POST します。
2. レスポンスが `200 OK` で JSON または SSE の場合、これは Streamable HTTP です。
3. レスポンスが `200 OK` で `Content-Type: text/event-stream` AND セカンダリエンドポイントを指す `Location` ヘッダーの場合、これはレガシー HTTP+SSE です；`Location` に従ってください。

### Cloudflare、ngrok、ホスティング

2026年のプロダクション リモート MCP サーバーは、Cloudflare Workers（MCP Agents SDK を使用）、Vercel Functions、またはコンテナ化された Node/Python で実行されます。重要なのは：ホスティングは SSE GET の長時間存続する HTTP 接続をサポートする必要があります。Vercel の無料枠は 10 秒でキャップされており、不適切です。Cloudflare Workers は無制限ストリームをサポートします。

### ゲートウェイ構成

複数の MCP サーバーをゲートウェイの前に置く場合（Phase 13 · 17）、ゲートウェイはシングルの Streamable HTTP エンドポイントで、セッション ID を書き直し、上流をマルチプレックスしています。ツールはゲートウェイレイヤーでマージされます；クライアントは単一の論理サーバーを見ます。

### トランスポート障害モード

- **stdio SIGPIPE.** 子プロセスが書き込み中に死ぬと SIGPIPE が発生します；サーバーはクリーンに終了する必要があります。クライアントは EOF を検出してセッションを dead とマークする必要があります。
- **HTTP 502 / 504.** Cloudflare、nginx、その他のプロキシが上流障害で発行します。Streamable HTTP クライアントは短い backoff 後に 1 回再試行する必要があります。
- **SSE 接続ドロップ。** TCP RST、プロキシタイムアウト、またはクライアントネットワーク変更がストリームを閉じます。クライアントは `Mcp-Session-Id` と optional な `last-event-id` で再接続して再開します。
- **セッション取り消し。** サーバーがセッション ID を無効化します；クライアントは次のリクエストで 404 を見ます。クライアントは再度ハンドシェイクする必要があります。
- **クロックスキュー。** リソース TTL 計算はクライアントとサーバーで分岐します。クライアントはサーバータイムスタンプを権限として扱うべきです。

### Streamable HTTP をバイパスする場合

一部のエンタープライズは MCP サーバーを gRPC またはメッセージキュートランスポートの背後に独自ネットワーク内にデプロイします。これは非標準です — MCP 仕様はこれらを正式に定義していません。ゲートウェイは gRPC を内部で使用しながら、MCP クライアントに Streamable HTTP サーフェスを公開できます。外部サーフェスは仕様準拠に保つ；ゲートウェイが翻訳を所有します。

## Use It

`code/main.py` は `http.server`（stdlib）を使用した最小限の Streamable HTTP エンドポイントを実装しています。POST、GET、DELETE を `/mcp` で処理し、最初のレスポンスで `Mcp-Session-Id` を設定し、`Origin` を検証し、許可されていないオリジンからのリクエストを拒否します。ハンドラーは Lesson 07 ノートサーバーのディスパッチロジックを再利用します。

見るべきこと：

- POST ハンドラーは JSON-RPC ボディを読み込み、ディスパッチし、JSON レスポンスを書き込みます（シングルレスポンスバリアント；SSE バリアントは構造的に類似）。
- `Origin` チェックはデフォルト `http://evil.example` プローブを拒否しますが `http://localhost` を受け入れます。
- セッション ID はランダムな 128ビット 16進数文字列です；サーバーはメモリ内にセッションごとの状態を保持します。

## Ship It

このレッスンは `outputs/skill-mcp-transport-migrator.md` を作成します。HTTP+SSE（レガシー）MCP サーバーが与えられた場合、スキルは Streamable HTTP への移行計画、セッション ID の継続性、Origin チェック、バックワード互換性プローブサポートを生成します。

## 演習

1. `code/main.py` を実行してください。`curl` から `initialize` を POST し、`Mcp-Session-Id` レスポンスヘッダーを観察してください。ヘッダーをエコーバックして 2 番目のリクエストを POST し、セッションの継続性を検証してください。

2. SSE ストリームを開く GET ハンドラーを追加してください。5 秒ごとに 1 つの `notifications/progress` イベントを送信してください。同じセッション ID で再 GET することで再接続し、サーバーがそれを受け入れることを確認してください。

3. `last-event-id` 再生ロジックを実装してください。再接続時に、その ID 以降に生成されたすべてのイベントを再生してください。

4. `Origin` 検証を拡張してワイルドカードパターン（`https://*.example.com`）をサポートし、`https://app.example.com` を受け入れるが `https://evil.example.com.attacker.net` を拒否することを確認してください。

5. 公式レジストリからレガシー HTTP+SSE サーバーを取得し（いくつか存在します）、移行をスケッチしてください：エンドポイント処理、セッション ID 生成、ヘッダーセマンティクスで何が変わるか。

## 主要用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| stdio トランスポート | 「ローカル子プロセス」 | JSON-RPC over stdin/stdout、改行区切り |
| Streamable HTTP | 「リモートトランスポート」 | シングルエンドポイント POST + GET + optional SSE、2025-03-26 仕様 |
| HTTP+SSE | 「レガシー」 | 2026年中盤に削除される 2 エンドポイントモデル |
| `Mcp-Session-Id` | 「セッションヘッダー」 | サーバー割り当てのランダム ID で後続のすべてのリクエストでエコーバック |
| `Origin` アローリスト | 「DNS リバインディング防御」 | Origin が承認されていない場合リクエストを拒否 |
| シングルエンドポイント | 「1 つの URL」 | `/mcp` が全セッション操作の POST / GET / DELETE を処理 |
| `last-event-id` | 「SSE 再生」 | ドロップされたストリームを再開するために使用されるヘッダー |
| バックワード互換性プローブ | 「古い vs 新しい検出」 | クライアント応答形状チェックで自動的にトランスポートを選択 |
| ロングライブ HTTP | 「SSE ストリーミング」 | サーバーが 1 つの TCP 接続で数分から数時間のイベントをプッシュ |
| セッション取り消し | 「強制再初期化」 | サーバーがセッション ID を無効化；クライアントは再度ハンドシェイク |
