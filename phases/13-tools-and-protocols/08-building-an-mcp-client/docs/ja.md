# MCPクライアント構築 — 発見、起動、セッション管理

> ほとんどのMCPコンテンツはサーバーチュートリアルを配布しクライアントで手を振る。クライアントコードは、ハード編成が住う場所：プロセス起動、キャパビリティネゴシエーション、複数サーバーを横断ツールリスト統合、サンプリングコールバック、再接続、および名前空間競合解決。このレッスンはマルチサーバークライアント構築し、3つの異なるMCPサーバーを1つ平ツール名前空間にリフト。

**タイプ:** ビルド
**言語:** Python（stdlib、マルチサーバーMCPクライアント）
**前提条件:** フェーズ13・07（MCPサーバー構築）
**所要時間:** 約75分

## 学習目標

- MCPサーバーを子プロセスとして起動し、`initialize`完了、`notifications/initialized`を送信。
- サーバーあたりセッション状態保持（キャパビリティ、ツールリスト、最後見通知ID）。
- 複数サーバーを横断ツールリストを1つの名前空間に統合し競合処理。
- ツール呼び出しをサーバーをそれが所有するにルーティングして応答を再組立。

## 問題

実エージェントホスト（Claude Desktop、Cursor、Goose、Gemini CLI）は一度複数MCPサーバーをロード。ユーザーはファイルシステムサーバー、Postgresサーバー、GitHubサーバーを同時実行することが持つかもしれません。クライアントの仕事：

1. 各サーバーを起動。
2. 各々独立にハンドシェイク。
3. 各々`tools/list`呼び出し結果を平坦化。
4. モデルが`notes_search`放出するとき、統合名前空間内それをルックアップしてサーバーに所属者へのルーティング。
5. いずれかのサーバーから通知（`tools/list_changed`）処理、ブロック無しに。
6. トランスポート失敗で再接続。

すべてそれを手では、「玩具」から「利用可能」を分離するもの。公開SDKはこれをラップしますが、メンタルモデルはあなたの方である必要。

## コンセプト

### 子プロセス起動

`subprocess.Popen`を持つ`stdin=PIPE, stdout=PIPE, stderr=PIPE`。`bufsize=1`セットし、行行ごとの読むためテキストモードを使用。各サーバーが1つのプロセス；クライアントはサーバーあたり1つの`Popen`ハンドル。

### サーバーあたりセッション状態

`Session`オブジェクトサーバーあたり保持：

- `process` — Popen ハンドル。
- `capabilities` — サーバーが`initialize`で宣言した。
- `tools` — 最後`tools/list`結果。
- `pending` — リクエスト ID からプロミス/フューチャー応答を待つためのマップ。

リクエストが非同期本質的；`tools/call`サーバーAに送られた間サーバーBが呼び出し中である必要があります、ブロック。スレッド持つキューまたは asyncio のいずれかを使用。

### 統合名前空間

クライアントは集計ツールリストを見るとき、名前が衝突することができます。2つのサーバーが両方露出`search`が可能。クライアントが3つのオプション：

1. **サーバー名でプレフィックス。** `notes/search`、`files/search`。明確しかし醜い。
2. **サイレント最初来。** 後者サーバーの`search`前者をオーバーライド。リスク；衝突隠す。
3. **衝突拒否。** 2番目サーバーをロード拒否；ユーザー通知。セキュリティ敏感ホストに安全。

Claude Desktop はサーバー別プレフィックス使用。Cursor は衝突拒否を使用明確エラー。VS Code MCP はサーバー別プレフィックスを採用。

### ルーティング

統合後、派遣テーブルマップ`tool_name -> session`。モデル名で呼び出し放出；クライアントはセッション見つけ、`tools/call`メッセージをそのサーバーのstdinに書く、その後応答を待つ。

### サンプリングコールバック

サーバーが`sampling`キャパビリティを`initialize`で宣言した場合、それは送信ことが`sampling/createMessage`クライアントをそのLLMを実行するようにリクエストして。クライアントは：

1. そのサーバーに対してさらなるリクエストをブロック、またはその実装が並行性サポート場合パイプライン。
2. LLMプロバイダー呼び出し。
3. サーバーに応答送信。

レッスン11はサンプリング。このレッスンは完全性ためスタブ。

### 通知処理

`notifications/tools/list_changed`は`tools/list`再呼び出しを意味。`notifications/resources/updated`はリソースが使用中場合は再読を意味。通知は応答を生成してはいけません — ack してはいけません。

一般的クライアントバグ：`tools/call`で読ループをブロックする通知がストリーム内に座っている間。使用バックグラウンドリーダースレッドはメイン処理がデキュー派遣中のすべてのメッセージをキューにプッシュ。

### 再接続

トランスポートが失敗ことができます：サーバークラッシュ、OSがプロセス殺害、stdioパイプ破れ。クライアントはstdout EOF 検出し、セッションを死タイトル扱う。オプション：

- サーバーをサイレント再起動、ハンドシェイク再度。読み取り専用サーバーのためOK。
- 失敗をユーザーに表面。ステートフル ユーザー可視セッション持つサーバーのためOK。

フェーズ13・09はStreamable HTTP再接続セマンティック；stdioはシンプル。

### キープアライブおよびセッション ID

Streamable HTTP は`Mcp-Session-Id`ヘッダを使用。Stdioはセッション ID ない — プロセスアイデンティティ IS セッション。キープアライブpingはオプション；stdioパイプは不活動の下破れない。

## 使用方法

`code/main.py`は3つのシミュレートされたMCPサーバーをサブプロセスとして起動し、各ハンドシェイク、ツールリストをマージし、ツール呼び出しを正しいもにルーティング。「サーバー」は実際他々のPython プロセスを実行する玩具応答（本物LLMなし）。実行して見：

- 3つの初期化、それぞれそれ自身のキャパビリティセット。
- 3つの`tools/list`結果は7ツール名前空間に統合。
- ツール名に基づく派遣判定。
- 名前空間プレフィックスで防止衝突。

見るべきもの：

- `Session`データクラスはサーバー単位状態きれい保持。
- バックグラウンドリーダースレッドはメインスレッドをブロックすることなくstdout の全行をデキュー。
- 派遣テーブルは単純`dict[str, Session]`。
- 衝突処理は明確：2つのサーバーが同名を宣言するとき、後者は名プレフィックスで名を変更。

## 出荷

このレッスンは`outputs/skill-mcp-client-harness.md`を生成します。MCPサーバーの宣言リスト（名、コマンド、args）が与えられた場合、スキルは衝突解決を持つそれらを起動、ツールリストマージ、ルーティング関数を配布するハーネスを生成。

## 演習

1. `code/main.py`を実行しサーバー起動ログを観察。シミュレートされたサーバープロセスの1つをSIGTERM で殺してクライアントがEOFを検出し、セッションを死でマークすることを観察。

2. 名前空間プレフィックスを実装。2つのサーバーが`search`露出するとき、2番目を`<server>/search`として名を変更。派遣テーブル更新、ツール呼び出しが正確ルーティングを確認。

3. サーバー再起動の接続プール形スタイルバックオフを追加：連続失敗の指数バックオフ、30秒でキャップ、3つの失敗後ユーザーに通知を放出。

4. 100の並行MCPサーバーをサポートクライアントをスケッチ。単純派遣dict を置き換えるデータ構造。（ヒント：プレフィックス名前空間のためのトライ、プラスメトリクスツール数-ごと-サーバー。）

5. 公開MCP Python SDKに対してクライアントをポート。SDKは`stdio_client`と`ClientSession`をラップ。コードは~200行から~40行に縮小しながらマルチサーバールーティング保持。

## キーターム

| 用語 | 人々が言うこと | それが実際に意味するもの |
|------|----------------|------------------------|
| MCPクライアント | 「エージェントホスト」 | サーバーを起動してツール呼び出し編成するプロセス |
| セッション | 「サーバーあたり状態」 | キャパビリティ、ツールリスト、ペンディングリクエスト簿記 |
| 統合名前空間 | 「1つのツールリスト」 | すべてのアクティブサーバー全体の平ツール名セット |
| 名前空間衝突 | 「2つのサーバー同じツール」 | クライアントはプレフィックス、拒否、または最初来する必要 |
| ルーティング | 「誰がこれ呼び出しを得るか。」 | ツール名からサーバーをしてルーティング所有 |
| バックグラウンドリーダー | 「非ブロックstdout」 | スレッドまたはタスク、キューにサーバーstdout の全行排出 |
| サンプリングコールバック | 「サービスとしてLLM」 | サーバーから`sampling/createMessage`のためのクライアントハンドラー |
| `notifications/*_changed` | 「プリミティブ変更」 | プリミティブリストがサーバーが変更クライアント必須シグナル再発見または再読 |
| 再接続ポリシー | 「サーバーが死ぬとき」 | トランスポート失敗したときの再起動セマンティック |
| Stdio セッション | 「プロセス = セッション」 | セッション ID なし；子プロセスライフタイムはセッション |
