# MCPゲートウェイとレジストリ — エンタープライズコントロールプレーン

> エンタープライズでは、すべての開発者が任意のMCPサーバーをインストールすることはできません。ゲートウェイは認証、RBAC、監査、レート制限、キャッシング、ツール中毒検出を一元化し、マージされたツールサーフェスを単一のMCPエンドポイントとして公開します。公式MCPレジストリ（Anthropic + GitHub + PulseMCP + Microsoft、名前空間検証済み）は正規の上流です。このレッスンではゲートウェイの位置付けを説明し、最小限の実装を実装し、2026年のベンダーランドスケープを調査します。

**タイプ:** Learn
**言語:** Python (stdlib, minimal gateway)
**前提条件:** Phase 13 · 15 (tool poisoning), Phase 13 · 16 (OAuth 2.1)
**所要時間:** 約45分

## 学習目標

- MCPゲートウェイがどこに位置するかを説明する（MCPクライアントと複数のバックエンドMCPサーバー間）。
- ゲートウェイの5つの責務を実装する：認証、RBAC、監査、レート制限、ポリシー。
- ゲートウェイレイヤーで固定ツールハッシュマニフェストを実装する。
- 公式MCPレジストリと多言語レジストリ（Glama、MCPMarket、MCP.so、Smithery、LobeHub）を区別する。

## 問題

フォーチュン500の企業は30個の承認されたMCPサーバー、5000人の開発者、コンプライアンスと監査要件、そして一元化されたポリシーを望むセキュリティチームを持っています。すべての開発者がIDEで任意のサーバーをインストールすることを許可することはできません。

ゲートウェイパターン：

1. ゲートウェイは、開発者が接続する単一のStreamable HTTPエンドポイントとして実行されます。
2. ゲートウェイは各バックエンドMCPサーバーの認証情報を保持します。
3. すべての開発者要求は、ゲートウェイ独自のOAuthを介して認証およびスコープ設定されます。
4. ゲートウェイは呼び出しをバックエンドサーバーにルーティングし、ポリシーを適用します。
5. すべての呼び出しが監査のためにログされます。

Cloudflare MCP Portals、Kong AI Gateway、IBM ContextForge、MintMCP、TrueFoundry、Envoy AI Gatewayはすべて2025-2026年にゲートウェイまたはゲートウェイ機能をリリースしました。

一方、公式MCPレジストリが正規の上流として起動しました：ゲートウェイが取得できる、厳選された、名前空間検証済みの、逆DNS名のサーバーです。多言語レジストリ（Glama、MCPMarket、MCP.so、Smithery、LobeHub）は複数のソースからサーバーを集約します。

## コンセプト

### ゲートウェイの5つの責務

1. **認証。** OAuth 2.1で開発者を識別し、ユーザーロールにマッピングします。
2. **RBAC。** ユーザー別ポリシー：どのサーバー、どのツール、どのスコープ。
3. **監査。** すべての呼び出しをログに記録し、誰が、何を、いつ、結果を記録します。
4. **レート制限。** ユーザー/ツール/サーバーあたりの上限を設定し、悪用を防止します。
5. **ポリシー。** 中毒された説明を拒否し、Rule of Twoを実装し、PIIを編集します。

### 単一エンドポイントとしてのゲートウェイ

開発者にとって、ゲートウェイは1つのMCPサーバーのように見えます。内部的には、Nのバックエンドにルーティングします。セッションID（Phase 13 · 09）は境界で書き換えられます。

### 認証情報ボルト

開発者はバックエンドトークンを見ることはありません。ゲートウェイがそれらを保持します（または、それを行うIDプロバイダーにプロキシします）。ゲートウェイで`notes:read`を持つ開発者は、ゲートウェイの独自のバックエンド認証情報を使用してノートMCPサーバーに推移的にアクセスできます。ただし、推移的アクセスをバインドするポリシーの下でのみです。

### ゲートウェイでのツールハッシュ固定

ゲートウェイは承認されたツール説明のマニフェスト（SHA256ハッシュ）を保持します。検出時に、各バックエンドの`tools/list`を取得し、ハッシュをマニフェストと比較し、説明が変更されたツールを削除します。これはPhase 13 · 15のrug-pull防御を一元化して適用したものです。

### ポリシー・アズ・コード

高度なゲートウェイは、OPA/Rego、Kyverno、またはStyraでポリシーを表現します。`user alice`が`github.open_pr`を`acme`組織のリポジトリでのみ呼び出せるなどのルールは、宣言的にエンコードされます。シンプルなゲートウェイは手動コード化されたPythonを使用します。両方の形式は有効です。

### セッション対応ルーティング

ユーザーのセッションに複数のサーバーが含まれている場合、ゲートウェイはマルチプレックス：開発者の単一MCPセッションはNのバックエンドセッションを保持し、サーバーあたり1つ。任意のバックエンドからの通知はゲートウェイを通じて開発者のセッションにルーティングされます。

### 名前空間マージ

ゲートウェイはすべてのバックエンドからツール名前空間をマージします。通常は衝突時にプレフィックスを付けます。`github.open_pr`、`notes.search`。これはルーティングを明確にします。

### レジストリ

- **公式MCPレジストリ（`registry.modelcontextprotocol.io`）。** Anthropic、GitHub、PulseMCP、Microsoftの管理下で起動。名前空間検証済み（逆DNS：`io.github.user/server`）。基本的な品質のための事前フィルタリング。
- **Glama。** 多くのソースを集約する検索中心のマルチレジストリ。
- **MCPMarket。** 商用志向のディレクトリとベンダーリスティング。
- **MCP.so。** コミュニティディレクトリ。オープン投稿。
- **Smithery。** パッケージマネージャースタイルのインストールフロー。
- **LobeHub。** LobeChat アプリのUI統合レジストリ。

エンタープライズゲートウェイはデフォルトで公式レジストリから取得し、管理者がキュレートした追加を多言語レジストリから許可し、固定されていないものを拒否します。

### 逆DNS命名

公式レジストリは公開サーバーに対して逆DNS名を必須にします：`io.github.alice/notes`。名前空間は不正使用を防ぎ、信頼委譲を明確にします。

## 使用方法

`code/main.py`は約150行の最小限のゲートウェイを出荷します：フェイクベアラートークンでユーザーを認証し、ユーザーごとのRBACポリシーを保持し、2つのバックエンドMCPサーバーへのリクエストをルーティングし、すべての呼び出しを監査ログに書き込み、レート制限を適用し、説明ハッシュが固定マニフェストと一致しないバックエンドツールを拒否します。

確認すべき点：

- `RBAC`辞書は`user_id`で索引付けされ、許可されている`server_tool`エントリを持っています。
- `AUDIT_LOG`は追加のみのイベントリストです。
- レート制限はユーザーあたりのトークンバケットを使用します。
- 固定マニフェストは`server::tool -> hash`の辞書です。

## 演習

1. `code/main.py`を実行します。許可されたユーザーとして呼び出しを行い、次に許可されていないユーザーとして、次にレート制限超過バーストとして。3つのフロー全てを確認します。

2. 結果をクライアントに返す前にPIIを編集するポリシーを追加します。SSN形状の文字列に対して単純な正規表現パスを使用します。ギャップ（メール、電話番号）に注意してください。

3. 監査ログを拡張してOpenTelemetry GenAI スパンを出力します。Phase 13 · 20は正確な属性をカバーします。

4. 5つのバックエンド（notes、github、postgres、jira、slack）を持つ50人の開発者チームのRBACポリシーを設計します。誰が各々に対して読み取り専用を取得しますか？誰が書き込みを取得しますか？

5. Cloudflareエンタープライズ MCP ポストを上から下まで読みます。このstdlibゲートウェイが持たない1つの機能をCloudflareがリリースしていることを確認します。

## キー用語

| 用語 | 人々が言うこと | それが実際に意味すること |
|------|------------------|---------------------------|
| Gateway | "MCP プロキシ" | クライアントとバックエンド間の一元化サーバー |
| Credential vaulting | "バックエンドトークンはサーバー側に留まる" | 開発者は上流トークンを見ることはありません |
| Session-aware routing | "マルチバックエンドセッション" | ゲートウェイは開発者セッションあたりNのバックエンドセッションをマルチプレックス |
| Tool-hash pinning | "承認マニフェスト" | すべての承認されたツール説明のSHA256。rug-pullsを一元化してブロック |
| RBAC | "ユーザー別ポリシー" | ツールとサーバーの役割ベースアクセス制御 |
| Policy-as-code | "宣言的ルール" | ゲートウェイで実装されるOPA/Rego、Kyverno、Styraポリシー |
| Audit log | "誰が、何を、いつ" | コンプライアンスのための追加のみイベントログ |
| Rate limit | "ユーザーあたりトークンバケット" | 悪用を防ぐための1分あたりの上限 |
| Official MCP Registry | "正規の上流" | `registry.modelcontextprotocol.io`、名前空間検証済み |
| Reverse-DNS naming | "レジストリ名前空間" | `io.github.user/server`規約 |
