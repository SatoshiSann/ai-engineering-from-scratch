# 構造化出力 — JSONスキーマ、Pydantic、Zod、制約付きデコーディング

> 「モデルに親切にJSONを返すように求める」は5～15％の時間に失敗し、最先端のモデルでもそうです。構造化出力は制約付きデコーディングでそのギャップを閉じます：モデルはスキーマに違反するトークンを放出することが文字通り防止されます。OpenAIの厳密モード、Anthropicのスキーマ型ツール使用、Geminiの`responseSchema`、Pydantic AIの`output_type`、およびZodの`.parse`は同じアイデアの5つのサーフェス形式です。このレッスンは、スキーマ検証子と厳密モードコントラクトをビルドします。学習者が本番抽出パイプラインごとに使用するでしょう。

**タイプ:** ビルド
**言語:** Python（stdlib、JSONスキーマ2020-12サブセット）
**前提条件:** フェーズ13・02（関数呼び出し詳細解説）
**所要時間:** 約75分

## 学習目標

- 右の制約（enum、min/max、required、pattern）を使用してJSONスキーマ2020-12を抽出目標のために記述する。
- 厳密モードと制約付きデコーディング「生成後検証」から異なる保証をなぜ与えるか説明する。
- 3つの障害モード区別：パース誤り、スキーマ違反、モデル拒否。
- 型付き修復と型付き拒否処理で抽出パイプラインを配布します。

## 問題

購買発注メールを読むエージェントは自由テキストを`{customer, line_items, total_usd}`に変える必要があります。3つのアプローチ。

**アプローチ1：JSONを促す。** 「フィールドcustomer、line_items、total_usdを持つJSONで返信してください。」最先端のモデル上では85～95％の時間機能します。6つの方法で失敗：括弧が欠落している、末尾のコンマ、間違った型、幻想化されたフィールド、トークン制限で切り詰め、「ここはあなたのJSONです」のような漏出散文。

**アプローチ2：生成後検証。** 自由に生成、パース、スキーマに対して検証、失敗時再試行。信頼できるが高価 — すべての再試行に対して支払い、トランケーション バグは発生ごとに1つの余分なターンをコストします。

**アプローチ3：制約付きデコーディング。** プロバイダーはデコード時にスキーマを適用します。無効なトークンはサンプリング分布から外されます。出力は解析保証され、スキーマの検証が保証されます。障害は1つのモードに折りたたまれます：拒否（モデルは入力がスキーマに適合しないと判断）。

2026の最先端プロバイダーすべてはアプローチ3のいくつかの形式を配布します。

- **OpenAI。** `response_format: {type: "json_schema", strict: true}`プラス拒否（モデルが辞退する場合）。
- **Anthropic。** `tool_use`入力のスキーマ適用；`stop_reason: "refusal"`は物ではありませんが、ツール呼び出しを持つ`end_turn`がシグナルです。
- **Gemini。** リクエストレベルの`responseSchema`；2026年Geminiは選択型のトークンレベル文法制約を配布します。
- **Pydantic AI。** `output_type=InvoiceModel`は型付き`InvoiceModel`構造化`RunResult`を放出します。
- **Zod（TypeScript）。** Zodスキーマに対してプロバイダー出力を検証するランタイムパーサー；OpenAIの`beta.chat.completions.parse`と対になります。

共通スレッド：スキーマを一度宣言し、エンドツーエンドで適用してください。

## コンセプト

### JSONスキーマ2020-12 — リングア・フランカ

すべてのプロバイダーはJSONスキーマ2020-12を受け入れます。最も使用する構成：

- `type`: `object`、`array`、`string`、`number`、`integer`、`boolean`、`null`うち1つ。
- `properties`: フィールド名からサブスキーマへの地図。
- `required`: 表示される必要があるフィールド名のリスト。
- `enum`: 許可される値の閉じたセット。
- `minimum` / `maximum`（数字）、`minLength` / `maxLength` / `pattern`（文字列）。
- `items`: すべての配列要素に適用されるサブスキーマ。
- `additionalProperties`: `false`はエクストラフィールド禁止（デフォルトはモードによって異なる）。

OpenAI厳密モード3つの要件を追加：すべてのプロパティは`required`にリストされる必要があります、`additionalProperties: false`どこでも、および無決議`$ref`なし。これらを破るとAPIはリクエスト時に400を返します。

### Pydantic、Pythonバインディング

Pydantic v2 は`model_json_schema()`経由でDataclass型モデルからJSONスキーマを生成します。Pydantic AIは次のことをラップするため記述：

```python
class Invoice(BaseModel):
    customer: str
    line_items: list[LineItem]
    total_usd: Decimal
```

そしてエージェントフレームワークはスキーマをOpenAI厳密モード、Anthropic`input_schema`、またはGemini`responseSchema`をエッジで変換します。モデルの出力は型付き`Invoice`インスタンスが戻ります。検証エラーは型付きエラーパスで`ValidationError`を上げます。

### Zod、TypeScriptバインディング

Zod（`z.object({customer: z.string(), ...})`）はTS等価です。OpenAIのNode SDKは`zodResponseFormat(Invoice)`を公開し、APIのJSONスキーマペイロードに変換します。

### 拒否

厳密モードはモデルが答えるように強制することはできません。入力がスキーマに適合できない場合（「メールは請求書ではなく詩でした」）、モデルは理由を含む`refusal`フィールドを放出します。コードはこれを第1級の結果として処理する必要があります。失敗ではなく。拒否は安全シグナルとしても有用です：保護コンテンツメールから信用枠番号を抽出するように求められたモデルは安全理由が付いた拒否を返します。

### オープンウェイト制約付きデコーディング

オープン重量実装は3つの手法を使用します。

1. **文法ベースデコーディング**（`outlines`、`guidance`、`lm-format-enforcer`）：スキーマから確定的有限オートマトンを構築；すべてのステップで、FSMを違反するトークンのロジットをマスク。
2. **JSON パーサーを使用したロジットマスク**：ストリーミングJSONパーサーをモデルと同期で実行；すべてのステップで、有効なネクストトークンセットを計算。
3. **検証者付き投機的デコーディング**：廉価なドラフトモデル提案トークン、検証者はスキーマを適用。

商用プロバイダーは背後にあるこれらのうち1つを選ぶ。2026年最先端短い構造化出力の場合は通常の生成より高速で、長い出力の場合、大体同じ速度です。

### 3つの障害モード

1. **パース誤り。** 出力は有効なJSONではありません。厳密モード下では発生できません。厳密でないプロバイダー外では発生することができます。
2. **スキーマ違反。** 出力パースが完了しますがスキーマを違反。厳密モード下では発生できません。それ外で一般的。
3. **拒否。** モデル辞退。型付き結果として処理される必要があります。

### 再試行戦略

厳密モード外の場合（Anthropicツール使用、非厳密OpenAI、古いGemini）、回復パターン：

```
generate -> parse -> validate -> if fail, inject error and retry, max 3x
```

1回再試行は通常十分です。3回再試行は弱いモデルの不具合を捕まえます。3つを超えると不良スキーマのシンです：モデルはいくつかの入力のためにそれを満たすことができず、プロンプトまたはスキーマ修正が必要です。

### 小さいモデルサポート

制約付きデコーディングは小さいモデルで機能します。文法適用を持つ3Bパラメータ開放モデルは構造化タスク上で生の促進を持つ70Bパラメータモデルをアウト実行します。これは構造化出力が本番に対して重要である主な理由です：信頼性からモデルサイズを切り離します。

## 使用方法

`code/main.py` stdlibの最小限JSONスキーマ2020-12バリデータを配布（型、必須、列挙、最小/最大、パターン、アイテム、additionalProperties）。`Invoice`スキーマをラップし、偽のLLM出力をバリデータ通じて実行し、パース誤り、スキーマ違反、拒否パスを示します。本番でプロバイダーの実際の応答でフェイクを入れ替え。

見るべきもの：

- バリデータはパスとメッセージを持つ型付き`[ValidationError]`リストを返します。それは再試行プロンプトに表示する形状です。
- 拒否ブランチは再試行しない。ログし、型付き拒否を返します。フェーズ14・09は拒否を安全シグナルとして使用。
- `additionalProperties: false`チェックは敵対的テスト入力上で火が上がり、厳密モードがなぜ幻想化されたフィールドにドアをシャットします。

## 出荷

このレッスンは`outputs/skill-structured-output-designer.md`を生成します。自由テキスト抽出目標が与えられた場合（請求書、サポートチケット、履歴書など）、スキルは厳密モード互換のJSONスキーマ2020-12とそれを鏡する型付き拒否とリトライ処理がスタブされたPydanticモデルを生成します。

## 演習

1. `code/main.py`を実行する。4番目のテストケースを追加し、そのケースの`total_usd`は負の数である。バリデータが`minimum`制約パスでそれを拒否することを確認。

2. `oneOf`を識別子で対応するようにバリデータを拡張。一般的なケース：`line_item`は製品またはサービスのいずれかであり、`kind`によってタグ付け。厳密モードは微妙なルールがあります；OpenAIの構造化出力ガイドをチェック。

3. Pydanticの同じInvoiceスキーマ BaseModelを記述し`model_json_schema()`出力を手書きスキーマと比較。手書きバージョンが誰のデフォルト設定である1つのフィールドPydanticセットは省略するか特定。

4. 拒否率を計測。抽出できない10の入力を構成（歌詞、数学証明、空白メール）し、厳密モードで実際のプロバイダーを通してそれらを実行。拒否と幻想化された出力をカウント。これは拒否対応再試行の地上真実です。

5. OpenAIの構造化出力ガイドをトップツーボトムで読みます。厳密モードで明示的に禁止する1つの構成を特定します。それは通常のJSONスキーマが許可。次に禁止された構成を使用するスキーマを設計し、厳密互換にリファクタリング。
