# LLMルーティング層 — LiteLLM、OpenRouter、Portkey

> プロバイダロックインは高くつきます。異なるツール呼び出しワークロードは異なるモデルに適しています。ルーティングゲートウェイは1つのAPI表面、再試行、フェイルオーバー、コスト追跡、ガードレールを提供します。3つのアーキタイプが2026年を支配します：LiteLLM（オープンソース自己ホスト）、OpenRouter（マネージドSaaS）、Portkey（本番グレード、2026年3月にオープンソース化）。このレッスンは決定基準に名前を付け、stdlibルーティングゲートウェイを説明します。

**タイプ:** Learn
**言語:** Python (stdlib, ルーティング + フェイルオーバー + コスト追跡)
**前提条件:** Phase 13 · 02 (関数呼び出し), Phase 13 · 17 (ゲートウェイ)
**所要時間:** 約45分

## 学習目標

- 自己ホスト、マネージド、本番グレードのルーティングオプションを区別します。
- 定義された優先順序でプロバイダ障害を再試行するフォールバックチェーンを実装します。
- プロバイダ全体のリクエストごとのコストとトークン使用量を追跡します。
- 特定の本番環境制約に対してLiteLLM、OpenRouter、Portkeyのいずれかを決定します。

## 問題

プロバイダルーティングが重要なシナリオ：

1. **コスト。** Claude Sonnetはハイクの3倍の費用がかかります。トリアージタスクの場合、ハイクで十分です。合成タスクの場合、Sonnetはそれの価値があります。リクエストごとにルーティングします。

2. **フェイルオーバー。** OpenAIは悪い時間を持っています。すべてのリクエストが失敗します。再デプロイなしでAnthropicに自動フォールバックを希望します。

3. **レイテンシー。** ライブチャットUIは高速な時間の必要があります。バッチサマライザーは必要ありません。レイテンシーSLAでルーティングします。

4. **コンプライアンス。** EUユーザーはEUリージョンに留まる必要があります。地域でルーティングします。

5. **実験。** 同じワークロード上で2つのモデルをA/Bします。テストバケットでルーティングします。

統合ごとにこのすべてを手動コード化することは反復的です。ルーティングゲートウェイは1つのOpenAI互換APIを提供し、残りを処理します。

## コンセプト

### OpenAI互換プロキシ形状

誰もOpenAI形状を話します。ルーティングゲートウェイは`/v1/chat/completions`を公開し、OpenAIスキーマを受け入れ、内部的にAnthropicに代理 / Gemini / Cohere / Ollama / 何かに代理します。クライアントは気になりません。

### モデルエイリアス

`claude-3-5-sonnet-20251022`の代わりに、あなたのコードは`our_smart_model`と言います。ゲートウェイはエイリアスを実際のモデルにマッピングします。Anthropicが Claude 4 を配信する場合、ゲートウェイがサーバー側でエイリアスを変更します。あなたのコードは物事に触れません。

### フォールバックチェーン

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

ゲートウェイはコンフィグで定義します。再試行はバジェットに対してカウントされるため、フォールバックカスケードはコストを爆発させません。

### セマンティックキャッシング

同じまたはほぼ同じプロンプトはプロバイダの代わりにキャッシュに当たります。リピートエージェントループの節約は30～60パーセントです。キーは埋め込みベース。ほぼ同じプロンプトはキャッシュスロットを共有します。

### ガードレール

ゲートウェイレベル：

- **PII編集。** プロンプト送信前の正規表現またはMLベースのパス。
- **ポリシー違反。** 禁止コンテンツを含むプロンプトを拒否します。
- **出力フィルター。** リークの完了をスクラブします。

PortkeyとKongは両方とも定見のあるガードレールを配信します。LiteLLMはそれらをオプションのままにします。

### キーごとのレート制限

1つのAPIキー = 1つのチーム。キーごとのバジェットは、1つのチームが共有クォータを消費することを防ぎます。ほとんどのゲートウェイはこれをサポートしています。

### 自己ホスト対マネージドトレードオフ

| ファクタ | LiteLLM (自己ホスト) | OpenRouter (マネージド) | Portkey (本番) |
|--------|---------------------|----------------------|-----------------|
| コード | オープンソース、Python | マネージドSaaS | オープンソース(Mar 2026) + マネージド |
| セットアップ | プロキシをデプロイ | サインアップ | いずれか |
| プロバイダ | 100以上 | 300以上 | 100以上 |
| 請求 | 独自キー | OpenRouter クレジット | 独自キー |
| 可観測性 | OpenTelemetry | ダッシュボード | 完全OTel + PII編集 |
| 最適 | 完全制御を望むチーム | 急速なプロトタイピング | コンプライアンス付き本番環境 |

LiteLLMは、SREチームがあり、データソブリンティを望む場合に勝ちます。OpenRouterは、1つのサブスクリプションとインフラなしで希望する場合に勝ちます。Portkeyは、すぐにガードレールとコンプライアンスが必要な場合に勝ちます。

### コスト追跡

すべてのリクエストは`provider`、`model`、`input_tokens`、`output_tokens`を実行します。モデルごとのモデルごとのトークン価格（ゲートウェイが保持する価格表から取得）で乗算します。ユーザー/チーム/プロジェクト集約ごと。

### MCPプラスルーティング

ゲートウェイはLLM呼び出しとMCPサンプリングリクエストの両方をルーティングできます。サンプリングリクエストのmodelPreferencesが特定のモデルを好む場合、ゲートウェイは右側バックエンドに変換します。これは、Phase 13 · 17（MCPゲートウェイ）とこのレッスンのルーティングゲートウェイが時々1つのサービスにマージされる場所です。

### ルーティング戦略

- **静的優先度。** リストの最初。エラー上のフォールバック。
- **負荷分散。** ラウンドロビンまたは重み付け。
- **コスト認識。** レイテンシー/品質を満たす最も安いモデルを選択します。
- **レイテンシー認識。** 最後のN分間で最速のモデルを選択します。
- **タスク認識。** プロンプト分類器は1つのモデルへのコーディングをルーティングし、別のモデルへの要約をルーティングします。

## 使用方法

`code/main.py`は約150行のルーティングゲートウェイを実装します：OpenAI形状のリクエストを受け入れ、プロバイダスタブに変換し、優先度フォールバックチェーンを実行し、リクエストごとのコストを追跡し、入力にPII編集パスを適用します。3つのシナリオで実行します：通常のリクエスト、プライマリプロバイダの停止によるフォールバック、編集に誘発されるPIIリーク。

確認すべき点：

- `ROUTES`辞書：エイリアス -> 優先度順序の具体的なプロバイダリスト。
- フォールバックループは5xxで再試行。
- コスト追跡ツールはトークン使用量をモデルごとの率で乗算します。
- PII編集では、転送前にSSN形状パターンをスクラブします。

## リリース

このレッスンは`outputs/skill-routing-config-designer.md`を生成します。ワークロードプロファイル（レイテンシー、コスト、コンプライアンス）が与えられた場合、スキルはLiteLLM / OpenRouter / Portkeyを選択し、ルーティング設定を生成します。

## 演習

1. `code/main.py`を実行します。停止シナリオをトリガーします。フォールバックが2番目のプロバイダに着地し、コストが正しく属性付けされることを確認します。

2. セマンティックキャッシングを追加：プロンプトのSHA256はルックアップキー。キャッシュヒットはすぐに返されます。繰り返し呼び出しのコスト節約を測定します。

3. プロンプト分類器を追加：「code ...」プロンプトを知能を好むエイリアスに、「summarize ...」プロンプトをスピードを好むエイリアスにルーティング。

4. チームごとのバジェットを設計：各チームは月間支出上限を持つ。ゲートウェイはキャップに達するとリクエストを拒否します。実施の粒度（リクエストごとまたはウィンドウ）を選択します。

5. LiteLLM、OpenRouter、Portkey docsを並べて読みます。各々が配信する1つの機能を他の2つが配信しないことに名前を付けます。

## キー用語

| 用語 | 人々が言うこと | それが実際に意味すること |
|------|------------------|---------------------------|
| Routing gateway | "LLMプロキシ" | 多くのプロバイダの前の1-API表面層 |
| OpenAI-compatible | "OpenAI スキーマを話す" | `/v1/chat/completions`形状を受け入れ、任意のバックエンドに変換 |
| Model alias | "our_smart_model" | ゲートウェイが具体的なモデルにマッピングするコード内の名前 |
| Fallback chain | "リトライリスト" | 障害時に試行されたプロバイダの順序リスト |
| Semantic caching | "プロンプト埋め込みキャッシュ" | キーはプロンプトの埋め込み。ほぼ複製はキャッシュヒットを共有 |
| Guardrails | "入力/出力フィルタ" | PII編集、ポリシー違反を拒否 |
| Per-key rate limit | "チームバジェット" | APIキーに範囲があるクォータ |
| Cost tracking | "リクエストごと支出" | トークン使用量集約 x モデルごと価格 |
| LiteLLM | "オープンプロキシ" | 自己ホスト可能なOSSルーティングゲートウェイ |
| OpenRouter | "マネージドSaaS" | クレジットベース請求を持つホストゲートウェイ |
| Portkey | "本番オプション" | ガードレール組み込みのオープンソース + マネージド |
