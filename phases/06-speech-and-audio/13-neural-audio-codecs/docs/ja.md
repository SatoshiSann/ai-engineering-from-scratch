# ニューラル音声コーデック — EnCodec、SNAC、Mimi、DAC とセマンティック・音響の分離

> 2026年の音声生成はほとんどすべてトークンだ。EnCodec、SNAC、Mimi、DAC は連続波形を、トランスフォーマーが予測できる離散シーケンスに変換する。セマンティック対音響トークンの分離 — 最初のコードブックをセマンティック、残りを音響とするもの — は、音声分野におけるトランスフォーマー以来、最も重要なアーキテクチャ上の転換だ。

**タイプ:** Learn
**言語:** Python
**前提条件:** Phase 6 · 02 (Spectrograms)、Phase 10 · 11 (Quantization)、Phase 5 · 19 (Subword Tokenization)
**所要時間:** 約60分

## 問題

言語モデルは離散トークンを扱う。音声は連続的だ。音声 / 音楽向けに LLM スタイルのモデル — MusicGen、Moshi、Sesame CSM、VibeVoice、Orpheus — が欲しければ、まず**ニューラル音声コーデック**が必要だ。これは音声を小さな語彙のトークンに離散化する学習済みエンコーダと、波形を再構成する対になったデコーダだ。

2つのファミリーが登場している。

1. **再構成優先コーデック** — EnCodec、DAC。知覚的な音声品質を最適化する。トークンは「音響的」であり、話者の同一性、音色、背景ノイズを含むすべてを捉える。
2. **セマンティック優先コーデック** — Mimi（Kyutai）、SpeechTokenizer。最初のコードブックに言語的 / 音素的な内容をエンコードさせる（多くは WavLM からの蒸留による）。後続のコードブックは音響的な詳細だ。

2024-2026年の洞察: **純粋な再構成コーデックは、テキストから生成しようとするとぼやけた音声を生む。** コーデックトークン上の LLM は、同じコードブック内で言語構造と音響構造の両方を学習しなければならず、これはスケールしない。それらを分離すること — セマンティックなコードブック0、音響的なコードブック1-N — が Moshi と Sesame CSM を機能させているものだ。

### 中核となる仕掛け: 残差ベクトル量子化（RVQ）

良い品質に数百万のコードが必要となる1つの巨大なコードブックではなく、すべての現代の音声コーデックは **RVQ** を使う。これは小さなコードブックのカスケードだ。最初のコードブックがエンコーダ出力を量子化し、2番目が残差を量子化する、という具合だ。各コードブックは1024コード。8つのコードブック = 実効語彙 1024^8 = 10^24。

推論時、デコーダはフレームごとに選択されたすべてのコードを合算して再構成する。

### 2026年に重要な4つのコーデック

**EnCodec（Meta, 2022）。** ベースライン。波形上のエンコーダ-デコーダ、RVQ ボトルネック。24 kHz、最大32コードブック可能、デフォルトは4コードブック @ 1.5 kbps。`1D conv + transformer + 1D conv` アーキテクチャを使う。MusicGen で使われている。

**DAC（Descript, 2023）。** L2正規化されたコードブック、周期的な活性化関数、改良された損失を伴う RVQ。あらゆるオープンコーデックの中で最高の再構成忠実度 — 12コードブックで元の音声と区別がつかないこともある。44.1 kHz フルバンド。

**SNAC（Hubert Siuzdak, 2024）。** マルチスケール RVQ — 粗いコードブックは細かいものより低いフレームレートで動作する。音声を階層的にモデル化する: 約12 Hz の粗い「スケッチ」と50 Hz の詳細。階層構造が LM ベースの生成によくマッピングされるため、Orpheus-3B で使われている。

**Mimi（Kyutai, 2024）。** 2026年のゲームチェンジャー。12.5 Hz のフレームレート（極めて低い）、8コードブック @ 4.4 kbps。コードブック0は **WavLM から蒸留** されている — WavLM の音声内容特徴を予測するよう訓練されている。コードブック1-7は音響的な残差だ。この分離が Moshi（レッスン15）と Sesame CSM を支えている。

### フレームレートは言語モデリングにとって重要だ

低いフレームレート = 短いシーケンス = 高速な LM。

| コーデック | フレームレート | 1秒 = N フレーム | 適した用途 |
|-------|-----------|----------------|---------|
| EnCodec-24k | 75 Hz | 75 | 音楽、汎用音声 |
| DAC-44.1k | 86 Hz | 86 | 高忠実度の音楽 |
| SNAC-24k（粗い） | 約12 Hz | 12 | AR-LM 効率的 |
| Mimi | 12.5 Hz | 12.5 | ストリーミング音声 |

12.5 Hz では、10秒の発話はわずか125コーデックフレームであり、トランスフォーマーは容易にそれらを予測できる。

### セマンティックトークン vs 音響トークン

```
frame_t → [semantic_token_t, acoustic_token_0_t, acoustic_token_1_t, ..., acoustic_token_6_t]
```

- **セマンティックトークン（Mimi のコードブック0）。** 何が言われたかをエンコードする — 音素、単語、内容。補助的な予測損失を介して WavLM から蒸留される。
- **音響トークン（コードブック1-7）。** 音色、話者の同一性、プロソディ、背景ノイズ、細かな詳細をエンコードする。

AR LM はまずセマンティックトークンを（テキストを条件として）予測し、次に音響トークンを（セマンティック + 話者リファレンスを条件として）予測する。この因数分解こそが、現代の TTS がゼロショットで声をクローンできる理由だ: セマンティックモデルが内容を扱い、音響モデルが音色を扱う。

### 2026年の再構成品質（bits per sec、ビットレートは低いほど良い）

| コーデック | ビットレート | PESQ | ViSQOL |
|-------|---------|------|--------|
| Opus-20kbps | 20 kbps | 4.0 | 4.3 |
| EnCodec-6kbps | 6 kbps | 3.2 | 3.8 |
| DAC-6kbps | 6 kbps | 3.5 | 4.0 |
| SNAC-3kbps | 3 kbps | 3.3 | 3.8 |
| Mimi-4.4kbps | 4.4 kbps | 3.1 | 3.7 |

Opus のような従来型コーデックは、知覚品質ではビットあたりでいまだに勝る。ニューラルコーデックは**離散トークン**（Opus は生成しない）と**生成モデルの品質**（LM がそれらのトークンで何ができるか）で勝る。

## 作ってみる

### ステップ1: EnCodec でエンコードする

```python
from encodec import EncodecModel
import torch

model = EncodecModel.encodec_model_24khz()
model.set_target_bandwidth(6.0)  # kbps

wav = torch.randn(1, 1, 24000)
with torch.no_grad():
    encoded = model.encode(wav)
codes, scale = encoded[0]
# codes: (1, n_codebooks, n_frames), dtype=int64
```

6 kbps で `n_codebooks=8`。各コードは0-1023（10ビット）。

### ステップ2: デコードして再構成を計測する

```python
with torch.no_grad():
    wav_recon = model.decode([(codes, scale)])

from torchaudio.functional import compute_deltas
import torch.nn.functional as F

mse = F.mse_loss(wav_recon[:, :, :wav.shape[-1]], wav).item()
```

### ステップ3: セマンティック・音響の分離（Mimi スタイル）

```python
from moshi.models import loaders
mimi = loaders.get_mimi()

with torch.no_grad():
    codes = mimi.encode(wav)  # shape (1, 8, frames@12.5Hz)

semantic = codes[:, 0]
acoustic = codes[:, 1:]
```

セマンティックなコードブック0は WavLM に整合している。テキストからセマンティックへのトランスフォーマーを訓練できる — 直接音声へ向かうよりもはるかに小さな語彙だ。その後、別の音響から波形へのデコーダが話者リファレンスを条件とする。

### ステップ4: なぜコーデックトークン上の AR LM が機能するか

Mimi の 12.5 Hz × 8コードブックでの10秒の音声クリップの場合:

```
N_tokens = 10 * 12.5 * 8 = 1000 tokens
```

1000トークンはトランスフォーマーにとって些細なコンテキストだ。2億5600万パラメータのトランスフォーマーは、現代の GPU 上でミリ秒単位で10秒の音声を生成できる。

## 使ってみる

問題 → コーデックの対応:

| タスク | コーデック |
|------|-------|
| 汎用音楽生成 | EnCodec-24k |
| 最高忠実度の再構成 | DAC-44.1k |
| 音声上の AR LM（TTS） | SNAC または Mimi |
| ストリーミング全二重音声 | Mimi（12.5 Hz） |
| テキスト付き効果音ライブラリ | EnCodec + T5 条件付け |
| きめ細かな音声編集 | DAC + インペインティング |

経験則: **生成モデルを構築するなら Mimi か SNAC から始める。圧縮パイプラインを構築するなら Opus を使う。**

## 落とし穴

- **コードブックが多すぎる。** コードブックを追加すると忠実度は線形に増えるが、LM のシーケンス長も線形に増える。8-12で止めること。
- **フレームレートの不一致。** 12.5 Hz の Mimi で LM を訓練した後、50 Hz の EnCodec でファインチューニングすると静かに失敗する。
- **すべてのコードブックが等しいと仮定する。** Mimi では、コードブック0が内容を担う。それを失うと明瞭度が破壊される。コードブック7を失ってもほとんど気づかない。
- **再構成品質を唯一の指標として使う。** コーデックは優れた再構成を持っていても、セマンティック構造が悪ければ LM ベースの生成には役立たないことがある。

## 演習

1. **易。** `code/main.py` を実行する。おもちゃのスカラ + 残差量子化器を実装し、コードブックを追加するにつれて再構成誤差を計測する。
2. **中。** `encodec` をインストールし、ホールドアウトの音声クリップで1、4、8、32コードブックを比較する。PESQ または MSE をビットレートに対してプロットする。
3. **難。** Mimi をロードする。クリップをエンコードする。コードブック0をランダムな整数に置き換えてデコードする。次にコードブック7を同様に置き換える。2つの破損を比較する — コードブック0の破損は明瞭度を破壊するはずだ。コードブック7の破損はほとんど何も変えないはずだ。

## 重要用語

| 用語 | 世間の言い方 | 実際の意味 |
|------|-----------------|-----------------------|
| RVQ | 残差量子化 | 小さなコードブックのカスケード。各々が前の残差を量子化する。 |
| フレームレート | コーデック速度 | 1秒あたりのトークンフレーム数。低いほど LM が高速。 |
| セマンティックコードブック | コードブック0（Mimi） | SSL 特徴から蒸留されたコードブック。内容をエンコードする。 |
| 音響コードブック | それ以外すべて | 音色、プロソディ、ノイズ、細かな詳細。 |
| PESQ / ViSQOL | 知覚品質 | MOS と相関する客観的指標。 |
| EnCodec | Meta コーデック | RVQ ベースライン。MusicGen で使われる。 |
| Mimi | Kyutai コーデック | 12.5 Hz フレームレート。セマンティック・音響分離。Moshi を支える。 |
