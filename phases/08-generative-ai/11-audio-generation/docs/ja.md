# オーディオ生成

> オーディオは16-48 kHzで1-Dシグナル。5秒クリップは80-240kサンプル。トランスフォーマーはその序列に直接注目しない。2026年のあらゆる本番オーディオモデルの解決策は同じ：ニューラルコーデック（Encodec、SoundStream、DAC）をオーディオを50-75 Hzで離散トークンに圧縮、トランスフォーマーまたは拡散モデルはトークンを生成。

**タイプ:** Build
**言語:** Python
**前提条件:** Phase 6 · 02 (オーディオ機能)、Phase 6 · 04 (ASR)、Phase 8 · 06 (DDPM)
**所要時間:** 約45分

## 問題

3つのオーディオ生成タスク：

1. **テキスト音声。** テキスト与えたら、音声を生成。クリーン音声は狭帯域で強いフォネティック構造がある — トランスフォーマー上トークンで十分に解く。VALL-E（マイクロソフト）、NaturalSpeech 3、ElevenLabs、OpenAI TTS。
2. **音楽生成。** プロンプト（テキスト、メロディー、和音進行、ジャンル）与えたら、音楽を生成。より広い分布。MusicGen（Meta）、Stable Audio 2.5、Suno v4、Udio、Riffusion。
3. **オーディオ効果/サウンドデザイン。** プロンプト与えたら、アンビエント音またはFoleyを生成。AudioGen、AudioLDM 2、Stable Audio Open。

3つすべてが同じ基盤で実行：ニューラルオーディオコーデック+トークンAR または拡散生成器。

### ニューラルオーディオコーデック

Encodec（Meta、2022）、SoundStream（Google、2021）、Descript Audio Codec（DAC、2023）。畳み込みエンコーダ波形をタイムステップベクトルに圧縮；残差ベクトル量子化（RVQ）各ベクトルをK コデックインデックスのカスケードに変換。デコーダは反転。24 kHz オーディオ 2 kbpsは、75 Hzで8 RVQコードブック = 600トークン/秒。

```
波形（16000サンプル/秒）
    └─ エンコーダ conv ─┐
                     ├─ RVQレイヤー1 → 75 Hzのインデックス
                     ├─ RVQレイヤー2 → 75 Hzのインデックス
                     ├─ ...
                     └─ RVQレイヤー8
```

### 上部の2つの生成パラダイム

**トークン自動回帰。** RVQトークンを序列に平坦化、デコーダのみトランスフォーマーを実行。MusicGenは「遅延並列」を使用K コードブック ストリーム並列で、ストリームごと相殺を出力。VALL-Eはテキストプロンプト+3秒音声サンプルから音声トークンを生成。

**潜在拡散。** コーデックトークンを連続潜在としてパック、またはカテゴリー拡散とモデル化。Stable Audio 2.5は連続オーディオ潜在上で流マッチングを使用。AudioLDM 2はテキスト-メル-オーディオ拡散を使用。

2024-2026トレンド：流マッチングは音楽で勝利（より速い推論、クリーンサンプル）、トークンAR言語は依然として音声を支配なぜなら本来的に因果的でストリーム十分。

## ビルドしよう

`code/main.py`コア思想をシミュレート：小さい次トークントランスフォーマーを合成「オーディオトークン」序列で訓練、2つの異なる「スタイル」（スタイルA交互低高トークン、スタイルB単調ランプ）から生成。スタイルで条件付けさまぷ。

### ステップ1：合成オーディオトークン

```python
def make_tokens(style, length, vocab_size, rng):
    if style == 0:  # 「音声的」：交互
        return [i % vocab_size for i in range(length)]
    # 「音楽的」：ランプ
    return [(i * 3) % vocab_size for i in range(length)]
```

### ステップ2：小さいトークン予測子を訓練

スタイルで条件付けされたバイグラムスタイル予測子。ポイントはパターン：コーデックトークン → クロスエントロピー訓練 → 自動回帰サンプリング。

### ステップ3：条件付けてサンプル

スタイルトークンおよび開始トークン与えたら、予測分布から次トークンをサンプル。20-40トークンに続け。

## ピットフォール

- **コーデック品質はキャップ出力品質。** コーデックが忠実に音を表現できない場合、生成器品質助けない。DACは現在のオープン最高。
- **RVQエラー蓄積。** 各RVQ層は前のもの残差をモデル化。レイヤー1のエラーが伝搬。高層で温度0でサンプリングが助け。
- **音楽構造。** 30秒トークンは75 Hzで20k+トークン。トランスフォーマーに難しい。MusicGenはスライディングウィンドウ+プロンプト続き使用；Stable Audioは短いクリップ+クロスフェード。
- **バウンダリアーティファクト。** 生成されたクリップ間クロスフェードは慎重重複追加必要。
- **クリーンデータ食欲。** 音楽生成者は数万時間ライセンス音楽が必要。Suno / Udio RIAA訴訟（2024）これを表面。
- **音声クローン倫理。** 3秒サンプル+テキストプロンプトはVALL-E / XTTS / ElevenLabsに音声をクローンするのに十分。すべての本番モデルは濫用検出+オプトアウトリストが必要。

## 演習

1. **簡単。** `code/main.py`を実行してスタイルを明示的に設定。生成された序列がスタイルのパターンと一致することを検証。
2. **中程度。** 遅延並列デコーディングを追加：1ステップでオフセットしたままいる必要がある2つのトークンストリームをシミュレート。共同予測子を訓練。
3. **難しい。** HuggingFaceトランスフォーマーを使用してローカルにMusicGen-smallを実行。3つの異なるプロンプントで10秒クリップを生成；スタイル準拠のためA/B。

## 本番注：オーディオはストリーミング問題

オーディオはユーザーが*すべて一度*ではなく*生成中に*到着するのを期待する唯一の出力モダリティ。本番用語では、これはTPOTが重要なことを意味する（出力トークン時間）ユーザーのリニング速度はターゲットスループット — 彼らの読む速度ではない。16kHzオーディオ~75トークン/秒でトークン化（Encodec）、サーバーはユーザーあたり再生スムーズ保つために≥75トークン/秒生成する必要。

2つのアーキテクチャ結果：

- **流マッチングオーディオモデルはストリーミングに簡単にできない。** Stable Audio 2.5とAudioCraft 2は固定クリップ長を1つのパスでレンダリング。ストリーミング、クリップをチャンク重複バウンダリ — スライディング窓拡散のように思う — vs コーデックAR モデルに対してレイテンシオーバーヘッド100-300ms追加。

製品が「ライブ音声チャット」または「リアルタイム音楽続き」の場合、コーデックAR パス選ぶ。「30秒クリップを提出レンダリング」の場合、流マッチングは品質とトータルレイテンシで勝つ。
