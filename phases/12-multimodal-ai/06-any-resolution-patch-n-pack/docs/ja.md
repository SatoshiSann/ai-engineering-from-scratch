# 任意解像度ビジョン：Patch-n'-Pack と NaFlex

> 実像は 224x224 の正方形ではない。レシートは 9:16、チャートは 16:9、医学スキャンは 4096x4096、モバイルスクリーンショットは 9:19.5。2024 前の VLM 答え — すべてを固定正方形にリサイズ — OCR、ドキュメント理解、高解像度シーン解析を機能させるシグナルを投げ棄てた。NaViT (Google、2023) は可変解像度パッチをブロック対角マスキング持つ単一トランスフォーマーバッチに詰めることができることを示した。Qwen2-VL の M-RoPE (2024) は絶対ポジショナルテーブル完全にドロップ。LLaVA-NeXT の AnyRes は高解像度画像をベース + サブイメージにタイル。SigLIP 2 の NaFlex バリアント (2025) は現在デフォルト単一チェックポイントを奉仕したいすべてのアスペクト比とすべて開く VLM 用。このレッスンは patch-n'-pack をエンドツーエンド実装。

**タイプ:** ビルド
**言語:** Python (標準ライブラリ、パッチパッカー + ブロック対角マスク)
**前提条件:** Phase 12 · 01 (ViT パッチ)、Phase 12 · 05 (LLaVA)
**所要時間:** 約120分

## 学習目標

- 可変解像度イメージのバッチからパッチを1つのシーケンスにパックし、ブロック対角アテンションマスク構築。
- 与えられたタスクの AnyRes タイリング (LLaVA-NeXT)、NaFlex (SigLIP 2)、M-RoPE (Qwen2-VL) 間で選択。
- OCR、チャート、リサイズなしフォトグラフィのトークン予算計算。
- 正方形リサイズの3つの失敗モード名付ける：つぶれたテキスト、クロップされたコンテンツ、パディングに浪費トークン。

## 問題

トランスフォーマーはシーケンス期待。バッチは同じ長さのシーケンススタック。イメージが 224x224 なら、毎回 196 パッチトークン取得、パディング不要、ジョブ完了。224 で訓練、224 で推論、解像度について2度と考えない。

世界は協力しない。ドキュメントは肖像 (8.5x11 インチ、2:3-ish)。チャートスクリーンショットは風景 (16:9)。レシートは背が高く薄い (1:3)。医学画像は 2048x2048 以上で配布。モバイルデバイススクリーンショットは 1170x2532 (0.46:1)。

3つの 2024 前のオプションと各なぜ失敗：

1. 固定正方形にリサイズ (224x224 または 336x336)。スクイッシュは顔をテキスト歪ませ。ダウンスケールはチャートラベルと OCR コンテンツ破壊。LLaVA-1.5 まで標準的実践。
2. 固定アスペクト比にクロップ。大部分の画像を投げ棄て、クロップ位置選択は独自のビジョン問題。
3. 最長辺にパド。歪み修正だが 50%+ トークン肖像イメージのパディング浪費。すべてのそれらpad トークン上二次アテンションコスト。

2024-2025 答え：トランスフォーマーはイメージのネイティブ解像度でパッチを食べさせ、図表は異種バッチを1つのシーケンスに詰める方法を計算するどうしよう浪費済みコンピュート。

## コンセプト

### NaViT と patch-n'-pack

NaViT (Dehghani et al.、2023) はこれがスケール機能を示したペーパー。アイデアは機械的：

1. バッチ内の各イメージについて、選択されたパッチサイズ (たとえば 14) でそのネイティブパッチグリッド計算。
2. 各イメージのパッチをそれ自体の可変長シーケンスにフラット。
3. すべてのイメージのパッチを1つの長シーケンスをバッチに連結。
4. ブロック対角アテンションマスク構築、イメージ A のパッチは A の内のみ注意。
5. パッチあたりの位置情報を実行 (2D RoPE または小数点ポジション埋め込み)。

3つのイメージのバッチ 336x336 (576 トークン)、224x224 (256 トークン)、448x336 (768 トークン) 1600 トークンシーケンス + 1600x1600 ブロック対角マスク。パディングなし。浪費済みコンピュートなし。トランスフォーマーは任意アスペクト比ハンドル。

NaViT は訓練中の小数点パッチドロップも導入 — バッチ横断で ランダム 50% パッチドロップ — 規則化して高速訓練の両方。SigLIP 2 は継承。

### AnyRes (LLaVA-NeXT)

LLaVA-NeXT の AnyRes は実用的代替。高解像度イメージ与えられ、固定エンコーダ (CLIP または SigLIP 336)、タイル イメージ：

1. 事前定義セットからグリッドレイアウト選択 — (1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2) など — イメージのアスペクト比に最も良くフィット。
2. グリッドに完全イメージタイル；各タイルは 336x336 クロップ。
3. また生成サムネイル：全イメージグローバルコンテキストトークンとして 336x336 にリサイズ。
4. すべてのタイルを凍結 336 エンコーダ経由でエンコード。タイルトークン + サムネイルトークン連結。

2x2 グリッド加えて 672x672 イメージ用サムネイル：4 * 576 + 576 = 2880 ビジュアルトークン。高価だが効果的 — LLM はローカル詳細とグローバルコンテキスト両方を見。

AnyRes は固定解像度だけサポート エンコーダでトークン予算重要 (長ビデオ、多くイメージ)。大イメージ (1344x1344 イメージ 4x4 グリッド 9216 + 576 ≈ 9800 トークン、これはほとんどの 8k LLM コンテキスト埋める) でトークン爆発。

### M-RoPE (Qwen2-VL)

Qwen2-VL は Multimodal Rotary Position Embedding を導入。NaViT の小数点ポジション代わりに、AnyRes のタイルサムネイル、各パッチは 3D ポジション (時間、高さ、幅) を実行。クエリ/キー回転は任意 H、W、時間長をハンドル。

M-RoPE はネイティブ動的解像度なし再訓練に配布。推論時 任意 HxW イメージを供給、パッチ埋め込み器は H/14 x W/14 トークン生成、各トークンは (t=0、r=行、c=列) ポジション取得、RoPE は正しい周波数でアテンション回転、完了。Qwen2.5-VL と Qwen3-VL はこれ続ける。InternVL3 の V2PE は同じアイデア、モダリティあたり可変エンコーディング。

AnyRes と異なり、M-RoPE は O(H x W / P^2) ネイティブ解像度トークン — 乗法的タイルオーバーヘッド。NaViT と異なり、それは単一イメージあたり前方期待まだ。バッチ横断で解像度をまだ必要 patch-n'-pack 上の。

### NaFlex (SigLIP 2)

NaFlex は SigLIP 2 チェックポイント のネイティブフレックスモード。単一モデル推論で複数シーケンス長 (256、729、1024 トークン) 奉仕。内部的に訓練中は NaViT スタイル patch-n'-pack を使用、絶対小数点ポジション/パッチ。販売ポイント：1つのチェックポイント、タスク基づきタスク推論時トークン予算ピック。

セマンティックタスク (分類、検索)、256 トークン。OCR またはチャート理解、1024 トークン。再訓練なし。

### パッキングマスク

ブロック対角マスクは多くの実装で最も落ちやすい位置。長さ `N_total` の詰めたシーケンス イメージ `i=0..B-1` 長さ `n_i` カバー、マスク `M` 形 `(N_total, N_total)` は1 両インデックスは同じイメージ ブロックで落ちる場合、別0。累積長リストから構築できる：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 iff there exists b where offsets[b] <= i < offsets[b+1] and offsets[b] <= j < offsets[b+1]
```

これは PyTorch で1行は `torch.block_diag` または明示的ギャザー。FlashAttention の可変長パス (`cu_seqlens`) マスク完全スキップ、累積長テンソル直接使用してシーケンス内で注意 — 典型的バッチ用に10倍高速。

### トークン予算

タスクで戦略を選択：

- OCR / ドキュメント：1024-4096 トークン。SigLIP 2 NaFlex 1024、または AnyRes 3x3 + サムネイル。
- チャートと UI：729-1024 トークン 384-448 ネイティブ。Qwen2.5-VL 動的解像度 max pixels キャップ。
- 自然フォト：256-576 トークンはいい。ダウンストリーム LLM は十分に見る。コンテンツ密度高いのにトークンに支払い。
- ビデオ：空間プール後のフレームあたり 64-128 トークン、2-8 FPS。Lesson 12.17 はこれをカバー。

2026 本番ルール：タスクあたり max-pixels キャップを選択、アクティブ率それのキャップまでネイティブアスペクト比にエンコード、バッチをパック、パディングをスキップ。Qwen2.5-VL は `min_pixels` と `max_pixels` ちょうどこのノブのために公開。

## これを使用する

`code/main.py` は可変解像度イメージの異種バッチ用に patch-n'-pack を実装。それは：

- イメージサイズのリスト (H、W) を取得。
- 各イメージのパッチシーケンス長計算パッチサイズ 14。
- それらを1つシーケンス合計長 `sum(n_i)` にパック。
- ブロック対角アテンションマスク構築 (密度、明確さのため)。
- 詰めたコスト vs 正方形リサイズと AnyRes タイリング比較。
- トークン予算テーブル印字、混合バッチ (レシート、チャート、スクリーンショット、写真)。

それを実行。出るナンバーはすべての 2026 オープン VLM が patch-n'-pack を使用する理由。

## 配布する

このレッスンは `outputs/skill-resolution-budget-planner.md` を生成。混合アスペクト比ワークロード (OCR、チャート、写真、ビデオフレーム) と合計トークン予算が与えられたとき、正しい戦略 (NaFlex、AnyRes、M-RoPE、または固定正方形) をピック、リクエストあたり設定を発行。スキルを使用 VLM をサイジング製品 — それはレイテンシー予算を殺す サイレント 10倍トークン噴出を防ぐとき。

## 演習

1. レシート 600x1500 (1:2.5)。パッチサイズ 14、ネイティブ解像度トークンいくつ？336 に正方形リサイズ後？どちらが実践で OCR 精度を失うより多く？

2. ブロック対角マスク構築4イメージバッチ長 256、576、729、1024。アテンションマトリックス 2585x2585 検証、正確に `256^2 + 576^2 + 729^2 + 1024^2` ゼロ以外エントリ。

3. 1792x896 イメージパッチ 14、比較：(a) 正方形リサイズ 336 次にエンコード、(b) AnyRes 2x1 + サムネイル、(c) M-RoPE ネイティブ。最も少ないトークンを使用？最も詳細を保持？

4. 小数点パッチドロップ実装：詰めたシーケンス与えられ、ドロップ 50% トークン均等ランダム、ブロック対角マスク更新。マスク疎度変更を測定。

5. Qwen2-VL ペーパー Section 3.2 読む (arXiv:2409.12191)。`min_pixels` と `max_pixels` は制御と両方なぜ限界は事項を2文で記述。

| 用語 | 人は何と言うか | 実際の意味 |
|------|----------------|----------|
| Patch-n'-pack | 「NaViT スタイルパッキング」 | 異なるイメージから可変長パッチシーケンスを1つバッチディメンションに連結 |
| ブロック対角マスク | 「パッキングマスク」 | アテンションマスク、各イメージのパッチをそれ自体だけに注意を閉じ込め、パック隣人ではなく |
| AnyRes | 「LLaVA-NeXT タイリング」 | 固定サイズタイルグリッド + グローバルサムネイルに高解像度イメージ分割；固定エンコーダですべてのタイルエンコード |
| NaFlex | 「SigLIP 2 ネイティブフレックス」 | 単一 SigLIP 2 チェックポイント 256/729/1024 トークン予算推論で奉仕再訓練なし |
| M-RoPE | 「マルチモーダル RoPE」 | 3D 回転ポジション符号化 (時間、行、列) 任意 H、W、T が位置テーブルなく処理 |
