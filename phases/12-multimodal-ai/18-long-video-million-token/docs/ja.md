# 百万トークンコンテキストでのロングビデオ理解

> 24FPS 4K動画の1時間は、パッチ化・埋め込み後に約6000万トークンを生成する。文字起こしされたポッドキャストエピソードの2時間分は30,000トークン。完全なブルーレイ映画は、積極的なプーリングで圧縮しても数十万トークンを必要とする。Googleの Gemini 1.5（2024年3月）は1000万トークンコンテキストでこの時代を開き、1時間の動画において確実な干し草の中の針検索（needle-in-a-haystack recall）を実現した。LWM（Liu et al., 2024年2月）はリング注意の拡張パスを示した。LongVILAとVideo-XLはさらに摂取を拡張した。VideoAgentは生のコンテキストをエージェント的検索に置き換えた。各アプローチは計算、再現率、エンジニアリング複雑度のトレードオフを異なる形で実現している。本レッスンは並べて読む。

**タイプ:** ビルド
**言語:** Python（stdlib、干し草の中の針検索シミュレータ + エージェント的検索ルーター）
**前提条件:** Phase 12 · 17（ビデオ時間トークン）
**所要時間:** 約180分

## 学習目標

- 異なるFPSとプーリングでのロングフォーム動画の合計ビジュアルトークン数を計算する。
- 3つのスケーリングパス：ブルートコンテキスト（Gemini 1.5）、リング注意（LWM）、トークン圧縮（LongVILA / Video-XL）を説明する。
- 生のコンテキストビデオVLMとエージェント的検索ビデオVLM（VideoAgent）を精度とレイテンシで比較する。
- 30分の動画に対する干し草の中の針検索テストを設計し、特定の分での再現率を測定する。

## 問題

Qwen2.5-VL サイズのパッチでの単一フレーム（384ネイティブ解像度）は約729トークン。3x3プーリングで81トークン/フレーム。30分クリップを1 FPS = 1800フレーム = 145,800トークン。2025年の開いたVLMで実現可能だが、タイトなコンテキスト。2FPSで291,600トークン — 最大のコンテキストのみが対応。

2時間映画を1FPSで583kトークン。ほとんどの2026年の開いたモデルを超える；Gemini 2.5 Pro または、より積極的なプーリングが必要。

3つのスケーリングパスが出現した。

## コンセプト

### パス1：ブルートコンテキスト（Gemini 1.5、Claude Opus）

ハードウェアで問題を解く。コンテキストを百万トークンにスケーリングし、すべてを1回の順伝播で処理。

Gemini 1.5 Proは1Mトークンで開始；Gemini 1.5 Ultraは10Mまで；Gemini 2.5 Proは2026年に時間単位の動画を確実に処理。論文（arXiv:2403.05530）は~9.5Mトークンまでで99.7%の干し草の中の針検索再現率を記録している。

エンジニアリング：メモリ階層（ローカル+グローバル+スパース）とMoEエキスパートルーティングを備えたカスタム注意実装。完全な詳細は発表されていない。オープンソースではない。

### パス2：リング注意（LWM、LongVILA）

リング注意は長いシーケンスをデバイスの「リング」に分散する。各デバイスがチャンクを保持し、リングパターンで次に送信し、部分注意を計算し、集約する。

LWM（Liu et al., 2024）はこの方法で1Mトークンコンテキストモデルを訓練した。訓練計算はコンテキストで線形にスケーリング。二次的なヒットは注意。二次的なヒットはリングのデバイス全体で分散。

LongVILA（arXiv:2408.10188）はパターンをVLMに適応。1400フレーム動画は192トークン/フレーム = 268k コンテキスト、8方向並列化でリング注意を訓練。

### パス3：トークン圧縮（Video-XL、LongVA）

ブルートコンテキストより安価：LLMがシーケンスを見る前に積極的に圧縮。

Video-XL（arXiv:2409.14485）はビジュアルサマリトークンを使用：N フレームのクリップは、N フレームに注意する単一の「サマリ」トークンを生成。推論時、LLMはクリップごとに1つのサマリトークンを見て、コンテキストを大幅に縮小。

LongVAは長文脈転送技術でLLMコンテキストを200k から2Mに拡張。長コンテキストテキストで訓練し、共有表現を通じて長コンテキストビデオに転送。

トークン圧縮は特定のタイムスタンプでの再現率をトレードオフするがスケーラビリティを獲得。モデルは一般に何が起きたかを知るが、時々正確なフレームを見逃す。

### パス4：エージェント的検索（VideoAgent）

完全な動画をLLMに供給しない。代わりに、動画をデータベースとして扱い、LLMを使って照会。

VideoAgent（arXiv:2403.10517）：

1. LLMが質問を読む。
2. LLMが検索ツールに関連クリップを要求（「猫がいるセグメントを表示」）。
3. ツールが一致するクリップのタイムスタンプを返す。
4. LLMがそのクリップをVLMを通じて読む。
5. LLMが回答を構成するか、フォローアップクエリを要求。

これはLLM-as-agent パターンをロング動画に適用。推論コスト低い（関連クリップのみエンコード）、エンジニアリング難しい（検索品質がボトルネック）。

### 干し草の中の針ベンチマーク

標準的なロングコンテキストテスト：動画のランダムポイントに一意のビジュアルまたはテキストマーカを挿入し、それを思い出すことが必要な質問をする。

メトリック：動画の長さとマーカ位置全体でのRecall@k。

Gemini 2.5 Proは最大90分動画で>99%再現率を達成。開いた72Bモデル（Qwen2.5-VL-72B、InternVL3-78B）は30分で~85-90%、60分を過ぎると低下。

VideoAgentは2時間以上でツールが良い場合、生のコンテキストモデルと同等またはそれ以上を実現。

### どのパスを選ぶか

15分クリップで先端精度：開いた72B + ネイティブコンテキストは通常動作。Qwen2.5-VL-72Bを選択。

30分から1時間のコンテンツ：開いたモデルではLongVILAまたはVideo-XL；クローズドではGemini 2.5 Pro。品質基準が重要 — 先端はクローズド。

2時間以上のコンテンツ：VideoAgentまたは類似の検索パターン。または、より小さなチャンクに要約し、階層的なサマリを供給。

### 2026年本番パターン

実際には、本番ロング動画パイプラインはハイブリッド：

1. 動的FPSサンプリング + 積極的なプーリングを全動画に実行（100k トークンのグローバル表現を獲得）。
2. 72B VLMに渡してグローバルサマリを生成。
3. ユーザーが詳細な質問をした場合、サマリをインデックスとして使用してエージェント的検索を実行。

これはグローバル理解のためのブルートコンテキストとローカル詳細のための検索を結合。

## 使用方法

`code/main.py`：

- 1分から3時間の動画の異なるFPS + プーリングでのトークン予算を計算。
- 干し草の中の針検索を実行：ランダムなタイムスタンプでマーカを注入、質問、再現率をスコア。
- エージェント的検索ルーターシミュレーターを含む。下流VLMに供給する特定のクリップを選択。

予算テーブルを実行してスケールギャップを感じる。

## 提出

本レッスンは`outputs/skill-long-video-strategy-planner.md`を生成。動画の期間とクエリ複雑度が与えられた場合、ブルートコンテキスト、圧縮、エージェント的検索を選択し、レイテンシ + 品質の期待を計算。

## 演習

1. 1FPS、フレームあたり81トークンでの45分講演。合計トークン？どのモデルのコンテキストに適合？

2. 干し草の中の針検索テストを設計：マーカをどの分に注入し、正確なクエリ形式は何か？

3. ブルートコンテキストQwen2.5-VL-72B（80kコンテキスト）とVideoAgent（Claude 3.5 + 検索）を1時間動画で比較。どちらが再現率で勝つ？どちらがレイテンシで勝つ？

4. リング注意のメモリコストはシーケンス長と線形、デバイス数と線形。なぜか説明し、リング回転フェーズを削除した場合に何が失敗するか。

5. Gemini 1.5 Section 5で干し草の中の針検索を読む。1Mトークン vs 10Mトークン境界での再現率について論文は何を見つけたか？

## 主要な用語

| 用語 | 人々が言う | 実際の意味 |
|------|------------|---------|
| ブルートコンテキスト | 「もっとトークン」 | LLMコンテキストを百万トークンにスケーリング；すべてを1回のパスで処理 |
| リング注意 | 「LWMスタイル並列」 | 各デバイスがチャンクを保持し、リングパターンで回転する分散注意パターン |
| トークン圧縮 | 「サマリトークン」 | LLMの前に学習コンプレッサー経由でクリップごとのトークンを削減 |
| 干し草の中の針検索 | 「NIH テスト」 | ランダムなポイントに一意のマーカを挿入、テスト時にモデルにそれを思い出させる |
| エージェント的検索 | 「LLM as クエリプランナー」 | LLMが関連クリップの検索ツールに要求、VLMで読む、回答を構成 |
| VideoAgent | 「ビデオの検索パターン」 | 標準的なエージェント的検索設計：質問 -> ツール -> クリップ -> 回答 |
