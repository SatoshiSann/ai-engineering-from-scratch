# Janus-Pro: 統一マルチモーダルモデルのためのデカップルされたエンコーダ

> 統一マルチモーダルモデルは避けられない緊張を持つ。理解は意味的フィーチャーを望む — SigLIPまたはDINOv2出力ベクトルは概念レベルの情報で豊か。生成は再構成親和的コードを望む — VQトークンは明確なピクセルに組み立て直す。2つの目標は単一エンコーダに互換性ではない。Janus（DeepSeek、2024年10月）とJanus-Pro（DeepSeek、2025年1月）は修正は試すことを停止することを議論：2つのエンコーダをデカップル。タスク間でトランスフォーマーボディを共有、しかしSigLIPを通して理解をルート、VQトークナイザーを通して生成。7Bで、Janus-ProはGenEvalで DALL-E 3を打ち負かしMMUMでLLaVAにマッチ。このレッスンは1つが失敗する場所で2つのエンコーダが働く理由を読む。

**タイプ:** ビルド
**言語:** Python (stdlib、デュアルエンコーダルーティング+共有ボディ信号)
**前提条件:** Phase 12 · 13 (Transfusion)、Phase 12 · 14 (Show-o)
**所要時間:** 約120分

## 学習目標

- 単一の共有エンコーダが理解または生成品質のいずれかを損なわせる理由を説明できる。
- Janus-Proのルーティングを説明できる：理解のための入力側のSigLIPフィーチャー、生成のための入力と出力両方のVQトークン。
Janus-Proが Janusがなかった所で成功させるデータミックススケーリングをトレース。
- デカップル（Janus-Pro）、結合連続（Transfusion）、結合離散（Show-o）アーキテクチャを比較できる。

## 問題

統一モデルは理解と生成を横切ってトランスフォーマーボディを共有。過去の試み（Chameleon、Show-o、Transfusion）はすべて両方向のために1つのビジュアルトークナイザーを使用。トークナイザーは妥協：

- 再構成のために最適化（生成）：VQ-VAEは細粒度ピクセル詳細をキャプチャするが、弱い意味的コヒーレンスを持つトークンを作成。
- 意味論のために最適化（理解）：SigLIP埋め込みは「猫」画像を「猫」トークン近くでグループしかし良好な再構成を許さない。

Show-oと Transfusionはこれを1つの方向で可視品質税で支払う。Janus-Proは聞く：タスクが異なる必要を持つときなぜ1つのトークナイザーを必要？

## コンセプト

### デカップルされたビジュアルエンコーディング

Janus-Proのアーキテクチャは2つのエンコーダを分離：

- 理解パス。入力画像 → SigLIP-SO400m → 2層MLP → トランスフォーマーボディ。
- 生成パス。入力画像（既存画像に条件の場合）→ VQトークナイザー → トークンID → トランスフォーマーボディ。
- 出力生成。トランスフォーマーで予測された画像トークン → VQデコーダ → ピクセル。

トランスフォーマーボディは共有。ボディの上流と下流のすべてはタスク固有。

入力はプロンプト形式で曖昧性を除去：`<understand>`タグはSigLIPを通してルート；`<generate>`はVQを通してルート。またはルーティングはタスクから黙示的。

### なぜこれが働くか

理解損失はSigLIPフィーチャーを得、CLIP風事前学習は意味的類似性のためにチューン。モデルの知覚ベンチマークはShow-o / Transfusion上で改善タスクのための入力フィーチャーが良いため。

生成損失はVQトークンを得、トークナイザーは再構成のためにチューン。画像品質はShow-oを上回るVQコードはピクセルにきれいに組み立て直すため。

共有トランスフォーマーボディは2つの入力分布（SigLIPと VQ）を見て両方で働くことを学ぶ。主張：十分データ+十分パラメータ、ボディはスイッチを吸収。

### データスケーリング — Janusと Janus-Pro

Janus（元々、arXiv 2410.13848）はデカップリングを導入しかし小さいスケールで（1.3Bパラメータ、限定データ）。Janus-Pro（arXiv 2501.17811）はスケール：

- 7Bパラメータ（対1.3B）。
- ステージ1（アライメント）のために90M画像テキストペア72Mから上昇。
- ステージ2（統一）のために72M 26Mから上昇。
- ステージ3のために200k画像生成命令サンプルを追加。

要点：Janus-Pro-7BはMMUMでLLaVAにマッチ（60.3対約58）GenEvalで DALL-E 3を打ち負かし（0.80対0.67）。1つのオープンモデル、統一スペクトルの両側で競争。

### JanusFlow — rectified フロー変異

JanusFlow（arXiv 2411.07975）は VQ生成パスを rectified フロー生成パス（連続）に交換。分割はSigLIP-理解のために+rectified フロー生成のために。品質上限はさらに持ち上げ。アーキテクチャはデカップルエンコーダ共有ボディのままで。

### 共有ボディの仕事

トランスフォーマーボディは統一シーケンスを処理するが2つの入力分布で。その仕事は：

- 理解のために：SigLIPフィーチャー+テキストトークンを消費 → テキスト自動回帰を出力。
- 生成のために：テキストトークン+（任意画像VQトークン）を消費 → 画像VQトークン自動回帰を出力。

ボディはブロックあたりのモダリティ固有重みなし。それはQwenまたはLlamaの内で見つけることを望む通常のテキストスタイルトランスフォーマー、加えて2つの入力アダプター。

興味深く、これはJanus-Proのボディは事前学習されたLLMから初期化できることを意味。Janus-Pro は DeepSeek-MoE-7Bから初期化する。その選択は重要：LLMはスクラッチから純粋に統一モデルが到達するのに苦労する推論能力に貢献。

### InternVL-Uと比較

InternVL-U（レッスン12.10）は2026フォローアップ。それは組み合わせ：

- ネイティブマルチモーダル事前学習（InternVL3バックボーン）。
- デカップルエンコーダルーティング（SigLIPイン、VQ+拡散ヘッドアウト）。
- 統一理解+生成+編集。

InternVL-Uはより大きいフレームワークにおけるデカップルエンコーダのアーキテクチャ選択を含有。デカップルエンコーダのアイデアは今スケールでの統一モデルのデフォルト。

### 制限

デカップルエンコーダはアーキテクチャ複雑性を追加。2つのトークナイザーを訓練、2つの入力パスを維持、2つのセット失敗モード。生成が必要でない製品のために、Janus-Proはオーバーエンジニア — LLaVAファミリー理解モデルをピック。

理解が必要でない製品のために、Janus-Proはオーバー適格 — Stable Diffusion 3 / Fluxモデルをピック。

両方が必要な製品のために、Janus-Proは今参照オープンアーキテクチャ。

## 使う

`code/main.py`はJanus-Proルーティングをシミュレート：

- 2つのモック エンコーダ：SigLIP型（256-暗い意味的ベクトルを作成）と VQ型（整数コードを作成）。
- タスクタグに基づくエンコーダを選ぶプロンプトルーター。
- トークンシーケンスを処理する共有ボディ（スタンドイン）関係なくどのエンコーダがそれを作成。
- ステージ1（アライメント）からステージ3（命令チューン）加重サンプルスケジュールへのスイッチ。

3つの例のためにルート化されたパス印刷：画像QA、T2I、画像編集。

## デプロイ

このレッスンは`outputs/skill-decoupled-encoder-picker.md`を作成。統一生成+理解を境界品質で望む製品を与えられて、ピック Janus-Pro、JanusFlow、または InternVL-U具体的なデータスケール推奨で。

## 演習

1. Janus-Pro-7BはGenEvalで DALL-E 3を打ち負かし。7Bオープンモデルが生成で境界固有モデルにマッチしながら理解ではマッチしない理由を説明。

2. ルーター関数を実装：プロンプトテキスト与えられて、`understand`または`generate`として分類。「説明それからスケッチ」のように曖昧なプロンプトをどのように処理？

3. JanusFlowは VQパスを rectified フロー で置き換え。トランスフォーマーボディは今出力どのおのれしあって、損失で何が変わる？

4. Janus-Proアーキテクチャが1つもっと多くのデカップルエンコーダで処理できた第4のタスクを提案。例：画像セグメンテーション（DINOスタイル）、深さ（MiDaSスタイル）。

5. Janus-Pro セクション4.2をデータスケーリングについて読む。どのデータステージが Janusと比較するとT2I品質ゲインに最も貢献？

## キー用語

| 用語 | 人が言うこと | それが実際に意味すること |
|------|-----------------|------------------------|
| デカップルエンコーディング | 「2つのビジュアルエンコーダ」 | 各方向ごとの別のトークナイザーまたはエンコーダ：理解のための意味的、生成のための再構成 |
| 共有ボディ | 「1つのトランスフォーマー」 | 単一トランスフォーマー関係なくどのエンコーダの出力を処理；モダリティ固有重みなし |
| 理解のためのSigLIP | 「意味的フィーチャー」 | 豊か概念的フィーチャーを提供する CLIP族ビジョンタワーしかし貧弱な再構成 |
| 生成のためのVQ | 「再構成コード」 | ベクトル量子化トークンはピクセルにきれいにデコード |
| JanusFlow | 「Rectified フロー変異」 | VQ代わりに継続的フロー一致生成ヘッド持つJanus-Pro |
| ルーティングタグ | 「タスクタグ」 | プロンプトマーカー（`<understand>` / `<generate>`）入力エンコーダをピック |
