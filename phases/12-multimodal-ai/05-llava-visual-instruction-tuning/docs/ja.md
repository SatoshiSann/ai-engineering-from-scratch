# LLaVA と視覚命令チューニング

> LLaVA (2023年4月) は世界で最も複製されたマルチモーダルアーキテクチャ。BLIP-2 の Q-Former を 2層 MLP に置き換え、Flamingo のゲートクロスアテンションをナイーブなトークン連結に置き換え、テキストのみのキャプションから GPT-4 によって生成された 158k 視覚命令ターンで訓練。2023年から2026年の間に VLM を構築した任意の実践者は LLaVA のバリアント構築。LLaVA-1.5 は AnyRes を追加。LLaVA-NeXT は解像度をバンプ。LLaVA-OneVision は画像、マルチ画像、ビデオを1つのレシピで統一。このレッスンはレシピを読み、プロジェクターを実装し、なぜ「シンプルが勝った」を説明。

**タイプ:** ビルド
**言語:** Python (標準ライブラリ、プロジェクター + 命令テンプレートビルダー)
**前提条件:** Phase 12 · 02 (CLIP)、Phase 11 (LLM エンジニアリング — 命令チューニング)
**所要時間:** 約180分

## 学習目標

- ViT パッチ埋め込み (次元 1024) を LLM 埋め込み次元 (次元 4096) にマップする 2層 MLP プロジェクター構築。
- LLaVA 2段階レシピを実行：(1) プロジェクター配置 558k キャプションペア、(2) 視覚命令チューニング 158k GPT-4 生成ターン。
- 画像トークンプレースホルダー、システムプロンプト、ユーザー/アシスタントターンを持つ LLaVA フォーマットプロンプト構築。
- コミュニティが Q-Former から MLP に移動した理由を説明、Q-Former のトークン予算勝利にもかかわらず。

## 問題

BLIP-2 の Q-Former (Lesson 12.03) は画像を 32 トークンに圧縮。きれい、効率的、ベンチマーク向け。だが2つの問題がある。

まず、Q-Former は訓練可能だが、その損失は最終タスクではない。段階1は ITC+ITM+ITG を訓練。段階2は LM 損失を訓練。クエリは LLM がそれからデコードする必要があるいくつか中間表現を学習。情報はボトルネックで失われる。

2番目、Q-Former は 188M パラメータをかかり、LLaVA の 2023 スケールで、ターゲット LLM と共同設計する必要。LLM を変更、Q-Former を再訓練。ビジョンエンコーダを変更、再訓練。すべての組み合わせは別々の R&D プロジェクト。

LLaVA の答えはそれの単純さで恥ずかしい：ViT の 576 パッチトークンをかかり、2層 MLP を介してそれぞれを通す (`1024 → 4096 → 4096`)、すべて 576 を LLM の入力シーケンスにダンプ。ボトルネックなし。奇妙な目的で段階1事前学習なし。直接 LM 損失でだけ MLP を訓練。

データはどこから来るか？LLaVA の2番目の洞察：GPT-4 (テキストのみ) を命令データ生成に使用。GPT-4 に COCO キャプションと画像のバウンディングボックスデータを供給、会話、記述、複雑推論質問を生成するよう求める。158k 命令応答ターン無料。人間による注釈なし。

結果：1日で 8 A100s で実行した VLM、Flamingo 上で MMMU を打ち、コミュニティが拡張できるオープンチェックポイントを配布。2023年後半までにそれは 50+ フォークを産出していた。

## コンセプト

### アーキテクチャ

LLaVA-1.5 13B：
- ビジョンエンコーダ：CLIP ViT-L/14 @ 336 (段階1中に凍結、段階2オプショナルアンフロ)。
- プロジェクター：2層 MLP GELU アクティベーション、`1024 → 4096 → 4096`。
- LLM：Vicuna-13B (後に Llama-3.1-8B)。

画像 + テキストプロンプト上の前方パス：

```
img -> ViT -> 次元 1024 の 576 パッチ
patches -> MLP -> 次元 4096 の 576 トークン
prompt: system + "<image>" placeholder + user question
<image> token を 576 投影トークンで置き換え
完全シーケンスを LLM にフィード
応答デコード
```

画像は 576 トークン LLM コンテキストを占有。2048 コンテキストで、それはテキスト用に 1472 トークン残す。32k コンテキストで、それは丸め誤差。

### 段階1：プロジェクター配置

ViT を凍結。LLM を凍結。2層 MLP だけをトレーニング。データセット：558k イメージキャプションペア (LAION-CC-SBU)。損失：キャプションの言語モデリング、投影されたイメージトークンで条件付け。

バッチ 128 での単一エポックで、これは数時間で行われる。プロジェクターは ViT-空間を LLM-空間にマップすることを学習。タスク固有教師ありなし。

### 段階2：視覚命令チューニング

プロジェクターをアンフロ (まだ訓練可能)。LLM をアンフロ (通常は完全、時々は LoRA)。158k 視覚命令ターンで訓練。

命令データはトリック。Liu et al. 生成によって：
1. COCO 画像を取得。
2. テキスト記述抽出 (5 人間キャプション + バウンディングボックスリスト)。
3. 3つのプロンプトテンプレート持つ GPT-4 に送信：
   - 会話：「この画像についてユーザーとアシスタント間のバックアンドフォース対話生成。」
   - 詳細記述：「画像のリッチで詳細な記述を与える。」
   - 複雑推論：「画像について推論が必要な質問を尋ねて、それに答える。」
4. GPT-4 の出力を (命令、応答) ペアに解析。

これはイメージに直接触れない — テキスト記述だけ。GPT-4 は妥当な画像コンテンツを幻覚。いくつかノイズ、だが機能：158k ターンは対話をアンロック十分。

### コミュニティがこれをなぜコピーしたか

- 段階1固有損失なし。全体で LM 損失。
- プロジェクター日で訓練、日ではなく。
- LLM をスワップ (LLaVA-Llama2、LLaVA-Mistral、LLaVA-Llama3) プロジェクターだけを再訓練。
- 視覚命令データパイプラインは GPT-4 を使用し、新規ドメイン用に再生成するのは安い。

### LLaVA-1.5 と LLaVA-NeXT

LLaVA-1.5 (2023年10月) 追加：
- アカデミックタスクデータ (VQA、OKVQA、RefCOCO) 命令チューニングに混入。
- より良いシステムプロンプト。
- 2048 → 32k コンテキスト。

LLaVA-NeXT (2024年1月) 追加：
- AnyRes：高解像度画像を 2x2 または 1x3 グリッド 336x336 クロップに分割、1つのグローバル低解像度サムネイル加えて。各クロップが 576 トークンになる；総計画像あたり約 2880 視覚トークン。OCR とチャートタスクは ジャンプ。
- より良い命令データミックス ShareGPT4V (高品質 GPT-4V キャプション)。
- より強いベース LLM (Mistral-7B、Yi-34B)。

### LLaVA-OneVision

Lesson 12.08 は OneVision を詳しく覆う。短いバージョン：同じプロジェクター、だが 単一イメージ、マルチイメージ、ビデオをカバーしているカリキュラムで訓練、共有ビジュアルトークン予算。

### Q-Former との比較

| | Q-Former (BLIP-2) | MLP (LLaVA) |
|---|---|---|
| イメージあたりビジュアルトークン | 32 | 576 (ベース) または 2880 (AnyRes) |
| 訓練可能パラメータ | 188M + LM | 40M + LM |
| 段階1損失 | ITC+ITM+ITG | LM のみ |
| LLM ドロップイン | 再訓練が必要 | 最小限の再訓練でスワップ |
| マルチイメージ | 気が進まない | 自然 (連結) |
| ビデオ | 気が進まない | 自然 (フレームあたり連結) |
| トークン予算 | 小 | 大 |

MLP はシンプリシティとトークン柔軟性で勝つ。Q-Former はトークン予算で勝つ。2023年後半までにトークン予算はもはやバインディング制約ではない (LLM コンテキスト 32k-128k+ に成長) そしてシンプリシティが支配的。

### プロンプトフォーマット

```
A chat between a curious human and an artificial intelligence assistant. The assistant gives helpful, detailed, and polite answers to the human's questions. USER: <image> Describe this image in detail. ASSISTANT: The image shows ...
```

`<image>` はプレースホルダートークン。トークン化の前、それは 576 ビジュアルトークンで置き換え (AnyRes で 2880)。トークナイザーは訓練されたより少し長いシーケンスを見る、だが LLM は段階1 がそれに新規入力を教えたため、ハンドル。

### パラメータ経済

LLaVA-1.5-7B ブレークダウン：
- CLIP ViT-L/14 @ 336：303M (段階1凍結、しばしば段階2アンフロ)。
- プロジェクター (2x 線形)：～22M 訓練可能。
- Llama-7B：7B。
- 合計：7.3B パラメータ。段階2中に訓練可能：完全 7B + 22M プロジェクター。

段階2 トレーニングコスト：～20 時間 8xA100。これはキーナンバー — 1日、1ノード、再現可能。これが LLaVA スプレッド理由。

## これを使用する

`code/main.py` は実装：

1. 2層 MLP プロジェクター (おもちゃスケール 次元 16 → 32 → 32) 純粋 Python。
2. プロンプト構築パイプライン：システムプロンプト + `<image>` 置き換え N 投影トークン + ユーザーターン + アシスタント生成プレースホルダー。
3. ビジュアライザー 576 トークンビジュアルブロック LLM コンテキストで何か見える (2k / 32k / 128k コンテキストの消費パーセント)。

## 配布する

このレッスンは `outputs/skill-llava-vibes-eval.md` を生成。LLaVA ファミリーチェックポイント与えられたとき、10 プロンプトバイブスエバル スイート (3 キャプショニング、3 VQA、2 推論、2 拒否) を実行し、人間が読み取り可能スコアカード報告。ベンチマークではない；プロジェクターと LLM が良く接続しているか確認スモークテスト。

## 演習

1. 2層 MLP プロジェクターの訓練可能パラメータカウント `1024 → 4096 → 4096` で計算。GELU とバイアス、LLaVA-13B の何分か表す？

2. LLaVA プロンプト「拒否」ケース構築 — 画像は個人プライベートが含む。期待されるアシスタント応答を書く。なぜ LLaVA ゼロショット拒否すべき、どのトレーニングデータが拒否を強化する必要？

3. LLaVA-NeXT のブログの AnyRes セクション読む。1344x672 画像の AnyRes でのビジュアルトークンカウント計算。ベース 336x336 で 576 トークンと比較。

4. LLaVA 段階1プロジェクターはキャプションで LM 損失で訓練。段階1をスキップ、段階2 (視覚命令チューニング) に直接行くと何が起こるか？Prismatic VLM アブレーション (arXiv:2402.07865) 答えでを引用。

5. LLaVA-Instruct-150k は GPT-4 で COCO キャプション使用して命令を生成。新規ドメイン (医学 X線、衛星画像)、ドメイン命令を生成する4段階データパイプライン記述。各段階で何が悪くなる？

## キーターム

| 用語 | 人は何と言うか | 実際の意味 |
|------|----------------|----------|
| プロジェクター | 「MLP ブリッジ」 | 2層 MLP GELU ViT 次元を LLM 次元にマップ |
| イメージトークン | 「<image> placeholder」 | プロンプトマーカー推論前に N 投影ビジュアルトークンで置き換え |
| 視覚命令チューニング | 「LLaVA 段階2」 | GPT-4 生成 (画像、命令、応答) トリプレットで訓練 |
| 段階1配置 | 「プロジェクター事前訓練」 | ViT と LLM を凍結、キャプションの LM 損失でプロジェクター訓練 |
| AnyRes | 「マルチクロップタイリング」 | 高解像度画像をタイルグリッドに分割、各タイルのビジュアルトークン連結 |
| LLaVA-Instruct | 「GPT-4 生成」 | 158k 命令応答ペア COCO キャプション + GPT-4 から合成 |
| ビジョンエンコーダフリーズ | 「バックボーンロック」 | CLIP 重みは段階1 更新しない、時々段階2 でもない |
| ShareGPT4V | 「より良いキャプション」 | 1M 密なキャプション GPT-4V で生成、より高品質配置 |
| VQA | 「視覚質問応答」 | 画像についての自由形式の質問に答えるタスク |
| Prismatic VLM | 「設計空間ペーパー」 | Karamcheti 2024 アブレーション計画的にテスト プロジェクターとデータ選択 |
