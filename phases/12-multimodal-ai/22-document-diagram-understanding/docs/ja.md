# ドキュメントとダイアグラム理解

> ドキュメントは写真ではない。PDF、科学論文、インボイス、または手書きフォームはレイアウト、表、ダイアグラム、脚注、ヘッダー、セマンティック構造を持つ。平文画像理解はキャプチャできない。前VLM スタックはパイプライン：Tesseract OCR + LayoutLMv3 + テーブル抽出ヒューリスティック。VLM波はそれをOCRフリーモデルで置き換え — Donut（2022）、Nougat（2023）、DocLLM（2023） —直接構造化マークアップを発行。2026年までに先端は「ページ画像をClaude Opus 4.7で2576pxネイティブに供給」し、構造化マークアップ出力は無料で来る。本レッスンはドキュメントAIの3時代のアーク を読む。

**タイプ:** ビルド
**言語:** Python（stdlib、レイアウト認識ドキュメント パーサー スケルトン）
**前提条件:** Phase 12 · 05（LLaVA）、Phase 5（NLP）
**所要時間:** 約180分

## 学習目標

- ドキュメント AI の3時代を説明：OCR パイプライン、OCRフリー、VLMネイティブ。
- LayoutLMv3 の3つの入力ストリームを説明：テキスト、レイアウト（bbox）、イメージ パッチ、統一マスキング。
- Donut（OCRフリー、イメージ -> マークアップ）、Nougat（科学論文 -> LaTeX）、DocLLM（レイアウト認識生成）、PaliGemma 2（VLMネイティブ）を比較。
- 新しいタスク（インボイス、科学論文、手書きフォーム、中国レシート）のドキュメント モデルを選択。

## 問題

「このPDFを理解」は欺くほど難しい。情報は座る：

- テキスト コンテンツ（90%のシグナル）。
- レイアウト（ヘッダー、脚注、サイドバー、2列フォーマット）。
- テーブル（行、列、マージされたセル）。
- フィギュアとダイアグラム。
- 手書き注釈。
- フォントとタイポグラフィ（タイトル対ボディ）。

生のOCRはテキストをダンプして残りを失う。インボイスケアするシステムは「Total: $1,245」が右下からきたか脚注からかを知る必要。

## コンセプト

### 時代1 — OCR パイプライン（2021年前）

古典的スタック：

1. PDF -> ページごとのイメージ。
2. Tesseract（または商用OCR）はワードバウンディング ボックス付きテキストを抽出。
3. レイアウト アナライザーはブロック（ヘッダー、テーブル、段落）を識別。
4. テーブル構造レコグナイザーはテーブルを解析。
5. ドメイン ルール + 正規表現はフィールドを抽出。

きれいな印刷テキストで動作。手書き、傾いたスキャン、複雑なテーブル、非英語スクリプトで破壊。すべての失敗モードはカスタム例外パスが必要。

### TrOCR（2021）

TrOCR（Li et al., arXiv:2109.10282）はTesseractの古典的CNN-CTCを変換エンコーダ・デコーダで置き換え。合成+実テキスト画像で訓練。手書きと多言語テキストで消しない勝利。もまだパイプライン（検出器その後TrOCRその後レイアウト）だが、OCRステップは劇的に改善。

### 時代2 — OCRフリー（2022-2023）

最初のOCRフリー モデルは言う：検出を完全にスキップ、イメージピクセルを構造化出力に直接マップ。

Donut（Kim et al., arXiv:2111.15664）：
- エンコーダ・デコーダ トランスフォーマー、エンコーダは Swin-B。
- 出力はフォーム理解用JSON、要約用マークダウン、または任意のタスク特定スキーマ。
- OCRなし、レイアウトなし、検出なし。

Nougat（Blecher et al., arXiv:2308.13418）：
- 特に科学論文で訓練。
- 出力はLaTeX / マークダウン。
- 方程式、マルチ列レイアウト、フィギュアを処理。
- すべてのarXivパーサーが呼ぶモデル。

これらは専門家、汎用ではない。科学論文上の Donut は失敗；インボイス上の Nougat は失敗。

### LayoutLMv3（2022）

別の軌道。LayoutLMv3（Huang et al., arXiv:2204.08387）はOCRを保持するがレイアウト理解を加える：

- 3つの入力ストリーム：OCRテキスト トークン、トークン当たり2D バウンディング ボックス、イメージ パッチ。
- マスクドトレーニング目標はすべての3つのモダリティ全体（マスクドテキスト、マスクドパッチ、マスクド レイアウト）。
- 下流：分類、エンティティ抽出、テーブルQA。

LayoutLMv3 はOCRベース ドキュメント理解のピーク。フォームとインボイスで強い。上流のOCRが必要。標準化ドキュメント ベンチマーク上で最良の事前VLM精度。

### DocLLM（2023）

DocLLM（Wang et al., arXiv:2401.00908）はLayoutLMの生成シブリング。レイアウト トークンに条件付けされた自由形式の回答を生成。ドキュメント上QAのためにより良い；まだOCR入力に依存。

### 時代3 — VLMネイティブ（2024年以降）

2024年 VLMは十分に良くなり、パイプライン全体を置き換え。全ページ イメージを高解像度でVLMに供給、質問、回答を得る。

- LLaVA-NeXT 336タイル AnyRes は小さいドキュメントで動作。
- Qwen2.5-VL 動的解像度はネイティブに2048+ピクセルを処理。
- Claude Opus 4.7 は2576pxドキュメントをサポート。
- PaliGemma 2（2025年4月）はドキュメント+手書きのために特に訓練。

VLMネイティブとOCRパイプラインの間のギャップは迅速に閉じた。2026年までに、VLMネイティブは勝つ：

- シーンテキスト（手書き+印刷、混合スクリプト）。
- マージされたセルの複雑なテーブル。
- テキストに埋め込まれた数学方程式。
- テキスト注釈付きの数字。

OCRパイプラインはまだ勝つ：

- 単純スキャン ワークロード大規模でページごとのレイテンシが重要。
- パイプライン信頼性（決定論的失敗対VLM幻覚）。
- VLM幻覚対 OCR出力の監査が必要な規制環境。

### Claude 4.7 / GPT-5 先端

2576ピクセルネイティブ入力で、先端VLMはほぼ人間精度でドキュメント理解をする。2026年初期からのベンチマーク数字：

- DocVQA：Claude 4.7 ~95.1、PaliGemma 2 ~88.4、Nougat ~77.3、パイプライン LayoutLMv3 ~83。
- ChartQA：Claude 4.7 ~92.2、GPT-4V ~78。
- VisualMRC：Claude 4.7 ~94。

クローズド モデル ギャップはほとんど解像度と基本LLMスケール。7B でのオープン モデルは数ポイント後ろだが追い捕る。

### 数学方程式とLaTeX出力

科学論文は方程式用に正確なLaTeX出力が必要。Nougat はこれで訓練された。LaTeX ターゲット（Qwen2.5-VL-Math、Nougat導関数）で訓練されたVLMは使用可能なLaTeX を生成。明確なLaTeX訓練なし、VLMは読み可能だが不正確な転写を生成。

2026年での科学論文パイプラインのため：PDFでNougat をチェーン、その後トリッキーなページでVLM。

### 手書き

まだ最難しいサブタスク。混合印刷+手書き（医師のノート、記入フォーム）はOCRパイプラインがVLMでまだコストで勝つところ。手書きのみVLMは改善（Claude 4.7、PaliGemma 2）。

### 2026年レシピ

新しいドキュメント AI プロジェクトのため：

- 規模でのピュア印刷インボイス：LayoutLMv3 + ルール、コスト効率。
- 混合ドキュメント（科学+手書き+フォーム）：VLMネイティブ（PaliGemma 2 またはQwen2.5-VL）。
- 完全なarXiv摂取：数学にはNougat、フィギュアにはVLM。
- 規制：OCR パイプライン + クロスチェック用VLM バリデータ。

## 使用方法

`code/main.py`：

- おもちゃレイアウト認識トークナイザー：（テキスト、bbox）ペアから、LayoutLMv3スタイル入力を生成。
- Donutスタイル タスク スキーマ ジェネレータ：フォーム用JSONテンプレート。
- トークン予算のページごとの比較：OCRパイプライン、Donut、Nougat、VLMネイティブ。

## 提出

本レッスンは`outputs/skill-document-ai-stack-picker.md`を生成。ドキュメント AI プロジェクト（ドメイン、スケール、品質、規制）が与えられた場合、OCRパイプライン、OCRフリー専門家、VLMネイティブの間で選択。

## 演習

1. プロジェクトは1日あたり1000万インボイス。精度なしでコストごとのページを最小化スタックはどれ？

2. なぜLayoutLMv3 はフォームQAで純CLIP VLMを上回るが、シーンテキストで下回る？bboxストリームは何をあきらめ？

3. Nougat が LaTeX を生成。VLMネイティブ出力がNougat上でLaTeX忠実性を打つテストケースを提案し、Nougat が勝つケース。

4. PaliGemma 2論文（Google、2024）を読む。PaliGemma 1 対ドキュメント精度を上げた主要訓練データ追加は何？

5. 規制的安全ハイブリッドを設計：OCRパイプラインを1次、VLMを2次クロスチェック。相違の解決方法？

## 主要な用語

| 用語 | 人々が言う | 実際の意味 |
|------|------------|---------|
| OCR パイプライン | 「Tesseract スタイル」 | 段階的スタック：検出 -> OCR -> レイアウト -> ルール；決定論的、壊れやすい |
| OCRフリー | 「Donutスタイル」 | 明確なOCRをスキップする画像から出力トランスフォーマー；単一モデル |
| レイアウト認識 | 「LayoutLM」 | 入力はトークン当たりbboxコーディネートを含む；統一マスキング全モダリティ全体 |
| VLMネイティブ | 「先端VLM」 | ページイメージをClaude/GPT/Qwen VLMで高解像度に直接供給；パイプラインなし |
| DocVQA | 「ドキュメント ベンチマーク」 | ドキュメント VQA 基準；最も引用されたスコア |
| マークアップ出力 | 「LaTeX / MD」 | 自由形式テキストの代わりの構造化出力フォーマット；下流自動化を有効化 |
