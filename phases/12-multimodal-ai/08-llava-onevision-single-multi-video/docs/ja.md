# LLaVA-OneVision：単一イメージ、マルチイメージ、ビデオ1つのモデルで

> LLaVA-OneVision (Li et al.、2024年8月) の前に、開く VLM 世界は別々の系統を持つ：単一イメージ向け LLaVA-1.5、マルチイメージモデル Mantis と VILA、ビデオモデル Video-LLaVA と Video-LLaMA のような。各がそのベンチマーク勝った、他で失敗。LLaVA-OneVision は、1つのカリキュラムが1つのモデルをすべての3つのシナリオで支配することができ、出現タスク転送効果 (単一イメージスキル ビデオへエクスポート、マルチイメージ推論エクスポート単一イメージ) スペシャリストの合計を打つことができると主張。レシピは欺くほど単純：シナリオ横断で定数であるビジュアルトークン予算、プラス単一イメージからビデオに移動明示的カリキュラム。このレッスンは予算、カリキュラム、出現動作を読む。

**タイプ:** ビルド
**言語:** Python (標準ライブラリ、トークン予算ソルバー + カリキュラムプランナー)
**前提条件:** Phase 12 · 05 (LLaVA)、Phase 12 · 06 (任意解像度)
**所要時間:** 約180分

## 学習目標

- 単一イメージ、マルチイメージ、ビデオ入力横断で定数を保つビジュアルトークン予算設計。
- 大喪失なしビデオにスキル転送する単一イメージからの訓練カリキュラムをオーダー。
- なぜ単一モデルが同じパラメータカウントのスペシャリストを打つか、カリキュラムが右のとき説明。
- 3つ出現機能 LLaVA-OneVision により報告命名：マルチカメラ推論、マーク設定プロンプト、iPhone スクリーンショットエージェント。

## 問題

画像、マルチイメージ、ビデオは各々モデル異なるストレス。

単一イメージは高解像度トークン (AnyRes、～2880 ビジュアルトークン) をしたいテキストと細粒詳細をキャッチ。サンプルあたり予算：1つのイメージ、2880 トークン。

マルチイメージは複数イメージを中程度解像度 (～576 トークン/イメージ) をしたい理由推論は通観に適合。サンプルあたり予算：4-8 イメージ、576 それぞれ、2300-4600 トークン。

ビデオは時間的ダイナミクスをキャッチする多くのフレーム低解像度 (～196 トークン/フレーム プール後) をしたい。サンプルあたり予算：8-32 フレーム、196 それぞれ、1600-6200 トークン。

別々モデルを訓練場合、1つの予算をピック。1つのモデルを訓練場合、あなたは文脈をクリアするシナリオを怒らすことなく予算をスケール必要。

OneVision 前、デフォルト答えは「トレーニング 1つのシナリオ、他のシナリオは無視。」Video-LLaVA は余分な訓練段階でビデオを画像モデルに改装。LLaVA-NeXT はマルチイメージサポートをタイリング加えた。いずれもきれいに3つが扱う。

## コンセプト

### OneVision トークン予算

LLaVA-OneVision は統一ビジュアルトークン予算の約 3000-4000 トークン/サンプルをピック、シナリオあたり異なるに配置：

- 単一イメージ：AnyRes-9 (3x3 タイル + サムネイル)、各タイル 384 729 パッチ、積極的な双線形プール 2x2 → 182 / タイル。合計：9 * 182 + 182 = 1820 トークン。または AnyRes-4 729 / タイル = 2916 + 729。
- マルチイメージ：各イメージを中程度解像度 (384、タイリングなし)、729 トークンプール なし。予算 6 イメージ → 4374 トークン。
- ビデオ：32 フレーム 384 解像度、積極的な 3x3 双線形プール → 81 トークン/フレーム。合計：32 * 81 = 2592 トークン。

配置は総計トークンをほぼ定数に保つ。LLM はバッチを見ることは決してない、その文脈を怒らす。エンコーダはシナリオあたり異なるジオメトリ生成、だが LLM は同じ予算をコンシューム。

### 3段階カリキュラム

LLaVA-OneVision は3段階で訓練：

1. 単一イメージ SFT (段階 SI)。すべてのデータは単一イメージ プラステキスト。高解像度 AnyRes 入力での訓練。これは知覚、OCR、細粒理解を教える。LLaVA-NeXT データプラス OneVision 固有単一イメージデータを使用。
2. OneVision SFT (段階 OV)。単一イメージ + マルチイメージ + ビデオ混合 (均等サンプリングフレーム)。統一トークン予算での訓練。これはモデルを異種バッチ形をハンドルすることを教える。ウェイトリセットなし — 段階 SI から続ける。
3. タスク転送 (段階 TT)。ターゲットタスク混合、典型的にマルチイメージまたはビデオについて より重い設定対象。オプショナルデプロイメント微調。

重要：カリキュラムオーダーは物質。ビデオファースト またはマルチイメージ最初のトレーニングは単一イメージのみ同じデータ悪いイメージ性能生成。ペーパーは明示的にこれを切る。

### カリキュラムなぜ機能

単一イメージトレーニングは知覚ベースを構築。パッチトークンは細粒ビジュアル特徴をキャリ；LLM はテキストと統合することを学習。マルチイメージとビデオは構造的チャレンジ導入 (どのイメージはどの、何が最初に起こったか)、強い知覚ベースなく学習することは困難。

すべてのシナリオを一緒からスクラッチでトレーニング場合、モデルは知覚アンダーフィット (バッチごとに限定単一イメージデータ) と過度合わせ構造 (多くのマルチイメージ / ビデオデータ)。結果：クロスイメージ推論パターンをフォローするモデルだが視覚的に浅い。

カリキュラムオーダーは段階 SI から知覚強度をあなたに提供、次に段階 OV からコンポジショナル/時間的推論、いずれも失わず。

### 出現クロスシナリオスキル

LLaVA-OneVision ペーパーは3つの出現機能報告：

1. マルチカメラ推論。別々に訓練マルチイメージ + ビデオ；推論時、マルチカメラ運転シーンについて理由するよう依頼。モデルは正しく訓練で正確なフォーマットを見ることは決してあったのに ビュー統合。
2. マーク設定プロンプト。ユーザー画像のオブジェクトをマークすることで注釈；モデルは理由 「マーク3は何をするマーク7に相対」。訓練マークもアノテーションのいずれも；空間グラウンディング + マルチイメージ参照の組み合わせから学習。
3. iPhone スクリーンショットエージェント。ユーザー iPhone スクリーン スクリーンショット提供，次のクリック計画する を依頼。訓練 UI スクリーンショット、ユーザーワークフロー ビデオ、マルチイメージ前後ペア。エージェント用件に一般化。

これらはトレーニング タスク ではない；それらは出現する カリキュラムのコンポジショナル構造。

### ビジュアルトークンプール

トークン予算はプール必要。OneVision は双線形補間を 2D パッチグリッド上で使用：24x24 = 576 パッチは 12x12 = 144 (2x 係数) または 8x8 = 64 (3x 係数) になる。プール パッチグリッド空間で行われ、トークン空間ではなく、局所性を保存。

シナリオあたりのプール係数選択は独自ハイパーパラメータ。より少ないプール = より多くトークン = より豊かな表現。より多くプール = より少ないトークン = より多くのフレーム / イメージが適合。

### LLaVA-OneVision-1.5

2025 フォローアップ (LLaVA-OneVision-1.5、arXiv 2509.23661) は「完全開く」トレーニング データ、モデル重み、コード。いくつかのベンチマーク上のプロプライエタリギャップマッチと民主化レシピ。同じカリキュラム、より多くデータ、より良いベース LLM。アーキテクチャ変更なし。

### Qwen2.5-VL との対比

Qwen2.5-VL (Lesson 12.09) は異なる選択を行う。それは M-RoPE と動的 FPS の代わりに使用固定プール。予算は入力をスケール — 1 分ビデオは 5 秒ビデオより多くトークン を使用。LLaVA-OneVision は予算を修正してプール をスケール。両方動作；それらは予測 可能性配置 交易可能。

## これを使用する

`code/main.py` は OneVision スタイル VLM 用カリキュラムと予算プランナー。トークン予算/サンプルとターゲットシナリオ混合与えられたとき (例えば 40% 単一イメージ、30% マルチイメージ、30% ビデオ)、それは：

- 各シナリオの解像度、プール係数、フレームを配置。
- すべてのシナリオが共有予算の内にフィット確認。
- 期待トークンカウント、LLM FLOP、どのシナリオが下トークン化報告。
- ステージバイステージ訓練スケジュール印字。

OneVision 微調またはサニティ確認 VLM デプロイメント/リクエスト コストをプランするときそれを使用。

## 配布する

このレッスンは `outputs/skill-onevision-budget-planner.md` を生成。ターゲットタスク分布，/サンプル予算与えられたとき、それは AnyRes 係数、/フレーム プール、ビデオフレーム カウント、カリキュラム段階ウェイトを発行。統一シナリオ VLM を訓練またはファインチューンするたびそれを使用。

## 演習

1. あなたの製品は 80% 単一イメージ、10% マルチイメージ (2-4 イメージ)、10% ビデオ (8-16 フレーム) をサポート。トークン予算設計。あなたが重いマルチイメージではなく保存する余分予算をどこに置く？

2. LLaVA-OneVision Section 4.3 (出現機能) を読む。カリキュラムは可能性があります第4出現スキル提案ペーパーが報告ではなかった。

3. カリキュラム オーダー スワップ — マルチイメージ最初のトレーニング、単一イメージ次、ビデオ次。どのベンチマーク劣化予測してなぜ。

4. ペーパーは、実にビデオベンチマーク報告フレーム/サンプルだけ8。推論30秒ビデオに一般化か？什么 第1を壊す — トークン予算またはテンポラル推論？

5. 24x24 パッチの双線形プール から 12x12 は 4x削減/ディム。stdliib Python でプール実装，各 2x2 ブロック上の平均が双線形出力マッチ確認。

## キーターム

| 用語 | 人は何と言うか | 実際の意味 |
|------|----------------|----------|
| OneVision シナリオ | 「単一イメージ、マルチイメージ、またはビデオ」 | 1つの3つ入力形は統一 VLM ハンドル；予算は横断定数 |
| トークン予算 | 「いくつトークン/サンプル」 | 総ビジュアルトークン LLM 見る/訓練 推論サンプル、典型的 3000-4000 |
| カリキュラム | 「訓練オーダー」 | ステージ順序 (単一イメージ → マルチイメージ → ビデオ) 出現転送用に選択 |
| 双線形プール | 「トークン削減」 | パッチグリッド (2D) への双線形補間適用トークン削減しつつ局所性保存 |
| 出現スキル | 「訓練されず、それでも機能」 | トレーニング データにマッチするなしで推論で表示 機能、カリキュラム構成。 |
| AnyRes-k | 「k-タイル設定」 | k サブタイル固定解像度プラス1つのサムネイル、典型的 k ∈ {4、9} |
| タスク転送 | 「クロスシナリオ一般化」 | 単一イメージで学習したスキルがビデオに適用 (そしてその逆) 共有バックボーン経由 |
