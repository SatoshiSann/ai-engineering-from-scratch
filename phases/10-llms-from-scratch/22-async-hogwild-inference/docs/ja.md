# 非同期とHogwild! 推論

> 投機的デコーディング（フェーズ 10・15）は1つのシーケンス内のトークンを並列化します。マルチエージェントフレームワークはシーケンス全体を並列化しますが、明示的な調整（投票、サブタスク分割）を強制します。Hogwild! 推論（Rodionov et al.、arXiv:2504.06261）は別のことを行います：共有キー値キャッシュに対してN個の同じLLMのインスタンスを並列実行します。各ワーカーは他のワーカーが生成したトークンをすぐに見ます。現代的な推論モデル—QwQ、DeepSeek-R1—はファインチューニングなしで共有キャッシュを通じて自己調整できます。このアプローチは実験的ですが、投機的デコードに直交する推論並列化の全く新しい軸を開きます。このレッスンはstdlib Pythonで2ワーカーのHogwild! シミュレーターを実装し、共有キャッシュの協調作用が既存モデルの推論能力から生じる理由を説明します。

**タイプ:** Build
**言語:** Python（stdlib）
**前提条件:** フェーズ 10・12（推論最適化）、フェーズ 10・15（投機的デコーディング）
**所要時間:** 約60分

## 学習目標

- 3つの共通の並列LLMトポロジー（投票、サブタスク、Hogwild!）を説明し、各々がターゲットとする問題を名前付ける。
- コアHogwild! セットアップを述べる：複数のワーカー、1つの共有KVキャッシュ、自己プロンプトを通じた生じた調整。
- Hogwild! の壁時間スピードアップをワーカー数`N`、タスクレベル並列化`p`、および調整オーバーヘッド`c`の関数として計算する。
- 2ワーカーHogwild! シミュレーターをトイ問題に実装し、生じたタスク分割を観察する。

## 問題

現代のLLMは長い推論チェーン—5000トークンのステップバイステップロジックが一般的であり、深い数学の問題では数万トークンが発生—を産生することで難しい問題を解きます。70Bモデルで35トークン/秒デコード、50kトークンは24分です。インタラクティブではないモデル。

投機的デコーディング（フェーズ 10・15）は1つのシーケンス内で並列化することで3～5倍のスピードアップを得られます。その先では、自動回帰デコーディングの順序依存は硬い天井です。各新しいトークンはすべての以前のトークンに依存します。

明白な質問：シーケンス全体で並列化できますか？同じ問題の同じモデルの複数のコピーを実行して、協力させ、仕事を分割させますか？

前の仕事：投票アンサンブル（N個のモデルを実行、多数回答を選ぶ）、思考の木（推論パスを分岐して再組み合わせ）、マルチエージェントフレームワーク（各エージェントにサブタスクを割り当て、コーディネーターを使用）。これらはすべて特定のタスク領域で役立ちます。また、すべて明示的な調整機械—投票ルール、分岐プルーン論理、エージェント間メッセージングプロトコルを導入します。

Hogwild! 推論は異なるアプローチを取ります。Nワーカーは単一のKVキャッシュを共有します。各ワーカーはあらゆる他のワーカーが生成したトークンをすぐに見、それがそれ自身のコンテキストであるかのように。ワーカー—訓練やファインチューニングなしで—仕事を分割する方法を理解します。現代的な推論モデル（QwQ、DeepSeek-R1、Claude-family推論モード）は共有キャッシュを読んで「ワーカー2がすでに基本ケースを処理したのを見ます、だから私は帰納的ステップで仕事します」のようなことを言うことができます。

スピードアップはワークロード依存であり、2026年4月の時点で実験的です。しかし、推論並列化の新しい軸を開くため、このアイデアは知る価値があります。

## コンセプト

### セットアップ

Nワーカープロセスを初期化し、すべて同じLLMを実行しています。ワーカーごとのKVキャッシュの代わりに、1つの共有キャッシュを維持します。ワーカー`i`がトークン`t_j`を生成すると、トークンは次の位置で共有キャッシュに書き込まれます。ワーカー`k`がその次のステップを行うとき、キャッシュの現在の状態を読みます（これまでのすべてのNワーカーが生成したすべてを含む）。

ステップ時に、ワーカーはトークンを書き込むために競争します。ワーカーごとの位置インデックスはありません—キャッシュは単一の成長シーケンスです。順序は書き込み到着時間で決定されます。

### なぜ調整が生じるか

ワーカーは提案を共有します。通常「この問題に一緒に取り組んでいるNインスタンスのいずれかです。各インスタンスは共有メモリを読んで、他のインスタンスが書いたものを見ることができます。冗長な仕事を避けてください。」のようなもの。提案と共有キャッシュが十分です。推論モデルはキャッシュを読み、問題のどの部分がすでに試みられたかに注目し、（常にではありませんが）未探索の部分にピボットします。

Hogwild! ペーパー（Rodionov et al.、2025）はしばしば以下のような観察をレポートします：

- ワーカーはキャッシュを通じて他のワーカーに計画を策定して通信します。
- ワーカーは他のワーカーの推論のエラーに注目してそれを呼び出します。
- ワーカーは計画が失敗したときに適応して代替案を提案します。
- 冗長性をチェックするよう促すと、ワーカーはそれを検出してピボットします。

これらはいずれもファインチューニングを必要としません。生じた行動はモデルがすでに持つ推論能力から来ます。

### 命名

ペーパーの名前はHogwild! SGD（Recht et al.、2011）、非同期更新オプティマイザーに言及します。類推：SGDの非同期ワーカーはすべて共有パラメータベクトルに書き込みます；Hogwild! 推論のワーカーはすべて共有KVキャッシュに書き込みます。両方とも同期保証よりも経験的な収束に依存します。

### RoPEはこれを扱いやすくします

回転位置埋め込み（RoPE、Su et al. 2021）はQおよびKベクトル内の回転を通じて位置情報をエンコードします。位置は回転であり、焼き込まれたオフセットではなく、トークンの位置はKVキャッシュエントリを再計算せずにシフトできます。ワーカー`i`が位置`p`で共有キャッシュに書き込むと、その位置を読む他のワーカーは、キャッシュエントリを直接使用できます—再ローテーション不要。

学習位置または絶対位置モデルでは、Hogwild! は各並行書き込みでキャッシュ無効化が必要です。RoPEはキャッシュを安定させておきます。

### 壁時間数学

`T_serial`を1ワーカーが問題を単独で解く時間にしましょう。`p`をタスクレベルで並列化可能な分数にしましょう。`c`をステップごとの調整オーバーヘッド（拡張キャッシュを読む、書き込む内容を決定する）にしましょう。

シングルワーカー時間：`T_serial`。
N-ワーカーHogwild! 時間、調整が無料の場合：`T_serial * ((1 - p) + p / N)`。古典的なAmdahl。
調整オーバーヘッド付き：`T_serial * ((1 - p) + p / N) + c * steps_per_worker`。

ワーカーが生産的であるため、`c`は1ステップあたりのデコード時間に対して小さくなければなりません。5k+トークンを生成する推論モデルでは、ワーカーは何百ものトークンの調整オーバーヘッドを余裕を持つことができ、それでも先に出ます。短いチャットタスクでは、調整が支配的であり、Hogwild! はシリアルより悪いです。

### 具体例

推論問題：10kトークンの思考チェーン。問題が`p = 0.7`の並列化可能なコンテンツ（異なる証明戦略、異なるケース分析）と`c = 200`トークンのワーカーごとの調整オーバーヘッドを持つと仮定します。N = 4ワーカーでは：

- シリアル時間：10000デコードステップ。
- Hogwild! 時間：10000 * (0.3 + 0.7 / 4) + 200 * 4 = 10000 * 0.475 + 800 = 5550デコードステップ。
- スピードアップ：10000 / 5550 = 1.8倍。

それは控えめです。しかし、より長い推論問題（50kトークン）では、調整オーバーヘッドは償却され、スピードアップは2.5～3倍にプッシュします。Hogwild! はマルチスレッドコードを自然に書くことができる言語のスレッドレベル並列化の推論同等です。

### Hogwild! に手を伸ばすとき

- 長い推論問題（数千トークン）、タスクが独立したサブゴール全体で並列化できる場合。
- ステップバイステップで考えるよう訓練された推論モデル。非推論モデルは自己調整をよくしません。
- 共有キャッシュとNワーカープロセスを保持するのに十分なVRAMを備えた単一ノード展開。キャッシュは共有されますが、各ワーカーはそれ自身のアクティベーションメモリを持ちます。

### そうしないときは

- 短いインタラクティブなチャット。調整オーバーヘッドが支配的です。
- 並列化しないタスク（単一の線形証明、単一のコンパイル）。N=1が最大です。
- 非推論モデル。調整は生じません。
- マルチノード展開。共有キャッシュはワーカー間の非常に高速な相互同期が必要です。ノード内はそれらです；ノード間はレイテンシ災害です。

### 実験的状態

2026年4月の時点で、Hogwild! はオープンソースPyTorch実装を備えた研究方法です。本番の採用は起きていません。3つのブロッカー：

1. 同時プロセス全体の共有KVキャッシュ管理は非自明なエンジニアリングです。
2. 生じた調整はタスク依存；ベンチマークはまだ構築されています。
3. スピードアップは投機的デコーディングが既に提供するものと比較して控えめ、そして2つを組み合わせることができますが、組み合わせたエンジニアリングは別のレイヤーです。

知る価値あり。実験する価値あり。まだ製品に賭ける価値あり。

## それを構築する

`code/main.py`はトイHogwild! シミュレーターを実装します：

- 2つのワーカープロセス、各々は決定論的「LLM」は、いくつかのトークンカテゴリーの1つ（仕事トークン、観察トークン、調整トークン）を既知の確率で産生します。
- 両方のワーカーが読み込みおよび書き込みする共有キャッシュ（トークンのリストだけ）。
- 簡単な調整論理：ワーカーが他がすでに十分な仕事トークンをカテゴリーに産生したのを見たとき、別のカテゴリーを選びます。

シミュレーターは固定ステップ予算で実行され、以下をレポートします：

- 産生された総仕事トークン。
- 総壁時間（ワーカーステップ数）。
- シングルワーカー上の有効スピードアップ。
- どのワーカーがどのトークンを書き込んだかのトレース。

### ステップ1：共有キャッシュ

両方のワーカーが追加するリスト。実装では簡単なロック（Python`threading.Lock`）；私たちはカウンターで模擬します。

### ステップ2：ワーカーループ

各ワーカー、各ステップ：

- 現在の共有キャッシュを読みます。
- すでにあるものに基づいて何のカテゴリーのトークンを書き込むかを決定します。
- 1つのトークンを書き込みます。

### ステップ3：調整ヒューリスティック

カテゴリーXがすでにキャッシュにKトークンを持ち、ワーカーの意図されたカテゴリーはXの場合、ワーカーはカテゴリーYに切り替えます。これは「これはすでにカバーされていることに注目、別の代わりにしてください」の推論モデルの行動のトイスタンドインです。

### ステップ4：測定されたスピードアップ

N=1ワーカーとN=2ワーカー、同じ合計ステップ予算でシミュレーターを実行します。産生された仕事トークンをカウントします。調整駆動タスク分割のため、N=2はおおよそ1.5～1.8倍多くの仕事トークンを産生する必要があります。

### ステップ5：調整を圧迫する

調整ヒューリスティックの感度を減らします。再度実行します。良い調整なしで、N=2は冗長にそれら同じトークンを産生し、スピードアップが1以下に落ちるのを観察します。これはペーパーの観察に一致します：トリックはワーカーが自己調整する推論容量を持つ場合のみ機能します。

## 使用方法

2026年4月の時点での本番のHogwild! 統合は研究グレードです。Yandex/HSE/ISTからのリファレンス実装はPyTorch-ベースでおよびシングルノード多進プロセスセットアップ、DeepSeek-R1およびQwQモデル上の目標をします。

実用的な採用パス：

1. 推論タスクワークロードをプロファイルしてください。トークンの画分が探索的（複数戦略、ケース分析、サーチ）対線形であることを測定してください。
2. 探索が支配的であれば、2ワーカーHogwild! 実験を実行してください。壁時間改善を測定します。
3. 改善が1.3倍未満の場合、調整支配レジーム内です。シングルワーカーに戻してください。
4. 改善が1.5倍を超える場合、N=4にプッシュして再度測定します。減収は通常N=4～8の周りに当たります。

投機的デコーディングと組み合わせる：各Hogwild! ワーカーは独立して投機デコードを使用できます。2つのスピードアップは乗算します（粗く）、3倍投機デコードと1.8倍Hogwild! を有能なシングルワーカーデコーディング上の有効5.4倍にもたらします。

## 提供

このレッスンでは`outputs/skill-parallel-inference-router.md`を生成します。推論ワークロードプロファイル（トークン予算、タスク並列化プロファイル、モデルファミリー、展開対象）が与えられた場合、投票、思考の木、マルチエージェント、Hogwild!、および投機的デコーディング戦略間でルーティングしてください。

## 演習

1. デフォルト設定で`code/main.py`を実行します。N=2 Hogwild! 構成がN=1ベースラインで同じ壁時間でより多くの仕事トークンを産生することを確認します。

2. 調整ヒューリスティックの強度を減らします（`coordination_weight=0.1`を設定）。再実行します。スピードアップが崩壊することを示します。理由を説明：ワーカーは調整できないとき努力を複製します。

3. `p=0.8, c=500`と4つのワーカーを持つ50k-トークン推論タスクのための期待されるHogwild! スピードアップを計算します。`p=0.3, c=200`と4つの1k-トークンチャットタスク用に同じことをしてください。なぜ1つは勝利で他方は損失ですか？

4. Hogwild! ペーパーのセクション4（予備評価）を読んでください。著者がレポートする2つのフェイルモードを特定します。より良い調整プロンプトがそれぞれをどのように緩和する可能性があるかを説明してください。

5. トイでのHogwild! と投機的デコーディングを組み合わせます：各ワーカーは内部で2トークン投機デコードを使用します。乗法スピードアップをレポートします。2つのワーカーが同じ共有キャッシュプレフィックスを拡張したいとき、どんな帳簿付けの問題が生じますか？

## キーターム

| 用語 | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| Hogwild! | 「並列ワーカー、共有キャッシュ」 | Nインスタンスの同じLLMは1つの共有KVキャッシュで同時に実行；自己プロンプトを通じた生じた調整 |
| 共有KVキャッシュ | 「調整媒体」 | すべてのワーカーが読み込みおよび書き込みする単一の成長するKVバッファ；ワーカー間での瞬間的なトークン可視化を有効化 |
| 生じた調整 | 「訓練が不要」 | 推論-能力LLMは共有キャッシュを読んで、ファインチューニングまたは明示的プロトコルなしで仕事を分割できます |
| 調整オーバーヘッド（c） | 「方向付けで費やしたトークン」 | 拡張キャッシュを読んでしようとすることを決定するワーカーごとの費用；総デコード時間対小さい滞在 |
| 並列化可能分数（p） | 「並列実行できるもの」 | タスクレベル並列化：本質的に順序的ではない総仕事の割合 |
| RopeはHogwild! を有効化 | 「回転位置はシフト不変」 | 位置は回転であるため、共有キャッシュに書き込むことは以前のトークンを再計算する必要がありません |
| 投票アンサンブル | 「N個を実行、多数を選ぶ」 | 最も簡単な並列推論トポロジー；分類に有用、長形式推論に減少 |
| 思考の木 | 「分岐とプルーン」 | 複数の分岐を探索してプルーン；明示的調整論理を行う推論戦略 |
| マルチエージェントフレームワーク | 「サブタスク割り当て」 | 各エージェントは役割を得ます；コーディネーターは管弦楽；重いプロトコルオーバーヘッド |
