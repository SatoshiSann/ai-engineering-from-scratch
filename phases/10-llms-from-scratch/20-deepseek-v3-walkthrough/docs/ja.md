# DeepSeek-V3 アーキテクチャウォークスルー

> フェーズ 10・レッスン 14 では、すべてのオープンモデルが調整する 6 つのアーキテクチャ パラメータを紹介しました。DeepSeek-V3（2024年12月、総パラメータ671B、アクティブパラメータ37B）はこれら6つすべてを調整し、さらに4つ追加します：Multi-Head Latent Attention、補助損失を使用しない負荷分散、Multi-Token Prediction、DualPipe訓練です。このレッスンでは、DeepSeek-V3のアーキテクチャを上から下まで読み進め、公開されているコンフィグからすべてのパラメータ数を導き出します。終わりまでに、671B/37Bの比率が正しい選択である理由と、MLA + MoEが単独での手法よりも前線規模で優れている理由を説明できます。

**タイプ:** Learn
**言語:** Python（stdlib、パラメータ計算機）
**前提条件:** フェーズ 10・14（オープンモデルウォークスルー）、フェーズ 10・17（NSA）、フェーズ 10・18（MTP）、フェーズ 10・19（DualPipe）
**所要時間:** 約75分

## 学習目標

- DeepSeek-V3コンフィグを上から下まで読み、6つのGPT-2パラメータと4つのDeepSeek固有の追加について各フィールドを説明する。
- 総パラメータ数（671B）、アクティブパラメータ数（37B）、および各成分への寄与度を導出する。
- 128kコンテキストでのMLAのKVキャッシュフットプリントを計算し、GQAを持つ同じアクティブパラメータの密集モデルが支払う場合と比較する。
- 4つのDeepSeek固有の革新（MLA、MTP、補助損失を使用しないルーティング、DualPipe）を述べ、アーキテクチャ/訓練スタックのどの部分が各々のターゲットであるかを名前付ける。

## 問題

DeepSeek-V3は、アーキテクチャがLlamaファミリーから意味を持って異なる最初のフロンティアオープンモデルです。Llama 3 405Bは「6つのパラメータを調整したGPT-2」です。DeepSeek-V3は、GPT-2のすべての6つのパラメータと4つ追加で調整しています。Llama 3のコンフィグを読むことはDeepSeekコンフィグを読むためのウォームアップですが、深い構造—アテンション層の形状、ルーティング論理、訓練時の目的—は十分に異なるため、別のウォークスルーが必要です。

これを学ぶことの価値：DeepSeek-V3のオープン重みリリースは、オープンモデルにおける「フロンティア機能」の意味をシフトさせました。アーキテクチャは、2026年の訓練実行の多くがコピーしているブループリントです。それを理解することは、フロンティアLLMの訓練または推論に関わる任意の役割に対して必須です。

## コンセプト

### 不変コア、再び

DeepSeek-V3は依然として自動回帰的です。それでもデコーダブロックをスタックしています。各ブロックには依然としてアテンション + MLP + 2つのRMSNormがあります。MLPではSwiGLUを使用しています。RoPEを使用しています。Pre-norm。重み結合埋め込み。Llamaまたはmistralのすべてと同じベースラインです。

### ツイスト：GQAの代わりにMLA

フェーズ 10・14からご存じのとおり、GQAはクエリ頭のグループ間でK と Vを共有することでKVキャッシュを縮小します。Multi-Head Latent Attention（MLA）はさらに進みます：K と Vは共有低ランク潜在表現（`kv_lora_rank`）に圧縮され、その後頭ごとにその場で解凍されます。KVキャッシュは潜在のみを保存します—通常、トークンあたり512フロート、8 x 128 = 1024フロートではなく、層あたり。

128kコンテキストでのDeepSeek-V3では、MLA（トークン単位、層単位の共有潜在`c^{KV}`；K と Vはどちらもこの潜在から導出され、後続の行列乗算に吸収できるアップ投影を介して）：

```
kv_cache = num_layers * kv_lora_rank * max_seq_len * bytes_per_element
         = 61 * 512 * 131072 * 2
         = 7.6 GB
```

仮説的なGQAベースライン（Llama 3 70B形状、8 KV頭、頭次元128）は以下のようになります：

```
kv_cache = 2 * 61 * 8 * 128 * 131072 * 2
         = 30.5 GB
```

MLAは128kコンテキストでのLlama-3-70B形式のGQAキャッシュより4倍小さいです。

トレードオフ：MLAはアテンション計算（頭ごと）ごとに解凍ステップを追加します。追加の計算は節約された帯域幅と比較して小さいです。長いコンテキスト推論のネット勝利。

### ルーティング：補助損失を使用しない負荷分散

MoEルーターは、各トークンを処理するトップk個の専門家を決定します。素朴なルーターは、少数の専門家に仕事を集中させ、他の専門家をアイドル状態のままにします。標準的な修正：負荷不均衡にペナルティを課す補助損失項を追加します。これは機能しますが、主要タスクのパフォーマンスはやや低下します。

DeepSeek-V3は補助損失を使用しないスキームを導入します。専門家単位のバイアス項がルーターロジットに追加され、簡単なルールで訓練中に調整されます：専門家`e`が過負荷の場合、`bias_e`を減少させます；アンダーロードされている場合、増加させます。追加の損失項なし。訓練はクリーンなままです。専門家負荷はバランスが取れたままです。

主な損失への影響：測定可能な影響はありません。MoEアーキテクチャへの影響：より清潔で、調整する補助損失ハイパーパラメータはありません。

### MTP：より密な訓練 + 無料ドラフト

フェーズ 10・18からご存じのとおり、DeepSeek-V3はD=1 MTPモジュールを追加して、2つの位置先のトークンを予測します。推論時には、訓練されたモジュールは80%以上の受け入れ率を持つ投機的デコーディングドラフトとして再利用されます。訓練では、各隠れ状態はD+1 = 2つのターゲットで教師あり学習され、より密な信号を提供します。

パラメータ：671B本体の上に14B。オーバーヘッド：2.1％。

### 訓練：DualPipe

フェーズ 10・19からご存じのとおり、DualPipeは、フォーワードとバックワードチャンクをクロスノードのall-to-all通信と重なる双方向パイプラインです。DeepSeek-V3の2,048-H800スケールでは、1F1Bがパイプラインバブルで失ったであろう約245kのGPU時間を回復します。

### コンフィグ、フィールドバイフィールド

DeepSeek-V3のコンフィグ（簡略版）は次のとおりです：

```
hidden_size: 7168
intermediate_size: 18432   (dense MLP hidden size, used on first few layers)
moe_intermediate_size: 2048 (expert MLP hidden size)
num_hidden_layers: 61
first_k_dense_layers: 3    (first 3 layers use dense MLP)
num_attention_heads: 128
num_key_value_heads: 128   (formally equal to num_heads under MLA, but
                           the real compression is in kv_lora_rank)
kv_lora_rank: 512          (MLA latent dimension)
num_experts: 256            (MoE expert count per block)
num_experts_per_tok: 8      (top-8 routing)
shared_experts: 1           (always-on shared expert per block)
max_position_embeddings: 163840
rope_theta: 10000.0
vocab_size: 129280
mtp_module: 1               (1 MTP module at depth 1)
```

これを解析します：

- `hidden_size=7168`：埋め込み次元。
- `num_hidden_layers=61`：総ブロック深さ。
- `first_k_dense_layers=3`：最初の3つのブロックはサイズ18432の密なMLPを使用します。残りの58はMoEを使用します。
- `num_attention_heads=128`：128のクエリ頭。
- `kv_lora_rank=512`：K と Vはこの潜在次元に圧縮され、頭ごとに解凍されます。
- `num_experts=256, num_experts_per_tok=8`：各MoEブロックには256人の専門家がいて、トップ8でルーティングします。
- `shared_experts=1`：256ルーティング専門家の上に、常時稼働する1人の専門家があらゆるトークンに寄与します。すべてのトークンが信頼できる何かを取得することを保証する「密な床」と考えてください。
- `moe_intermediate_size=2048`：各専門家のMLPハイドサイズ。256人がいるため、密なMLPより小さいです。

### パラメータ会計

完全な計算は`code/main.py`に存在します。見出し：

- 埋め込み：`vocab * hidden = 129280 * 7168 = ~0.93B`。
- 最初の3つの密なブロック：MLA（約144M/ブロック）+ 密なMLP（約260M/ブロック）+ ノルム付きアテンション。合計約1.2B。
- 58 MoEブロック：MLA（約144M）+ 256個の専門家（各30M）+ 1つの共有専門家（30M）+ ノルム付きアテンション。合計 ~7.95B/ブロック、すべての専門家を含む。58 MoEブロックの合計461B。
- MTPモジュール：14B。

総計：コアアーキテクチャの～476B + 14B MTP + 公開された671B数では、追加の構造パラメータ（バイアステンソル、専門家固有のコンポーネント、共有専門家スケーリングなど）を明確に説明します。計算機で再現する数値は公開されたものの3～5％以内です—デルタはDeepSeekのレポートがセクション2付録でドキュメント化した微粒度アカウンティングから発生します。

フォワード当たりのアクティブパラメータ：

- アテンション：層あたり144M * 61 = 8.8B（すべてのレイヤーが発火）。
- MLPアクティブ：最初の3層は密（3 * 260M = 780M）、58 MoE層は各アクティブ8ルーティング + 1共有 + ルーティングオーバーヘッド。レイヤーあたりのアクティブMLP：～260M。合計：3 * 260M + 58 * 260M = ～15.9B。
- 埋め込み + ノルム：1.2B。
- アクティブ合計：約26Bコア + 14B MTP（訓練されているが推論時に常に実行されるわけではない）≈ 37B。

### 671B / 37B比率

18倍のスパース性比（アクティブパラメータは総数の5.5％）。DeepSeek-V3は、オープン重みを提供している最もスパースなフロンティアMoEモデルです。Mixtral 8x7BのRatio 13/47（28％）ははるかに密集しています。Llama 4 Maverickのレート17B/400B（4.25％）と比較可能です。DeepSeekの賭け：フロンティアスケールでは、より多くの専門家と低い活性化比率はアクティブFLOPあたりの品質を向上させます。

### DeepSeek-V3がどこにあるか

| モデル | 総計 | アクティブ | 比率 | アテンション | 新しいアイデア |
|-------|------|-------|-------|-----------|-------------|
| Llama 3 70B | 70B | 70B | 100% | GQA 64/8 | — |
| Llama 4 Maverick | 400B | 17B | 4.25% | GQA | — |
| Mixtral 8x22B | 141B | 39B | 27% | GQA | — |
| DeepSeek V3 | 671B | 37B | 5.5% | MLA 512 | MLA + MTP + aux-free + DualPipe |
| Qwen 2.5 72B | 72B | 72B | 100% | GQA 64/8 | YaRN extension |

### フォローオン：R1、V4

DeepSeek-R1（2025）はV3バックボーン上の推論訓練実行です。R1は同じアーキテクチャを使用しています。変わったのは事前訓練後のレシピ（検証可能なタスク上の大規模RL）であり、事前訓練アーキテクチャではありません。

DeepSeek-V4（提供された場合）は、MLA + MoE + MTPを保持し、フェーズ 10・17のNSAの後継者であるDSA（DeepSeek Sparse Attention）を追加することが予想されます。系統は安定しています：アーキテクチャレベルの革新は蓄積されます；各バージョンは追加のパラメータを調整します。

## 使用方法

`code/main.py`はDeepSeek-V3の形状に特化したパラメータ計算機です。実行して、その出力をペーパーの数値と比較し、仮説的なバリアント（256個の専門家対512個、トップ8対トップ16、MLAランク512対1024）で使用します。

見てみることは：

- 総パラメータ数対公開671B。
- アクティブパラメータ数対公開37B。
- 128kコンテキストでのKVキャッシュ—MLAとGQAの比較。
- パラメータ予算が実際にどこに行くのかを見るために、層ごとの分析。

## 提供

このレッスンでは`outputs/skill-deepseek-v3-reader.md`を生成します。DeepSeekファミリーモデル（V3、R1、または任意の将来のバリアント）が指定されたら、コンポーネント単位のアーキテクチャ読み取りを生成します。これは、コンフィグの各フィールドに名前を付け、コンポーネント単位でパラメータ数を導出し、モデルが使用する4つのDeepSeek固有の革新を特定します。

## 演習

1. `code/main.py`を実行します。計算機の総パラメータ推定値を公開671Bと比較し、デルタがどこから来ているかを特定します。ペーパーのセクション2には完全な詳細があります。

2. コンフィグを修正して、MLAランクを512の代わりに256を使用します。128kコンテキストでの結果のKVキャッシュサイズを計算します。どのような割合の削減を購入でき、頭ごとの表現力にどのような費用がかかりますか？

3. DeepSeek-V3の（256人の専門家、トップ8）ルーティングを仮説的な（512人の専門家、トップ8）バリアントと比較します。総パラメータが増加します。アクティブパラメータは同じままです。追加の専門家容量は理論的には何を購入し、推論ではどのような費用がかかりますか？

4. DeepSeek-V3技術レポート（arXiv:2412.19437）のセクション2.1を読むMLAに関するもの。3文で、推論時の効率のために、K と Vの解凍マトリックスが後続の行列乗算に「吸収」されることができる理由を説明してください。

5. DeepSeek-V3は、ほとんどの操作にFP8訓練を使用します。671Bの重みを保存するためのFP8対BF16のメモリ節約を計算します。これは14.8T-トークン訓練予算とどのように交差しますか？

## キーターム

| 用語 | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| MLA | 「Multi-Head Latent Attention」 | K と Vを共有低ランク潜在（kv_lora_rank、通常512）に圧縮し、その場で頭ごとに解凍；KVキャッシュは潜在のみを保存 |
| kv_lora_rank | 「MLA圧縮次元」 | K と Vの共有潜在のサイズ；DeepSeek-V3は512を使用 |
| 最初のk密なレイヤー | 「初期レイヤーは密なままです」 | MoEモデルの最初のいくつかのレイヤーはMoEルーターをスキップし、安定性のために密なMLPを実行 |
| num_experts_per_tok | 「トップkルーティング」 | トークンあたりいくつのルーティング専門家が発火するか；DeepSeek-V3は8を使用 |
| 共有専門家 | 「常時稼働の専門家」 | ルーティングに関係なくあらゆるトークンを処理する専門家；DeepSeek-V3は1を使用 |
| 補助損失を使用しないルーティング | 「バイアス調整済み負荷バランス」 | 専門家単位のバイアス項は訓練中に調整され、損失項を追加せずに専門家負荷のバランスを保つ |
| MTPモジュール | 「追加予測頭」 | トランスフォーマーブロックはh^(1)とE(t+1)からt+2を予測；より密な訓練、無料の投機的デコーディングドラフト |
| DualPipe | 「双方向パイプライン」 | フォーワード/バックワード計算をクロスノードall-to-all通信と重なる訓練スケジュール |
| アクティブパラメータ比率 | 「スパース性」 | active_params / total_params；DeepSeek-V3は5.5％に達する |
| FP8訓練 | 「8ビット訓練」 | FP8での訓練ストレージと多くの計算操作；BF16と比較してメモリはほぼ半分で、わずかな品質低下 |
