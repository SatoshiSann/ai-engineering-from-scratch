# Jamba — ハイブリッド SSM-トランスフォーマー

> 状態空間モデル（SSM）とトランスフォーマーは異なるもの求めています。トランスフォーマーは二次コストを通じた注目によって品質を購入します。SSMは定数メモリを通じた線形時間推論を購入しますが、品質に遅れます。AI21のJamba（2024年3月）とJamba 1.5（2024年8月）は同じモデルに入れています：7つのMambaレイヤーごとに1つのトランスフォーマーレイヤー、すべての隔日ブロック上のMoE、そして単一の80GB GPUに適合する256kコンテキストウィンドウ。Mamba-3（ICLR 2026）は複素値状態空間とMIMO投影を備えたSSM側を圧縮しています。このレッスンでは両方のアーキテクチャを終わりまで読み、純粋なSSMおよび純粋なトランスフォーマーの長いコンテキスト試行が成功しなかった場合、ハイブリッドレシピが3年のスケーリングを生き残っている理由を説明します。

**タイプ:** Learn
**言語:** Python（stdlib、レイヤーミックス計算機）
**前提条件:** フェーズ 10・14（オープンモデルアーキテクチャ）、フェーズ 10・17（ネイティブスパースアテンション）
**所要時間:** 約60分

## 学習目標

- Jambaブロック内の3つのプリミティブ—トランスフォーマーレイヤー、Mambaレイヤー、MoE—と1:7:偶数インターリーブレシピを説明する。
- SSMの再発が高レベルでどのように見えるかを述べ、定数メモリ推論を可能にする理由を述べる。
- 256kコンテキストでのJambaモデルのKVキャッシュフットプリントを計算し、純粋なトランスフォーマーモデルが必要とするものと比較する。
- 3つのMamba-3イノベーション（指数曲線台形離散化、複素値状態更新、MIMO）を名前付けし、各々がターゲットとする問題を述べる。

## 問題

アテンションはシーケンス長の二次数です。状態空間モデルは線形です。その差は複合します：256kトークンでは、トランスフォーマーアテンションマップはヘッドあたり65Bエントリです；SSMの再発状態はシーケンス長に関わらず固定サイズです。

純粋なSSMモデル（Mamba、Mamba-2）は小規模でトランスフォーマーの困惑度に一致しますが、状態追跡タスク上に遅れ、コンテキスト内検索のいくつかのカテゴリで失敗します。直感：SSMは履歴を固定状態に圧縮し、履歴が長いと情報が漏洩します。アテンションはすべてを正確に覚えていますが、二次コストを支払います。

明白な修正：両方を使用してください。正確なリコールが必要なところにトランスフォーマーレイヤーを置きます。SSMレイヤーを他のところで使用します。比率を調整します。Jambaは、このハイブリッドレシピをスケールで提供する最初の本番グレードモデルです（52B総、12Bアクティブ、256kコンテキスト、単一80GB GPU）。Jamba 1.5はファミリーを398B総/ 94Bアクティブに拡張します。Mamba-3（ICLR 2026）は、ハイブリッドが再構築できる現在のベストな純粋SSMベースラインです。

このレッスンでは3つのペーパーをすべて読み、「正しい比率を選ぶ」ための精神モデルを生成します。

## コンセプト

### 1ページ以内のSSM

状態空間モデルは固定サイズの状態`h`を通じてシーケンス`x_1, ..., x_N`を処理します：

```
h_t = A h_{t-1} + B x_t
y_t = C h_t
```

各ステップで状態は線形ダイナミクス`A`を通じて進化し、入力`B x_t`を受け取り、出力`C h_t`を発行します。`A, B, C`を学習できます。重要な特性に注意してください：`y_t`の計算には`h_{t-1}`と`x_t`のみが必要で、以前の`x`は必要ありません。メモリは定数です。推論はトークンあたりO(1)です。

モデリング品質のトリックは`A`の構造です。S4（Gu 2021）は、訓練中に長い畳み込みとして効率的に評価できる高度に構造化された行列を使用しました。Mamba（Gu、Dao 2023）は固定`A, B, C`をデータ依存のもの（「選択的」部分）に置き換えました。Mamba-2（2024）はさらに構造を簡略化しました。Mamba-3（2026）は特定の場所で複雑さを再追加します。

重要な特性：デコーダLLMの場合、SSMレイヤーはアテンションレイヤーのドロップイン置き換え、増加するKVキャッシュの代わりに固定サイズのペアレイヤー状態で。

### Jambaブロック

Jambaブロックは2つの数字に応じてレイヤーをインターリーブします：

- `l`：アテンション対Mamba比。Jambaは`l = 8`を使用、つまり7つのMambaレイヤーごとに1つのトランスフォーマーレイヤー（7 Mamba + 1 Attention = グループあたり8レイヤー）。
- `e`：MoE頻度。Jambaは`e = 2`を使用、つまりすべての隔日レイヤーはMoEを適用します。

ブロック内のレイヤーシーケンス：

```
M  M  M  M  M  M  M  A    (7 Mamba + 1 Attention)
|  M  |  M  |  M  |  M    (where | marks MoE applied)
```

各Jambaブロックは8レイヤーです。4ブロック深く（合計32レイヤー）、28 Mambaおよび4アテンションレイヤーを取得します。16がMoEを使用します。

### なぜ1:7比率なのか

AI21はアブレーション実験を実行しました：彼らの長いコンテキスト評価のアテンション対Mamba比は、パラメータあたり最高の困惑度とコンテキスト内リコールを与えますか？

- 注意が多すぎる（1:1）：品質は上がりますが、メモリと速度は低下します。
- 注意が少なすぎる（1:15）：メモリは優れていますがコンテキスト内検索は失敗します。
- スウィートスポット：1:7または1:8。

直感：トランスフォーマーレイヤーは正確なリコールと状態追跡を処理します。Mambaレイヤーは処理の安い大部分を処理します。

### 位置エンコーディング

Mambaレイヤーはそれ自体が位置認識（再発を通じて）。元のMambaベースハイブリッドのアテンションレイヤーはRoPEを使用しませんでした—SSMレイヤーが位置情報を提供しました。Jamba 1.5は、経験的な長いコンテキスト評価に基づく事後精緻化として、アテンションレイヤーにRoPEを追加します。

### メモリ予算

Jamba-1形状の場合（32レイヤー：28 Mamba + 4アテンション、隠れ4096、32アテンションヘッド）：

- KVキャッシュ（アテンションレイヤーのみ）：256k BF16で`2 * 4 * 32 * 128 * 256k * 2 = 8.4 GB`。4つのアテンションレイヤーのみが寄与します。
- SSMの状態：トークンプレフィックスあたり`28 * hidden * state_size`、しかしこれはシーケンス長とスケーリングしない、レイヤーごとの固定サイズです。典型的なMamba状態は機能あたり16、隠れ4096：`28 * 4096 * 16 * 2 = 3.7 MB`合計。

256k BF16での32レイヤーの純粋なトランスフォーマー、同じ隠れ、完全MHAが32ヘッド：`2 * 32 * 32 * 128 * 256k * 2 = 128 GB`と比較してください。KVキャッシュで8倍削減。最も2024年モデルが使用するGQA(8)ベースライン（`2 * 32 * 8 * 128 * 256k * 2 = 32 GB`）に対しても、16GBでのJambaの1:7ハイブリッドはまだ2倍小さいです。

これはAI21が「単一80GB GPU上の256kコンテキスト」で意味することです。完全MHAの純粋なトランスフォーマーのKVキャッシュはフィットしません；GQAベースラインでさえ重みとアクティベーションのための余地を残しません；Jambaのはします。

### Mamba-3：2026年の純粋SSMベースライン

Mamba-3（ICLR 2026、arXiv:2603.15569）は純粋なSSM側に3つのイノベーションを導入します：

1. **指数曲線台形離散化。** Mamba-2のオイラー法離散化を、より表現力豊かな再発に置き換えます。畳み込みのような操作は、外側の畳み込みとしてではなく、コア再発内の状態入力に適用されます。

2. **複素値状態更新。** 以前のMambaは状態行列を複素（S4）から実数対角（Mamba）を段階的にスケールされたアイデンティティ（Mamba-2）に削減しました。Mamba-3は複素値を再追加します—状態上のデータ依存回転埋め込みに相当します。これは以前の実数値簡略化がコストする状態追跡機能を復元します。

3. **多入力多出力（MIMO）投影。** フィーチャーごとのスカラー投影の代わりに、行列値投影を使用してください。デコードレイテンシを増加させずに、モデリングパワーと推論時ハードウェア利用を向上させます。

1.5Bパラメータでは、Mamba-3はゲート付きDeltaNetに対する平均ダウンストリーム精度を0.6ポイント改善します；MIMOバリアントは追加の1.2を加えて総1.8ポイント利得。同じ状態サイズで、Mamba-3は半分の状態を持つMamba-2と一致します。

Mamba-3はまだスケールでの本番ハイブリッドで提供されていません—しかし、次のJambaクラスモデルのSSM側が再構築する明白な候補です。

### ハイブリッドに手を伸ばすとき

ハイブリッドは以下の場合に勝利します：

- コンテキストは、純粋なトランスフォーマーKVキャッシュが痛みになるのに十分なほど長い（64k+）。
- タスクが短距離構造（SSMに良い）と長距離リコール（トランスフォーマーが必要）を混在させます。
- トランスフォーマーKVキャッシュ単独がフィットしない単一GPUメモリ予算上での展開が必要です。

ハイブリッドは以下の場合に失敗します：

- コンテキストは短い（16k未満）。SSMオーバーヘッドは無駄です；純粋なトランスフォーマーは細かいです。
- タスクは至るところへのあらゆる場所へのアテンション（深い推論、マルチドキュメント相互参照）が必要です。ハイブリッド内のアテンションレイヤーのスパース性は傷つけます。
- 兆パラメータのフロンティアモデルへのスケーリング。純粋トランスフォーマー + MLA + MoE（DeepSeek-V3スタイル）は現在機能競争に勝っています。

### 競争風景

| モデル | ファミリー | スケール | ユニークなクレーム |
|-------|--------|------|-------------|
| Mamba-2 | pure SSM | 3B | 線形時間、定数メモリ |
| Jamba | ハイブリッド | 52B/12B | 80GB上の256k |
| Jamba 1.5 Large | ハイブリッド | 398B/94B | エンタープライズグレード長いコンテキスト |
| Mamba-3 | pure SSM | 1.5B（ペーパー） | 状態追跡復元 |
| DeepSeek-V3 | pure Transformer + MoE | 671B/37B | フロンティア機能 |

2026年の風景：純粋トランスフォーマーMoEはフロンティアを支配しますが、ハイブリッドは256k-plus コンテキストニッシュを所有しています。Mamba-3の状態追跡の勝利は、次の世代でハイブリッド比をより低く（SSMが多く、注目が少なく）プッシュする可能性があります。

## 使用方法

`code/main.py`はハイブリッドアーキテクチャ用のメモリ計算機です。SSM-トランスフォーマー比と隠れサイズ/レイヤーカウントコンフィグが与えられた場合、計算します：

- ターゲットコンテキストでのKVキャッシュ。
- SSM状態メモリ。
- コンテキストNでの合計メモリ、モデル形状の範囲内。

計算機は以下をサポートしています：

- 純粋トランスフォーマーベースライン（KVキャッシュはNで成長）。
- Jambaスタイル1:7ハイブリッド。
- 純粋なSSM（KVキャッシュはない）。

数値は、公開形状のJamba-1およびJamba-1.5ペーパーから直接で、仮説バリアントに対して外挿されます。

実際の展開のための統合考慮事項：

- ほとんどの本番推論サーバー（vLLM、SGLang）はJambaとMambaをサポートしています。特定のバージョンを確認します。
- 256kコンテキストでは、Jambaのメモリ利点は並行リクエストスループットに表示されます。同じVRAM上では、トランスフォーマーシーケンスよりも多くのJambaシーケンスをフィットさせます。
- Mamba-3スタンドアロンモデルはまだ本番で提供されていません—1.5Bで研究プレビュー。

## 提供

このレッスンでは`outputs/skill-hybrid-picker.md`を生成します。ワークロード仕様（コンテキスト長プロファイル、タスクミックス、メモリ予算）が与えられた場合、純粋トランスフォーマー、Jambaスタイルハイブリッド、および純粋SSM間を推奨し、メモリと品質トレードオフに対する明確な推論を持つ。

## 演習

1. `code/main.py`を実行して、32レイヤー純粋トランスフォーマー（隠れ4096、32ヘッド）および同じ形状のJamba-1ハイブリッドでの256kコンテキストでのKVキャッシュを計算します。AI21ペーパーが要求する～8倍のメモリ削減を検証します。

2. 計算機を修正してモデル1:3ハイブリッド（4 Mamba：1アテンション）および1:15ハイブリッド（14 Mamba：1アテンション）。プロットKVキャッシュ対比率。KVキャッシュがSSM状態メモリと等しい比率で何があるか？

3. Jambaペーパー（arXiv:2403.19887）のセクション3を読んでください。AI21がMamba-2がより高速であるにもかかわらずMamba-2の代わりにMamba-1を使用する理由を説明してください。ヒント：ハイブリッドアブレーションセクションはこれを文書化します。

4. Jamba 1.5 Large（398B総、94Bアクティブ）でのMoEエブリアイザーレイヤーのパラメータオーバーヘッドを計算します。アクティブ比をDeepSeek-V3（37B/671B）と比較し、Jambaのアーキテクチャがアクティブ比をより高く押す理由を説明してください。

5. Mamba-3ペーパー（arXiv:2603.15569）のセクション3を読んでください。複素値状態更新がデータ依存回転埋め込みに相当する理由を3文で説明してください。フェーズ 7・レッスン04のRoPE導出に答えを結びます。

## キーターム

| 用語 | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| 状態空間モデル（SSM） | 「固定状態を持つ再発」 | 学習された再発`h_t = A h_{t-1} + B x_t`を持つレイヤー；トークンあたりの定数メモリ |
| 選択的SSM | 「Mambaのトリック」 | データ依存A、B、C パラメータは、線形時間でゲーティングのような選択性をモデルに与える |
| アテンション対Mamba比 | 「どのくらいのアテンションレイヤーか」 | Jambaで、`l = 8`は7つのMambaレイヤーあたり1つのアテンションレイヤーを意味 |
| Jambaブロック | 「8レイヤーグループ」 | 1つのアテンション + 7つのMamba + 交互位置でのMoE |
| SSM状態 | 「隠れバッファ」 | MambaレイヤーのためのKVキャッシュを置き換えるレイヤーごとの固定サイズ状態 |
| 256kコンテキスト | 「Jambaのフラッグシップ数」 | Jamba-1が単一80GB GPUに適合するシーケンス長；純粋なトランスフォーマーはそのサイズではできない |
| Mamba-3 | 「2026年純粋SSM」 | 複素状態 + MIMO付き現在のベスト純粋SSMアーキテクチャ；ハイブリッドが再構築するベースライン |
| MIMO | 「多入力多出力」 | スカラーフィーチャーごとの代わりに行列値投影を使用するMamba-3イノベーション |
| 指数曲線台形離散化 | 「Mamba-3の再発」 | Mamba-2のオイラー法離散化を包含するより表現力豊かな再発 |
| ハイブリッドアーキテクチャ | 「アテンションとSSMを混ぜる」 | トランスフォーマーとSSMレイヤーをインターリーブするあらゆるモデル；Jambaは本番原型 |
