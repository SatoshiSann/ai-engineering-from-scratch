# チェックポイントとロールバック

> グラフ状態遷移はすべて永続化されます。ワーカーがクラッシュすると、そのリースが満了し、別のワーカーが最新のチェックポイントから復帰します。Cloudflare Durable Objectsは数時間から数週間の状態を保持します。先行提案後コミット(レッスン15)は、アクション毎にロールバック計画を定義します。事後検証はループを閉じます。EU AI法第14条により、高リスクシステムに対しては実効的な人間の監視が義務付けられています。実務的には、チェックポイントはクエリ可能である必要があり、ロールバックはリハーサルされ、監査証跡はデプロイを生き残る必要があります。鋭い失敗モード:冪等性キーと前提条件チェックがないと、一時的な障害後の再試行により、既に承認されたアクションが二重実行される可能性があります。事後検証がそれを検出します。

**タイプ:** 学習
**言語:** Python(標準ライブラリ、チェックポイントとロールバック状態機械)
**前提条件:** Phase 15 · 12(Durable実行)、Phase 15 · 15(先行提案後コミット)
**所要時間:** 約60分

## 問題

Durable実行(レッスン12)により、クラッシュしたエージェントは再開可能になります。先行提案後コミット(レッスン15)により、承認されたアクションは監査可能になります。このレッスンはそれらを結合します:承認されたアクションが部分的に実行され、クラッシュして再開した場合は何が起きるのか?ロールバックはいつ実行され、どのような状態に対して実行されるのか?

実際のシステムはこれを異なる方法でワイヤリングします:

- **LangGraph** グラフ状態遷移をPostgreSQLの各チェックポイントに永続化します。ワーカークラッシュ時にリースが解放され、別のワーカーが最新のチェックポイントから再開します。ワークフローは`interrupt()`で一時停止します。これ自体も永続化されます。
- **Cloudflare Durable Objects** キー毎の状態を数時間から数週間保持します。承認されたアクションのコンピュテーションをストレージと共存させます。
- **Microsoft Agent Framework** ワークフロー API内で`Checkpoint`プリミティブを公開します。リプレイと冪等性により再試行をカバーします。

すべての場合において、実際に機能する組み合わせは:冪等性キー(二重実行を防ぐ) + 前提条件チェック(状態が承認されたものと矛盾していないか) + 事後検証(副作用が実際に発生したか) + 検証失敗時のロールバックです。

## コンセプト

### すべての遷移が永続化される

グラフ状態遷移とは、ワークフローを1つの名前付き状態から別の状態に移す任意のステップです。素朴な実装では特定のコミットポイントでのみ永続化しますが、本番実装ではすべての遷移を永続化します。コスト(わずかな追加の書き込み)は信頼性ゲイン(リプレイはどこにでも着地でき、リース復帰は精密)に対して小さいです。

### リース復帰

ワーカーがクラッシュするとき、ワークフローは失われません。リース(このワーカーがこの実行を実行しているという短期の主張)が単に満了するだけです。別のワーカーが最新のチェックポイントを取得して再開します。リースメカニズムにより、本番システムはローリングデプロイ中に進行中の作業を失わずに生き残ることができます。

### 冪等性と前提条件

冪等性だけでは不十分です。例を考えてください:ワークフローが「残高 > $1000のとき、Aから Bへ$100を転送」するよう承認されています。ワークフローがコミットされ、実行中にクラッシュして再開します。冪等性キーのみがチェックされ、実行が再開される場合、転送は1回実行されます(正しい)。しかし、クラッシュと再開の間にAの残高が別のワークフローを経由して$500に低下したと考えてください。冪等性チェックはまだパスします。前提条件はしません。前提条件チェックなしで、私たちはオーバードラフトを配送します。

重要なすべてのアクションには両方が必要です:

- **冪等性キー**: 二重実行を防ぎます。
- **前提条件チェック**: 状態が承認されたときと矛盾していることを確認します。

### 事後検証

「ツールが200を返した」は検証ではありません。実際の検証は対象の状態を再度読み込み、副作用が実際に発生したことを確認します。パターン:

- データベース更新:`UPDATE ... RETURNING *`その後返されたロウが意図された状態と一致することを主張します。
- メール送信:送信後に送信フォルダでメッセージ IDをチェックします。
- ファイル書き込み:ファイルを読み戻してハッシュします。
- API呼び出し:対象リソースでの後続`GET`。

検証が失敗すると、ワークフローは既知の不具合な状態にあります。ロールバックが発動します。

### ロールバック計画

先行提案後コミット(レッスン15)の重要なすべてのアクションは、ロールバック計画を伴います。タイプ:

- **インバンドロールバック**: 副作用を直接逆転させます(`INSERT`後に`DELETE`、送信後に`Send-correction-email`)。
- **補償トランザクション**: 元のものを無効にする新しいアクション(標準 SAGA パターン)。
- **アウトオブバンドロールバック**: 人間に警告し、ワークフローを一時停止し、不具合な状態を調査のために残す。

ノーオップロールバック(「これは元に戻せません」)は提案で名前付けされる必要があります。ロールバックがないアクションはコミット時にはより強い HITL が必要です(レッスン15チャレンジアンドレスポンス)。

### EU AI法第14条の運用的読み取り

第14条は「高リスクシステムに対する実効的な人間の監視」を要求します。運用上の用語では、実装者はそれを次のように読みます:

- チェックポイントは監査人によってクエリ可能です。
- ロールバックはリハーサルされます(少なくとも1回はエンドツーエンドテストされます)。
- 監査証跡はデプロイを生き残ります(チェックポイントバックエンドは短命ではありません)。
- 検証の失敗は警告されます。サイレントにログされません。

コミット中にクラッシュし、再開して副作用をしていなく、検証 + ロールバックパスウェイなしで完了するワークフローは、第14条テストを生き残りません。

### 鋭い失敗モード:二重実行

このスペースで最も一般的な本番インシデント:

1. アクション承認、冪等性キー k。
2. コミット開始、実行、200を返します。
3. ワークフローは「コミット済み」ステータスの永続化前にクラッシュします。
4. ワークフロー再開。「承認されたがコミットされていない」と見ます。再実行。
5. 副作用が2回発火します。

緩和: 実行前に「進行中」の意図を永続化し、冪等性キーで実行してから、事後検証の成功後にのみ「コミット済み」をマークします。アクションが発火し、ステータス書き込みが失敗した場合、検証を知り、必要に応じて再発火します。ステータス書き込みが成功してアクションが失敗した場合、検証して復帰パス経由で正確に1回発火します。

## それを使う

`code/main.py`は冪等性、前提条件、検証、ロールバックを備えたチェックポイント付きワークフローを実装します。ドライバーは4つのシナリオをシミュレートします:クリーン実行、クラッシュ後の再試行(冪等性が検出)、前提条件失敗(ワークフロー発火なしで中止)、検証失敗(ロールバック発火)。

## それを配送する

`outputs/skill-rollback-rehearsal.md`は提案されたワークフローのロールバックリハーサルテストを設計し、監査証跡の永続性のためにチェックポイントバックエンドを監査します。

## 演習

1. `code/main.py`を実行します。4つのシナリオを確認します。クラッシュ中コミットの場合、アクションが再試行全体で正確に1回発火することを確認します。

2. 「最初にマークしてから実行する」パターンを変更して、ステータス書き込みが実行後に発火します。クラッシュシナリオを再実行します。重複アクションが何回発火するかを測定します。

3. 特定の本番アクション(例: 「Slackチャネルに投稿」)のロールバック計画を設計します。インバンド、補償、またはアウトオブバンドとして分類します。選択を正当化します。

4. あなたが知っているワークフローを1つ取ります。すべての状態遷移を識別します。各を耐久性要件(永続化/永続化しない)でマークします。現在永続化していないものをカウントします。

5. リハーサルロールバックテスト:実際のワークフローを実行し、クラッシュさせ、ロールバックパスが発火することを確認するエンドツーエンドテストを設計します。テストは何を主張しますか?

## 重要な用語

| 用語 | 人が言うこと | 実際の意味 |
|---|---|---|
| チェックポイント | 「セーブポイント」 | すべてのグラフ状態遷移が耐久性ストアに永続化されます |
| リース | 「ワーカー主張」 | ワーカーが実行を実行しているという短期の主張。クラッシュで満了します |
| 前提条件 | 「状態ゲート」 | 状態がまだ承認されたアクションと矛盾していることの主張 |
| 事後検証 | 「再読チェック」 | 副作用が対象システムで実際に発生したことを確認します |
| インバンドロールバック | 「直接元に戻す」 | 逆演算で副作用を逆転させます |
| 補償トランザクション | 「SAGA元に戻す」 | 元のものを無効にする新しいアクション |
| 最初にマークしてから実行 | 「ステータス書き込み順」 | コミットから戻る前にコミット済みステータスを永続化します |
| 第14条 | 「EU AI法人間監視」 | 運用: クエリ可能なチェックポイント、リハーサルロールバック、監査可能な証跡 |
