# 長期実行バックグラウンドエージェント：耐久的な実行

> 本番環境の長地平エージェントは`while True`で実行されません。すべてのLLMコールは、チェックポイント、再試行、リプレイを備えたアクティビティになります。TemporalのOpenAI Agents SDK統合は2026年3月にGA化しました。Claude Code Routines（Anthropic）は、永続的なローカルプロセスなしでスケジュールされたClaude Codeの呼び出しを実行します。セッションは人間入力で一時停止し、デプロイの存続期間を長くし、`thread_id`でキー設定された最新のチェックポイントから再開されます。新しい人間工学の背後には、古いパターン（ワークフロー調整）が1つの新しい入力で座っています。確定的にリプレイされなければならない非決定論的なアクティビティとしてのLLMコール。

**タイプ:** 学習
**言語:** Python（標準ライブラリ、最小限の耐久実行状態マシン）
**前提条件:** Phase 15 · 10 (権限モード)、Phase 15 · 01 (長地平エージェント)
**所要時間:** 約60分

## 問題

4時間実行されるエージェントを考えてください。3つのツールを呼び出し、ユーザーに2回プロンプトを出し、40回のLLMコールを行います。途中で、実行しているホストが再起動します。どうなりますか？

- ナイーブな`while True`ループで：すべてが失われます。実行はゼロから再開されます。3つのツール呼び出し（実副作用を持つ）は再度実行されます。ユーザーは既に承認したものについて再度プロンプトされます。40のLLMコールは再度請求されます。
- 耐久的な実行を使用：実行は最新のチェックポイントから再開されます。既に完了したアクティビティは再度実行されません。その結果は耐久ログからリプレイされます。ユーザーは既に承認したものを再度承認しません。既に作成されたLLMコールは再度請求されません。

これは、ワークフローエンジンが10年間出荷したのと同じパターンです（Temporal、Cadence、UberのCherami）。新しいのは、LLMコールが非決定論的で、高価で、副作用を持つアクティビティの一種になり、このパターンにきれいに適合することです。

レッスンの反復的なテーマ：長地平の信頼性は減衰します（METR「35分間の劣化」を観察します。成功率はほぼ二次的に水平線で低下します）。耐久的な実行により、信頼性プロファイルがサポートするより長い実行が可能になります。これは、設計が正しい場合は安全に失敗し、設計が間違っている場合は安全でない新しい方法です。

## コンセプト

### アクティビティ、ワークフロー、リプレイ

- **ワークフロー**：決定論的なオーケストレーションコード。アクティビティの順序、ブランチ、待機を定義します。イベントログからリプレイされる場合、驚く分岐なしに再生できるように決定論的である必要があります。
- **アクティビティ**：非決定論的で、潜在的に失敗する作業単位。LLMコール、ツール呼び出し、ファイル書き込み、HTTPリクエスト。各アクティビティはその入力を使ってログされ、（完了したら）その出力。
- **イベントログ**：耐久バッキングストア。すべてのアクティビティ開始、完了、失敗、再試行、およびすべてのワークフロー決定が記録されます。
- **リプレイ**：復旧時に、ワークフローコードは開始から再実行されます。既に完了したすべてのアクティビティは、再実行することなくログされた結果を返します。完了していなかったアクティビティのみが実際に実行されます。

これはReactが仮想DOMに対して再レンダリングするのと同じ形、またはGitがコミットからワーキングツリーを再構築するのと同じです。オーケストレータの決定論は、耐久性を安価にするものです。

### LLMコールがパターンに適合する理由

LLMコールは：
- 非決定論的（温度> 0。温度0でもモデル版間でドリフト）。
- 高価（お金とレイテンシ）。
- 潜在的に失敗（レート制限、タイムアウト）。
- 副作用的（ツールを呼び出す場合）。

これはまさにアクティビティプロファイルです。すべてのLLMコールをアクティビティとしてラップすると、指数バックオフで再試行、再起動間でのチェックポイント、デバッグ用の再生可能トレースが得られます。

### `thread_id`でキー設定されたチェックポイント

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects、Claude Code Routinesはすべて同じAPIシェイプに収束しました。`thread_id`（または同等）はセッションを識別します。各状態遷移はバックエンド（PostgreSQL default、dev用SQLite、キャッシュ用Redis）に永続化されます。再開は最新のチェックポイントを読みます。

バックエンド選択は重要：

- **PostgreSQL**：耐久性、クエリ可能、デプロイの存続期間を長くします。LangGraphのデフォルト。
- **SQLite**：ローカル開発のみ。ホスト間でデータを失う。
- **Redis**：高速ですが、AOF/スナップショットが構成されていない限り短命。
- **Cloudflare Durable Objects**：透過的に分散。一意のキーでスコープ。数時間から数週間存続。

### 最初のクラス状態としての人間入力

Propose-then-commit（レッスン15）には、耐久性のある「人間を待機」状態が必要です。ワークフローが一時停止し、外部キューは保留中のリクエストを保持し、承認は正確にその点から再開されます。耐久性なしで、これはベストエフォード。耐久性を使って、一晩の承認は到着し、ワークフローは朝に取り上げます。

### 35分間の劣化

METRは、測定されたすべてのエージェントクラスが連続操作の約35分を超えて信頼性の減衰を示すことを観察しました。タスク時間を2倍にするとほぼ失敗率が4倍になります。耐久的な実行はこれを修正しません。信頼性プロファイルがサポートするより長く実行できます。安全なパターンは、再入時に新しいHITLを必要とするチェックポイントでの耐久性と、総コンピュートを上限とする予算キルスイッチ（レッスン13）の壁時計時間を組み合わせることです。

### 耐久的な実行が間違った答える場合

- 人間入力なしで数分以下の実行。オーバーヘッド>利点。
- 厳密に読み取り専用の情報取得。
- 正確性が1つのコンテキストウィンドウ内のエンドツーエンドを必要とするタスク（いくつかの推論タスク。いくつかのワンショット生成）。

## 使用方法

`code/main.py`は、標準ライブラリPythonで最小限の耐久実行エンジンを実装します。サポート：

- 入力と出力をJSONイベントログにログする`@activity`デコレータ。
- アクティビティをシーケンスするワークフロー関数。
- 完了したアクティビティを再度実行せずにリプレイする`run_or_replay(workflow, event_log)`関数。

ドライバーは3エクティビティワークフローをシミュレートし、途中でクラッシュし、（a）ナイーブな再試行がすべてを再実行するのに対して（b）リプレイが欠落しているアクティビティのみを実行することを示しています。

## 運用

`outputs/skill-durable-execution-review.md`は、提案されたlong-runningエージェントのデプロイを正しい耐久実行シェイプのためにレビューします。アクティビティ、決定論、チェックポイントバックエンド、人間入力状態、およびHITL-on-resume方針。

## 演習

1. `code/main.py`を実行します。ナイーブな再試行とリプレイの間のアクティビティ実行数の違いを観察します。クラッシュポイントを変更して、リプレイ数がそれに応じて変化することを示します。

2. おもちゃエンジンを明示的に`thread_id`を使用するように変換します。エンジンを共有する2つの同時セッションをシミュレートし、それらのイベントログが衝突しないことを確認します。

3. おもちゃエンジンのいずれかのアクティビティを使用します。非決定論を導入（ワークフロー決定内の壁時計タイムスタンプ）。リプレイで分岐を実証します。実際のエンジンがこれを処理する方法を説明します（副作用登録、`Workflow.now()` API）。

4. LangChain「本番環境の深いエージェントの背後にあるランタイム」投稿を読みます。ランタイムが永続化するすべての状態をリストアップし、各状態がカバーする失敗モードに名前を付けます。

5. 6時間自律コーディングタスクのチェックポイントポリシーを設計します。どこでチェックポイント？リプレイ・オン・クラッシュはどのように見えますか？新しいHITLが必要なのはどれですか？

## キーターム

| タイプ | 人々が言うこと | それが実際に意味するもの |
|---|---|---|
| ワークフロー | 「エージェントのスクリプト」 | 決定論的なオーケストレーションコード。イベントログからリプレイ可能 |
| アクティビティ | 「ステップ」 | 非決定論的なユニット（LLMコール、ツール呼び出し）。前後にログされる |
| イベントログ | 「バッキングストア」 | すべての状態遷移の耐久記録 |
| リプレイ | 「再開」 | ワークフローを再実行。完了したアクティビティは再度実行することなくログされた結果を返します |
| チェックポイント | 「セーブポイント」 | thread_idでキーが設定された永続化状態。再開時に最後の勝利 |
| thread_id | 「セッションキー」 | 耐久状態をスコープするアイデンティファイア |
| 35分間の劣化 | 「信頼性の減衰」 | METR：成功率は水平線で約二次的に低下 |
| 非決定論 | 「リプレイでドリフト」 | 壁時計、ランダム、LLM出力。副作用として登録される必要があります |
