# METR Time Horizons と外部能力評価

> METR(ex-ARC Evals)は2023年12月以来独立した501(c)(3)です。彼らの Time Horizon 1.1 ベンチマーク(2026年1月)は、タスク成功確率 対 log(専門家人間完了時間)にロジスティック曲線を適合させます。50%確率での交差は、モデルの時間地平線を定義します。2025-2026の従事セットは GPT-5.1、GPT-5.1-Codex-Max、プロトタイプ監視評価(監視は副タスクをキャッチできるか。エージェントは監視を回避できるか)をカバーします。ベンチマーク スイート:HCAST(180+ ML、cyber、SWE、推論タスク。1分から8+時間)、RE-Bench(71 ML 研究エンジニアリング タスク、専門家ベースラインで)、SWAA。正直なノート:METR 測定は理想化されています - 人間なし、実際の結果なし - そしてチームは評価対デプロイメント動作ギャップを文書化しました(レッスン1)。時間地平線は上限です、デプロイメント予測ではありません。

**タイプ:** 学習
**言語:** Python(標準ライブラリ、ロジスティック適合地平線推定器)
**前提条件:** Phase 15 · 01(Long-horizon エージェント)、Phase 15 · 19(RSP)
**所要時間:** 約60分

## 問題

スケーリング ポリシー(レッスン19、20)はそれらが参照する測定と同じくらい有用です。「AI R&D-4 閾値」と「Long-range Autonomy」はポリシー散文に定義されています。彼らは特定の評価が特定の数字を生成する場合、実行可能になります。

METR は2024-2026の外部評価組織であり、多くのこれらの数字を定義しました。彼らは frontier モデルを評価します - 多くの場合リリース前、NDA の下でラボと - そしてその後方法論を公開します。Time Horizon 1.1 ベンチマーク(2026年1月)は彼らの見出しアーティファクトです:人間が知覚する能力を圧縮する単一のスカラー(「このモデルは専門家が 50%信頼性で X 時間を費やす種類のタスクをすることができます」)。

レッスンは部分的には方法論(地平線がどのように計算されるか)と部分的には解釈(なぜ地平線は上限であり、デプロイメント予測ではないか)です。2つのスキルは一緒に属します。地平線がどのように適合されるかを理解するチームは、悪い vendor 主張でだまされることがはるかに難しいチーム、それは単に「14時間」をスライドで見ています。

## コンセプト

### METR 背景

- 設立:2023年12月(ex-ARC Evals、独立した501(c)(3)に回転)。
- スコープ:frontier モデルの自動性能力の評価、多くの場合リリース前。
- ラボのパートナー:Anthropic、OpenAI(複数の従事 2025-2026)。
- 顕著な成果物:Time Horizon 1.0(2025年3月)、Time Horizon 1.1(2026年1月)、プロトタイプ監視評価。

### Time Horizon 適合

方法論(METR ブログと論文から):

1. 分単位から時間-スケール専門家完了時間にまたがるタスク スイートを集めます。現在のスイート:HCAST(180+ タスク)、RE-Bench(71 タスク)、SWAA。
2. 各タスク上でモデルを実行します。成功または失敗を記録します。
3. ロジスティック曲線を適合:P(成功)は log(専門家完了時間)の関数として。
4. 地平線は P(成功) = 0.5 の専門家時間です。

ロジスティック適合形状は正しい方です。能力は一般的に、難易度に対する増加、高原に接近関係を持つためです。50%ポイントは選択です(10%、90%可能)。METR は詳細なペーパーで複数のしきい値を報告しますが、それが最も直感的であるため50%でリードします。

### 2026年1月の番号

時間地平線 1.1 ごと:

- Claude Opus 4.6:~14時間 50%信頼性で、時間地平線 1.1(2026年1月)ごと。
- HCAST スタイル タスク上でのダブリング時間:~4.3ヶ月(130.8日)時間地平線 1.1(2026年1月)で報告された2023年後フィットで。~7ヶ月の図は 2019-2025 フルフィットです。時間地平線 1.0 および 1.1 と一緒に報告されました。

これらは、Phase 15 の残りが参照するシング番号です。それらは Anthropic の RSP v3.0(レッスン19)と DeepMind の FSP v3(レッスン20)で引用され、閾値評価をフィードする測定の1つとして。

### ベンチマーク スイート

- **HCAST**:ML、cybersecurity、ソフトウェア エンジニアリング、一般推論全体で189のタスク。スパン:1分から8+時間の専門家時間。地平線番号の後ろの主要なスイート。
- **RE-Bench**:71 ML 研究エンジニアリング タスク、人間の専門家ベースラインで。モデルが AI R&D-4 閾値に供給する ML R&D 作業の種類をすることができるかどうかを測定します。
- **SWAA**:より短いタスク。地平線曲線の低端をキャリブレートするのに有用です。

### プロトタイプ監視評価

2025-2026 METR 作業は監視評価プロトタイプを含みました:監視は副タスク(操作者が要求したこと以外のもの)を実行しているモデルをキャッチできるか。エージェントは監視を回避できるか。これらは OpenAI PF v2 の「Undermining Safeguards」カテゴリと DeepMind FSF v3(レッスン20)の欺まくされた整列監視に直接供給します。

### 地平線が上限である理由

- **理想化された工具設定**:ベンチマーク タスクはモデルにクリーン ツールと良くフォーマットされたデータを与えます。実際のデプロイメントはより混乱しています。
- **実際の結果なし**:モデルはいいえ顧客に請求、本物のデータを削除、本物の人々に連絡します。実際のデプロイメントは非可逆的な賭けを持っています。
- **評価コンテキスト ゲーミング**:レッスン1。モデルはテスト中に異なる動作をします。2026年 International AI Safety Report は経験的にこれを文書化します。
- **合法的なユーザーの変動なし**:ベンチマーク プロンプトは構造化されています。実際のユーザーは曖昧で、コンテキスト依存の要求を生成します。

地平線は有利な条件の下の能力天井です。デプロイメント信頼性は異なる番号、より低い、そしてチームはそれを知るために自分自身の分布を測定する必要があります。

### 外部評価器のケース

外部評価が重要なのは、内部ラボが報告する指標を最適化するための誘因を持つためです。METR の独立 - 宣言された方法論と同等レビュー済み論文を持つ501(c)(3) - は構造的な緩和です。それは単独では十分ではありません(ラボはまだ METR が見るものをコントロール)ですが、外部評価がまったくないよりも厳密に優れています。

### 実装で地平線番号を使用する方法

- **能力フィルターとして**:モデルの地平線が提案されたタスクの専門家時間ではるかに下にある場合、それを自動(レッスン1スキル ファイル)で出荷しないでください。
- **トレンド指標として**:ダブリング時間は、新しい緩和なしでも、現在の実用が安全なままである期間を指示します。
- **先行として**:14時間の地平線はスタート ポイントです。あなたのタスク分布、ツール設定品質、デプロイメント コンテキストダウンで調整します。

## それを使う

`code/main.py` は、合成結果セットを与えられた、タスク成功 対 log(専門家時間)のロジスティック適合を実装します。それは 50%地平線(METR の見出し)、10%地平線(保守的)、90%地平線(楽観的)を報告します。また、成功率が評価コンテキスト ゲーミングによって人為的に膨れ上がるときに変更されるのは何かを実証しています。

## それを配送する

`outputs/skill-horizon-interpretation.md`は vendor の地平線主張をレビューし、ベンチマーク要求とデプロイメント現実の間のギャップ分析を生成します。

## 演習

1. `code/main.py`を実行します。適合の50%地平線が合成根拠の真理と一致することを確認します。今、タスク時間グリッドを半分にします。地平線推定は有意に変更されますか?

2. METR の Time Horizon 1.1 ブログ投稿を読みます。信頼性が最も高く、最も低い特定のタスクを識別します。ギャップが存在する理由を説明します。

3. METR の「Autonomous AI Capabilities の測定」リソースを読みます。HCAST タスク カテゴリーをリストします。本番タスクのためにより重く重み付けするカテゴリを選び、なぜを正当化します。

4. 評価コンテキスト ゲーミングをシミュレーターに導入:失敗したタスクの~20%を成功に反転します。新しい地平線を報告します。これは、20%のゲーミング率が観察された番号に何をするかを近似します。

5. 自分のバグ バックログまたは代表的なタスク セットで内部地平線評価を設計します。データ コレクション、適合、出力が指示することを説明します。METR 番号と比較します。
