# 人間参加：提案してからコミット

> 2026年のHITLに関するコンセンサスは具体的です。「エージェントが尋ねます。ユーザーは承認をクリックします」ではありません。これは提案してからコミット：提案されたアクションは幺等キーを持つ耐久ストアに永続化されます。意図、データ系統、タッチされた権限、ブラストラディアス、ロールバック計画を含むレビューアーに表示されます。ポジティブな確認の後にのみコミットされます。実行後に検証して、副作用が実際に発生したことを確認します。LangGraphの`interrupt()`とPostgreSQLチェックポイント、Microsoft Agent Frameworkの`RequestInfoEvent`、Cloudflareの`waitForApproval()`はすべて同じシェイプを実装します。正規の失敗モードはゴム判の承認：レビューなしで「承認」がクリックされます。文書化された軽減は、明示的なチェックリストを使用したチャレンジアンドレスポンスです。

**タイプ:** 学習
**言語:** Python（標準ライブラリ、幺等性を持つ提案してからコミット状態マシン）
**前提条件:** Phase 15 · 12 (耐久実行)、Phase 15 · 14 (トリップワイア)
**所要時間:** 約60分

## 問題

エージェントはアクションを実行します。ユーザーは決定する必要があります。承認するか、しないか。決定が即座である場合、それはおそらくレビューではありません。決定が構造化されている場合、それは遅いが信頼できます。エンジニアリング上の質問は、構造化されたレビューを最小抵抗パスにする方法です。

2023年代のHITLパターンは、同期的なプロンプト：「エージェントは本文Yを含むメールをXに送信したい-承認してください？」ユーザーは承認をクリックします。みんなシステムは安全だと感じています。実際には、このサーフェスは大いにゴム印されます。ユーザーは高速で承認し、承認はほとんど予測し、エージェントが間違っていくと、監査証跡はユーザーが思い出せない承認の長い履歴を示しています。

2026年のパターン（提案してからコミット）は、HITLを耐久基板に移動し、構造化されたメタデータを接続し、ポジティブなコミットが必要です。すべてのマネージドエージェントSDKはバージョンを出荷します。LangGraph`interrupt()`、Microsoft Agent Framework`RequestInfoEvent`、Cloudflare`waitForApproval()`。APIの名前は異なります。シェイプではありません。

## コンセプト

### 提案してからコミット状態マシン

1. **提案。** エージェントが提案されたアクションを生成します。耐久ストア（PostgreSQL、Redis、Durable Object）に永続化されます。含む：
   - 意図（エージェントがこれをしている理由）
   - データ系統（この提案につながった特定のソース）
   - タッチされた権限（スコープ/ファイル/エンドポイント）
   - ブラストラディアス（最悪の場合）
   - ロールバック計画（コミットされた場合、どのようにして元に戻すか）
   - 幺等キー（提案ごとに一意。再提出は同じレコードを返します）
2. **サーフェス。** レビューアーはすべてのメタデータを含む提案を見ます。レビューアーは人間です（エージェントが自分自身をレビューしていません）。
3. **コミット。** ポジティブな確認。アクションが実行されます。
4. **検証。** 実行後、副作用は読み取られて確認されます。検証ステップが失敗した場合、システムは既知の悪い状態にあり、アラートが関わります。

### 幺等キー

幺等キーがなければ、一時的な障害後の再試行は承認されたアクションを2回実行できます。具体的な例：ユーザーは「AからBへ100ドルを転送」を承認します。ネットワークのブリップ。ワークフローは再試行します。ユーザーは1回承認しましたが、転送は2回実行されます。幺等キーは、承認を単一の一意の副作用に結び付けます。2番目の実行はno-opです。

これはStripeとAWS APIが使用するのと同じ幺等パターンです。エージェント承認でそれを再利用することは、Microsoft Agent Frameworkドキュメント内で明示的です。

### 耐久性：なぜ承認はプロセスを上回る

承認待機室は、エージェントが所有しない状態です。ワークフローが一時停止されます（レッスン12）。承認が到着したら、ワークフローはその正確なポイントから再開されます。これはLangGraphが`interrupt()`をPostgreSQLチェックポイント化と対にするのはなぜか。単なるメモリ内状態ではなく、2日後の承認でさえワークフロー整全に見つかります。

### ゴム判の承認とチャレンジアンドレスポンス軽減

HITL用のデフォルトUI（「承認」/「拒否」ボタン）は、本当のレビューなしで高速な承認を生成します。文書化された軽減：チャレンジアンドレスポンスチェックリストは、承認ボタンが有効にされる前に特定の質問に対してポジティブな回答が必要です。具体的なシェイプ：

- 「これがタッチするリソースを理解していますか？ [ ]」
- 「ブラストラディアスが受け入れられることを確認しましたか？ [ ]」
- 「これが失敗した場合のロールバック計画がありますか？ [ ]」

それ自体のための官僚制度ではなく、強制的な機能です。ボックスをティックできないレビューアーは、明確化（エスカレーション）を要求するか、辞任（安全なデフォルト）のいずれかです。Anthropic agent-safetyの研究は、チェックリスト駆動型のHITLをゴム判の承認パターンの軽減として明示的に引用しています。

### 結果のある契約

すべてのアクションが提案してからコミットする必要はありません。2026年のガイダンス：

- **結果のあるアクション**（常にHITL）：不可逆的な書き込み、金融取引、アウトバウンド通信、本番環境データベースの変更、破壊的なファイルシステム操作。
- **可逆的なアクション**（時々HITL）：ローカルファイルへの編集、ステージング環境の変更、明確なロールバックとの可逆的な書き込み。
- **読み取りと検査**（決してHITL）：ファイルの読み取り、リソースのリスト、読み取り専用APIの呼び出し。

### アクション後の検証

「コミットが実行された」は「副作用が起こった」と同じではありません。ネットワークパーティションと競争条件は、バックエンドが永続化しなかった間、ワークフローが成功したと考えるワークフローを生成できます。検証ステップは、コミット後にターゲットリソースを読み取って確認して確認します。これはデータベーストランザクションと同じパターンで、`RETURNING`句またはAWS`GetObject`は`PutObject`の後です。

### EU AI法第14条

第14条は、EUの高リスクAIシステムに対する効果的な人間の監視を義務付けています。「有効」は装飾的ではありません。規制言語は特にゴム印のパターンを除外しています。提案してからのチャレンジアンドレスポンス付きでコミットすることは、Microsoft Agent Governance Toolkit準拠ドキュメントで第14条の精査を生き残る形です。

## 使用方法

`code/main.py`は、標準ライブラリPythonで提案してからコミット状態マシンを実装します。耐久ストアはJSONファイルです。幺等キーは（thread_id、action_signature）のハッシュです。ドライバーは3つのケースをシミュレートします。クリーンな承認フロー、一時的な障害後の再試行（再度実行してはいけません）、ゴム判デフォルトとチャレンジアンドレスポンスフロー。

## 運用

`outputs/skill-hitl-design.md`は、提案してからのコミットシェイプのための提案されたHITLワークフローをレビューし、欠落しているメタデータ、幺等性、検証、またはチャレンジアンドレスポンスレイヤーをフラグします。

## 演習

1. `code/main.py`を実行します。承認された提案の再試行が耐久レコードを使用し、再度実行しないことを確認します。これで、幺等キーにタイムスタンプを含めるようにタイムスタンプを変更し、再試行が2回実行されることを示します。

2. `rollback`フィールドを含む提案レコードを拡張します。検証ステップが失敗する実行をシミュレートします。ロールバックが自動的に起動することを示します。

3. Microsoft Agent FrameworkのRequestInfoEventドキュメントを読みます。APIが含むが、おもちゃのエンジンが欠落している1つのメタデータフィールドを特定します。それを追加して、それが何から保護するかを説明します。

4. 特定のアクション（例：「公開Twitterアカウントに投稿」）のためにチャレンジアンドレスポンスチェックリストを設計します。レビューアーが回答する必要がある3つの質問は何ですか？なぜこれら3つですか？

5. 同期「承認？」プロンプトが十分であろう1つのケースを選択（耐久ストアは必要ありません）。なぜなのかを説明し、受け入れている危険クラスに名前を付けます。
