# スーパーバイザー / オーケストレーター・ワーカー・パターン

> 1つのリード・エージェントが計画して委任；特化したワーカーが並列コンテキストで実行してレポート戻す。これはAnthropicのResearchシステムの背後のパターン（リードとしてClaude Opus 4、サブエージェント用のSonnet 4）、内部研究評価上で単一エージェントOpus 4の上で+90.2%で測定。Anthropic エンジニアリング投稿は80% BrowseComp上の分散はトークン使用だけで説明される — マルチエージェント・ウィンはおおむねそれぞれのサブエージェントが新しいコンテキスト・ウィンドウを得るため。このレッスンは原始要素からスーパーバイザー・パターンを構築し、本番展開から2026年エンジニアリング・レッスンをカバーする。

**タイプ:** Learn + Build
**言語:** Python (stdlib, `threading`)
**前提条件:** Phase 16 · 04 (Primitive Model)
**所要時間:** ~75 minutes

## 問題

研究は単一エージェント・システムが失敗する典型的なタスク。あなたが「2023年と2026年の間のマルチエージェント・システムで何が変わった」かと質問する。1つのエージェントは5つの論文を連続して読み、その文で文脈の半分を埋めてから、すべてについて一緒に理由を付ける必要がある。5番目に達するまでに最初を忘れる。パラレライズできない。

スーパーバイザー・パターンはこれを修正。1つのリード・エージェントは検索を計画し、各サブ質問をワーカーに委任し、合成。各ワーカーは1つの狭い質問用の独自の200k-トークン・ウィンドウを得る。リードは生論文を見ない — ワーカー・サマリーだけ。

Anthropic本番Researchシステムは内部研究評価上でsvs 1つのOpus 4で+90.2%をレポート。同じ投稿は80% BrowseComp分散がトークン使用だけで説明されることに注意 *。 新しいコンテキストあたりのサブエージェントが主なメカニズムである。

## コンセプト

### パターン

```
                 ┌──────────────┐
                 │   Lead       │  計画、分解、
                 │  (Opus 4)    │  合成
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         新規       新規        新規
         コンテキスト コンテキスト コンテキスト
```

リードは生マテリアルを読まない。ワーカーはリードが合成するまで互いの作業を見ない。各矢印は狭い人工物でハンドオフ。

### なぜそれがウィンするか

3つのメカニズム。

1. **サブエージェント単位の新規コンテキスト。** 「FIPA-ACLへリテージ」を探索するワーカーはリードが計画に費やした40kトークンを運ばない。1つの質問用に200kウィンドウを得る。
2. **プロンプト経由の特化。** リードのプロンプトは「分解と合成」であり、「研究」ではない。各ワーカーのプロンプトは狭い。「Xで何が変わったか見つける」。集中プロンプトは集中出力を生成。
3. **パラレリズム。** ワーカーは並行して実行。ウォールクロック・タイムは大ざっぱに`max(worker_times) + plan + synthesis`であり、`sum(worker_times)`ではない。

### エンジニアリング・レッスン（Anthropic 2025）

Anthropic投稿は本番で関連性のある2026年のいくつかのレッスンをリスト。

- **クエリ複雑性に取り組みをスケール。** 単純クエリ。1つのエージェント、3-10ツール呼び出し。複雑なクエリ。10+エージェント。リードはこれを推定する必要がある、呼び出し側ではなく。
- **広それから狭。** 最初に広いサブ質問に分解してから、答えが保証する場合、より多くのワーカー単位のサブ質問あたりスポーン。
- **虹展開。** エージェントは長時間実行とステートフル。従来のブルー・グリーンは機能しない。Anthropic使用虹。徐々に古いドレイン中に新しいバージョンのロールアウト。
- **トークン使用が支配。** マルチエージェントは単一エージェントのトークンの〜15倍。タスク値がコストを正当化するときだけ実行。

### LangGraph ターン

LangGraphは元々高レベルの`create_supervisor`ヘルパーを持つ`langgraph-supervisor`ライブラリをシップ。2025年にLangChainは推奨を変更、スーパーバイザー・パターンをツール呼び出し経由で実装、ツール呼び出しが*スーパーバイザーが見ている*より多くの制御（コンテキスト・エンジニアリング）を与えるため。ライブラリはまだ機能する。ドキュメントは今のツール呼び出し形式を推奨。

### 失敗モード

- **リード幻覚計画。** リードが分解する実質的な質問を分解しないサブ質問を生成、ワーカーは誤ったターゲットで正確な研究をする。
- **ワーカー・オーバー・エクスプロア。** 明示的なスコープ・バウンダリなしで、ワーカーは割り当てられたサブ質問を超えてドリフト、合成ステップを汚す。
- **合成競合。** 2つのワーカーが矛盾した事実を返す。リードは再質問（ラウンドを追加）またはそれを明示的に注記する必要がある。静かに1つの側を選んでいることは悪い失敗。ユーザーは同意が起こったことを決して知ら。

### スーパーバイザーが誤っているとき

- **連続タスク。** ステップ2がリテラルにステップ1の出力が必要、パラレリズムは何も買わない。パイプラインを使う（CrewAI Sequential、LangGraph直線グラフ）。
- **単純クエリ。** 単一エージェント・ハンドルはより速く安い。スーパーバイザーの「スケール取り組み」チェック前にワーカーをスポーン。
- **厳格決定性。** スーパーバイザーは LLM 選択委任を使用。監査/リプレイがアダプタビリティより重要なとき静的グラフはより良い。

## 作成する

`code/main.py`は`threading`を使用して3つの並列ワーカーのスーパーバイザーを実装。リード分解クエリをサブ質問、ワーカーが並行して各サブ質問で実行、リードが合成。リアルLLMなし — ワーカーはスクリプト化、フェッチ・アンド・サマリーをシミュレート。

主なストラクチャ。

- `Lead.plan(query)`は3つのサブ質問にクエリを分割。
- `Worker.run(sub_q)`は偽サマリーを返す（本番のすべてのツール使用エージェントである可能性がある）。
- `Lead.run(query)`がワーカーを開始スレッド、参加、合成。

実行する。

```
python3 code/main.py
```

出力は計画、開始/終了タイムスタンプとの並列ワーカー・トレース、最終合成を示す。あなたはウォールクロック・ウィンを見ることができる。3つの0.3秒のワーカーは〜0.35秒で実行、0.9ではない。

## 使用する

`outputs/skill-supervisor-designer.md`はユーザー・クエリを取り、スーパーバイザー・パターン設計を生成。リード・システムプロンプト、ワーカー・ロール、サブ質問分解ルール、合成テンプレート。新しい研究スタイル・エージェント・システムを構築する前に使用。

## 出荷する

スーパーバイザー・パターンを展開する前にチェックリスト。

- **モデル・ペアリング。** リード上の推論層モデル（Opus クラス、`o3` クラス）。ワーカーより速い安いモデル上（Sonnet、`o4-mini`）。
- **ワーカー・タイムアウト。** 中央実行時間の2×を超えるすべてのワーカーが殺される。リード再スポーン狭いスコープで、またはそれなしで進め。
- **ワーカー単位トークン・キャップ。** 厳密リミット（予想合成入力の10×を言う）はランアウェイ・ワーカーが予算を吹く防ぐ。
- **可観測性。** リード計画、各ワーカーの ツール呼び出し、合成をトレース。これは任意のアド・ホック・デバッグのベース。
- **虹ロールアウト。** ステートフル長実行エージェントは段階的なバージョン転移が必要、ホット・スワップではなく。

## 演習

1. `code/main.py`を実行してから、5つのワーカーをスポーンするようにリードを変更。並列効果を観察。ワーカー・カウントはどこでスポーン・オーバーヘッドが並列貯蓄を超すか?
2. ワーカー・タイムアウトを実装。0.5秒以上実行するすべてのワーカーを殺してリード残りの結果を合成。ワーカーが切れたことを知るために何の可観測性が必要か?
3. リード合成にコンフリクト検出ステップを追加。2つのワーカーが矛盾した答えを返すとき、リードが1つを静かに選ぶのではなく、同意を注記。LLMを呼ばずに矛盾を検出する方法?
4. Anthropic Research-system エンジニアリング投稿を読む。この玩具デモが本番で実行するために採用する必要がある3つの実装をリスト。
5. LangGraph `create_supervisor`（レガシー）を新しいツール呼び出し推奨と比較。どれがスーパーバイザーが見ているもっと良いコントロールを与えるか? なぜAnthropicは明示的にサブ答えだけを合成に入れ、生ワーカー・コンテキストではないか?

## キーターム

| ターム | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| スーパーバイザー | 「リード・エージェント」 | 計画、委任、合成するオーケストレーター・エージェント。自分で作業をしない。 |
| ワーカー | 「サブエージェント」 | スーパーバイザーによって狭いスコープと独自のコンテキスト・ウィンドウで呼び出される集中エージェント。 |
| オーケストレーター・ワーカー | 「スーパーバイザー・パターン」 | 同じこと、異なる名前。2026文献は両方を使用。 |
| 新規コンテキスト | 「クリーン・ウィンドウ」 | ワーカーのコンテキストはそのシステムプロンプトと割り当てられた質問から始まり、リード履歴ではない。 |
| 虹展開 | 「段階的ロールアウト」 | 長実行ステートフル・エージェントは段階版ドレイン・アンド・リプレイスが必要、ブルー・グリーンではなく。 |
| トークン・ドミナンス | 「コンテキストが変数」 | 80% 研究評価分散はトークン使用から来、Anthropic当たりのモデル選択ではなく。 |
| スケール取り組み | 「複雑性に合わせエージェント・カウント」 | リード推定クエリ難度、相応に1 vs 10+ ワーカー。 |
| 合成コンフリクト | 「ワーカーは同意しない」 | 2つのワーカーが矛盾した事実を返す。リードは必ずサーフェス同意、1つを静かに選ばず。 |
