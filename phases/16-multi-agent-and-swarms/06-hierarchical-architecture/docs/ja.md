# 階級的アーキテクチャと失敗モード

> 階級的はスーパーバイザー・ネスト。マネージャー・エージェント・サブマネージャー・ワーカー。CrewAI `Process.hierarchical`はテキストブック版。マネージャーLLMは動的にタスクを委任して出力を検証。LangGraph同等は`create_supervisor(create_supervisor(...))`。タスクが本当のオルグ・チャートのときそれは自然なパターン。これはまたマネージャー・ループに崩壊する可能性が最も高いパターン — マネージャー・エージェント・割り当ていが悪い、サブ出力を誤解、または合意を達する失敗。 連続にはしばしばそれを打つ。

**タイプ:** Learn + Build
**言語:** Python (stdlib)
**前提条件:** Phase 16 · 05 (Supervisor Pattern)
**所要時間:** ~60 minutes

## 問題

スーパーバイザー・パターンがクリックすると、自然な次のステップは「ワーカー自体がスーパーバイザーであるとしたら?」チームはサブ・チーム。企業はサブ部門。階級的アーキテクチャはそれをミラーする。

問題。LLMマネージャーは人間マネージャーと同じではない。人間マネージャーはそれのレポートが知ることに関する安定した前の知識を持っている。LLMマネージャーは何がコンテキストにあるがその関数をオルグから再推理する。そのコンテキストでの小さなドリフト、整っぱいツリーは误置く。

## コンセプト

### 形

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

すべての内部ノードが計画、委任、合成。葉だけが作業。

### どこに輝くか

- **明確なオルグ・マッピング。** 本当のタスクが部門的（「法令査察、財務查察、エンジニアリング查察、エクゼク向けサマリーしてしたら」）階級は明確。
- **ローカル合成。** 各サブマネージャーはトップ・マネージャーがそれを見ないトップ・マネージャー・サマリーの前に，チーム出力を合成。トップマネージャーは15ワーカー出力ではなく3つのサブマネージャー・サマリーを見える。

### どこに壊れるか

3つの失敗モード 2026 アップリケーション・レポートが見つけるように。

1. **タスク・割り当て・エラー。** マネージャーがゴール読み、幻覚分解、誤ったサブマネージャーに委任。サブマネージャーは従順に与えられたで作業をするため、エラーはトップ合成でのみサーフェス — 1つのレベルが人間が持つことができた削除。
2. **出力誤解釈。** サブマネージャーは「クレーム X を検証できない」を返す。トップマネージャーは「クレーム X は確認されない」に合成。意味はすべてのレベルでドリフト。
3. **コンセンサス・ループ。** 2つのサブマネージャーが同意しない。トップマネージャーはそれらに調和するよう求める。彼らはダウン・デリゲート。ワーカーは再実行。サブマネージャーはわずかに異なる答えを返す。ループ。CrewAIの`Process.hierarchical`ステップ・リミットでこれに対する保護、ただしリミット自体は現在ハイパーパラメータ。

### 決定質問

連続（直線パイプライン）vs 階級的。あなたのタスクは本当に独立したサブ・チームまたははその1つの直線フロー、树のふりをしている?後者の場合、連続を使う。前者である場合、階級的を使うが明示的な調和ルールを予算。

### CrewAIの実装

`Process.hierarchical`はマネージャーLLMを専門クルーの上にワイヤ。マネージャーは。

- トップレベルのタスクを受け取る
- クルーにサブタスクを割り当てる
- クルー出力を評価する
- 受け入れるかどうか決定、再委任、繰り返す。

ドキュメンテーション。https://docs.crewai.com/en/introduction（コア・コンセプト下で「Hierarchical Process」見。

### LangGraphの実装

LangGraphは`create_supervisor`呼び出しをネスト。内部スーパーバイザーはそれ自身のグラフを持つ。外部スーパーバイザーは内部グラフを不透明なノードに扱う。これはCrewAIより創造より清潔（あなたはそれぞれのグラフをステップできる）ただし木を動的に再成形するのが難しい。

参照。https://reference.langchain.com/python/langgraph-supervisor。

## 作成する

`code/main.py`は3レベル階級を実行。

- トップマネージャー。タスクを「エンジニアリング」と「法律」ブランチに分割。
- エンジニアリング・サブマネージャー。「フロントエンド」と「バックエンド」ワーカーに分割。
- 法律サブマネージャー。1つのワーカー。

デモコントラストは幸せなパス（すべてが同意）に対して、**摂動パス**トップ・マネージャーの分解が「法律」を「財務」に誤ラベルして、エラー・カスケードを見ている — サブマネージャーは従順に財務作業をする、トップ合成が財務結果をレポート、オリジナル法律クエスは答えなし。

実行する。

```
python3 code/main.py
```

出力は両方のパスを明確なサイド・バイ・サイドで示す「何が質問された」vs「何が配信された」。

## 使用する

`outputs/skill-hierarchy-fitness.md`与えられたタスクが階級的、連続、または平坦なスーパーバイザーを使うべきかを評価。インプット。タスク説明、オルグ構造、調和予算。出力。パターン推奨、保護すべき具体的な失敗モード。

## 出荷する

階級的をシップするとき。

- **ツリー深さをキャップ2。** 3つのレベルはすでにほとんどのエラーを可観測性から隠す。
- **明示的な調和予算。** トップマネージャーが提出する前に最大ラウンドを設定。通常2。
- **すべての合成上のプロバンス。** 各ノードのサマリーはどの葉出力がそれを生成したかを引用。
- **分解ドリフト上のアラート。** マネージャーの分解を論文/ステップ単位でログに記録。ユーザー・クエリに対して diff。分解がもはやクエリをカバーしないとき、アラートを起動。

## 演習

1. `code/main.py`を実行して、幸せ vs 摂動を比較。トップ出力が完全にユーザーのクエリから発散するまで、マネージャー・ハンド・オフの何レベル?
2. 3番目のレベル（トップ→サブ→サブサブ→ワーカー）を追加。摂動パスが深さが大きくなるのに従って自分で修正するか、完全に発散するのか計測。
3. 「キャナリ」ワーカーを各サブマネージャーで実装、変わらないユーザー・クエリで常に質問。キャナリ答えを使用して分解ドリフトを検出。マネージャーがキャナリがシンセサイズされた答えに同意しないとき何をすべきか?
4. CrewAIの`Process.hierarchical`ドキュメントを読む。1つの具体的なガードレイル CrewAI 適用（ステップ・リミット、マネージャーLLM・コンストレイント）を識別して、それがどの失敗モードを対象とするか説明。
5. ネストしたLangGraph スーパーバイザーをCrewAI階級的と比較。どれが調和ループをより安く検出できるか?

## キーターム

| ターム | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| 階級的 | 「オルグ・チャート・パターン」 | スーパーバイザー・スーパーバイザー。葉だけが作業。 |
| マネージャーLLM | 「ボス」 | 内部ノードで分解、割り当て、検証するLLM。 |
| 分解ドリフト | 「ボスがプロットを失った」 | トップ・マネージャーの分割がもはやオリジナル・クエリをカバーしない。 |
| 調和ループ | 「終わりのない会議」 | サブマネージャーが同意しない。トップが再委任。ワーカーは再実行。ループ予算枯渇まで。 |
| 深さ2天井 | 「2つのレベルより深く行かない」 | 経験的ガードレイル。3+ レベル可観測性を崩す。 |
| キャナリ質問 | 「すべてのレベルの真実」 | すべてのレベルで常にオリジナル・クエリで質問されるワーカー、ドリフトを検出。 |
| プロバンス・チェイン | 「誰が何を言った」 | 各合成から葉出力を生成した追跡。 |
