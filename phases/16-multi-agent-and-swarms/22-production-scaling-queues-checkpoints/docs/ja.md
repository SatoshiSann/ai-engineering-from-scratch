# 本番環境スケーリング—キューズ、チェックポイント、耐久性

> 数千の並行実行にマルチエージェントシステムをスケーリングするには**耐久実行**が必要です。LangGraph ランタイムは`thread_id`でキーを付けた各スーパーステップ後にチェックポイントを書き込みます（デフォルトでPostgres）。ワーカークラッシュはリースを解放し、別のワーカーを再開します。エージェントは人間入力を無限に待ちながら眠ることができます。**MegaAgent** (arXiv:2408.09955) は3つの状態（Idle/Processing/Response）と2層協調（グループ内チャット+グループ間管理チャット）を持つエージェントごとのプロデューサーコンシューマーキューを実行しました。**ファイバー/async**はLLMストリーミングのためのスレッドごとのジョブを打ちます：スレッドはトークンを待つ間99%の時間アイドルし、ファイバーI/Oを協力的に譲ります。反論：Ashpreet Bediの「Scaling Agentic Software」は、負荷がそれ以外を証明するまで**FastAPI + Postgres +何もないも**を主張します—簡単なアーキテクチャは期待より遠いです。このレッスンは耐久チェックポイントログを構築し、状態遷移を持つエージェントごとの作業キューを構築し、非同期対スレッドのデモを構築し、実用的な「シンプルに開始」ルールを着地させます。

**タイプ:** 学習 + 構築
**言語:** Python (stdlib、`asyncio`、`sqlite3`)
**前提条件:** フェーズ16 · 09 (並列群ネットワーク)、フェーズ16 · 13 (共有メモリ)
**所要時間:** 約75分

## 問題

プロトタイプマルチエージェントシステムは1つのラップトップで3つのエージェントとメモリ内イベントループで機能します。本番環境に移動します：

- エージェントは時々時間を実行（長い研究、人間がループを待つ）。
- ワーカープロセスはクラッシュします。再起動は状態を失います。
- ピーク負荷は10倍平均。水平スケーリングが必要です。
- ユーザーはエージェント実行ごとに支払い。正確に一度のセマンティクスが必要です。

メモリ内イベントループはこれらのいずれも行いません。下側に耐久実行層が必要です。2026年の正規オプションは：

1. チェックポイント付きワークフローエンジン（Temporal、LangGraph runtime）。
2. 状態ストア付きメッセージキュー（Postgres + SQS/RabbitMQ）。
3. アクターモデルフレームワーク（MegaAgentのプロデューサーコンシューマー）。
4. 手書きFastAPI + Postgres（Bediの議論）。

このレッスンは各の縮小版を構築します。

## コンセプト

### 耐久実行、パターン

耐久実行エンジンは、各「ステップ」（LangGraphの言語でのスーパーステップ）の後の完全なプログラム状態を永続化します。クラッシュ時：

```
ワーカーはステップの中盤にクラッシュ
  -> リースタイムアウト
  -> 別のワーカーが thread_id を選択
  -> 最後のチェックポイントから再開
  -> 重複の副作用なし
```

これが機能するための要件：

- **シリアライズ可能な状態。** すべてのエージェント状態は永続化可能である必要があります。ライブ連携を持つ関数クロージャはサバイブされません。
- **決定論的な再開。** 同じ状態と同じ入力を与えられて、エージェントは同じアクションを生成します（またはLLM呼び出しの外部決定論的オラクルに延期）。
- **冪等副作用。** 外部呼び出し（ツール呼び出し、支払い）は冪等であるか除外キーを使用する必要があります。

LangGraph は各スーパーステップ後にチェックポイントを書き込みます。Temporal はアクティビティの後に書き込み。Restate はイベントソースジャーナルを使用します。すべての3つが同じパターンを実装します。

### LangGraph のランタイム

各エージェントには`thread_id`があります。状態は型付きdict です。各スーパーステップはチェックポイントテーブルに行を書き込みます。再開時に、ランタイムは最後のチェックポイントから再生し、最初からではありません。エージェントは人間入力を待機して`interrupt()`できます。ランタイムは永続化してワーカーを解放します。入力が到達すると、任意のワーカーが再開できます。

これは2026年4月のリファレンス本番デザインです。

### MegaAgent のエージェントごとのキュー

arXiv:2408.09955 はスケール実験を説明します：1つのクラスター内の数千の並行エージェント。アーキテクチャ：

```
エージェント i:
  状態 ∈ {Idle、Processing、Response}
  in_queue   <- エージェント i に宛てたメッセージ
  out_queue  -> 返信+副作用

コーディネーター：
  グループ内チャット  (同じグループの中のエージェント)
  グループ間管理チャット  (高レベルルーティング)
```

2層協調は、グループ内会話が密に起こるように許可しますが、グループ間はスパース—数千のエージェント内でコスト直線を保つために使用するパターン。

### 非同期対スレッドごとのジョブ

LLM呼び出しはI/O結合です。次のトークンを待っているスレッドはアイドル99%の時間です。スレッドはスタックごとに約1MB費用がかかります。10,000の並行呼び出しで、それはスタック用の10GBです。

ファイバー（Python `asyncio`、Go goroutines、Rust `tokio`）I/Oで協力的に譲ります。同じ10,000呼び出しはプロセスに快適に適しています。LLMエージェントスケールで、非同期は最適化ではなく—それはアーキテクチャです。

例外：CPU結合ポストプロセッシング（埋め込み、トークナイザートリック）依然としてスレッドまたはプロセスが必要です。I/OレイヤーをCPUレイヤーから分離します。

### Bediの反論

「Scaling Agentic Software」(Ashpreet Bedi、2026) は、ほとんどのチームが測定された負荷を持つ前にエンジニアリングします。実用的なデフォルト：

- FastAPI + Postgres。
- 各エージェント実行は行です。状態は楽観的な同等性で更新の場所にあります。
- `pg_notify`または簡単なCelery ワーカーを介したバックグラウンドジョブ。
- アプリケーションコードでのポリシーを再試行します。

管理可能なタスクで~100の並行エージェント実行の負荷の場合、これはしばしば必要なすべてです。測定されたそれが失敗するときアップグレード。

ルール：簡単なアーキテクチャが解決できない具体的な問題に当たったときに耐久実行フレームワークを採用してください。早期採用は支払いのない儀式で時間を燃やします。

### 正確に一度のセマンティクス

有料のエージェント実行のため、「正確に一度の効果的」が必要（少なくとも一度配信+冪等コンシューマー）。エンジニアリングの動き：

- **dedupキーごとの実行。** 各副作用呼び出しに含まれます。
- **Outboxパターン。** 副作用は最初にテーブルに書き込み、その後別のプロセスが実行します。両方のステップは冪等です。
- **補正トランザクション。** 副作用成功するがその追跡書きが失敗する場合、補正をスケジュール。

これらはデータベースエンジニアリングパターンで、LLM固有ではありません。LLMの税は、LLM呼び出しが遅いだけであり、他はすべて標準分散システムです。

### レインボー配置

Anthropic のマルチエージェント研究システムは「レインボー配置」を使用します：複数のバージョンのエージェント実行時が長時間実行エージェントが展開ごとにすべてを殺す必要がない場合、並行して実行されます。新しいバージョンをトラフィックのスライスにカナリア。古いバージョンはエージェントが終わるとき退職します。

これは長時間実行のステートフルシステムの標準です。2026年の調整は、エージェントが数時間の場合、展開サイクルはそれに対応する必要があります。

### 正規本番チェックリスト

- 耐久状態（チェックポイント、スナップショット、またはoutbox+再生可能ログ）。
- 冪等副作用。
- LLM呼び出しのための非同期I/Oレイヤー。
- 除外付きで少なくとも一度配信。
- ステートフルな労働負荷のためのレインボー/カナリア配置。
- 観測可能性：エージェントごとのトレース、スーパーステップ監査、再試行カウンター。

## 構築する

`code/main.py`を実装：

- `CheckpointStore` — thread-idキーを持つSQLiteバックアップのチェックポイントログ。各スーパーステップが行を追加します。
- `run_with_checkpoint(agent, thread_id)` — 実行の中盤クラッシュをシミュレート。第2のワーカーは最後のチェックポイントから再開します。
- `AgentQueue` — Idle/Processing/Response状態マシンを持つエージェントごとの小さな作業キュー。
- `demo_async_vs_threads()` — asyncioとスレッド経由の500並行シミュレート「LLM呼び出し」を実行。ウォールクロックと最大メモリを報告（近似）。

実行：

```
python3 code/main.py
```

予想される出力：チェックポイント再開はシミュレートクラッシュ後成功します。非同期バージョンは<1sで500の並行呼び出しを処理します。スレッドバージョンは数秒かかり、並行ユニットごとに桁でより多くのメモリを使用します。

## 使用方法

`outputs/skill-scaling-advisor.md`は耐久実行選択に助言します：FastAPI + Postgres、LangGraph runtime、Temporal、またはカスタム。負荷、状態保持ニーズ、展開頻度によって調整されます。

## 出荷する

正規本番硬化：

- **シンプルに開始（Bediのルール）。** FastAPI + Postgres測定が失敗するまで。
- **最適化する前に計器すべてのものを。** ラン当たりのレイテンシヒストグラム、ステップ当たりの時間、再試行カウント、失敗分類。
- **副作用用のOutboxパターン。** 特に支払いおよび外部API呼び出し。
- **レインボー配置。** デプロイ中に飛行中のエージェント実行を殺しないこと。
- **耐久実行エンジン（Temporal / LangGraph / Restate）を採用してください**具体的な問題に当たったときに：長時間人間がループを待つ、クロスリージョン協調、複雑なリトライ/補正ポリシー。
- **I/O層用に非同期。** スレッドのみCPU結合ポストプロセッシング用。

## 演習

1. `code/main.py`を実行します。チェックポイント再開が機能することを確認。非同期対スレッド同時実行差を測定します。
2. **アウトボックス**テーブルを実装：すべてのツール呼び出しが最初にアウトボックスに書き込み、それから別のゴルーチン/タスク実行。冪等性を二倍のツール呼び出しで確認。
3. **レインボー配置**をシミュレート：2つの並行ランタイムバージョン。各々に新しい thread_ids の半分をルート。旧バージョンで飛行中のスレッドが中断されないことを確認。
4. LangGraph の runtime doc を読んでください（下にリンク）。どのランタイムの機能が手書き FastAPI + Postgres バージョンで複製に最長時間要するかを識別。それは採用の理由か、延期できるか？
5. MegaAgent (arXiv:2408.09955) セクション3を読んでください。2層協調（グループ内+グループ間管理チャット）は明示的です。これをメッセージキューに2つのキューファミリーでマップする方法を描きます。

## 主要用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| 耐久実行 | 「プログラム状態を永続化」 | エンジンは各スーパーステップ後に状態を書き込み。クラッシュ復旧は決定論的。 |
| スーパーステップ | 「トランザクション境界」 | チェックポイント間の作業単位。LangGraphの用語。 |
| thread_id | 「エージェント実行識別子」 | チェックポイントと再開ロジックを結ぶキー。 |
| 冪等性 | 「再試行しても安全」 | 副作用の繰り返しは1つの試みと同じ結果を生成します。 |
| Outboxパターン | 「副作用を分離」 | 意図をテーブルに書き込む。別のエクゼキューターが実行してマークします。 |
| 少なくとも一度配信 | 「可能な重複」 | メッセージキューセマンティクス。除外キーはコンシューマーを効果的に一度作成します。 |
| レインボー配置 | 「重複バージョン」 | 長時間実行労働負荷中に複数のランタイムバージョンが並行。 |
| 非同期ファイバー | 「協力的譲渡」 | ユーザーモード同時実行。I/O結合負荷の場合スレッドより安い。 |
| チェックポイント | 「状態スナップショット」 | スーパーステップ境界で設定的な状態。再開用キー。 |
