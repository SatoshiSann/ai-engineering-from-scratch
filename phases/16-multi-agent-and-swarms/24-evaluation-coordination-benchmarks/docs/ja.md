# 評価とコーディネーションベンチマーク

> 5つの2025-2026ベンチマークはマルチエージェント評価スペースをカバーします。**MultiAgentBench / MARBLE** (ACL 2025, arXiv:2503.01935) はマイルストーンKPI でスター/チェーン/ツリー/グラフトポロジを評価します。**研究のためのグラフが最も良く**、認知計画は~3%マイルストーン達成を追加します。**COMMA**はマルチモーダル非対称情報協調を評価します。GPT-4oを含む最先端モデルはランダムベースラインを打つのに苦しみます。**MedAgentBoard** (arXiv:2505.12371) は4つの医学タスクカテゴリをカバーし、マルチエージェントが単一LLMを支配しないことをしばしば発見します。**AgentArch** (arXiv:2509.10769) はツール使用+メモリ+オーケストレーションを組み合わせるエンタープライズエージェントアーキテクチャをベンチマークします。**SWE-bench Pro** ([arXiv:2509.16941](https://arxiv.org/abs/2509.16941)) は41個のレポジトリ全体で1865問題でビジネスアプリケーション、B2Bサービス、開発者ツールをスパン。フロンティアモデルはVerified上で70%+対Pro上で~23%にスコア—汚染信号。Claude Opus 4.7（2026年4月）は、明示的なエージェントチーム協調でPro上で**64.3%**と報告されます（Anthropicの主要源は公開されていません—予備として処理します）。Verdent（エージェントスキャフォード）は、Verified上で**76.1% pass@1**を打ちます（[Verdentの技術レポート](https://www.verdent.ai/blog/swe-bench-verified-technical-report)）。**AAAI 2026 ブリッジプログラムWMAC** (https://multiagents.org/2026/) は2026年のコミュニティの焦点です。このレッスンはMARBLEのメトリクスの上で構築し、トポロジ対メトリクススイープを実行し、「ただSWE-benchの検証をパスしているのは一般化の証拠ではない」ルールをピンします。

**タイプ:** 学習
**言語:** Python (stdlib)
**前提条件:** フェーズ16 · 15 (投票と討論の位相)、フェーズ16 · 23 (失敗モード)
**所要時間:** 約75分

## 問題

紙が「私たちのマルチエージェントシステムが良いです」と主張するとき、質問は、より良い何、、どのように測定ですか？2023-2024エラはマルチエージェント評価が混乱していました—みんなは独自のメトリクス、独自のベースラインと、独自のタスクセットを選びました。2025-2026ベンチマークが構造を強制しました。

共有ベンチマークなしで、2つのマルチエージェントシステムは意味をなして比較することはできません。悪い、ホールドアウトベンチマークなし、フロンティアモデルは汚染できます。SWE-bench検証は訓練コーパによって2025年半ばに部分的に汚染されました。フロンティアスコアは膨張しました。Proは汚染されていない現実チェックとして設計されました。

このレッスンは5つの正規2026ベンチマークを列挙し、各測定を命名し、ベンチマーク請求を懐疑的に読む方法を教えます。

## コンセプト

### MultiAgentBench (MARBLE) — ACL 2025

arXiv:2503.01935。4つのコーディネーション位相（スター、チェーン、ツリー、グラフ）を研究、コーディング、計画タスク上で評価します。マイルストーンベースのKPI は部分進捗を追跡するのではなく、最終成功のみを追跡します。

測定結果：

- **グラフ**位相は研究シナリオで最も良い。任意の相互批評をサポート。
- **チェーン**は段階的に洗練されたコーディングに最も良い。
- **スター**は高速事実統合に最も良い。
- **協調税**グラフ上~4エージェント過去に表示されます。
- **認知計画**トポロジ全体で~3%マイルストーン達成を追加。

使用：トポロジのコーディネーションを比較リンゴ対リンゴしたいとき。MARBLEレポ(https://github.com/ulab-uiuc/MARBLE)は評価を提供します。

### COMMA—マルチモーダルの非対称情報

エージェント異なる観察モーダルを持ち、情報共有を完全にしないで協調する必要があるタスクをカバーします。報告結果は不快です：GPT-4oを含むフロンティアモデルはCOMMAで**ランダムベースライン**を打つのに苦しみます。信号はマルチエージェントモダリティーが下練習され、低評価です—LLMは単一モーダル協調を妥当に扱います。マルチ-モーダル協調は崩壊します。

使用：システムにマルチモーダルまたは非対称情報協調がある場合。COMMAからのヌル結果は請求する前に測定する警告です。

### MedAgentBoard — ドメイン応力テスト

arXiv:2505.12371。4つの医学タスクカテゴリ：診断、治療計画、レポート生成、患者通信。マルチエージェント対単一LLM対従来のルールベースシステムを比較します。

発見：マルチエージェント**しません**ほとんどのカテゴリで単一LLMを支配します。マルチエージェント利点は狭い—タスク分解は副タスクが明確に分離可能な場合に役立ちます（診断+治療）。協調オーバーヘッド専門知識ゲイン超過を傷つけるとき（レポート生成）。

使用：ドメインが明確な単一LLMベースラインを持つとき。MedAgentBoardのレッスンが一般化する場合、多くの提案マルチエージェントシステムはエンジニアリングされています。

### AgentArch — エンタープライズアーキテクチャ

arXiv:2509.10769。ツール使用、メモリ、オーケストレーションを積層するエンタープライズ設定。ベンチマークは各レイヤーの貢献を分離：ツール追加はどのくらい助けますか？メモリを追加？マルチエージェントオーケストレーションを追加？

使用：エンタープライズエージェントスタックを設計してそれぞれのレイヤーを正当化する必要がある場合。AgentArchは測定できない機能の購入を避けるのに役立ちます。

### SWE-bench Pro —現実チェック

arXiv:2509.16941。41リポジトリ全体で1865問題でビジネスアプリケーション、B2Bサービス、開発者ツールをスパン。**汚染されていない**とするために設計してから後の訓練カットオフ。フロンティアモデルはPro上で~23%対検証上で70%+にスコア。ギャップは汚染信号です。

2026年4月スコア：
- Claude Opus 4.7 on Pro：**64.3%**（報告された明示的なエージェントチームコーディネーション。Anthropicプライマリソース公開なし—予備として処理）。
- Verdent（エージェントスキャフォード）on Verified：**76.1% pass@1** ([technical report](https://www.verdent.ai/blog/swe-bench-verified-technical-report))。
- フロンティアは低い Pro 上でエージェントスキャフォルディングなし：~23-35% ([SWE-bench Pro紙](https://arxiv.org/abs/2509.16941))。

要点：「我々はSWE-bench Verified を打ったこと」はもはや能力の証拠ではありません。Proは現在のゲーティングテストです。エージェントチームスキャフォルディングはProで測定可能なゲイン（~30-40ポイントデルタ）を生成し、マルチエージェント協調に対する2026年で最も強い経験的な議論の1つです。

### AAAI 2026 WMAC

AAAI 2026 ブリッジプログラム— マルチエージェント協調のワークショップ(https://multiagents.org/2026/)。マルチエージェント AI 研究用2026年のコミュニティ焦点。受け入れたペーパーとワークショップ会議は新しい方法を評価するための正規会場です。本番決定のための arXiv 前版よりWMAC受け入れた請求に延期。

### ベンチマーク請求を懐疑的に読む—2026チェックリスト

マルチエージェント結果があると主張するとき：

1. **どのベンチマーク、どのスプリット？** SWE-bench検証対Pro は多くの問題。間違ったスプリットで報告のある数はほぼ無価値。
2. **汚染チェック。** ベンチマークはモデルのトレーニングカットオフ後リリースされましたか？そうでなければ、注意深く処理。
3. **ベースライン比較。** 対単一LLMベースラインは、ランダム対、マルチエージェント作業の前。「同じシステムのチューニングされていないバージョンとの対」ではなく。
4. **統計的な重要性。** Nトライアル、p値、信頼区間。フロンティアモデルはハイバリアンスです。1つの実行は誤解します。
5. **タスク多様性。** 1つのタスクか多くか？本番の汎化の問題。
6. **コスト開示。** タスク当たりのトークン、壁時計。90%ソリューション 20 倍コスト能力の請求ではなく、ビジネス決定です。

### ベンチマークが良く測定しないもの

- **長期計画協調。** 壁時計の日。すべての現在のベンチマークは短く実行。
- **敵対弾力性。** 1つのエージェントが悪意があるか損なわれている場合に何が起こりますか？
- **展開下のドリフト。** ベンチマークは静的です。本番配布はシフト。
- **費用正規化パフォーマンス。** ほとんどのベンチマークは生の正確さを報告しますが、ドル対正確。

あなたが実際に気にする軸用のあなた自身の内部ベンチマークを構築することはしばしば正しい動きです。

## 構築する

`code/main.py`は非インタラクティブなウォークスルーです：

- 3つのマルチエージェントシステムを玩具タスク上でシミュレート。
- 各のためにMARBLEスタイルマイルストーンメトリクスを計算。
- 訓練セットからタスクを保留することで汚染チェック実行。
- 明示的にランダムベースラインと比較。
- ベンチマーク請求スコアカードを印刷。

実行：

```bash
python3 code/main.py
```

予想される出力：生の正確さ、マイルストーン達成、タスク当たりコスト、ランダムベースラインデルタ対、汚染チェック注でシステムスコアカード。

## 使用方法

`outputs/skill-benchmark-reader.md`はマルチエージェントベンチマーク請求を読み、厳しさチェックリストを適用します。出力：グレードと注意事項。

## 出荷する

本番環境評価規律：

- **実際の本番分布を反映するあなた自身の内部ベンチマーク構築。** 公開ベンチマークは知らせるが代替しません。
- **すべての比較に含めてランダムベースラインを。** 協調タスクでランダムを大きなマージンで打つことができない場合、タスクは悪く造られているかもしれません。
- **コストをレポート側の正確さ。** トークンコストと壁時計。Opsチームは両方が必要。
- **四半期ごとにベンチマーク再構築。** 本番配布はシフト。古いベンチマークは誤解します。
- **公開ベンチマーク過フィッティングを避ける。** チームはSWE-bench Proの数を最適化している場合、本番環境で回帰されます。

## 演習

1. `code/main.py`を実行します。3つのシミュレートシステムのうち、タスク当たり最高のコスト対マイルストーンのはどちらですか？それは最高の生の正確さシステムに一致していますか？
2. MultiAgentBench (arXiv:2503.01935)を読んでください。独自のタスクドメイン用に、MARBLEは4つのトポロジのどれを推奨するか決定。紙の結果から正当化します。
3. SWE-bench Proペーパーを読んでください。具体的には何が汚染耐性を作りますか？同じテクニックは他のベンチマークに適用できますか？
4. COMMSのマルチモーダル協調発見を読んでください。あなたの内部ベンチマークに追加できる単純なマルチモーダル協調タスク設計。有用な信号はどのようなカウントでしょう？
5. ベンチマーク請求チェックリストを1つの最近のマルチエージェント紙の見出しの結果に適用。請求にあなたはどのようなグレードを与えるでしょう？
