# MARL — MADDPG、QMIX、MAPPO

> マルチエージェント協調の強化学習の遺産、2026年のLLMエージェントシステムに依然として知らせています。**MADDPG** (Lowe et al., NeurIPS 2017, arXiv:1706.02275) は集中型訓練、分散実行（CTDE）を導入しました：各批評家は訓練中にすべてのエージェントの状態とアクションを見ます。テスト時のみローカルアクターが実行されます。協力的、競争的、および混合設定で機能します。**QMIX** (Rashid et al., ICML 2018, arXiv:1803.11485) は単調な混合ネットワークでの価値分解です。エージェントごとのQsは共同Qに結合し、`argmax`はきれいに分布します—StarCraft Multi-Agent Challenge（SMAC）で優位。**MAPPO** (Yu et al., NeurIPS 2022, arXiv:2103.01955) は集中型価値関数を持つPPOです。パーティクルワールド、SMAC、Google Research Football、Hanabiで最小限のチューニングでサプライズ効果的」。これらは、分散的に動作する必要があるエージェントチームの訓練方針を支えています。MAPPOは**デフォルト2026協力的-MARLベースライン**です。このレッスンは小さなグリッドワールドのおもちゃから3つのそれぞれを構築し、LLMエージェント訓練に触れる前に3つのアイデアを筋肉記憶に着地させます。

**タイプ:** 学習
**言語:** Python (stdlib、小さなNumPy無料の実装)
**前提条件:** フェーズ09 (強化学習)、フェーズ16 · 09 (並列群ネットワーク)
**所要時間:** 約90分

## 問題

LLMエージェントシステムはエージェント間協調のための方針を訓練します：いつに延期するか、いつに行動するか、どの仲間に呼ぶか。そのような方針を訓練する方法を教えてくれる文献はマルチエージェント強化学習（MARL）で、LLM波の前に来ていて、支配的なアルゴリズムの小さなセットを持っています。

パターン語彙なしでMARLペーパーを読むことは痛いです。集中型訓練と分散実行（CTDE）、価値分解、および集中型批評家は流行語ではなく、特定の問題への特定の回答です：

- 独立したRL（各エージェントは単独で学ぶ）は各エージェントの観点から非定常です。悪い。
- 集中型RL（1つのエージェントがすべてを制御）はスケールしず、実行制約に違反します。
- CTDE は最高を得る両方：グローバル情報で訓練し、ローカル方針で展開します。

## コンセプト

### 3つのペーパーが使用する環境

- **パーティクルワールド（マルチエージェントパーティクル環境）。** 協力的/競争的タスク付きシンプル2D物理。MADDPGのオリジナルのテストベッド。
- **StarCraft Multi-Agent Challenge (SMAC)。** 協力的な微管理、部分的な観察。QMIXのテストベッド。離散アクション、連続状態。
- **Google Research Football、Hanabi、MPE。** MAPPOベースライン。

異なる環境には異なるアクション/観察型があります。アルゴリズムはそれぞれ選択します。

### MADDPG (2017) — CTDEパターン

各エージェント`i`には、独自の観察をアクションにマップするアクター`mu_i(o_i)`があります。各エージェントには、訓練中に全観察とすべてのアクションを見ている批評家`Q_i(x, a_1, ..., a_n)`もあります。アクターは批評家の評価に対する方針勾配で更新されます。

```
アクター更新:    grad_theta_i J = E[grad_theta mu_i(o_i) * grad_a_i Q_i(x, a_1..n) at a_i=mu_i(o_i)]
批評家更新:   次状態の共同推定を与えたTD on Q_i(x, a_1..n)
```

CTDEなぜ：訓練時に、皆のアクションを知っています。それを使用して各批評家の分散を削減します。デプロイ時に、各エージェントは`o_i`のみを見て`mu_i(o_i)`を呼び出します。

失敗モード：批評家はNエージェント（入力はすべてのアクションを含む）で成長します。近似なしに~10エージェントを超えてスケールしません。

### QMIX (2018) — 価値分解

協力的のみ。グローバル報酬は単調関数のエージェントごとのQ値の合計です：

```
Q_tot(tau, a) = f(Q_1(tau_1, a_1), ..., Q_n(tau_n, a_n)),   df/dQ_i >= 0
```

単調性は`argmax_a Q_tot`を各エージェント独立して`argmax_{a_i} Q_i`を選択して計算できることを保証します。それは**あなたが必要とする分散実行プロパティ**です。訓練時に、混合ネットワークは`Q_tot`を エージェントごとのQsから生成します。

なぜQMIXはSMAC上で勝つ：協力的なStarCraft 微管理は同種のエージェント、ローカル観察、グローバル報酬—価値分解のための完璧なフィット。

失敗モード：単調性制約は制限的です。一部のタスクは単調に分解できない報酬構造があります（1つのエージェント、チームを犠牲にしています）。拡張機能（QTRAN、QPLEX）このをリラックスさせます。

### MAPPO (2022) — 見落とされたデフォルト

マルチエージェント PPO：集中型価値関数を持つPPO。各エージェントは独自の方針を持っています。すべてのエージェントは完全な状態を見ている共有（またはエージェントごと）価値関数を持っています。Yu et al. 2022は5つのベンチマークでMADDPG、QMIX、およびそれらの拡張に対してMAPPOをベンチマークし、発見しました：

- MAPPOはパーティクルワールド、SMAC、Google Research Football、Hanabi、MPEでオフ方策MARL方法を一致または打ちます。
- 最小限のハイパーパラメーターチューニングが必要です。
- 安定した訓練。種の上で再現可能。

このペーパーまで、コミュニティはオンポリシーMARLを過小評価しました。2026年には、MAPPOは協力的なMARLのデフォルトベースラインです。新しい方法は別々にそれを打つ必要があります。

### なぜLLMエージェントエンジニアが気にするべきか

3つの直接的な用途：

1. **ルーター訓練。** メタエージェントはどのサブエージェントがタスクを処理するかを選択します。これはN個の分散サブエージェントと1つの集中型ルーター付きMARLの問題です。MAPPOが適しています。
2. **役割の出現。** 生成型エージェントシミュレーションで、エージェントを補完的な役割を採用するように訓練することは、変装のMARLの問題です。QMIX値の分解は補完性を構築で強制します。
3. **マルチエージェントツール使用。** エージェントがツールを共有し、予算を競争するときに、CTDEを介してそれらを訓練することでリソース制約を尊重する展開可能なローカル方針を生成します。

実用的な注意：2026年に、ほとんどの本番LLMエージェントシステムは彼らの方針をプロンプトするのではなく訓練します。MARLは（a）多くの相互作用データ、（b）明確な報酬信号、（c）訓練インフラストラクチャへの意思がある場合に入ります。

### RL を超えた設計パターンのような CTDE

訓練なしでも、CTDEは有用なアーキテクチャパターンです：

- **設計の間に**完全なチーム可視性を仮定します。
- **ランタイムで、分散実行を強制します：各エージェントは`o_i`のみを見ます。

パターンは、エージェントごとの状態を明示的に保つことを強制し、初期段階で部分的な可観測性について考えることを考えさせます。多くの本番マルチエージェントシステムは、どこでも共有状態を無言で仮定します—CTDEの規律がそれを防ぎます。

### 非定常の問題

複数のエージェントが同時に学ぶとき、各エージェントの環境（他者の方針を含む）は非定常です。古典的なシングルエージェントRL証拠は壊れます。このレッスンのMARL アルゴリズムはすべてこれに対処します：

- MADDPG：グローバル批評家はすべてのアクションを見るため、その価値推定は定常です。
- QMIX：価値分解は学習を共同Q空間に移動します。そこで最適性が定義されます。
- MAPPO：集中型価値関数は他者の方針変更からの分散を柔和します。

LLMエージェントシステムでは、非定常性は「先月私のエージェントが働いた、今月その上流のエージェントが変更され、鉱夫は悪い行動」として現れます。CTDE付きMARLを訓練することは原則的な修正です。プロンプトレベルの修正は速いですが、耐久性が低いです。

### このレッスンがカバーしていないもの

実際のネットワークの訓練はフェーズ09トピックです。このレッスンは、勾配アップデートなしのパターンデモを示すスクリプト化されたポリシーバージョンを構築します。目標は、完全なMARLライブラリ（PyMARL、MARLlib、RLlib multi-agent）を取り上げる前にパターンを内在させることです。

## 構築する

`code/main.py`は3つのパターンのデモを実装し、すべて小さい2エージェント協力的なグリッドワールド上：

- 環境：4x4グリッド上の2エージェント、1つの報酬ペレット。報酬=任意のエージェントがペレットに到達する場合は1。タスク完了。
- `IndependentAgents` — 各エージェントは他を環境として扱う。ベースライン。
- `MADDPGStyle` — 集中型批評家は共同価値を計算します。アクター方針がそこから更新されます。スクリプト化されたポリシーの改善。
- `QMIXStyle` — 単調なミキサーの値の分解。
- `MAPPOStyle` — 集中型価値関数。方針は共有ベースラインに対して更新されます。

すべて4つは同じエピソードを実行し、目標への平均ステップを報告します。CTDE変数は独立したベースラインより短いパスに収束します。

実行：

```
python3 code/main.py
```

予想される出力：独立したエージェントは平均~6ステップを取得します。CTDE変数は~3.5ステップに向かって収束（4x4グリッドの最適は3）。パターン差がスクリプト化されたポリシーにもかかわらず表示されます。

## 使用方法

`outputs/skill-marl-picker.md`は、特定のマルチエージェントタスクのためにMARLアルゴリズムを選択するスキルです：協力的対競争的、同種対異質、アクション空間型、スケール、報酬信号。

## 出荷する

本番環境のMARLはまれです。これを使用する場合：

- **MAPPOで開始します。** 2022年のペーパーがこれをベースラインとして確立しました。それを再現することは最初に数週間を費やすことを節約します。
- **すべてのエージェントの観察とアクションストリームをログします。** エージェントトレースのないMARLをデバッグすることは絶望的です。
- **訓練コードから実行コードを分離します。** CTDEは規律です。実行パスは本当に`o_i`のみを見てください。
- **報酬形成の警告。** MARLは報酬設計に非常に敏感です。形成のコーディネーション1つのバグとエージェント学習がそれを悪用します。敵対的なテストを実行します。
- **LLMエージェント、**プロンプトレベルの方針を考慮します最初。相互作用データ+報酬信号+インフラストラクチャがすべて存在する場合のみMARLの訓練に投資します。

## 演習

1. `code/main.py`を実行します。独立したおよびMAPPOスタイルのエージェント間のステップアンティゴアルギャップを測定します。ギャップは6x6グリッドで成長または縮小していますか？
2. 競争的な変数を実装します：2つのエージェント、1つのペレット、最初に到達することで報酬のみを取得します。どのパターンが競争きれいに処理していますか？MAPPOは歴史的です。
3. MADDPG (arXiv:1706.02275) セクション3を読んでください。正確な批評家更新ルール疑似コードで象徴的に実装します。
4. MAPPO (arXiv:2103.01955)を読んでください。なぜ著者らは集中型価値+PPOは彼らのベンチマークでオフポリシーMARLに勝つと主張するのですか？3つの最強の主張をリストアップします。
5. CTDEを設計パターンとして仮説のLLMエージェントシステムに適用します（例えば、研究エージェント+サマライザー+コーダー）。デザイン時に利用可能な共同情報はランタイムで利用できないものですか？
