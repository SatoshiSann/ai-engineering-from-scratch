# A2A — エージェント間プロトコル

> Googleは2025年4月にA2Aを発表しました。2026年4月までに、仕様は https://a2a-protocol.org/latest/specification/ にあり、150以上の組織がそれを支持しています。A2AはMCP（レッスン13）の水平補完です。MCPは垂直（エージェント↔ツール）、A2Aはピアツーピア（エージェント↔エージェント）です。エージェントカード（ディスカバリー）、アーティファクト（テキスト、構造化データ、ビデオ）を持つタスク、不透明なタスクライフサイクル、認証を定義します。本番システムはますますMCPとA2Aをペアリングしています。Google CloudはVertex AI Agent Builder中に2025-2026年A2Aサポートをロールアウトしました。

**タイプ:** Learn + Build  
**言語:** Python (stdlib, `http.server`, `json`)  
**前提条件:** Phase 16 · 04 (Primitive Model)  
**所要時間:** 約75分

## 問題

あなたのエージェントは別のシステムの別のエージェントを呼び出す必要があります。どうする? HTTPエンドポイントを公開し、カスタムJSONスキーマを定義し、反対側が話すことを望むことができます。エージェントの各ペアがカスタム統合になります。

A2Aはその呼び出しの通用的ワイアプロトコルです。標準ディスカバリー、標準タスクモデル、標準トランスポート、標準アーティファクト。HTTPとRESTのようですが、エージェントを第一級市民として。

## コンセプト

### 4つの要素

**エージェントカード。** エージェントを説明するJSONドキュメント。`/.well-known/agent.json`：名前、スキル、エンドポイント、サポートされるモダリティ、認証要件。ディスカバリーはカードを読むことで発生します。

```
GET https://agent.example.com/.well-known/agent.json
→ {
    "name": "code-review-agent",
    "skills": ["review-python", "review-typescript"],
    "endpoints": {
      "tasks": "https://agent.example.com/tasks"
    },
    "auth": {"type": "bearer"},
    "modalities": ["text", "structured"]
  }
```

**タスク。** 作業の単位。非同期、ステートフルなオブジェクト。ライフサイクル：`submitted → working → completed / failed / canceled`。クライアントはタスクを送信し、更新をポーリングまたはサブスクライブします。

**アーティファクト。** タスクで生成される結果タイプ。テキスト、構造化JSON、画像、ビデオ、オーディオ。アーティファクトは型付けされるので、異なるモダリティは第一級です。

**不透明なライフサイクル。** A2Aはリモートエージェントがタスクをどのように解くかを規定しません。クライアントは状態遷移とアーティファクトを見ます。実装は任意のフレームワークを自由に使用できます。

### MCP/A2A分割

- **MCP**（レッスン13）：エージェント↔ツール。エージェントはツールサーバーへのJSON-RPCで読み書きします。デフォルトでステートレス。
- **A2A**：エージェント↔エージェント。ピアプロトコル。両側はそれ自身の推論を持つエージェント。

本番マルチエージェントシステムは両方を使用します。A2Aピアはその側のMCPツールを呼び出します。分割は2つの懸念を清潔に保ちます。

### ディスカバリーフロー

```
Client                     Agent server
  ├──GET /.well-known/agent.json──>
  <──Agent Card JSON─────────────
  ├──POST /tasks {skill, input}──>
  <──201 task_id, state=submitted
  ├──GET /tasks/{id}──────────────>
  <──state=working, 42% done──────
  ├──GET /tasks/{id}──────────────>
  <──state=completed, artifacts──
```

またはストリーミング使用：`/tasks/{id}/events`へのSSEサブスクリプション。プッシュ更新。

### 認証

A2Aは3つの一般的なパターンをサポート：

- **ベアラートークン** — OAuth2またはオペーク。
- **mTLS** — 相互TLS。組織は互いに身元を証明します。
- **署名されたリクエスト** — ペイロード上のHMAC。

認証はエージェントカードで宣言される。クライアントは発見して準拠します。

### 2026年4月までに150以上の組織

エンタープライズ採用がA2Aスケールを駆動しました。ヘッドライン：A2Aは、エンタープライズエージェントシステムが信頼境界を越える方法になりました。Google CloudはVertex AI Agent Builder A2Aサポートを出荷した。Microsoft Agent Frameworkはそれをサポートしている。ほとんどの主要フレームワーク（LangGraph、CrewAI、AutoGen）はA2Aアダプターを出荷しています。

### A2Aが勝つ場所

- **組織間呼び出し。** 会社Aのエージェントは会社Bのエージェントを呼び出します。A2Aなしで、各ペアはカスタムコントラクトです。
- **異種フレームワーク。** LangGraphエージェントはCrewAIエージェントをカスタムPythonエージェントを呼び出します。A2Aが正規化します。
- **型付きアーティファクト。** ビデオ結果、構造化JSON、オーディオ — すべて第一級。
- **長時間実行タスク。** 不透明なライフサイクル+ポーリングは時間単位のタスクを簡単にします。

### A2Aが苦労する場所

- **遅延敏感なマイクロ呼び出し。** A2Aのライフサイクルは非同期です。サブミリ秒のエージェント間は適合しない。直接RPCを使用します。
- **同じPythonプロセスの密結合エージェント。** 両エージェントが同じPythonプロセスで実行される場合、A2AのHTTPラウンドトリップはやり過ぎです。
- **小さいチーム。** 仕様オーバーヘッドは実数。内部のみのエージェントは形式性が必要ない場合があります。

### A2A対ACP、ANP、NLIP

2024-2026に複数の関連仕様が浮上：

- **ACP**（IBM/Linux Foundation）— A2Aの前身。より狭いスコープ。
- **ANP**（Agent Network Protocol）— ピアディスカバリー重い。分散優先。
- **NLIP**（Ecma自然言語相互作用プロトコル、2025年12月標準化）— 自然言語コンテンツタイプ。

A2Aは2026年4月現在、最も採用されたピアプロトコルです。比較のためarXiv:2505.02279（Liu et al。、「エージェント相互運用性プロトコルの調査」）を見る。

## ビルド

`code/main.py`は`http.server`とJSONを使用してA2Aミニマルサーバーとクライアントを実装します。サーバー：

- `/.well-known/agent.json`を公開、
- `POST /tasks`を受け入れ、
- タスク状態を管理、
- `GET /tasks/{id}`でアーティファクトを返す。

クライアント：

- エージェントカードを取得、
- タスクを送信、
- 完了までポーリング、
- アーティファクトを読む。

実行：

```
python3 code/main.py
```

スクリプトはバックグラウンドスレッドでサーバーを起動し、それに対してクライアントを実行します。完全なフローが表示される：ディスカバリー、送信、ポーリング、アーティファクト。

## 使用

`outputs/skill-a2a-integrator.md`はA2A統合を設計します。エージェントカードコンテンツ、タスクスキーマ、認証選択、ストリーミング対ポーリング。

## 出荷

チェックリスト：

- **仕様バージョンをピン留めします。** A2Aはまだ進化中。エージェントカードはプロトコルバージョンを宣言する必要があります。
- **冪等タスク作成。** 重複送信（ネットワークリトライ）は1つのタスクを生成する必要があります。
- **アーティファクトスキーマ。** エージェントが返す形状を宣言。コンシューマーが検証する必要があります。
- **レート制限+認証。** A2Aは公開面。標準ウェブセキュリティを適用します。
- **失敗したタスク用のデッドレター。** 繰り返し失敗タイプをスキャンします。

## 演習

1. `code/main.py`を実行します。クライアントがサーバーを発見し、正しいアーティファクトを受け取ることを確認します。
2. 2番目のスキルをサーバーに追加（例：「要約」）。エージェントカードを更新します。タスクタイプに基づいてスキルを選ぶクライアントを書きます。
3. SSEストリーミングエンドポイント`/tasks/{id}/events`を実装。状態変化を放出します。クライアントは何を異なる行う必要があります?
4. A2A仕様（https://a2a-protocol.org/latest/specification/）を読む。このデモが実装していない仕様が委任する3つのことを特定します。
5. A2A（エージェントカードディスカバリー）とMCP（`listTools`経由のサーバー側ケイパビリティリスト）を比較します。自己記述エージェントとケイパビリティプローブの間のトレードオフは?

## キーターム

| 用語 | 人々が言うこと | 実際の意味 |
|------|---|---|
| A2A | 「エージェント間」 | エージェントがシステム間で別のエージェントを呼び出すピアプロトコル。Google 2025。 |
| エージェントカード | 「エージェントの名刺」 | `/.well-known/agent.json`でのJSON。スキル、エンドポイント、認証を説明します。 |
| タスク | 「作業の単位」 | 非同期ステートフルオブジェクト。ライフサイクル。完了時にアーティファクト生成。 |
| アーティファクト | 「結果」 | 型付き出力：テキスト、構造化JSON、画像、ビデオ、オーディオ。第一級メディア。 |
| 不透明なライフサイクル | 「どのように解くかはエージェントの事業」 | クライアントは状態遷移を見る。サーバーはフレームワーク/ツールを選ぶ自由。 |
| ディスカバリー | 「エージェントを見つける」 | `GET /.well-known/agent.json`はカードを返します。 |
| MCP対A2A | 「ツール対ピア」 | MCP：垂直エージェント↔ツール。A2A：水平エージェント↔エージェント。 |
| ACP / ANP / NLIP | 「姉妹プロトコル」 | 隣接仕様。A2Aは2026年の最も採用。 |
