# 共有メモリとブラックボードパターン

> 2026年マルチエージェントシステムに2つのアプローチが共存：**メッセージプール**（すべてが全員のメッセージを見る、AutoGen GroupChatまたはMetaGPTのように）と**サブスクリプション付きブラックボード**（エージェントが関連イベントにサブスクライブ、Context-Aware MCPまたはMatrixフレームワークのように）。両方がマルチエージェントシステムの唯一のステートフル部分です。つまり、両方が面白いバグが住む場所です。参照フェイルモードは**メモリポイズニング**：1つのエージェントが「事実」を幻覚、他のエージェントがそれを確認済みとして扱う、精度が段階的に減衰する。それは即座のクラッシュより、はるかデバッグが難しい。このレッスンはstdlibから両方の構造を構築し、ポイズニング攻撃を注入し、本番で実際に機能する3つの軽減策を示します。

**タイプ:** Learn + Build  
**言語:** Python (stdlib, `threading`)  
**前提条件:** Phase 16 · 04 (Primitive Model), Phase 16 · 09 (Parallel Swarm Networks)  
**所要時間:** 約75分

## 問題

マルチエージェントシステムはエージェントが事実を共有する場所を必要とします。文字通りのオプションは「すべてをメッセージで渡す」ですが、それは余分なコピーで共有状態を再発明します。もう1つは「全員にグローバルログを与える」ですが、グローバルログは無制限に成長し、簡単にポイズニングします。3番目は「エージェントごとにビューを投影」ですが、スケーラブルですがスキーマ集約的です。

エージェントの1つが幻覚し、幻覚を共有状態に書き込むと、その状態を読む下流のすべてのエージェントが幻覚を事実として採用します。人間が気付いた時点で、推論鎖は5ステップ深く、根本原因は最初のメッセージでした。マルチエージェント精度減衰のデバッグはクラッシュのデバッグより難しい。

これはメモリポイズニングです。これはMAST分類法（Cemri et al。、arXiv:2503.13657）で2番目にドキュメント化された失敗ファミリーであり、それは構造的です。検証可能な不写入ベリファイアなしで真実を持つ共有メモリ設計は最終的にそれを展示します。

## コンセプト

### 2つの主要トポロジー

**フルメッセージプール。** すべてのエージェントがすべてのメッセージを読む。AutoGen GroupChatとMetaGPTはこれを使用します。シンプル、透明、検査可能ですが、~10を超えるエージェントでスケーリングしません。各エージェントのコンテキストが他のエージェントの作業で埋められるから。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**サブスクリプション付きブラックボード。** エージェントが関連トピックを宣言し、基質が関連メッセージのみをルーティングします。CA-MCP（arXiv:2601.11595）とMatrixの分散フレームワーク（arXiv:2511.21686）はこれを使用します。さらにスケーリング、しかしサブスクリプションを意味あるものにするため事前スキーマ設計が必要。

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 各ウィン場合

- **フルプール**はエージェントが少ない（< 10）、異種、会話が短地平のとき勝ちます。誰が何を言ったかについて推論は、全員が全員を見るときトリビアルです。
- **ブラックボード**はエージェントが多い、役割に同質だが大量なインスタンス（スワーム）、会話が長時間実行のときに勝ちます。ルーティングはトークンコストとコンテキスト汚染を節約します。

本番システムはしばしば混合：小さいフルプール上位（計画層）、ブラックボード下位（ワーカー層）。

### メモリポイズニング、1つのシナリオ

3つのエージェントは研究タスクに取り組みます。エージェントAは検索エージェント。エージェントBは要約者。エージェントCはアナリスト。

1. Aはページを取得し、共有状態にメッセージを書き込む。「研究は42%精度改善を報告する。」
2. 取得されたページは実際「4.2%改善」と言いました。Aは小数を幻覚した。
3. Bは共有状態を読み、書き込む：「大42%精度ゲイン報告（ソース：A）。」
4. Cは共有状態を読み、書き込む：「採用を推奨 — 42%リフトは変革的。」
5. 最終報告は決して存在しなかった42%数を引用する。

エージェントのクラッシュなし。テストフェイルなし。システムは「機能」しました。幻覚はワンエージェントのコンテキストからすべての下流エージェントのすべての推論を越えて共有状態経由で越えました。

### なぜこれは構造的か

共有状態なしで、エージェントAの幻覚はAのコンテキストに留まります。下流エージェントは再取得または再導出し、エラーをキャッチするかもしれません。ナイーブな共有状態で、Aのコンテキストは全員のコンテキストになり、幻覚が事実にロンダリングされます。

問題は共有状態それ自体ではなく、**検証可能な不写入ベリファイアなしの共有状態**です。3つの軽減策はこれに対応：

1. **すべての書き込みに検証可能を属する。** 共有状態のすべてのエントリは誰が書いたか、いつ、どのプロンプト下で、(適用可能な場合) エージェントが引用したどのソース。下流エージェントは検証可能にキー付きスケプティシズムで読む。
2. **書き込みをバージョン化。追記のように扱う。** 修正は古いを廃止する新しいエントリ。監査証跡は保持される。
3. **少なくとも1つのエージェントが共有状態に書き込めない保つ。** 読み取り専用ベリファイアエージェントはエントリをサンプリング、ソースを再取得、矛盾をフラグします。それは共有状態に書き込めないから、それはプールにポイズニングされない。

### ブラックボード前例（Hayes-Roth、1985）

ブラックボードパターンはLLMエージェントで4十年を前倒しします。Hayes-Roth（1985年、「制御のためのブラックボードアーキテクチャ」）は、グローバルブラックボードを観察し、部分解を貢献し、他のソースをトリガーする専門Knowledge Sourcesを説明しました。2026年のブラックボード（CA-MCP、Matrix）は同じパターンですがLLMエージェントをKnowledge Sourcesとしてとjsonブロブを部分解として。古い文献は、書き込み論争、機会主義制御、一貫性で文書化しました。モダンシステムが再発見する解決法。

### 投影対フルビュー

純粋なブラックボードは全サブスクライバーに同じ投影（トピックスコープ）を与えます。より積極的な設計は**エージェント毎投影**。各エージェントはその役割にカスタマイズされたビューを得ます。LangGraphの状態リデューサーは標準2026年実装です。リデューサー関数はグローバル状態を役割特定スライスに折ります。

エージェント毎投影はさらにスケーリング、しかし1つのスキーマを必要とします。1つなしで、すべてのエージェントのプロンプトでアドホック投影を再構築します。

### 書き込み論争パターン

複数のエージェント同時書き込みは同時実行問題で、LLM問題ではなく。3つのパターンが機能：

- **シーケンシャルライター（シングルプロデューサー）。** すべての書き込みは、シーケンシャルな1つのコーディネーターエージェント経由。シンプルですが、ボトルネック。
- **楽観的同時実行バージョン付き。** 各エントリにバージョン。ライターはバージョンミスマッチで失敗し、再試行。古典的データベーステクニック。
- **トピックパーティション。** 異なるエージェント異なるトピックを所有。クロストピック論争なし。設計パーティション境界を必要とします。

最も2026年フレームワークはシーケンシャルライターをデフォルトにします。LLM呼び出しは十分に遅いから、論争は稀で、ボトルネックは傷つけない。

### 不写入ベリファイア

最も負荷がかかる軽減は読み取り専用ベリファイアです。実装ルール：

- ベリファイアはチームと共有状態（ブラックボードまたはプールを読む）。
- ベリファイアは共有状態への書き込みハンドルなし — 別の検証チャネルへのみ。
- ベリファイアは独立的に書き込みで引用されるソースを取得。不一致をフラグします。
- ベリファイアの独自出力は、別の決定エージェント、またはプールへ戻りフィードなし。

この分離なし、ベリファイアの出力はプール内の新しいエントリになる。これはポイズニングされたプールがベリファイアをポイズニングし、その検証をポイズニングすることを意味する。

## ビルド

`code/main.py`はstdlib Pythonの両方のトポロジーとおもちゃのポイズニング攻撃と3つの軽減策を実装します。

- `MessagePool` — スレッドセーフな追記のみログ。フル読み出し。
- `Blackboard` — トピックキー付きpub/subエージェント毎サブスクリプション。
- `ProvenanceEntry` — すべての書き込みはレコード（ライター、タイムスタンプ、prompt_hash、source_uri）。
- `PoisoningScenario` — 3つのエージェント研究タスクを実行。エージェントAが小数を幻覚。最終報告を出力。
- `Verifier` — 読み取り専用エージェント。ソースを再取得し、矛盾をフラグします。検証者が存在する同じシナリオを実行。

実行：

```
python3 code/main.py
```

期待出力：
- 実行1（検証者なし）：幻覚42%が最終報告に伝播。
- 実行2（検証者あり）：検証者が矛盾をフラグ、プール「フラグ付き」、最終報告に撤回を含む。

## 使用

`outputs/skill-memory-auditor.md`はマルチエージェントシステムの共有メモリ設計を検証可能、バージョン化、ベリファイア分離のために監査するスキルです。本番前に新しいマルチエージェントアーキテクチャで実行します。

## 出荷

任意の共有メモリ設計：

- すべての書き込みで検証可能を記録：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- ログを追記のみにします。修正は廃止されたものを参照する新しいエントリ。
- 独立したソースアクセスを持つ少なくとも1つの読み取り専用ベリファイアエージェントをデプロイ。
- ベリファイア出力を別のチャネルにルーティング、共有プールへ戻りなし。
- スパースエッジデントの比率をログ — 上昇比率は幻覚パターンの早期証拠。

## 演習

1. `code/main.py`を実行。実行1が幻覚を伝播し、実行2がそれをキャッチすることを確認。
2. 2番目の幻覚を追加：エージェントBがデータセットサイズを発明。ベリファイアはハンドチューンなしの両方をキャッチするべき。
3. フルプールをブラックボードに切り替え（トピックパーティション：`prices`、`summaries`、`analyses`）。トピックパーティショニングはどのポイズニングシナリオをプルするのが難しくし、どれが助けないか?
4. Hayes-Roth（1985年、「制御のためのブラックボードアーキテクチャ」）を読む。このレッスンで議論されなかった論文から2つの制御パターンを特定。2026年システムが恩恵を受けるだろう。
5. CA-MCP（arXiv:2601.11595）を読む。その共有コンテキストストアを`code/main.py`のMessagePoolまたはBlackboardクラスに対応。CA-MCPが上位に追加するプリミティブは?

## キーターム

| 用語 | 人々が言うこと | 実際の意味 |
|------|---|---|
| メッセージプール | 「共有チャット履歴」 | 追記のみログがすべてのエージェント読む。完全透明、悪いスケーリング。 |
| ブラックボード | 「共有ワークスペース」 | トピックキー付きpub/sub。エージェントが関連トピックにサブスクライブ。スケール遠い。 |
| 検証可能 | 「誰が何を書いたか」 | 各書き込みのメタデータ：ライター、タイムスタンプ、プロンプト、ソース。 |
| メモリポイズニング | 「幻覚が広がる」 | 1つのエージェントのエラーが共有状態に入る、下流エージェントが事実として採用。 |
| 追記のみ | 「インプレース更新なし」 | 修正は新しいエントリが廃止。監査証跡保持。 |
| 不写入ベリファイア | 「独立した監査人」 | 読み取り専用エージェント。ソースを再取得し、矛盾をフラグ。 |
| 投影 | 「スコープビュー」 | グローバル状態から計算エージェント毎ビュー。LangGraphリデューサーは標準ケース。 |
| Knowledge Source | 「スペシャリストエージェント」 | Hayes-Rothの1985年用語。ブラックボード参加者。 |
