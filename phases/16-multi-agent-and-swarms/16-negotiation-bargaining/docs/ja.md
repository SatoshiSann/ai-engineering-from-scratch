# 交渉と取引

> エージェントはリソース、価格、タスク割り当て、条件を交渉する。2026年ベンチマークセットは明確：NegotiationArena（arXiv:2402.05863）はLLMが~20%ペイオフを改善できることを示す（ペルソナ操作「絶望」経由）。「言語モデルの交渉能力測定」（arXiv:2402.15813）は買手が売手より難しい、スケール助けない。彼ら**OG-Narrator**（決定論的提案ジェネレータ+LLMナレータ）26.67%から88.88%取引率を押し。大規模自律交渉コンペ（arXiv:2503.06416）~180k交渉を実行し、**鎖の思考隠蔽**エージェントが推論を相手方から隠すことで勝つ。Bhattacharya et al。2025年ハーバード交渉プロジェクトメトリクスに対応ランク Llama-3最も効果的、Claude-3アグレッシブ、GPT-4最も公正。このレッスンはコントラクト ネット プロトコル（FIPA先祖、レッスン02）を実装、LLMスタイル買手/売手をワイア、OG-Narratorスタイル分解を実行、取引率がどう各構造選択で変化するかを測定。

**タイプ:** Learn + Build  
**言語:** Python (stdlib)  
**前提条件:** Phase 16 · 02 (FIPA-ACL Heritage), Phase 16 · 09 (Parallel Swarm Networks)  
**所要時間:** 約75分

## 問題

2つのエージェントは価格に同意する必要があります。純粋言語プロンプトに左、2024-2026年LLM取引を閉じる驚くほど低い率（~27%で しっかりパラメータ化取引arXiv:2402.15813）。スケールは修正しない：GPT-4はGPT-3.5より構造的に交渉では優れていない。それはより言語交渉で優れています。

根本の問題はLLMが2つのジョブ混同 — 提案を決定して提案をナレート。OG-Narratorこれを分離：決定論的提案ジェネレータは数値の動きを計算。LLMのみナレート。取引率は~89%にジャンプ。

これはクラシカルなマルチエージェント発見を反映：メカニズムから通信層を分離が勝つ。コントラクトネットプロトコル（FIPA、1996; Smith、1980）は参照タスク市場メカニズム。LLMをナレーションスロットにプラグインし、最新LLM駆動タスク市場を得る。

## コンセプト

### コントラクトネット、1つの段落

Smith1980コントラクトネットプロトコル：**マネージャー**ブロードキャスト**提案の呼び出し（cfp）**。**ビッダー**は応答**提案**メッセージ彼ら提供を含むと。マネージャーはウィナーを選んで**受理提案**をウィナーと**拒否提案**を敗者に送る。ウィナーが仕事を実行。オプショナルメッセージ：**拒否**（ビッダー提案を提供を拒否）。FIPAはこれを`fipa-contract-net`相互作用プロトコルとしてコーデ。

### なぜOG-Narratorが勝つか

「言語モデルの交渉能力測定」（arXiv:2402.15813）観察：

- LLMはしばしば交渉ルールを破る（ナンセンス価格で提案、反対側ZOPA無視）。
- 彼ら不十分にアンカー（悪い最初提案を受け入れ；戦略的ではなく象徴的な量で逆提案）。
- スケール単独ではこれら修正しない。大きいモデルはより言語に見える。類似戦略エラー。

OG-Narrator分解：

```
           ┌──────────────────┐        ┌──────────────────┐
  state  → │ offer generator  │ price → │  LLM narrator    │ → message
           │  (deterministic) │        │  (writes the     │
           │                  │        │   human-style    │
           └──────────────────┘        │   accompaniment) │
                                       └──────────────────┘
```

提案ジェネレータはクラシカルな交渉戦略：ルビンシュタイン交渉モデル、ゼウテン戦略、価格ティット-フォー-タット簡単。LLMナレート。メッセージは決定論的価格と自然言語フレーミングを含む。

取引率はジャンプします理由：
- 価格は交渉ゾーンに止まる。
- アンカーは戦略的、感情的でない。
- LLMはそれが良いで行う：書く。

### NegotiationArenaの検出

arXiv:2402.05863はベンチマーク標準を提供。見出し検出：

- LLMは~20%ペイオフを改善できる（「金曜までにこれを売るのに絶望」のようなペルソナを採用 — ペルソナ操作は実タクティック。
- 公正/協力エージェントは敵対的によって搾取される。防防は明示的カウンターポスチャリングを必要とする。
- 対称ペアアップはベンチマークシナリオの約40%に不平等な結果に転換。

これは「LLMは悪い交渉者」ではない。これは「LLMはあまりに人間のように交渉する。搾取可能な部分を含む。」

### 鎖の思考隠蔽

大規模自律交渉コンペ（arXiv:2503.06416）は~180k交渉を多くのLLM戦略にわたって実行。ウィナーは推論を相手方から隠蔽した：

- エージェント「私は$75のみに行く。マイ予約価格$70」を公開見えるスクラップパッドに印刷して場合、相手はそれを読む。
- ウィナーはプライベート戦略を計算。出力チャネルは提案と最小必須ナレーションのみを含む。

これは古典的ゲーム理論への2026年エコー（Aumann 1976合理性と情報上）：プライベート評価を明かすペイオフをコスト。LLMはこれを直感せず、公然と相手方に見える推論トレースにそれら予約を愉快に型。

エンジニアリング助言：プライベートスクラップパッドコンテキストを公開メッセージコンテキストから分離。オプションでない。

### Bhattacharya et al。2025年 — モデルランキング

ハーバード交渉プロジェクトメトリクス上（原理ある交渉、BATNA尊重、関心相互性）：

- **Llama-3**は取引ストライキで最も効果的だった（取引率+ペイオフ）。
- **Claude-3**は最も攻撃的な交渉者だった（高いアンカー、後期譲歩）。
- **GPT-4**はフェアレストだった（ペアリング全体で最小分散ペイオフ）。

これは2025年スナップショット。ポイントは4月2026年どのモデルが勝つか ではなく。異なるベースモデルは永続的な交渉スタイルを持つ。異種アンサンブル（レッスン15）は多様性ソースとして含む。

### コントラクトネット+LLMviaタスク割り当て

マルチエージェント用コントラクトネットの最新再利用：

1. マネージャーエージェント分解タスク単位に。
2. ワーカーエージェント`cfp`ブロードキャスト（タスク説明を含む）。
3. 各ワーカーは提案を返す：`(price, eta, confidence)`ここで価格はトークン、計算単位、またはドルであるかもしれない。
4. マネージャーはウィナー選ぶ（シングルまたはマルチプル、タスクに依存して）オファーを。
5. 拒否されたワーカーは他のタスクに入札無料。

これは100以上のワーカーよりスケーリングします。調整は同期チャットではなく放送と応答。本番で使用：Microsoft Agent Frameworkオーケストレーションパターン、LangGraph実装その一部。

### LLM-ステークホルダー相互作用的交渉

NeurIPS 2024（https://proceedings.neurips.cc/paper_files/paper/2024/file/984dd3db213db2d1454a163b65b84d08-Paper-Datasets_and_Benchmarks_Track.pdf）は**シークレットスコア**と**最小受理閾値**を持つマルチパーティ採点可能ゲームを導入。各ステークホルダーはプライベートユーティリティを持つ。LLMはメッセージから彼ら推論する必要。これは2パーティ取引の一般化Nパーティ連合形成。異種ワーカーケイパビリティを持つ本番タスク市場に関連。

### ナレーション対メカニズムルール

2024-2026年交渉ベンチマークでは、一貫したエンジニアリングルール：

> LLMがナレートしましょう。LLMに提案を計算させない。

提案が数値を必要とする場合（価格、ETA、数量）、交渉状態から決定論的に生成し、LLMが枠取りを生成。提案が構造を必要とする場合（タスク分解、役割割り当て）、LLMが草案を許す。しかしスキーマに対して検証し、送信前に制約チェック。

## ビルド

`code/main.py`実装：

- `ContractNetManager`、`ContractNetTask`、`Bid` — マネージャー+ビッダー、放送cfp、提案を集める、オファー。
- `og_narrator_bargain(state, rng)` — OG-Narratorバイア：決定論的ゼウテン譲歩中点へ。
- `seller_response(state, rng)` — 決定論的売手反提案ポリシー（両スタイルのための構造的確定真実）。
- `naive_llm_bargain(state, rng)` — シミュレート全-LLM交渉者：高分散で価格を選ぶ、しばしばZOPAの外。
- 測定：1000試験にわたる取引率。フレッシュ予約価格は試験毎にサンプリング。

実行：

```
python3 code/main.py
```

期待出力：ナイーブLLM取引率~65-75%。OG-Narrator取引率~85-95%。15-25ポイント間隙は提案生成からナレーションの分解の構造的優位。プラス3つのビッダーと1つのタスクを持つコントラクトネットタスク市場割り当て例。

## 使用

`outputs/skill-bargainer-designer.md`交渉プロトコルを設計：誰が提案生成（決定論的またはLLM）、誰がナレート、どのようにプライベートスクラップパッドはパブリックメッセージから分離、取引率はどう監視。

## 出荷

本番取引チェックリスト：

- **スクラップパッドを分離。** プライベート状態は相手方コンテキストに到達しない。これは交渉不可能。
- **決定論的提案生成。** 価格、数量、ETA：計算、プロンプトしない。
- **すべての着信提案をスキーマに対して検証。** ZOPAの外提案をプロトコル境界で拒否。
- **ラウンドを制限。** 3-5ラウンド最大。仲裁者にデッドロックでエスカレート。
- **取引率とペイオフ分散を継続的に測定。** 落ちる取引率は症状 — しばしばプロンプト漂流または相手方攻撃。
- **すべての拒否提案をログ決定論的根拠を持つ。** コントラクトネットマネージャーについては、敗者ビッダーはなぜ理解する必要。

## 演習

1. `code/main.py`を実行。OG-Narratorがナイーブ-LLM取引率で勝つことを確認。いくら?
2. **ペルソナベースペイオフ改善**実装（arXiv:2402.05863）— バイアはナレーションのみ（「この週末に買うのに絶望」ペルソナを採用）、提案ジェネレータ変更なし。取引率またはペイオフが変化するか?
3. **鎖の思考隠蔽**実装：プライベートスクラップパッド文字列を相手方に通すべきでないを維持。それを誤って漏らした場合何が起こるか（チャネル交換をシミュレート）?
4. 予約価格でN-ビッダー競売にコントラクトネットを拡張。ビッドが全て予約を超える場合、マネージャーはどのように低価格と高品質を決定するか? どのオファードルルを選んで、なぜ?
5. Bhattacharya et al。2025年ハーバード交渉プロジェクトメトリクスを読む。2つのバーガイナー異なるスタイル（攻撃対公正）を実装。対称と非対称ペアアップでペイオフ分散を測定。
