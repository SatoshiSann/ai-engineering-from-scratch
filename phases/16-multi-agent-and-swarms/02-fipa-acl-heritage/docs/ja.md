# FIPA-ACLとスピーチアクトの遺産

> MCP以前、A2A以前、そこはFIPA-ACLの時代だった。2000年、IEEE Foundation for Intelligent Physical Agentsは20の遂行文(performative)、2つのコンテンツ言語、相互作用プロトコル（コントラクトネット、サブスクライブ・ノーティファイ、リクエスト・ウェン）の集合を有するエージェント通信言語を批准した。オントロジーのオーバーヘッドがウェブには重すぎたために業界から消えてしまったが、LLMによる多エージェントシステムの復興は、形式意味論のない同じアイデアを静かに再実装している。JSONコントラクトが遂行文の代わりとなり、自然言語がオントロジーの代わりとなっている。このレッスンではFIPA-ACLを真摯に読むことで、2026年のプロトコル決定のどれが再発明で、どれが新規性を持ち、現在の波がどこで2000年代が既に解決した問題を再発見するのかが見えるようになる。

**タイプ:** Learn
**言語:** Python (stdlib)
**前提条件:** Phase 16 · 01 (Why Multi-Agent)
**所要時間:** ~60 minutes

## 問題

2026年のエージェント・プロトコルのランドスケープは忙しい。ツール用のMCP、エージェント用のA2A、エンタープライズ監査用のACP、分散型信頼用のANP、自然言語コンテンツ用のNLIP、さらにCA-MCPと数十の研究提案がある。各仕様は自分が基盤だと発表している。

正直に見るなら、ほとんどは非常に具体的な20年前の決定木を再発見している。Austin（1962）とSearle（1969）のスピーチアクト理論は「発話は行為である」をもたらした。KQML（1993）はそれをワイヤプロトコルに変えた。FIPA-ACL（2000年に批准）は参照標準化を生成した。20の遂行文、コンテンツ言語SL0/SL1、コントラクトネットとサブスクライブ・ノーティファイ用の相互作用プロトコル。JADEとJACKはJavaの参照プラットフォームだった。この取り組みは2010年前後に、オントロジーのオーバーヘッドが重すぎて、ウェブが勝っていたために消えてしまった。

MCPの`tools/call`、A2Aのタスクライフサイクル、またはCA-MCPの共有コンテキストストアを見ると、あなたはFIPA決定の柔らかい、JSON-ネイティブな焼き直しを見ている。この遺産を知ることで2つのことがわかる。どの新しい「革新」が実は再発明であり、新しい仕様がどの古い失敗モードを再発見するのか。

## コンセプト

### スピーチアクト、1段落で

Austinは、いくつかの文は世界を説明するのではなく、それを変えることに気づいた。「I promise.」「I request.」「I declare.」彼はこれらを遂行文的発話(performative utterances)と呼んだ。Searleは5つのカテゴリーを形式化した。断定的(assertive)、指令的(directive)、約束的(commissive)、表現的(expressive)、宣言的(declarative)。KQML（Finin et al., 1993）はこれをソフトウェアエージェント用に運用可能にした。メッセージは遂行文（アクション）とコンテンツ（アクションの対象）である。FIPA-ACLはKQMLのギャップをクリーンアップし、約20の遂行文の周りを標準化した。

### 20のFIPA遂行文（部分リスト）

| 遂行文 | 意図 |
|---|---|
| `inform` | 「Pがtrueであることをあなたに伝える」 |
| `request` | 「Xを実行するよう求める」 |
| `query-if` | 「Pはtrueか?」 |
| `query-ref` | 「Xの値は何か?」 |
| `propose` | 「Xを実行することを提案する」 |
| `accept-proposal` | 「提案を受け入れる」 |
| `reject-proposal` | 「提案を却下する」 |
| `agree` | 「Xを実行することに同意する」 |
| `refuse` | 「Xを実行することを拒否する」 |
| `confirm` | 「Pがtrueであることを確認する」 |
| `disconfirm` | 「Pを否定する」 |
| `not-understood` | 「あなたのメッセージが解析されませんでした」 |
| `cfp` | 「Xに関する提案の募集」 |
| `subscribe` | 「Xが変わったときに通知してください」 |
| `cancel` | 「進行中のXをキャンセルする」 |
| `failure` | 「Xを試みたが失敗した」 |

完全なリストは`fipa00037.pdf`（FIPA ACLメッセージ構造）にある。ポイントはそれを暗記することではなく、これらのすべてがLLMプロトコルが最終的に再追加する原始要素に対応していることである。

### 正規FIPA-ACLメッセージ

```
(inform
  :sender       agent1@platform
  :receiver     agent2@platform
  :content      "((price IBM 83))"
  :language     SL0
  :ontology     finance
  :protocol     fipa-request
  :conversation-id   conv-42
  :reply-with   msg-17
)
```

7つのフィールドがプロトコルエンベロープを運ぶ。1つのフィールド（`content`）がペイロードを運ぶ。残りのフィールドはリトライ、スレッド化、オントロジーをJSONプロトコルにボルトオンするたびに再発明する正確に同じものである。

### 2つのレガシープラットフォーム

**JADE**（Java Agent DEvelopment framework, 1999–2020s）は最も多く使用されたFIPA準拠ランタイムだった。エージェントはベースクラスを拡張し、ACLメッセージを交換し、コンテナ内で実行され、「行動(behavior)」を使用して調整された。相互作用プロトコルライブラリはコントラクトネット、サブスクライブ・ノーティファイ、リクエスト・ウェン、提案・受諾と共に出荷された。

**JACK**（Agent Oriented Software, commercial）はFIPAメッセージの上のBDI（Belief-Desire-Intention）推論を強調した。より形式的で、採用が少ない。

ウェブスタックが多エージェントユースケースを食べた一度、両方が減少した。MCPとA2Aは2026年のランタイム「コンテナ」である。

### FIPAが消えた理由

- **オントロジーオーバーヘッド。** FIPAは`content`を解析するための共有オントロジーを要求した。オントロジーに同意することは長年の標準プロセスである。ウェブはHTTP+JSONを使用した。
- **誰も使用しなかった形式意味論。** SL（Semantic Language）は厳密な真理条件を与えたが、ほとんどの本番システムは自由形式コンテンツを使用し、形式を無視した。
- **ツーリングロックイン。** JADEはJavaのみ。JACKは商用。ポリグロット・チームは両方の周りを迂回した。
- **インターネットが勝った。** REST、その後JSON-RPC、その後gRPCはACLのトランスポートを置き換えた。

### LLM復興はFIPA-Lite

FIPA `request`をMCP `tools/call`と比較する。

```
(request                                {
  :sender  agent1                         "jsonrpc": "2.0",
  :receiver tool-server                   "method":  "tools/call",
  :content "(lookup stock IBM)"           "params":  {"name":"lookup_stock",
  :ontology finance                                   "arguments":{"symbol":"IBM"}},
  :conversation-id c42                    "id": 42
)                                        }
```

同じエンベロープ、異なる構文。両方が運ぶもの。誰が、誰に、意図、ペイロード、相関id。どちらも相手に対する革命ではなく、同じ設計での異なるトレードオフである。

2025年のLiu et al.による調査（「エージェント相互運用プロトコルの調査：MCP、ACP、A2A、ANP」、arXiv:2505.02279）はこの系統を明示的にする。MCPはツール使用スピーチアクトに対応し、A2Aはエージェント・ピア・スピーチアクトに対応し、ACPは監査証跡スピーチアクトに対応し、ANPは分散ID拡張に対応する。新しい仕様はJSON構文とより柔軟な意味論を持つACLの子孫である。

### トレードオフ、明確に述べる

**FIPAが与えて現代の仕様が落とすもの:**

- 形式意味論 — `inform`は送信者がコンテンツを信じることを意味することを証明できる。
- 遂行文の正規カタログ — 「`cancel`を持つべきか?」を再議論する必要がない。
- 数十年の相互作用プロトコルパターン — コントラクトネット、サブスクライブ・ノーティファイ、提案・受諾 — 既知の正しさの性質を持つ。

**現代の仕様が与えてFIPAが持たなかったもの:**

- JSONネイティブペイロード、すべての最新ツールと互換性がある。
- LLMが手書きオントロジーなしで解釈できる自然言語コンテンツ。
- Webスタック・トランスポート（HTTP、SSE、WebSocket）。
- 自己記述ドキュメント（MCP `listTools`、A2A Agent Card）によるカパビリティ検出。

より容易な実装のためにより緩い意図の意味論。それが正確なトレードオフである。

### 移植する価値のある相互作用プロトコル

FIPAは約15の相互作用プロトコルをシップした。3つはLLM多エージェントシステムに移植する価値がある。

1. **コントラクト・ネット・プロトコル（CNP）。** マネージャーが`cfp`（提案の募集）を発行し、入札者が`propose`で応答し、マネージャーが受け入れ/拒否する。これは正規タスク市場パターンである（Phase 16 · 16 Negotiation）。
2. **サブスクライブ・ノーティファイ。** サブスクライバーが`subscribe`を送信し、パブリッシャーはトピックが変わるたびに`inform`を送信する。これは2026年のすべてのイベントバスである。
3. **リクエスト・ウェン。** 「条件Yが成り立つときXを実行する」。条件付き遅延アクション。2026年のアナログは耐久ワークフローエンジンの遅延タスクである（Phase 16 · 22 Production Scaling）。

それぞれはモダンメッセージキュー、HTTP+ポーリング、またはSSEストリーミングにきれいにマップされます。

### オントロジーを落とすと何が壊れるか

共有オントロジーなしに、エージェントは自然言語コンテンツから意味を推測する。記録された2026年の失敗モードは**意味的ドリフト**である。2つのエージェントが同じ単語（`"customer"`）をわずかに異なる概念に使用し、受信側のエージェントが誤った解釈で行動し、スキーマバリデータがそれをキャッチしない。FIPAのオントロジー要件はメッセージを解析時に却下していたであろう。

フルオントロジーに行かずに軽減する。

- `content`上のJSON Schema — ワイヤで構造エラーを却下する。
- 型付きアーティファクト（A2A） — 誤ったモダリティを却下する。
- エンベロープの明示的な遂行文 — コンテンツが自然言語であっても意図を明確にする。

### 2026年仕様、スピーチアクト遺産にマップされた

| 現代の仕様 | FIPAアナログ | 何を保つか | 何を落とすか |
|---|---|---|---|
| MCP `tools/call` | `request` | 明示的な意図、相関id | 形式意味論、オントロジー |
| MCP `resources/read` | `query-ref` | 明示的な意図、相関id | 形式意味論 |
| A2A タスクライフサイクル | コントラクトネット+リクエスト・ウェン | 非同期ライフサイクル、状態遷移 | 形式完全性保証 |
| A2A ストリーミングイベント | サブスクライブ・ノーティファイ | 非同期プッシュ | 型付き述語サブスクリプション |
| CA-MCP 共有コンテキスト | ブラックボード（Hayes-Roth 1985） | マルチライター共有メモリ | 論理一貫性モデル |
| NLIP | 自然言語コンテンツ | LLMネイティブ | スキーマ |

テーブルを上から下に読むと、パターンはこうだ。構造的原始要素を保つ、形式を落とす、LLMに曖昧さをカバーさせる。

## 作成する

`code/main.py`は純粋なstdlib FIPA-ACL翻訳機を実装する。正規ACLエンベロープをエンコードおよびデコードし、すべてのMCP/A2Aメッセージの形状が同じ7つのフィールドに還元される方法を示す。デモ。

- 5つのMCPスタイルおよびA2AスタイルのメッセージをFIPA-ACLとしてエンコードする。
- FIPA-ACLを最新のものにデコードする。
- 1つのマネージャーと3つの入札者を使用して`cfp`、`propose`、`accept-proposal`、`reject-proposal`を使用してコントラクトネット入札の玩具交渉を実行する。

実行する。

```
python3 code/main.py
```

出力は各モダンメッセージを2026年のJSON形式と FIPA-ACL形式の両方で示すサイドバイサイドトレース、その後コントラクトネット入札のラウンドトリップである。同じプロトコル原始要素がラウンドトリップに残る。構文のみが異なる。

## 使用する

`outputs/skill-fipa-mapper.md`は任意のエージェント・プロトコル仕様を読み、FIPA-ACLマッピングを生成するスキルである。新しいプロトコルを採用する前に使用して、「これは本当に新しいのか、それともJSON構文を持つ`inform`か?」に答える。

## 出荷する

FIPA-ACLを戻さない。そのチェックリストを戻す。

- 各メッセージの意図の原始要素（遂行文）は何か?
- リクエスト・レスポンスとキャンセルのための相関idがあるか?
- 明示的なコンテンツ言語があるか（JSON-RPC、プレーンテキスト、構造化型アーティファクト）?
- 相互作用プロトコルはファーストクラスか、それともスクラッチからコントラクトネットを再実装しているのか?
- 2つのエージェントがコンテンツの意味について同意しないときに何が起こるか（意味的ドリフト）?

本番環境に新しいプロトコルをシップする前に、これらの5つの質問をドキュメント化する。

## 演習

1. `code/main.py`を実行する。ラウンドトリップ・エンコーディングを観察する。どのFIPA遂行文が`tools/call`、`resources/read`、A2Aタスク作成に対応しているかを特定する。
2. マネージャーが入札中にタスクを取り消すことができる`cancel`遂行文でコントラクトネット・デモを拡張する。リトライのみでは解決しない`cancel`はどの失敗ケースを解決するか?
3. FIPA ACLメッセージ構造（http://www.fipa.org/specs/fipa00037/）セクション4.1–4.3を読む。このレッスンで取り上げられていない1つの遂行文を選び、その最新JSON-RPC類似体を説明する。
4. Liu et al., arXiv:2505.02279を読む。MCP、A2A、ACP、ANPのそれぞれについて、保つFIPA遂行文ファミリーと落とすものをリストします。
5. あなた自身のシステムの`request`遂行文の`content`フィールド用の最小JSON-Schemaを設計する。そのスキーマは純粋な自然言語が与えないもので何を与え、それは何を費やすか?

## キーターム

| ターム | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| スピーチアクト | 「何かを実行する発話」 | Austin/Searle: 行為としての発話。ACLの理論的親。 |
| FIPA | 「その古いXMLのもの」 | IEEE Foundation for Intelligent Physical Agents。2000年にACLを標準化。 |
| ACL | 「エージェント通信言語」 | FIPAのエンベロープ形式。遂行文+コンテンツ+メタデータ。 |
| 遂行文 | 「動詞」 | メッセージの意図クラス。`inform`、`request`、`propose`、`cfp`など。 |
| KQML | 「FIPAの前身」 | Knowledge Query and Manipulation Language (1993)。より単純で狭い。 |
| オントロジー | 「共有語彙」 | コンテンツ言語が話す概念の形式的定義。 |
| SL0 / SL1 | 「FIPAコンテンツ言語」 | セマンティック言語レベル0と1 — 形式的コンテンツ言語ファミリー。 |
| コントラクト・ネット | 「タスク市場」 | マネージャーが cfp を発行、入札者が提案、マネージャーが受け入れ/拒否。正規相互作用プロトコル。 |
| 相互作用プロトコル | 「メッセージパターン」 | 既知の正しさを持つ遂行文のシーケンス。リクエスト・ウェン、サブスクライブ・ノーティファイなど。 |
