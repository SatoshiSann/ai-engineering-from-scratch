# ロール・スペシャリゼーション — プランナー、クリティック、エクゼキューター、ベリファイア

> 2026年で最も一般的なマルチエージェント分解。1つのエージェント計画、1つ実行、1つは批判するか検証。MetaGPT（arXiv:2308.00352）はロール・プロンプトとしてエンコードされたSOPとして正式化 — Product Manager、Architect、Project Manager、Engineer、QA Engineer — `Code = SOP(Team)`に従う。ChatDev（arXiv:2307.07924）はデザイナー、プログラマー、レビュアー、テスターを「コミュニケーティブ・デハルュシネーション」を持つ「チャット・チェイン」経由で鎖（エージェント明示的に要求欠けている詳細）。ベリファイアは荷重のベアリング。Cemri et al.（MAST、arXiv:2503.13657）はすべてのマルチエージェント失敗を欠けているか壊れた検証にトレース。PwCは構造化バリデーション・ループから7× 精度ゲイン（10%→70%）をレポート CrewAIの。

**タイプ:** Learn + Build
**言語:** Python (stdlib)
**前提条件:** Phase 16 · 04 (Primitive Model), Phase 16 · 05 (Supervisor)
**所要時間:** ~60 minutes

## 問題

一般的なマルチエージェント・システムは一般的な出力を生成。グループチャットの3つのコーダーは同じ平凡なコードの3つのフレーバーを書く。より多くのエージェントを追加、ラウンドをさらに追加、まだ品質閾値を横断しない。

修正はより多くのエージェントではない — その*異なる*エージェント。個別のロールを割り当てる。プランナーが持たないクリティックにツールを与える。ベリファイアに客観的テスト・スイートを与える。現在のシステムは並列当てずっぽうではなく、根拠のある修正を持つ内部不同意がある。

## コンセプト

### 4つの正規ロール

**プランナー。** ゴールを読み、ステップ・リストまたはスペックを生成。ツール。知識取得、ドキュメント。出力。構造化計画。

**エクゼキューター。** 1つのプラン・ステップを一度に読み、アーティファクトを生成。ツール。実際の作業ツール（コード・コンパイラ、シェル、APIクライアント）。出力。アーティファクト。

**クリティック。** エクゼキューター出力をプランナーの意図に対して読む。ツール。アーティファクトへの読むだけアクセス、静的分析。出力。受け入れ/拒否に理由で。

**ベリファイア。** アーティファクト読み、決定的なチェックを実行。ツール。テスト・ランナー、タイプ・チェッカー、スキーマ・バリデータ。出力。パス/失敗に証拠で。

クリティックは主観的、意見的、しばしば LLM ベース。ベリファイアは客観的、決定的、しばしばコード・ベース。彼らは同じロールではない。

### MetaGPT の SOP パターン

MetaGPT（arXiv:2308.00352）がエンジニアリング SOP をロール・プロンプトとしてエンコード。

- **Product Manager**はPRDを書く。
- **Architect**がシステム設計を生成。
- **Project Manager**タスク分割。
- **Engineer**実装。
- **QA Engineer**テスト実行。

各ロールは厳密な入出力スキーマを持つ。ロール・プロンプトはロールが*である*ことと何を*生成する必要がある*ものを言う。`Code = SOP(Team)`定式化 — 決定的 SOP LLM のチームを予測可能パイプラインに変える。

### ChatDev のコミュニケーティブ・デハルュシネーション

ChatDev は主なムーブを追加。エクゼキューターがプランで与えられなかった具体的な詳細が必要なとき、それは続ける前にデザイナーに名前で明示的に尋ねる。これは詳細を説得的に作り上げる LLM 失敗の古典を防ぐ。

実装。ロール・プロンプトは「あなたが与えられなかった具体的な情報が必要なとき、出力を生成する前に名前でロール要求」を含む。

### ベリファイアがなぜ最も重要か

Cemri et al.（MAST）は1642マルチエージェント実行失敗をトレース。21.3% は検証ギャップだった — システム・シップはだれも確認していない答え。残り79% は「失敗をサイレント実行またははじめて実行されなかった」にしばしばトレース。検証は荷重のベアリング・ロール。

PwCはレポート（CrewAI展開、2025）構造化バリデーション・ループを追加することが精度を10%から70%に移動。7× 1つのロールからのゲイン。

### クリティック vs ベリファイア

- クリティックは品質のアーティファクトをレビューするLLM。主観的。説得的な散文でだまされることができる。
- ベリファイアはアーティファクトで実行する決定的なプログラム。客観的。パス/失敗に証拠を与える。

両方を使う。クリティックはベリファイアが表現できない味問題を捕捉。ベリファイアはランタイムでのみ現れるため、クリティックが見られないバグを捕捉。

### アンチ・パターン

あなたのシステムのすべてのロールはLLM、すべてのロールの出力は「見える良い」。クラシック MAST 失敗モード。少なくとも1つのベリファイア、パス/失敗がLLMではなくコードで決定されることを追加。

### フレームワーク・マッピング

- **CrewAI** — `Agent(role, goal, backstory)`がテキストブック・スペシャリゼーション・サーフェス。
- **LangGraph** — ノードは特化プロンプトを持つことができる。エッジがパイプラインを強制。
- **AutoGen** — ロール固有の ConversableAgent グループチャットの 1 ワード名。
- **OpenAI Agents SDK** — ロール特化エージェント間のハンドオフ・ツール。

## 作成する

`code/main.py`は簡単なPython関数を構築する4ロール・パイプラインを実装。

- **プランナー**仕様を生成。
- **エクゼキューター**コード文字列を生成。
- **クリティック**（LLMシミュレート）明白な問題をフラグ。
- **ベリファイア**テスト・ケース（サンドボックス `exec`）に対して生成されたコードを実行。

デモは2回実行。1回でエクゼキューターが正しいコード（クリティック+ベリファイアの両方がパス）、1回でエクゼキューターがオフスペック・コード（クリティックはバグを見ている説得的に見えるので、ベリファイアはテストが失敗するためそれを捕捉）を生成。

実行する。

```
python3 code/main.py
```

## 使用する

`outputs/skill-role-designer.md`タスクを取り、ロール・ロスター（3-5ロール）、ロール当たりの入出力スキーマ、ベリファイア・チェックを生成。エージェントをフレームワークに配線する前に使用。

## 出荷する

チェックリスト。

- **少なくとも1つの決定的ベリファイア。** 全く LLM ではなく。
- **ロール当たりの明示的なI/O スキーマ。** プランナーは散文ではなくスペックを返す。エクゼキューターはそのスキーマを読む。
- **コミュニケーティブ・デハルュシネーション。** 情報が欠けているとき、エクゼキューターはプランナーに求める。決して作り上げない。
- **クリティック/ベリファイア順序。** クリティックを最初に実行（安い、デザイン問題を捕捉）、ベリファイア2番目（遅い、バグ捕捉）。
- **ループ予算。** 2つのクリティック・エクゼキューター改正ラウンドの前に人間に昇格。最大。

## 演習

1. `code/main.py`を実行してベリファイアがクリティックが逃した方法を観察。`return`の出現をカウント追加の静的分析チェック（ベリファイア）を追加。ランタイム・テストが逃すものを何が捕捉?
2. 5番目のロール追加。「要件アナリスト」ユーザー希望をプランナー・レディ・スペックに翻訳。どの通信デハルュシネーション要求がそれまで流れるべきか?
3. MetaGPT セクション3（「エージェント」）を読む。MetaGPT の5ロールそれぞれの入出力スキーマをリスト。
4. ChatDev のチャット・チェイン図を読む（arXiv:2307.07924図3）。通信デハルュシネーションが別にループをそれは無限ループになる周辺を打つ場所を識別。
5. PwC の 7× 精度ゲインはベリケーション・ループから来た。3つのタスクを推測してベリファイアを追加は助けないどこ — どこ決定的な正しさの確認は不可能または禁止的に費用がかかる。

## キーターム

| ターム | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| ロール・スペシャリゼーション | 「異なるエージェント、異なるジョブ」 | プランナー/エクゼキューター/クリティック/ベリファイア・ロール用チューニング・システム・プロンプト異なる。 |
| SOP パターン | 「エンコード標準操作手順」 | MetaGPT フレーミング。厳密な I/O スキーマ/ロールはチームをパイプラインに変える。 |
| コミュニケーティブ・デハルュシネーション | 「作り上げる前に尋ねる」 | ChatDev パターン。エクゼキューターは詳細が欠けているときプランナーに尋ねず1つを作り上げ。 |
| クリティック | 「LLM レビュアー」 | 主観的、意見的なレビュアー。味問題を捕捉。説得的な散文でだまされることができる。 |
| ベリファイア | 「決定的チェック」 | コード・ベースパス/失敗。テスト・ランナー、タイプ・チェッカー、スキーマ・バリデータ。だまされることはできない。 |
| 検証ギャップ | 「だれも確認していない」 | MAST 失敗の21.3%。答えはバグを捕捉するだろう確認なしに出荷。 |
| 改正ループ | 「クリティックはそれを戻す」 | クリティック否定がエクゼキューター再実行をフィードバックとトリガー。予算が必要。 |
| 全 LLM アンチ・パターン | 「見える良い」 | すべてのロールがLLM、決定的なチェックなし。クラシック MAST 失敗。 |
