# パラレル / スワーム / ネットワーク・アーキテクチャ

> スーパーバイザーとコントラスト。中央決定者なし。エージェントは共有イベント・バスを読、非同期に仕事を選び、結果を書き戻す。LangGraph は明示的に「Swarm Architecture」を分散、動的環境向けサポート。Matrix（arXiv:2511.21686）は分散キューを通じて渡される連続メッセージとして制御とデータ・フローの両方を表す、オーケストレーター・ボトルネックを排除。トレードオフは明示的。決定性とトレーサビリティのためのスケーラビリティ。Swarm は多くの独立サブ問題のタスクに適合。1つの一貫した計画が必要なタスクに適合しない。

**タイプ:** Learn + Build
**言語:** Python (stdlib, `threading`, `queue`)
**前提条件:** Phase 16 · 05 (Supervisor Pattern), Phase 16 · 04 (Primitive Model)
**所要時間:** ~75 minutes

## 問題

スーパーバイザーはいくつかのワーカーにスケール。100 について? スーパーバイザー自体はボトルネック。誰が何をするかについてのあらゆる決定は1つのエージェントを通じてファンネル。1つの遅いプラン・ステップがシステム全体を停止。

Swarm アーキテクチャはデザインを反転。中央プランナーが作業をディスパッチする代わりに、ワーカーは共有キューから作業を選び。「調整」はイベント・バス・セマンティクスにベイク。オーケストレーターなし。キューはシステムがスケール。

## コンセプト

### 形

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

オーケストレーターなし。各ワーカーは繰り返す。仕事を引く、処理、結果を書く（そして optional にフォローアップ・エンキュー）。

### Swarm がフィットするとき

- **多くの独立タスク。** スクレイピング、変形、分類。タスクは互いに依存しない。
- **可変期間作業。** いくつかのタスクが100ms だけ取り、他は10秒、swarm は自動的に負荷バランス — 速いワーカーは次のジョブを引く。スーパーバイザーは期間を期待する必要。
- **決定性より処理量。** あなたは厳密な順序ではなく、総完了時間を気にする。

### Swarm が失敗するとき

- **命令ワークフロー。** ステップ3がステップ2の出力を必要とするとき、swarm はステップ3を生火する危険ステップ2は完了する前に。
- **グローバル計画タスク。** 複雑な研究クエスティオンはプランナーから利益。独立事実のswarm は一貫性のあるレポートを生成。
- **デバッギング。** 中央ログなしで非同期作業で、バグを再現することが費用。

### Matrix（arXiv:2511.21686）

Matrix は swarm をその自然な結論に持つ2025論文。制御フロー両方とデータ・フローは分散キューで連続メッセージ。中央コーディネータなし。フォルト・トレランスはメッセージ耐久性から来。スケーラビリティはメッセージ・ブローカーの問題、システムではない。

貢献。プログラミング・モデルマルチエージェント・調整は「このエージェントが署名するメッセージ・トピックは何か?」むしろ「スーパーバイザーがピックする次のエージェント?」これはシステムをパブ/サブ・イベント・メッシュのように見える。

### LangGraph の Swarm アーキテクチャ

LangGraph 2025ドキュメント明示的に「Swarm Architecture」マルチエージェント・パターンの1つとして説明。エージェントはノード、エッジはサイクルと形成される、どんなノード・プールからサイクルを形成。ワーカーはスーパーバイザー割り当てにより条件により使用可能な作業を選ぶ。

### 失敗モード。飢餓と熱スポット

すべてのワーカーが最もファスト利用可能タスクを引くなら、長時間実行タスク決してだけのとき選ばれ。クラシック・キュー飢餓。

軽減。

- 優先度キューが明示的な経過（年代とともに優先度を増やす）。
- ワーカー特化。いくつかのワーカーのみ「長い」タスク。
- バック圧。速いタスクがどれぐらい多く、キューに入るかをリミット。

### コンテンツ・ベース・ルーティング・リンク

Swarm は自然とコンテンツ・ベース・ルーティング・ペアリング（Lesson 22）。一般的なキューではなく、1つのメッセージ型ごとのキューを持つ。スペシャリスト・ワーカーは型だけをサブスクライブ。これがスケール・メッセージ・バス・アーキテクチャの多くのエージェントのベース。

## 作成する

`code/main.py`は共有`queue.Queue`から引く4ワーカー・スレッドの swarm を実装。タスクの可変期間（いくつかは速い、いくつかは遅い）。デモは比較。

- **連続ベースライン。** 1つのワーカー・プロセス・すべて・タスク・連続。
- **固定割り当て。** 各タスク特定ワーカーに事前割り当て（スーパーバイザー・スタイル）。
- **Swarm。** ワーカーは共有キューから引く。

Swarm は自動的に負荷バランスしない均等に最適。固定割り当ては速いワーカーがアイドル割り当てられたタスク遅いとき葉。

実行する。

```
python3 code/main.py
```

出力は可変期間ワークロード上で swarm がで、ワーカー・タスク・カウント（swarm 不均等にしかし最適に分配）、ウォール・クロック・タイムを示す。

## 使用する

`outputs/skill-swarm-fit.md`がタスク swarm vs スーパーバイザーを使用すべきかを評価。インプット。タスク独立性、期間分散、順序要件、デバッギング・ニード。

## 出荷する

チェックリスト。

- **優先度キューと経過。** ロングタスク飢餓防ぐ。
- **ワーカー冪等性。** タスクが2回以上引かれるかもしれない場合ワーカー・クラッシュ・ミッド・実行。ワーカーは冪等である必要がある。
- **耐久キュー。** 本番用にKafka、Redis Streams、またはデータベース・バックアップ・キューを使用。`queue.Queue` はメモリのみ。
- **タスク/タスク当たりの可観測性。** すべてのタスクがトレース ID を持つ。すべてのワーカーはそれで開始/終了をログに記録。
- **バック圧。** キューがワーカーが排気するより速くプロデューサーが成長するなら、遅いプロデューサー。

## 演習

1. `code/main.py`を実行。swarm は可変期間ワークロード上で連続よりどれぐらい速いか? 固定割り当てより? 
2. 優先度キュー変種を追加（`queue.PriorityQueue`を使用）。タスク「重要性」フィールドで優先度割り当て。連続ロードの下に低優先度タスクは決してスターブするか観察。
3. 熱スポット検出機を実装。ログが最もスローイ・ワーカーより3× より多くタスク・プロセスするあるワーカー。タスク期間分配について何それはインジケート?
4. Matrix 紙（arXiv:2511.21686）抽象とセクション3を読む。Matrix がアクセプト（スケーラビリティ・ゲイン）ということ1つの具体的なトレードオフと1つそれが与え（トレーサビリティ、決定性）を識別。
5. swarm デモを変換`queue.Queue`の（タスク・タイプ、ペイロード）タプル、ワーカーが型だけをサブスクライブ。どんなルーティング・ルール異質なタスクが理にかなっているとき?

## キーターム

| ターム | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| Swarm アーキテクチャ | 「分散エージェント」 | ワーカーが共有キューから引く。中央オーケストレーターなし。 |
| イベント・バス | 「エージェント・トピック・サブスクライブ」 | メッセージ・ブローカー・ルート・タスク・タイプまたはコンテンツ・ワーカーに。 |
| 飢餓 | 「タスク・ネバー・ラン」 | 低優先度タスク決してより高い優先度作業が連続して到着するため選ばれ。 |
| 熱スポット | 「1つのワーカー・ドローンズ」 | 1つのワーカーが最もタスク得ロード・アンバランス。 |
| バック圧 | 「遅いプロデューサー」 | キューが埋まるときアップストリーム・シグナルのメカニズムを止める。 |
| 冪等ワーカー | 「再実行に安全」 | 2回タスク・プロセス・サムと同じ結果。ワーカー・クラッシュ・ミッド・ラン必須。 |
| 耐久キュー | 「ワーカー・クラッシュ・サーバイブ」 | ディスク・また複製・ストレージでバック・キュー。ワーカー・クラッシュ・ロスト・タスクなし。 |
| Matrix フレームワーク | 「完全メッセージ・パッシング・スワーム」 | データとコントロール・フロー両方分散キューで連続メッセージ。 |
