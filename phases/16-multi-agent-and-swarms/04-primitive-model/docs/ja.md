# マルチエージェント原始モデル

> 2026年にシップしているすべてのマルチエージェント・フレームワーク — AutoGen、LangGraph、CrewAI、OpenAI Agents SDK、Microsoft Agent Framework — は4次元設計空間の1点である。4つの原始要素、それ以上：エージェント、ハンドオフ、共有状態、オーケストレーター。このレッスンはゼロからそれらを構築し、4つすべての玩具システムを実行してから、すべての主要フレームワークを同じ軸にマップし、1段落で新しいリリースを読むことができるようにする。

**タイプ:** Learn
**言語:** Python (stdlib)
**前提条件:** Phase 14 (Agent Engineering), Phase 16 · 01 (Why Multi-Agent)
**所要時間:** ~60 minutes

## 問題

6ヶ月ごとに新しいマルチエージェント・フレームワークが出荷される。2023年のAutoGen。2024年のCrewAI。2024年のLangGraphとOpenAI Swarm。2025年4月のGoogle ADK。2026年2月のMicrosoft Agent Framework RC。各プレスリリースは「正しい抽象化」であると請求する。

一度に1つ学ぶことを試みたら、あなたは燃え尽きる。APIは異なる。ドキュメントは「エージェント」の定義について同意していない。1つのフレームワークは共有メモリを「ブラックボード」と呼び、別のそれを「メッセージプール」と呼び、3番目は「StateGraph」と呼ぶ。あなたは分野が単に解雇していることを疑い始める。

ではない。マーケティングの下、4つの原始要素は安定している。一度それらを学び、1段落で新しいフレームワークを読む。

## コンセプト

### 4つの原始要素

1. **エージェント** — システムプロンプトとツールリスト。ステートレス。すべての実行はそのシステムプロンプトと現在のメッセージ履歴から始まる。
2. **ハンドオフ** — あるエージェントから別のエージェントへの制御の構造化転送。機械的には、新しいエージェントを返すツール呼び出しまたは条件に従うグラフエッジ。
3. **共有状態** — 複数のエージェントが読む（時々書く）ことができるデータ構造。メッセージプール、ブラックボード、キー・バリュー・ストア、ベクトル・メモリ。
4. **オーケストレーター** — 次に誰が話すかを決定する人。オプション。明示的なグラフ（確定的）、LLMスピーカー・セレクタ（柔軟）、最後のスピーカーのハンドオフ呼び出し（OpenAI Swarm）、キューでのスケジューラ（swarm architecture）。

それは全設計空間である。すべてのフレームワークは各軸のデフォルトを選択する。残りはサーフェス構文である。

### すべての2026フレームワークがそれにマップする方法

| フレームワーク | エージェント | ハンドオフ | 共有状態 | オーケストレーター |
|-----------|-------|---------|--------------|--------------|
| OpenAI Swarm / Agents SDK | `Agent(instructions, tools)` | ツールがエージェント返す | 呼び出し側の問題 | LLMの次のハンドオフ呼び出し |
| AutoGen v0.4 / AG2 | `ConversableAgent` | GroupChatでスピーカー・セレクタ | メッセージプール | セレクタ関数（LLMまたはラウンドロビン） |
| CrewAI | `Agent(role, goal, backstory)` | `Process.Sequential / Hierarchical` | チェーンされたタスク出力 | マネージャーLLMまたは静的順序 |
| LangGraph | ノード関数 | グラフエッジ+条件 | `StateGraph` レデューサ | グラフ、確定的 |
| Microsoft Agent Framework | エージェント+オーケストレーション・パターン | パターン固有 | スレッド/コンテキスト | パターン固有 |
| Google ADK | エージェント+A2Aカード | A2Aタスク | A2Aアーティファクト | ホストが決定 |

サーフェス違いは巨大に見える。下：同じ4つのノブ。

### なぜこれが重要か

原始要素を見ると、フレームワーク比較は短いチェックリストになる。

- オーケストレーターはLLMにルーティングを信頼するか（Swarm）またはコードにルーティングを固定するか（LangGraph）?
- 共有状態は完全履歴（GroupChat）か投影されるか（StateGraphレデューサ）?
- エージェントは互いのプロンプトを変更できるか（CrewAIマネージャー）または単にハンドオフするか（Swarm）?

これら3つの質問はどのフレームワークが与えられた問題に適合するかの80%に答える。あなたは「最高のマルチエージェント・フレームワーク」を買うのをやめ、実際に気になるアクシスのために設計を始める。

### ステートレス・インサイト

共有状態を除く、すべての原始要素はステートレスである。エージェントは（プロンプト、ツール）の関数である。ハンドオフは関数呼び出しである。オーケストレーターはスケジューラである。**唯一のステートフル・ものはシステムの共有状態である。** すべての興味深いバグがここに生きる。メモリ中毒（Lesson 15）、メッセージ順序、バージョン、書き込み競合。

共有状態を隠すフレームワーク（Swarm）は問題を呼び出し側に押す。それを集約するフレームワーク（LangGraph チェックポイント、AutoGen プール）はそれを検査可能にするが調整コストを共有状態実装にシフトさせる。

### 単一の原始要素の解剖

#### エージェント

```
Agent = (system_prompt, tools, model, optional_name)
```

メモリなし。状態なし。同じシステムプロンプトとツールを持つ2つのエージェントは相互交換可能である。ペアごとのエージェント状態のように見えるすべてのものは、実際には共有状態またはハンドオフプロトコルの中にある。

#### ハンドオフ

```
Handoff = (from_agent, to_agent, reason, payload)
```

3つの実装が支配する。

- **関数返回** — ツールは次のエージェントを返す。これはOpenAI Swarmパターンである。エージェントはツールスキーマでルーティングを運ぶ。
- **グラフエッジ** — LangGraph。エッジは宣言的である。LLMは値を生成。条件は次のノードを選択する。
- **スピーカー・セレクション** — AutoGen GroupChat。セレクタ関数（時々LLM呼び出し自体）はプールを読み、次のスピーカーを選ぶ。

#### 共有状態

```
SharedState = { messages: [], artifacts: {}, context: {} }
```

最小限では、メッセージのリスト。しばしばより多く。構造化アーティファクト（CrewAIタスク出力）、型付きコンテキスト（LangGraphレデューサ）、外部メモリ（MCP、ベクトルDB）。

2つのトポロジー。**フルプール**（すべてのエージェントがすべてのメッセージを見る）と**投影**（エージェントはロール・スコープビュー）。フルプールは単純でスケール悪い。投影プールはスケールが、前もってスキーマ設計が必要。

#### オーケストレーター

```
Orchestrator = ({state, last_speaker}) -> next_agent
```

4つの味。

- **静的** — グラフはビルド時に固定（LangGraph確定的、CrewAI Sequential）。
- **LLM選択** — LLMはプールを読み、次のスピーカーを選ぶ（AutoGen、CrewAI Hierarchical）。
- **ハンドオフ駆動** — 現在のエージェントはハンドオフ・ツール呼び出しで決める（Swarm）。
- **キュー駆動** — ワーカーは共有キューから引く。明示的な次スピーカーなし（swarm architectures、Matrix）。

### フレームワーク間で何が変わるか

原始要素が固定されると、残っている設計決定はこれ。

- **メモリ戦略** — エフェメラル vs 耐久チェックポイント（LangGraph checkpointer）。
- **セーフティ・バウンダリ** — ハンドオフを承認できる人（人間・ループ）。
- **コスト会計** — エージェントごとのトークン予算。
- **可観測性** — ハンドオフの追跡、リプレイのための状態の永続化。

すべて原始要素の上に実装可能。これらのいずれも新しい原始要素ではない。

## 作成する

`code/main.py`は約150行のstdlib Pythonで4つの原始要素を実装する。実リLLMなし — 各エージェントはスクリプト化ポリシーなので焦点は調整構造に留まる。

ファイルはこれをエクスポート。

- `Agent` — 名前、システムプロンプト、ツール、ポリシー関数のデータクラス。
- `Handoff` — 新しいエージェントを返す関数。
- `SharedState` — スレッドセーフなメッセージプール。
- `Orchestrator` — 3つの変種。`StaticOrchestrator`、`HandoffOrchestrator`、`LLMSelectorOrchestrator`（シミュレート）。

デモは同じ3エージェント・パイプライン（研究→書く→レビュー）をすべての3つのオーケストレーター・タイプ経由で実行し、最後のメッセージプールを印刷する。出力は*次の誰が選ぶか*のみで異なることが見える。エージェントと共有状態は実行全体で同じである。

実行する。

```
python3 code/main.py
```

予想出力。3つのオーケストレーター実行、1つずつパターンあたり。各々は最終メッセージプールを印刷する。ハンドオフ駆動実行は研究者がそれがした早く決める場合より少ないエージェントに達する — LLMルーティング・トレードオフは小さい。

## 使用する

`outputs/skill-primitive-mapper.md`は任意のマルチエージェント・コードベース上のスキルまたはフレームワーク・ドキュメント読み4原始要素マッピングを返す。新しいフレームワーク・リリース上で実行してドキュメントを深く読む前に1段落の理解を得る。

## 出荷する

新しいフレームワークを採用する前に、原始要素マッピングを書く。できなければ、ドキュメントは不完全またはフレームワークは5番目の原始要素を発明している（稀 — 見たことのない共有状態味をチェック）。

マッピングをアーキテクチャ・ドキュメントにピン留めする。新しいチーム・メンバーが参加するとき、APIドキュメント前にそれらにマッピングを送信する。フレームワーク・バージョンが変わるとき、チェンジログではなくマッピングをdiff。

## 演習

1. `code/main.py`を3回異なるエージェント・ポリシーで実行する。オーケストレーター選択が実行するエージェントを変えるのを観察する。
2. 4番目のオーケストレーター・タイプを実装。共有状態から作業をポーリングするキュー駆動。どのデッドロックが起こることができ、どのようにあなたはそれを検出するか?
3. LangGraph クイックスタート（https://docs.langchain.com/oss/python/langgraph/workflows-agents）を取り4つの原始要素として書き直す。LangGraphの抽象化の4つのうち、どれが1：1でマップしおよびどれが便利ラッパーか?
4. OpenAI Swarm クックブック（https://developers.openai.com/cookbook/examples/orchestrating_agents）を読む。Swarmが最もエルゴノミック4つの原始要素を識別し、どれがそれを呼び出し側に押す。
5. このテーブルで共有状態を完全に隠すフレームワークを1つ見つける。ハンドオフ再読むことなく調整する必要があるときに何が壊れるか説明する。

## キーターム

| ターム | 人々が言うこと | 実際には何を意味するか |
|------|----------------|------------------------|
| エージェント | 「ツール付きLLM」 | `(system_prompt, tools, model)`トリプル。ステートレス。 |
| ハンドオフ | 「制御転送」 | 次のエージェントと optional ペイロードを名前にする構造化呼び出し。3つの実装。関数返回、グラフエッジ、スピーカー・セレクション。 |
| 共有状態 | 「メモリ」/「コンテキスト」 | マルチエージェント・システムの唯一のステートフル部分。メッセージプールまたはブラックボード。 |
| オーケストレーター | 「コーディネーター」 | 次に誰が実行するかを決める人。静的グラフ、LLMセレクタ、ハンドオフ駆動、またはキュー駆動。 |
| 原始要素 | 「抽象化」 | すべてのフレームワークがパラメータ化する4軸の1つ。フレームワーク・フィーチャーではない。 |
| メッセージプール | 「共有チャット履歴」 | 完全履歴共有状態。理由付けは容易、スケール悪い。 |
| 投影状態 | 「スコープビュー」 | 共有状態へのロール固有ビュー。スケール、スキーマ設計が必要。 |
| スピーカー・セレクション | 「次は誰が話すか」 | グループからオーケストレーター・パターンで関数（しばしばLLM）が次のエージェントを選ぶ。 |
