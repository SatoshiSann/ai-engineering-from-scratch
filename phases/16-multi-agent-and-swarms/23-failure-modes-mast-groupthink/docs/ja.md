# 失敗モード — MAST、グループシンク、モノカルチャー、カスケード エラー

> 2026年の参照分類は**MAST** (Cemri et al., NeurIPS 2025, arXiv:2503.13657)、7つの最先端のオープンソースMAS全体で1642実行トレースから派生し、**41–86.7%失敗レート**を示しています。3つのルートカテゴリー：**仕様の問題**（41.77%）—役割の曖昧さ、不明確なタスク定義。**協調失敗**（36.94%）—通信の破裂、状態の非同期。**検証ギャップ**（21.30%）—検証の欠落、品質チェックの不在。**グループシンク**ファミリー(arXiv:2508.05687)を追加：モノカルチャー崩壊（同じベースモデル→相関エラー）、適合バイアス（エージェントは他者のエラーを強化）、欠陥心の理論、混合動機ダイナミクス、カスケード信頼性の失敗。カスケード例：再試行嵐ここで支払い失敗トリガー順序の再試行、在庫の再試行、在庫サービスを圧倒する（10倍負荷秒内—回路ブレーカーが必要です）。メモリ毒：1つのエージェントの幻覚が共有メモリに入り、下流エージェントは事実として扱う。精度は段階的に低下し、ルート原因の診断は痛い。**STRATUS** (NeurIPS 2025) は、特化した検出/診断/検証エージェント経由で1.5倍の軽減成功の改善を報告します。このレッスンは失敗モードを第1級のエンジニアリング目標として扱います。

**タイプ:** 学習
**言語:** Python (stdlib)
**前提条件:** フェーズ16 · 13 (共有メモリ)、フェーズ16 · 14 (合意とBFT)、フェーズ16 · 15 (投票と討論の位相)
**所要時間:** 約75分

## 問題

マルチエージェントシステムは実際のタスクで41-86.7%の時間失敗します（Cemri et al. 2025は7つのオープンソースMAS全体でこれを測定）。それは「さらに多くのエージェントを追加するだけ」でデバッグ可能ではありません。失敗は構造的な原因があります。MAST分類は、カテゴリーを与えます。このレッスンは各カテゴリーを具体的な検出、診断、軽減パターンにマップしていただき、数字は任意のように見えることを停止します。

2026年の本番練習は失敗モードを設計入力として扱うことです。アーキテクチャは「十分」ではなく、各MASカテゴリーと展開した軽減を指名できるまで。

## コンセプト

### MASTカテゴリ

**仕様の問題（失敗の41.77%）。** エージェントのタスクは十分に厳密に定義されていません。例：

- 役割の曖昧さ：2つのエージェントは両方がレビューアーと思っています。
- タスク under-specified：「これを要約」ユーザーが特定の角度を望んだとき。
- 成功基準隠し：エージェントはそれが成功したかどうかを確認することはできません。

軽減：
- 明示的なロールコントラクトを書く。各エージェントのプロンプトはそれが何をするか**をしないか**述べます。
- 各タスクの受け入れテスト。エージェントが開始する前に、「完了は x に見える」を定義します。
- 飛行前仕様チェック：別のエージェントは配置前にタスク定義をレビュー。

**協調失敗（36.94%）。** 通信またはステートの崩壊。

例：
- 2つのエージェントは同期なしに共有状態を更新します。
- メッセージはエージェント間で失われました（キュー失敗、タイムアウト）。
- 状態ドリフト：エージェントAはタスクが完了。エージェントB は依然として実行しています。

軽減：
- バージョン付き共有状態で楽観的な同時実行。
- 重要なメッセージのための明示的な確認（acked まで再試行）。
- 定期的な状態シンクチェックポイント。早くドリフトを検出します。

**検証ギャップ（21.30%）。** 出力での独立した確認なし。

例：
- 1つのエージェントは成功を請求します。誰も検証しません。
- エージェントの連鎖は各前のを信頼します。
- テスト注射は出現する組成行動を見落としていますか。

軽減：
- 独立した検証エージェント（Lesson 13）。読み取り専用、独立したソースアクセス。
- 明示的な相渡契約：「Aの出力は Bが開始する前にチェッカー Cをパスするべき」。
- 結果ログを事後分析用に記録します。

### グループシンク家族 (arXiv:2508.05687)

エージェントが同種化または模倣するとき、関連する5つの失敗：

**モノカルチャー崩壊。** 同じベースモデルまたは訓練データ→相関エラー。3つのエージェントが LLM を共有するときに、彼らはその幻覚を共有します。

**適合バイアス。** エージェントは最も大声または最も自信の仲間に向かって調整、誤った場合でも。

**欠陥ToM。** エージェントは相互の信念をモデル化に失敗。協調は崩壊します（Lesson 18）。

**混合動機ダイナミクス。** 部分的に整列インセンティブを持つエージェントは誰も満たさない妥協中央に向かってドリフト。

**カスケード信頼性の失敗。** 1つのコンポーネントのエラーパターンは従属コンポーネント内のエラーパターンをトリガー。

### カスケード例—再試行嵐

古典的な2026年インシデントパターン：

```
支払いサービスは要求の10%が失敗
   ↓
オーダーエージェントが支払いを再試行（指数バックオフしかし素朴）
   ↓
各再試行は新しいオーダーインベントリチェック
   ↓
在庫サービスが通常負荷の2倍を見ます
   ↓
在庫サービスはタイムアウトを開始
   ↓
すべてのオーダーが在庫チェックを再試行
   ↓
在庫サービスは通常負荷の10倍を見ます
   ↓
クラスターが下がります
```

修正は古典的です：**回路ブレーカー。** 下流エラーレートが閾値を超えても短い回路でキャッシュまたはデフォルト結果で。さらに再試行予算は要求あたりキャップ。

回路ブレーカーは、ほんの少数の多エージェント失敗の軽減の1つは修正なしで分散システムから直接借ります。

### メモリ毒（再訪）

Lesson 13から：1つのエージェント幻覚は共有メモリの事実になります。下流エージェントは毒事実で推論します。MAST用語では、これは共有メモリ層での検証ギャップです。

段階的な精度低下は症状です。クラッシュをしません。遅いドリフトを取得する事前根本原因診断が困難です。

軽減：追記ログ、発信元、書き込み不可能検証機。Lesson 13でカバー。

### STRATUS — 失敗検出用の特化したエージェント

STRATUS (NeurIPS 2025) は軽減成功の1.5倍向上を報告する場合：

- **検出エージェント。** 症状パターンを監視（高い不同意、再試行スパイク、精度ドリフト）。
- **診断エージェント。** 症状を与えられて、MASTの分類から推測可能な根本原因。
- **検証エージェント。** 軽減が適用された後、症状がクリアされることを確認。

これはSREスタイルのインシデント応答で、エージェントシステムに適用されます。3つのロールはすべて特化したプロンプトを持つLLMエージェントできます。

### 失敗モード監査

2026年のベストプラクティスは年次（または主要リリースごと）の失敗モード監査です：

1. **トレースサンプル。** 実行トレース~1000をコレクト。
2. **分類。** 各トレースの失敗のため、MASTおよびグループシンクカテゴリにマップします。
3. **カテゴリー別失敗率を計算。** どのカテゴリーがシステムを支配していますか？
4. **ランク軽減。** どの修正が最も失敗を排除していますか？
5. **2-3軽減を選択。** 実装。次の四半期にアウディット。

規律は特定の選択より重要です。監査なしで、失敗はノイズに混ざり、体系的に対処されることはありません。

### システムが無声で失敗するとき

最も危険な失敗カテゴリーは無声正確性失敗です。システムは大声で失敗（クラッシュ、例外、警告）監視できます。システムが妥当なが誤った出力を生成するのは検出できません。これはなぜ検証ギャップが失敗カウント21.30%だけであるとしても最も高価な種理由です。

投資：
- サンプルベースの人間のレビュー。
- ゴールデンデータセット回帰テスト。
- 重要な出力の相互エージェント相互検査。

### 失敗対遅い失敗

一部の失敗は即座。一部は遅い。即座の失敗（タイムアウト、スキーマ不一致、auth エラー）は検出が安いです。遅い失敗（メモリ毒、モノカルチャードリフト、役割曖昧さ）は検出と防止に高価です。

2026年のエンジニアリングの動き：遅い失敗プロキシを計器して、目に見えるエラー前にドリフトをキャッチできます。同意率、再試行率、出力長分布、連続エージェントバージョン間の編集距離はすべて有用なプロキシです。

## 構築する

`code/main.py`を実装：

- `FailureTaxonomy` — シミュレートされたインシデントをMASTおよびグループシンクカテゴリに分類します。
- `CircuitBreaker` — 古典的なパターン。エラーレートが閾値を超えても開きます。
- `RetryStormSimulator` — カスケード失敗を示す。回路ブレーカー開/閉を切り替えます。
- `DetectionAgent` — STRATUSスタイル症状マッチャースクリプト。

実行：

```
python3 code/main.py
```

予想される出力：
- 回路ブレーカーなし再試行嵐：在庫エラーが吹き上がります（シミュレート）。
- 回路ブレーカー付き：閾値をキャップ。低下モード応答を提供されました。
- 検出エージェントはパターンにフラグを立てMASTカテゴリーに名前を付けます。

## 使用方法

`outputs/skill-mast-auditor.md`はマルチエージェントシステムでMASTスタイル失敗モード監査を実行します。トレース→分類→軽減ランキング。

## 出荷する

本番環境の失敗モード規律：

- **MAST監査ごとの四半期。** 年次ではありません。カテゴリーはシステムが成長したようにシフト。
- **回路ブレーカーはいたるところ。** 各アウトバウンド呼び出しは任意の従属サービス。デフォルトは開く閾値5-10%エラーレート。
- **ゴールデンデータセット。** 小さい、高品質、手で監査。週ごとに回帰テストして対抗。
- **STRATUSトリオ。** 検出+診断+検証エージェント本番を監視。最初は検出エージェントのみで開始。診断を追加症状がノイズがあるとき。
- **失敗予算。** カテゴリー別失敗レートのための明示的なSLO。予算超過のトリガーは出荷停止会話。

## 演習

1. `code/main.py`を実行します。回路ブレーカーが再試行嵐をキャップすることを確認。失敗閾値を変更して相互作用を観察。
2. **遅い失敗プロキシ**を実装：3つの並行エージェント全体で同意レート。それが急落するとき、アラートトリガー。モノカルチャードリフトシミュレーション段階的にエージェント出力を関連させます。
3. Cemri et al. (arXiv:2503.13657)を読んでください。彼らの7 MASシステムの1つを選択し、トップ3の失敗カテゴリにマップ。これらはMAST予測とどのように比較していますか？
4. グループシンク紙(arXiv:2508.05687)を読んでください。5つのパターンのどれが本番環境で検出するのが最も難しいかを識別。プロキシメトリクスを提案。
5. 特定のマルチエージェントシステムでSTRATUSスタイル検出診断検証トリオを設計してください。どのような症状を検出しますか？どのような軽減について診断はお勧めしますか？検証はそれらがどのように機能を確認しますか？
