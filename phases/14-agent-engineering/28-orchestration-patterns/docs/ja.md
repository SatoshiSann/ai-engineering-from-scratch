# オーケストレーション パターン: スーパーバイザー、スワーム、階層型

> 4 つのオーケストレーション パターンが 2026 年フレームワークを通じて繰り返されます: スーパーバイザー ワーカー、スワーム / ピア ツー ピア、階層型、ディベート。Anthropic のガイダンス: 「あなたのニーズに合う適切なシステムを構築することです」。シンプルに開始; 単一エージェントに 5 つのワークフロー パターンで不十分な場合のみトポロジーを追加します。

**タイプ:** Learn + Build
**言語:** Python (stdlib)
**前提条件:** Phase 14 · 12 (Workflow Patterns)、Phase 14 · 25 (Multi-Agent Debate)
**所要時間:** 約 60 分

## 学習目標

- 4 つの反復するオーケストレーション パターンにそれぞれいつ適用するかに名前を付けることができる。
- 2026 年の LangChain 推奨事項: ツール呼び出しベースのスーパービジョンとスーパーバイザー ライブラリを説明できる。
- Anthropic の「適切なシステムを構築」ルール、およびそれがトポロジー選択をどのようにゲートするかを説明できる。
- すべての 4 つを stdlib に対して共通のスクリプト化された LLM に対して実装できる。

## 問題

チームは必要になる前に「マルチエージェント」に手を出します。 4 つのパターンがフレームワークを通じて繰り返されます; それらに名前を付けることができたら、正しいパターンを選択できます - またはトポロジーを完全にスキップできます。

## コンセプト

### スーパーバイザー ワーカー

- 中央ルーティング LLM がスペシャリスト エージェントにディスパッチします。
- 判定: 自分にループバック、スペシャリストにハンドオフ、終了。
- スペシャリストは互いに話しません; すべてのルーティングはスーパーバイザーを通じて行きます。

フレームワーク: LangGraph `create_supervisor`、Anthropic オーケストレーター ワーカー、CrewAI Hierarchical Process。

**2026 LangChain 推奨事項:** `create_supervisor` ではなく直接ツール呼び出しでスーパービジョンを実行します。より細かなコンテキスト エンジニアリング制御を提供 - 各スペシャリストが正確に何を見るかを決定できます。

### スワーム / ピア ツー ピア

- エージェントは共有ツール面を通じて直接ハンドオフします。
- 中央ルーターなし。
- スーパーバイザーより低レイテンシー (ホップが少ない)。
- 推論が難しい (単一制御ポイントがない)。

フレームワーク: LangGraph スワーム トポロジー、OpenAI Agents SDK ハンドオフ (すべてのエージェントが他のすべてのエージェントにハンドオフできる場合)。

### 階層型

- サブスーパーバイザーを管理するスーパーバイザーがワーカーを管理。
- LangGraph ネストされたサブグラフとして実装; CrewAI ネストされたクルーとして。
- 運用の複雑さの代償で、大規模なエージェント母集団にスケール。

必要な場合: 単一スーパーバイザーのコンテキスト予算がすべてのスペシャリストの説明を保持できない場合。

### ディベート

- 並列提案者 + 反復的なクロス クリティーク (Lesson 25)。
- 本当のオーケストレーションではない - むしろ検証 - しかしフレームワークのトポロジー選択として表示されます。

### CrewAI Crew vs Flow

CrewAI は 2 つのデプロイメント モードを形式化します:

- **Flow** - 決定論的イベント駆動オートメーション用 (本番環境用の推奨スタート ポイント)。
- **Crew** - 自律的なロールベースコラボレーション用。

これは上記の 4 つのパターンに直交していますが、トポロジーにマップします: Flow は通常スーパーバイザーまたは階層型; Crew は通常 LLM ルーター付きのスーパーバイザーです。

### Anthropic のガイダンス

「LLM スペースでの成功は、最も洗練されたシステムの構築についてではありません。あなたのニーズに合う適切なシステムの構築についてです」。

決定順序:

1. 単一エージェント + ワークフロー パターン (Lesson 12) - ここから開始します。
2. スーパーバイザー ワーカー - 2-4 人のスペシャリストがいる場合。
3. スワーム - レイテンシーが推論の明確さより重要な場合。
4. 階層型 - スーパーバイザー コンテキスト予算が失敗する場合のみ。
5. ディベート - 精度がコストより重要な場合。

### このパターンが失敗する場所

- **トポロジー優先の思考。** マルチエージェントが解決する問題を特定する前に「マルチエージェントが必要」。
- **スワームでのバウンシング ハンドオフ。** A -> B -> A -> B。ホップ カウンターを使用します。
- **フェイク階層。** 「エンタープライズ」だから 3 層; 2 つの実際のチーム。 折りたたむ。

## Build It

`code/main.py` はスクリプト化された LLM に対して stdlib のすべての 4 つのパターンを実装します:

- `Supervisor` - 中央ルーター。
- `Swarm` - ピア ツー ピア直接ハンドオフ。
- `Hierarchical` - スーパーバイザーのスーパーバイザー。
- `Debate` - 並列提案者 + クリティーク。

各パターンは同じ 3 つの意図タスク (払い戻し / バグ / 営業) を処理します。トレース形状が異なります。

実行:

```
python3 code/main.py
```

出力: パターンごとのトレース + オペレーション カウント。 スーパーバイザーが最もクリーン; スワームが最短; 階層型が最も深い; ディベートが最も高価。

## Use It

- **LangGraph** - スーパーバイザーと階層型 (ネストされたサブグラフ)。
- **OpenAI Agents SDK** - ツールとしてのハンドオフ (スーパーバイザー形)。
- **CrewAI Flow** - 本番環境の決定論的。
- **Custom** - ディベート用またはあなたが正確な制御を望む場合。

## 演習

1. スーパーバイザー ワーカーをスワームに変換します。ルーターを削除します。 何が壊れますか? 何が改善されますか?
2. スワームにホップ カウンターを追加: 3 つのハンドオフ後に拒否。 A->B->A のバウンシングをキャッチしますか?
3. 12 スペシャリスト ドメイン用の 2 レベルの階層型システムを構築します。 ネスティングなしでコンテキスト予算が失敗する場所はどこですか?
4. 本番環境形のワークロードで 4 つのパターンをプロファイルします。 どれが各メトリック (レイテンシー、コスト、精度、デバッグ可能性) で勝ちますか?
5. Anthropic の「Building Effective Agents」投稿を読みます。 各本番環境フローを 4 つのうち 1 つにマップします。 きれいにマップしないものはありますか?

## 主要用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|----------------|----------|
| スーパーバイザー ワーカー | 「ルーター + スペシャリスト」 | 中央 LLM がスペシャリストにディスパッチ; 互いに話さない |
| スワーム | 「ピア ツー ピア」 | 共有ツール経由の直接ハンドオフ; 中央ルーターなし |
| 階層型 | 「スーパーバイザーのスーパーバイザー」 | 大規模な母集団用のネストされたサブグラフ |
| ディベート | 「提案者 + クリティーク」 | 並列提案者、クロス クリティーク (Lesson 25) |
| ツール呼び出しベースのスーパービジョン | 「ライブラリなしのスーパーバイザー」 | コンテキスト制御用の直接ツール呼び出しとしてスーパーバイザーを実装 |
| Crew | 「自律的なチーム」 | CrewAI のロールベースコラボレーション モード |
| Flow | 「決定論的ワークフロー」 | CrewAI のイベント駆動本番環境モード |
