# LangGraph：ステートフルグラフと永続実行

> LangGraphは低レベルステートフルオーケストレーションの2026年参照。エージェントは状態機械；ノードは関数；エッジは遷移；状態は不変ですべてのステップ後にチェックポイントされる。失敗からちょうどそこから正確に再開。

**タイプ:** 学習 + ビルド
**言語:** Python（stdlib）
**前提条件:** Phase 14 · 01（エージェントループ）、Phase 14 · 12（ワークフローパターン）
**所要時間:** 約75分

## 学習目標

- LangGraphの中核モデルを説明できる：不変状態、関数ノード、条件付きエッジ、ステップ後チェックポイントの状態機械。
- ドキュメントが強調する4つの機能を名前付けできる：永続実行、ストリーミング、人間ループ内、包括的メモリ。
- LangGraphが支持する3つのオーケストレーション位相を説明できる：スーパーバイザー、ピアツーピア（スワーム）、階層的（ネストされたサブグラフ）。
- 不変状態、条件付きエッジ、チェックポイント/再開サイクルを持つstdlibステートグラフを実装できる。

## 問題

エージェントとワークフローは問題を共有：40ステップ実行がステップ38で失敗するとき、ステップ38から再開を望む、開始ではなく。2次的な状態モデルは、フレッシュ実行を想定するライブラリの周辺に再試行をハック化する。

LangGraphの設計答え：状態は1次クラスのタイプオブジェクト、変異は明示的、すべてのノード後にチェックポイント永続化。再開は`load_state(session_id)`呼び出し。

## コンセプト

### グラフ

グラフは以下で定義される:

- **状態タイプ。** 型付きdict（またはPydanticモデル）、すべてのノードが読み取りと変異。
- **ノード。** 純関数`(state) -> state_update`。更新は復帰後の状態にマージ。
- **エッジ。** ノード間の条件付きまたは直接遷移。
- **入出力。** `START`と`END`センチネルノードが境界をマーク。

例：`classify`、`refund`、`bug`、`sales`、`done`ノードを持つエージェント—ワークフローとしてのルーティング。

### 永続実行

各ノード復帰後、ランタイムは状態をシリアル化し、チェックポインタに書き込む（SQLite、Postgres、Redis、カスタム）。ステップNで失敗した場合、ランタイムは`resume(session_id)`でき、正確な状態でステップN+1から選ぶ。

LangGraphドキュメントは明示的に本番ユーザーをハイライト、これが重要：Klarna、Uber、J.P. Morgan。主張はグラフ形状ではない；グラフ形状とチェックポイント作成は回復を安価にする。

### ストリーミング

すべてのノードが部分出力を獲得できる。グラフはノードデルタイベントごとにストリーム、呼び出し側へのUIはグラフ実行として更新。

### 人間ループ内

ノード間の検査と修正状態。実装：重要なノード前にポーズ、状態を人間に表面化、修正を受け入れ、再開。チェックポインタはこれを容易にする状態が既にシリアル化されているため。

### メモリ

短期（実行内—会話履歴状態）と長期（実行間—チェックポインタ経由永続化＋別の長期ストア）。LangGraphは外部メモリシステム（Mem0、カスタム）とツール経由で統合。

### 3つの位相

1. **スーパーバイザー。** 中央ルータLLMが専門サブエージェントにディスパッチ。`langgraph-supervisor`の`create_supervisor()`（ただし、LangChainチーム2026は、より多くのコンテキスト制御のためにツール呼び出し経由で直接これを行うことを推奨）。
2. **スワーム / ピアツーピア。** エージェントが共有ツール表面経由で直接ハンドオフ。中央ルータなし。
3. **階層的。** サブスーパーバイザーを管理するスーパーバイザー、ネストされたサブグラフとして実装。

### このパターンがうまくいかない場合

- **チェックポイント小さすぎる。** 会話ターンのみチェックポイント化はツール状態とメモリ書き込みを回復不可にする。完全状態が必要にシリアル化。
- **非決定的なノード。** 再開はノード入力が同じ状態更新を生成する想定。ランダムシード、ウォールクロック、外部APIはキャプチャする必要がある。
- **条件付きエッジの過度使用。** すべてのエッジが条件付きグラフは推論できない状態機械。線形チェーンにしばしば枝とともにプリファー。

## ビルドする

`code/main.py`はstdlibステートフルグラフを実装：

- `State` — `messages`、`step`、`route`、`output`、`human_approval`を持つ型付きdict。
- `Node` — 状態を取り返す、更新dictを返す呼び出し可能。
- `StateGraph` — ノード＋エッジ＋条件付きエッジ＋実行＋再開。
- `SQLiteCheckpointer`（イン・メモリ偽） — すべてのノード後に状態をシリアル化；`load(session_id)`が復帰。
- デモグラフ：classify -> branch(refund / bug / sales) -> 人間ゲート -> send。

実行する:

```
python3 code/main.py
```

トレースは人間ゲートで失敗する最初の実行、永続化、次に最終出力を生じる再開を示す。

## 使う

- **LangGraph** — 参照、本番準備。`create_react_agent`、`create_supervisor`、または自分のグラフを構築を使用。
- **AutoGen v0.4**（レッスン14）— 高並行シナリオのためのアクターモデル代替。
- **Claude Agent SDK**（レッスン17）— 組み込みセッションストアを持つ管理harness。
- **カスタム** — 状態形状またはチェックポインタバックエンド上の正確な制御が必要なとき。

## 出荷

`outputs/skill-state-graph.md`はLangGraph形状ステートグラフをあらゆるターゲットランタイムで生成、チェックポイントと再開を接続。

## 演習

1. `classify`から`end`への条件付きエッジを追加、分類信頼が閾値を下回るとき。人間がセッションが手動で`route`をセットした後の実行を再開。
2. SQLiteのような偽をリアルSQLiteチェックポインタに交換。ステップごとのシリアル化オーバーヘッドを測定。
3. 並列エッジを実装：2つのノードは同時実行、カスタム減速機でマージ。不変状態がここで何を獲得するか？
4. `langgraph-supervisor`参照を読む。おもちゃを`create_supervisor`にポート。トレース形状を比較。
5. ストリーミングを追加：各ノードは実行しながら部分状態を獲得。デルタが着到する際に印刷。

## キー用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| ステートグラフ | 「状態機械としてエージェント」 | 型付き状態＋ノード＋エッジ＋減速機 |
| チェックポインタ | 「永続バックエンド」 | すべてのノード後に状態をシリアル化；再開を有効化 |
| 減速機 | 「状態マージャー」 | 現在の状態とノードの更新を結合する関数 |
| 条件付きエッジ | 「分岐」 | 状態の関数で選択されるエッジ |
| サブグラフ | 「ネストグラフ」 | 別のグラフ内のノードとして使用されるグラフ |
| 永続実行 | 「失敗から再開」 | 正確な状態で最後の成功ノードで再開 |
| スーパーバイザー | 「ルータLLM」 | 専門サブエージェント向けの中央ディスパッチャー |
| スワーム | 「P2Pエージェント」 | エージェントが共有ツール経由でハンドオフ；中央ルータなし |
