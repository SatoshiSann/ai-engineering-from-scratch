# Anthropicのワークフローパターン：シンプル優先複雑

> Schluntz and Zhang（Anthropic、2024年12月）はワークフロー（事前定義パス）とエージェント（動的ツール使用）を区別する。5つのワークフローパターンがほとんどの場合をカバー。直接API呼び出しで開始。ステップが予測できないときのみエージェントを追加。

**タイプ:** 学習 + ビルド
**言語:** Python（stdlib）
**前提条件:** Phase 14 · 01（エージェントループ）
**所要時間:** 約60分

## 学習目標

- Anthropicの5つのワークフローパターンを名前付けできる：プロンプトチェーン、ルーティング、並列化、オーケストレータ-ワーカー、評価器-オプティマイザー。
- エージェント対ワークフロー区別と各エンジニアリングコストを説明できる。
- ワークフロー対エージェントをいつ選ぶか（その逆）を識別できる。
- スクリプト化されたLLMに対してstdlibで5つのパターンすべてを実装できる。

## 問題

チームは複数エージェントフレームワークに、単一の関数呼び出しを望む問題に到達する。コストは本物だ：フレームワークはプロンプトを曖昧にし、制御フロー隠し、時期尚早な複雑さを招くレイヤーを追加する。Schluntz and Zhangの2024年12月の投稿は最も引用される業界のプッシュバック：シンプルで開始し、複雑さはそれがコストを獲得するときのみ追加。

## コンセプト

### ワークフロー対エージェント

- **ワークフロー。** LLMとツール、事前定義されたコードパスを通じてオーケストレート。エンジニアはグラフを所有。
- **エージェント。** LLMが動的にツールを指示し、自分たちのステップを取る。モデルはグラフを所有。

両方に場所がある。ワークフローはより安価、高速、デバッグしやすい。エージェントは開放性の高い問題を解き放つが、失敗モードを推論するのが難しくなる。

### 拡張LLM

5つのパターンすべての基礎：検索（検索）、ツール（アクション）、メモリ（永続性）の3つの機能が接続されたLLM。すべてのAPI呼び出しはこれらを使用できる。

### 5つのパターン

1. **プロンプトチェーン。** 呼び出し1の出力は呼び出し2への入力。ステップが清潔な線形分解を持つときに使用。ステップ間のオプションプログラム的ゲート。

2. **ルーティング。** 分類器LLMが、下流のどのLLMまたはツールを呼び出すかを選ぶ。カテゴリ的に異なる入力が異なる処理が必要な場合に使用（ティア1サポート対払い戻し対バグ対セールス）。

3. **並列化。** N個のLLM呼び出しを同時に実行し、結果を集約。2つの形：セクション化（異なるチャンク）と投票（同じプロンプト、N実行、多数決/合成）。

4. **オーケストレータ-ワーカー。** オーケストレータLLMが動的にどのワーカー（また、LLM）を実行するかを決定し、その出力を合成。エージェントループに類似だがオーケストレータは無限にループしない。

5. **評価器-オプティマイザー。** 1つのLLMが答えを提案、別のLLMがそれを評価。評価器がパスするまで反復。これはSelf-Refine（レッスン05）を一般化。

### ワークフローがエージェントを打つ場所

- **予測可能なタスク。** ステップを列挙できるなら、そうすべき。
- **コストバウンドなタスク。** ワークフローはバウンドされたステップ数；エージェントは螺旋することができる。
- **コンプライアンスバウンドなタスク。** 監査役は軌跡から推論するのではなく、グラフを読みたい。

### エージェントがワークフローを打つ場所

- **開放的な研究。** 次のステップが前のステップが返したものに依存する場合。
- **変数長タスク。** 分間から時間の仕事で、ステップ数が不明。
- **新しいドメイン。** 正しいワークフローが何かまだわからない場合—先に探索、後でコード化。

### コンテキストエンジニアリング仲間

「AI エージェント向け効果的コンテキストエンジニアリング」（Anthropic 2025）は隣接する規律を形式化：200kウィンドウは容器ではなく予算。何を含めるか、いつコンパクト化するか、コンテキストがいつ成長させるか。詳細はPhase 14のコンテキスト圧縮に関するレッスンでカバーされている（このカリキュラムの前にPhase 14の以前のレッスン06）。

## ビルドする

`code/main.py`はスクリプト化されたLLMに対して5つのワークフローパターンすべてを実装：

- `prompt_chain(input, steps)` — 順序付け。
- `route(input, classifier, handlers)` — 分類＋ディスパッチ。
- `parallel_vote(prompt, n, aggregator)` — N実行、集約。
- `orchestrator_workers(task, workers)` — オーケストレータはワーカーを選ぶ。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)` — パスまでループ。

実行する:

```
python3 code/main.py
```

各パターンそのトレースを印刷。パターンごとのコード行数は〜10-15；フレームワークのコストは数千で測定される。

## 使う

- ほとんどのタスクは直接API呼び出し。
- フレームワークのみパターンが真に永続的状態を必要とするとき（LangGraph）、アクターモデル並行性（AutoGen v0.4）、またはロールテンプレート化（CrewAI）。
- Claude Code harness形状を望むときClaude Agent SDKに到達するが、それを再構築しない。

## 出荷

`outputs/skill-workflow-picker.md`は与えられたタスク説明のための正しいパターンを選ぶ、決定論理とワークフローが足りなくなったときのリファクタパスを含む。

## 演習

1. 信頼閾値でルーティングを実装。その下—人間にエスカレート。ティア1サポートユースケースのための閾値はどこに着地するか？
2. `parallel_vote`へのタイムアウトを追加。1つの呼び出しが中止するとどうなる？不足している投票で集約はどう？
3. `evaluator_optimizer`をバンディットに変える：反復全体で上位2つの出力を保持し、遅い良い結果が遅い悪い結果で上書きされない。
4. プロンプトチェーンをルーティングと結合：ルータが3つのチェーンの1つを選ぶ。トークンコストを単一の大きいプロンプト代替に対して測定。
5. あなたの本番機能の1つを選ぶ。ワークフローグラフを描く。ステップをカウント。エージェントが実際により良いだろうか？

## キー用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| ワークフロー | 「事前定義フロー」 | エンジニア所有のLLMとツール呼び出しのグラフ |
| エージェント | 「自律AI」 | モデル所有のグラフ；動的ツール方向 |
| 拡張LLM | 「ツール付きLLM」 | LLM + 検索 + ツール + メモリ；アトミック単位 |
| プロンプトチェーン | 「順序付き呼び出し」 | 呼び出しN の出力は呼び出しN+1への入力 |
| ルーティング | 「分類器ディスパッチ」 | 入力を処理する、チェーン/モデルを選ぶ |
| 並列化 | 「ファンアウト」 | N同時呼び出し；セクション化または投票で集約 |
| オーケストレータ-ワーカー | 「ディスパッチャーエージェント」 | オーケストレータLLMが専門LLMを動的に選ぶ |
| 評価器-オプティマイザー | 「提案者＋判事」 | 評価器がパスするまで反復；Self-Refineを一般化 |
