# 障害モード: エージェントが失敗する理由

> MASFT (Berkeley, 2025) は 3 つのカテゴリーにおいて 14 のマルチエージェント障害モードをカタログ化しています。Microsoft のタクソノミーは、既存の AI 障害がエージェント設定でどのように増幅されるかを文書化しています。業界フィールドデータは、5 つの反復する障害モードに集約されています: ハルシネーション(幻覚)されたアクション、スコープ クリープ、カスケード エラー、コンテキスト損失、ツール誤用。

**タイプ:** Learn + Build
**言語:** Python (stdlib)
**前提条件:** Phase 14 · 05 (Self-Refine and CRITIC)、Phase 14 · 24 (Observability)
**所要時間:** 約 60 分

## 学習目標

- MASFT の 3 つの障害カテゴリーと各カテゴリーの少なくとも 4 つの特定の障害モードを述べることができる。
- エージェント設定で既存の AI 障害モード (バイアス、ハルシネーション) がどのように増幅されるのかを説明できる。
- 5 つの業界で繰り返される障害モードとそれらの軽減策を説明できる。
- エージェント トレースに障害モード ラベルをタグ付けする stdlib ディテクターを実装できる。

## 問題

チームは、トレースの 90% で機能するエージェントをデプロイします。10% の障害はランダムノイズではなく、小数の反復するカテゴリーに分類されます。それらに名前を付けることができれば、監視して修正することができます。

## コンセプト

### MASFT (Berkeley, arXiv:2503.13657)

マルチエージェント システム障害タクソノミー。3 つのカテゴリーにクラスタ化された 14 の障害モード。評価者間 Cohen の Kappa は 0.88 - カテゴリーは確実に区別できます。

中心的主張: 障害はより良い基礎モデルで修正すべき LLM の制限ではなく、マルチエージェント システムの根本的な設計欠陥です。

### Microsoft: エージェント AI システムにおける障害モードのタクソノミー

- 既存の AI 障害 (バイアス、ハルシネーション、データ漏洩) はエージェント設定で増幅されます。
- 自律性から新しい障害が生じます: スケールでの意図しないアクション、ツール誤用、ミッション ドリフト。
- ホワイトペーパーはエージェント製品のリスク レジスターです。

### エージェント AI における障害の特性評価 (arXiv:2603.06847)

- 障害はオーケストレーション、内部状態の進化、環境相互作用から生じます。
- 単なる「悪いコード」または「悪いモデル出力」ではありません。

### LLM エージェント ハルシネーション調査 (arXiv:2509.18970)

2 つの主な表れ方:

1. **命令追従の逸脱** - エージェントがシステム プロンプトに従いません。
2. **長距離文脈の誤用** - エージェントが以前のターンからコンテキストを忘れるか、誤用します。

副目的エラー: 省略 (ステップ欠落)、冗長性 (ステップ繰り返し)、無秩序 (ステップ順序間違い)。

### 5 つの業界で繰り返される障害モード

Arize、Galileo、NimbleBrain の 2024-2026 フィールド分析は以下に集約されています:

1. **ハルシネーション(幻覚)されたアクション** - エージェントが存在しないツールを呼び出すか、引数を捏造します。
2. **スコープ クリープ** - エージェントがユーザーのリクエストを超えてタスクを拡張します (余分な PR を作成、余分なメール送信)。
3. **カスケード エラー** - 1 つの誤った呼び出しがダウンストリーム効果をトリガーします。幽霊 SKU のハルシネーション は 4 つの API 呼び出しをトリガー - マルチシステム インシデント。
4. **コンテキスト損失** - 長期タスクは初期ターンの制約を忘れます。
5. **ツール誤用** - 正しいツールを間違った引数で呼び出すか、完全に間違ったツールを呼び出します。

カスケードは殺傷力があります。エージェントは「失敗した」と「タスクは不可能」を区別できず、多くの場合 400 エラーの成功メッセージをハルシネーション(幻覚)してループを閉じます。

### 軽減: すべてのステップでゲート

推論チェーンのすべてのステップで、環境状態に対して事実的根拠を確認する自動検証ゲート。具体的には:

- ステップごとの安全分類器 (Lesson 21)。
- ツール呼び出し引数の検証 (Lesson 06)。
- 取得したコンテンツと既知の事実を相互チェック (Lesson 05、CRITIC)。
- 状態を再プロービングして成功ハルシネーション(幻覚)を検出 (ファイルは実際に作成されたか?)。

### 障害監視が失敗する場所

- **クラッシュのみをタグ付け** - ほとんどのエージェント障害は有効に見える出力を生成します。コンテンツレベルのチェックが必要です。
- **ベースラインなし** - ドリフト検出には最後に既知の良好な状態が必要です; それなしではこれが「悪化している」と言えません。
- **過度なアラート** - すべての障害がページを生成します。クラスター化とレート制限が必要です。

## Build It

`code/main.py` は stdlib 障害モード タガーを実装します:

- 5 つのモードをカバーする合成トレース データセット。
- モードごとのディテクター関数 (ツール呼び出しの署名パターン、出力、アクション繰り返し)。
- 各トレースにラベルをタグ付けし、モード分布をレポートするタガー。

実行:

```
python3 code/main.py
```

出力: トレースごとのラベル + 集計分布、Phoenix のトレース クラスタリングが表示するものの廉価な再現。

## Use It

- **Phoenix** - 本番環境ドリフト クラスタリング用 (Lesson 24)。
- **Langfuse** - セッション リプレイ + アノテーション用。
- **Custom** - オブザーバビリティ プラットフォームが検出できないドメイン固有の署名用。

## Ship It

`outputs/skill-failure-detector.md` はあなたのドメインに合わせた障害モード ディテクターを生成し、トレース ストアに接続します。

## 演習

1. 「成功ハルシネーション(幻覚)」のためのディテクターを追加: エージェントは成功を返しますが、ターゲット状態は変更されていません。
2. 構築した製品から 100 の実際のトレースをタグ付けします。どのモードが支配的ですか? それを修正するコストはいくらですか?
3. 「カスケード 半径」メトリックを実装: ステップ N での障害が与えられた場合、それは何個のダウンストリーム ステップに影響を与えましたか?
4. MASFT の 14 の障害モードを読みます。あなたの製品に適用される 3 つを選択します。ディテクターを記述します。
5. 1 つのディテクターを CI ジョブに接続: トレースの >= 5% が障害モードをタグ付けする場合、ビルドを失敗させます。

## 主要用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|----------------|----------|
| MASFT | 「マルチエージェント障害タクソノミー」 | Berkeley の 14 モード カテゴリー化 |
| カスケード エラー | 「リップル障害」 | 1 つの初期ミスが N 個のステップを通じて伝播 |
| コンテキスト損失 | 「制約を忘れた」 | 長期ターンが初期ターンの事実を削除 |
| ツール誤用 | 「間違ったツール / 間違った引数」 | 有効な呼び出し、間違った実行 |
| 成功ハルシネーション(幻覚) | 「完了を偽造」 | エージェントは 400 の成功を主張; 状態は変更されません |
| スコープ クリープ | 「オーバーリーチ」 | エージェントはリクエスト以上のことを実行 |
| 命令追従の逸脱 | 「不従順」 | システム プロンプトまたはユーザー制約を無視 |
| 副目的エラー | 「計画バグ」 | 計画実行の省略、冗長性、無秩序 |
