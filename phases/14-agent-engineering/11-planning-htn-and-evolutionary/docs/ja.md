# HTNと進化探索を使ったプランニング

> 記号計画は計画が証明可能に正しい場合を扱う。進化コード探索は適応度関数がマシンチェック可能な場合を扱う。ChatHTN（2025）とAlphaEvolve（2025）は、各LLMと対になったときにそれぞれが何を解き放つかを示す。

**タイプ:** ビルド
**言語:** Python（stdlib）
**前提条件:** Phase 14 · 02（ReWOOとPlan-and-Execute）
**所要時間:** 約75分

## 学習目標

- 階層的タスクネットワークを説明できる：タスク、メソッド、オペレータ、前提条件、効果。
- ChatHTNのハイブリッドループを説明できる—記号探索とLLMフォールバック分解。
- AlphaEvolveの進化ループを説明し、なぜプログラム的な評価器でのみ動作するかを説明できる。
- stdlibでおもちゃのHTNプランナーとおもちゃの進化探索を実装できる。

## 問題

ReWOO（レッスン02）、Plan-and-Execute、ReActはほとんどのエージェントプランニングをカバーする。2つ、そこまでカバーしない場合：

1. **証明可能に正しい計画。** スケジューリング、飛行経路、コンプライアンスワークフロー—計画は構成によって健全である必要がある。時々ステップを幻覚するフローエントなLLMプランは受け入れられない。
2. **マシンチェック可能な適応度関数を持つ最適化。** 行列乗法、スケジューリングヒューリスティック、コンパイラパス—目標は「正しいプラン」ではなく「最良のプラン」。

HTNプランニングとAlphaEvolveは2つの異なる問題を解く。両方ともLLMを増幅器として、置き換えとしてではなく使う。

## コンセプト

### 階層的タスクネットワーク

HTNは:

- **タスク** — 複合（分解される）とプリミティブ（直接実行可能）。
- **メソッド** — 複合タスクをサブタスクに分解する方法、前提条件付き。
- **オペレータ** — 前提条件と効果を持つプリミティブアクション。
- **状態** — 事実のセット。

プランニング：目標タスクと初期状態が与えられたとき、前提条件が順序付けられて満たされるプリミティブオペレータへの分解を見つける。

HTNはLLMより古く、証明可能に正しいプランのための参照である。

### ChatHTN（Gopalakrishnan et al., 2025）

ChatHTN（arXiv:2505.11814）は記号HTNとLLMクエリを交互に実行する:

1. 既存メソッドで現在の複合タスクを分解しようとする。
2. メソッドが適用されない場合、LLMに質問：「状態`s`でタスク`task`をどう分解する？」
3. LLM応答をタスク候補に翻訳する。
4. オペレータスキーマに対して検証；無効な分解を拒否。
5. 再帰。

論文の中心的主張：すべての生成計画は証明可能に健全である—LLMの提案は候補分解としてのみ入力し、直接計画編集には決してならない。記号層が正確性を所有；LLMはメソッドライブラリを拡大。

オンラインメソッド学習（OpenReview `gwYEDY9j2x`、2025フォローアップ）は、LLM生成分解を回帰で一般化する学習器を追加—LLMクエリ頻度を最大75%削減。

### AlphaEvolve（Novikov et al., 2025）

AlphaEvolve（arXiv:2506.13131、DeepMind、2025年6月）は別の獣：Gemini 2.0 Flash/Proアンサンブルで編成された進化コード探索。

ループ:

1. シードプログラム＋プログラム的評価器（適応度スコアを返す）で開始。
2. LLMアンサンブルが変異を提案。
3. 評価器で変異を実行。
4. 最良を保持；再び変異。

公開された勝利:

- 56年で初めて4x4複素行列乗法でStrassenを超える（48スカラー乗算）。
- Google計算の0.7%をBorgスケジューリングヒューリスティックで回収。
- 限界のワークロードでFlashAttentionが32%高速化。

厳密な制約：適応度関数はマシンチェック可能である必要がある。散文答案上の進化探索は収束しない。

### どれを使うか

| 問題クラス | 使う | 理由 |
|---------------|-----|-----|
| ハード制約でのスケジューリング | HTN + ChatHTN | 証明可能な正確性 |
| コンパイラ最適化 | AlphaEvolve | マシンチェック可能な適応度 |
| マルチステップタスク実行 | ReAct / ReWOO | ループ内LLM、正式保証なし |
| テストでのコード改善 | AlphaEvolve | テストは評価器 |
| ポリシーバウンド自動化 | HTN | 前提条件はポリシーをエンコード |

### このパターンがうまくいかない場合

- **オペレータなしのHTN。** 前提条件/効果スキーマなしで、正確性主張は崩壊する。ChatHTNの「LLMが分解を提案」はスキーマが無効な移動を拒否することが必要。
- **本当の評価器なしのAlphaEvolve。** 「LLMがコードが良いか聞く」は適応度関数ではない。評価器は決定的で高速である必要がある。
- **過度なエンジニアリング。** ほとんどのエージェントタスクはどちらも必要としない。まずReActまたはReWOOに到達。

## ビルドする

`code/main.py`は2つのおもちゃを実装する:

- オペレータ、メソッド、前提条件、効果、複合タスクがメソッドに一致しないときにキックインする`LLMFallback`を持つstdlib HTNプランナー。「LLM」はスクリプト化された分解子なのでプランナーはオフラインで動作する。
- `|f(x) - target|`をテストセットで最小化する出力を持つ算術プログラム上の進化探索。評価器は決定的。

実行する:

```
python3 code/main.py
```

トレースはHTNプランナーが複合タスクを分解（計画中盤のLLMフォールバック付き）および進化ループが目標表現に収束するのを示す。

## 使う

- **HTNプランナー** — `pyhop`、`SHOP3`、またはドメイン固有ポリシー実行のために自分のものを構築。
- **ChatHTN** — 研究コード；パターン（記号＋LLMフォールバック）はあらゆるHTNプランナーにクリーンにポート。
- **AlphaEvolve** — DeepMind論文；パターン（アンサンブル＋評価器）は再現可能。OpenEvolveおよび同様のオープンソースフォークが出現している。
- **エージェントフレームワーク** — どれもまずクラスのHTNまたはAlphaEvolveを出荷しない。サブエージェントまたはバックグラウンドワーカーとして構築。

## 出荷

`outputs/skill-hybrid-planner.md`はハイブリッドプランナースキャフォールド（HTNまたは進化）を生成し、LLMの役割が明示的にスコープされる。

## 演習

1. HTNプランナーをバックトラッキングで拡張：オペレータの後条件が実行時に失敗するとき、ロールバックして次のメソッドを試す。
2. ChatHTNにLLMメソッドキャッシュを追加：LLMがタスク`T`を状態パターン`P`で分解するとき、結果を保存。次の呼び出しで最初にメソッドライブラリを再確認。
3. 進化探索評価器を本当のテストスイートに交換。20テストケースを通過するソート関数を進化；収束世代を報告。
4. AlphaEvolveの評価器設計ノートを読む。あなたが気にするドメイン向けの評価器を設計（SQL問い合わせ最適化、テストスイート最小化、デプロイYAML）。
5. 結合：HTNを使って複合タスクをサブタスクに分解、次に進化探索を各サブタスクのプリミティブオペレータに使う。どこで光り、どこで過度なエンジニアリングするか？

## キー用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| HTN | 「階層的プランナー」 | オペレータ、前提条件、効果を持つタスク分解 |
| メソッド | 「分解ルール」 | 複合タスクをサブタスクに分ける方法 |
| オペレータ | 「プリミティブアクション」 | 前提条件と効果を持つ具体的ステップ |
| ChatHTN | 「LLM + HTN」 | 記号プランナーはメソッドがマッチしないときLLMを問う |
| AlphaEvolve | 「進化コード探索」 | アンサンブルLLMがコードを変異；決定的評価器が選択 |
| 適応度関数 | 「評価器」 | 出力上の決定的、マシンチェック可能なスコア |
| オンラインメソッド学習 | 「キャッシュLLM分解」 | LLMプランを保存して一般化し、クエリコストを削減 |
