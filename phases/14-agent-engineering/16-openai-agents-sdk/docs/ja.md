# OpenAI Agents SDK：ハンドオフ、ガードレール、トレーシング

> OpenAI Agents SDKはResponses APIの上に構築されたライトウェイトマルチエージェントフレームワーク。5つのプリミティブ：Agent、Handoff、Guardrail、Session、Tracing。ハンドオフは`transfer_to_<agent>`という名前のツール。ガードレールは入力または出力でトリップ。トレーシングはデフォルトオン。

**タイプ:** 学習 + ビルド
**言語:** Python（stdlib）
**前提条件:** Phase 14 · 01（エージェントループ）、Phase 14 · 06（ツール使用）
**所要時間:** 約75分

## 学習目標

- OpenAI Agents SDKの5つのプリミティブを名前付けできる。
- ハンドオフを説明できる：なぜツールとしてモデル化されるか、モデルが見る名前形、コンテキストがどう移行するか。
- 入力ガードレール、出力ガードレール、ツールガードレールを区別；`run_in_parallel` vs ブロッキングモード説明。
- ハンドオフ＋ガードレール＋スパン形式トレーシングを持つstdlibランタイムを実装。

## 問題

クリーンにデリゲートできないエージェントは、すべてをプロンプトに詰め込む。ガードレールなしエージェントはPII、ポリシー違反出力、または無限ループを出荷。OpenAIのSDKはマルチエージェントを扱い易くする3つのプリミティブをコード化。

## コンセプト

### 5つのプリミティブ

1. **Agent。** LLM + 指示 + ツール + ハンドオフ。
2. **Handoff。** 別のエージェントへのデリゲーション。ランタイムに`transfer_to_<agent_name>`という名前のツール として表現。
3. **Guardrail。** 入力（最初エージェントのみ）、出力（最後エージェントのみ）、またはツール呼び出し（関数ツールごと）での検証。
4. **Session。** ターンをまたいで自動会話履歴。
5. **Tracing。** LLM生成、ツール呼び出し、ハンドオフ、ガードレール上のスパン内蔵。

### ツールとしてのハンドオフ

モデルは`transfer_to_billing_agent`をツールリストに見る。呼び出すことは、ランタイムに信号を送る：

1. 会話コンテキストを引写（または`nest_handoff_history`ベタ経由でコラプス）。
2. ターゲットエージェント初期化、その指示。
3. ターゲットエージェント継続実行。

これはスーパーバイザーパターン（レッスン13 / レッスン28）製品化。

### ガードレール

3つのフレーバー:

- **入力ガードレール。** 最初のエージェントの入力で実行。危険なまたはスコープ外の要求をLLM呼び出しの前に拒否。
- **出力ガードレール。** 最後のエージェントの出力で実行。PIIリーク、ポリシー違反、形式不正応答を捕捉。
- **ツールガードレール。** 関数ツールごと実行。引数検証、許可をチェック、実行を監査。

モード:

- **並列**（デフォルト）。ガードレールLLMはメインLLMに並行実行。低いテール遅延。トリップするとメインLLMの仕事は廃棄（トークン浪費）。
- **ブロッキング**（`run_in_parallel=False`）。ガードレールLLMが最初実行。トリップするとメイン呼び出しの浪費なし。

トリップはraiseする `InputGuardrailTripwireTriggered` / `OutputGuardrailTripwireTriggered`。

### トレーシング

デフォルトオン。すべてのLLM生成、ツール呼び出し、ハンドオフ、ガードレールはスパンを出力。`OPENAI_AGENTS_DISABLE_TRACING=1`でオプトアウト。`add_trace_processor(processor)`はスパンをファンしてOpenAIの隣にあなたのバックエンドに。

### セッション

`Session`はバックエンド（SQLite、Redis、カスタム）に会話履歴を格納。`Runner.run(agent, input, session=session)`は自動ロードして追加。

### このパターンがうまくいかない場合

- **ハンドオフドリフト。** エージェントAがエージェントBにハンドオフ、BがAに戻す。ホップカウンタを追加。
- **ガードレールバイパス。** ツールガードレールは関数ツールのみ発火；組み込みツール（ファイルリーダー、ウェブ取得）は別のポリシーが必要。
- **過度トレーシング。** スパンの機密コンテンツ。OTel GenAIコンテンツキャプチャルール（レッスン23）とペアで—外部に格納、IDで参照。

## ビルドする

`code/main.py`はstdlibでSDK形状を実装：

- `Agent`、`FunctionTool`、`Handoff`（ハンドオフセマンティクス付き関数ツール）。
- `Runner`が入力/出力/ツールガードレール、ハンドオフディスパッチ、ホップカウンタ。
- シンプルなスパン出力器、本当のSDKが出力するトレース形を示す。
- トリアージエージェント、ユーザーのクエリに基づいて請求またはサポートにハンドオフ；ガードレールが1つの入力でトリップ。

実行する:

```
python3 code/main.py
```

トレースは2つの成功ハンドオフ、1つの入力ガードレールトリップ、本当のSDKが出力するものをミラーリングするスパンツリー。

## 演習

1. ハンドオフホップカウンタを追加：Nトランスフォー後に拒否。振る舞いをトレース。
2. `nest_handoff_history`をオプション実装—転送前に概要に先のメッセージをコラプス。
3. ブロッキング出力ガードレール書く。トリップするプロンプト上のレイテンシ対パスするものを比較。
4. `add_trace_processor`をJSONロギングにワイヤー。スパンごとにどの形を出力するか？
5. SDKドキュメント読む。stdlibおもちゃを`openai-agents-python`にポート。何をモデル誤ったか？

## キー用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| エージェント | 「LLM + 指示」 | SDK内のエージェント型；ツールとハンドオフを所有 |
| ハンドオフ | 「トランスファー」 | 別のエージェントにデリゲートするモデルが呼ぶツール |
| ガードレール | 「ポリシーチェック」 | 入力 / 出力 / ツール呼び出しでの検証 |
| トリップワイヤー | 「ガードレールトリップ」 | ガードレールが拒否するとき上げられた例外 |
| セッション | 「履歴ストア」 | 実行間に永続化された会話メモリ |
| トレーシング | 「スパン」 | LLM + ツール + ハンドオフ + ガードレール上の内蔵観測可能性 |
| ブロッキングガードレール | 「順序チェック」 | ガードレール最初実行；トリップ時のトークン浪費なし |
| 並列ガードレール | 「同時チェック」 | ガードレール並行実行；低レイテンシ、トリップ時浪費トークン |
