# ReWOOとプラン・アンド・エグゼキュート：分離されたプランニング

> ReActは思考と行動を1つのストリームの中に組み込みます。ReWOOはそれらを分離します：最初に1つの大きなプランを立て、その後実行します。トークン数は5分の1、HotpotQAでは精度が+4%向上し、プランナーを7Bモデルに蒸留できます。Plan-and-Executeはこれを一般化し、Plan-and-Actはウェブナビゲーションにスケーリングしました。

**タイプ:** ビルド
**言語:** Python (stdlib)
**前提条件:** Phase 14 · 01 (Agent Loop)
**所要時間:** 約60分

## 学習目標

- ReWOOのプランナー/ワーカー/ソルバー分割がなぜReActのインターリーブループよりもトークン数を節約し堅牢性を改善するのかを説明する。
- プランDAG、依存順序を考慮した実行器、ワーカー出力を組み合わせるソルバーをすべて標準ライブラリで実装する。
- 2026年の「5つのワークフローパターン」フレーミング(Anthropic)を使用して、タスクをプラン・ザン・エグゼキュートとしてすべきか、またはインターリーブReActとしてすべきかを判断する。
- 長期的なウェブまたはモバイルタスクでPlan-and-Actの合成プランデータが必要な場合を認識する。

## 問題

ReActのインターリーブ思考・行動・観測ループはシンプルで柔軟ですが、各ツール呼び出しは完全な先行コンテキストを持つ必要があります。これには以前のすべての思考が含まれます。トークン使用量は深さとともに二次的に増加します。さらに悪いことに、ツール呼び出しがループの中途で失敗した場合、モデルはエラー観測から全体的なプランを再導出する必要があります。

ReWOO (Xu et al., arXiv:2305.18323, 2023年5月)はこれに気づき、全体をあらかじめプランニングして、証拠を並行して取得し、最後に答えを組み合わせるという賭けに出ました。プランニングに1回のLLM呼び出し、証拠用にN回のツール呼び出し(並行可能)、最後に1回のLLM呼び出しで解くのです。トレードオフは柔軟性の低下(プランは静的)と引き換えに、はるかに優れたトークン効率と明確な失敗モードです。

## コンセプト

### 3つの役割

```
プランナー:  user_question -> [plan_dag]
ワーカー:    [plan_dag]     -> [evidence]        (ツール呼び出し、並行可能)
ソルバー:    user_question, plan_dag, evidence -> final_answer
```

プランナーはDAGを生成します。各ノードはツール、その引数、および依存する先行ノード(#E1、#E2などの参照)を名前付けます。ワーカーは位相順序でノードを実行します。ソルバーはすべてを組み合わせます。

### なぜ5分の1のトークン数か

ReActはステップ数とともにプロンプト長が線形に増加します。ステップ10では、プロンプトは思考1と行動1と観測1と思考2と行動2と観測2などを含みます。各中間ステップはまた冗長に元のプロンプトを含みます。

ReWOOは1つのプランナープロンプト(大きい)、N個の小さいワーカープロンプト(各々はツール呼び出しだけで、チェーンなし)、および1つのソルバープロンプトを支払います。HotpotQAでは論文は約5分の1のトークン数を測定しながら+4の絶対精度を記録します。

### なぜそれはより堅牢か

ReActではワーカー3が失敗した場合、ループはストリームの中途でエラーから推論する必要があります。ReWOOではワーカー3はエラー文字列を返し、ソルバーはそれを元のプランの文脈と一緒に見て、優雅に低下させることができます。失敗の局所化はステップごとではなくノードごとです。

### プランナー蒸留

論文の2番目の結果：プランナーは観測を見ないため、175Bの教師からのプランナー出力で7Bモデルをファインチューニングできます。小さいモデルがプランニングを処理し、大きいモデルは推論では不要です。これは現在標準です。多くの2026年の本番エージェントは小さいプランナーと大きい実行器、またはその逆を使用します。

### Plan-and-Execute (LangChain, 2023)

LangChainチームの2023年8月投稿はReWOOをパターン名に一般化しました：Plan-and-Execute。事前プランナーはステップリストを発行し、実行器各ステップを実行し、オプションの再プランナーは結果を観測した後に改訂できます。これはReActに近い(再プランナーは観測をプランニングに戻す)がトークン節約を保持します。

### Plan-and-Act (Erdogan et al., arXiv:2503.09572, ICML 2025)

Plan-and-Actはパターンを長期的なウェブおよびモバイルエージェントにスケーリングします。主な貢献は合成プランデータです：ラベル付きトラジェクトリジェネレータは、プランが明示的なトレーニングデータを生成します。プランナーモデルをファインチューニングして、単一のReActトラジェクトリが一貫性を失うWebArena様タスクで30-50ステップを超えて動作し続けるようにするために使用されます。

### どれを選ぶか

| パターン | タイミング |
|---------|------------|
| ReAct | 短いタスク、未知の環境、リアクティブ例外処理が必要 |
| ReWOO | 既知のツールで構造化されたタスク、トークン感度、並行可能な証拠 |
| Plan-and-Execute | ReWOOと同様だが部分的実行後に再プランニング |
| Plan-and-Act | 長期的(>30ステップ)、ウェブ/モバイル/コンピュータ使用 |
| Tree of Thoughts | 検索に価値がある(レッスン04) |

Anthropicの2024年12月ガイダンス：最もシンプルなもので始めます。タスクが1つのツール呼び出しと要約である場合、ReWOOを構築しないでください。タスクが40ステップの研究課題である場合、ReActだけでは行わないでください。

## ビルド

`code/main.py`はおもちゃのReWOOを実装します：

- `Planner` — プロンプトからプランDAGを発行するスクリプト化されたポリシー。
- `Worker` — レジストリを介して各ノードのツール呼び出しをディスパッチします。
- `Solver` — 証拠を読み、最終的な答えを生成するスクリプト化された構成。
- 依存性解決 — #E1などの参照は先行するワーカー出力で置き換えられます。

デモは「フランスの首都の人口は、100万単位で四捨五入するといくつか？」という質問に、2ステップのプラン(1)首都を調べる、(2)人口を調べる、その後解くことで答えます。

実行：

```
python3 code/main.py
```

トレースは完全なプランを最初に表示し、その後ワーカー結果、その後ソルバー構成を表示します。トークン数(概算の文字数を出力)をReAct風インターリーブ実行と比較します。ReWOOはこの種の構造化タスクで勝ちます。

## 使う

LangGraphはPlan-and-Executeを提供します(ReAct用`create_react_agent`、プラン・エグゼキュート用のカスタムグラフ)。CrewAIのFlowsはパターンを直接エンコードします：タスクを事前に定義し、Flow DAGがそれらを実行します。Plan-and-Actの合成データアプローチはまだほぼ研究段階です。ランタイムパターン(明示的なプランDAG)はLangGraphとCrewAI Flowsを通じて本番環境に出荷されます。

## 出荷

`outputs/skill-rewoo-planner.md`はユーザーリクエストからツールカタログが与えられたReWOOプランDAGを生成します。実行器に引き渡す前に、プランを検証します(非環形、すべての参照が解決済み、すべてのツールが存在する)。

## 演習

1. 独立したプランノードのワーカー実行を並行化します。2つの並行グループを持つ6ノードDAGで得られるのは何ですか？
2. いずれかのワーカーがエラーを返した場合に起動する再プランナーノードを追加します。ReWOOをPlan-and-Executeにする最小限の変更は何ですか？
3. `Planner`を小さいモデル(7Bクラス)に置き換え、`Solver`をフロンティアモデルに保ちます。端末から端末までの品質を比較します。分割が失敗するのはどこですか？
4. ReWOO論文のセクション4を読み、プランナー蒸留について読みます。175B -> 7B結果を概念的に再現します：どのようなトレーニングデータが必要で、プラン品質をどのようにスコアリングしますか？
5. おもちゃをPlan-and-Actのトラジェクトリ形にポートします：プランはDAGではなくシーケンスです。トレードオフはどう変わりますか？

## キーターム

| ターム | 人が言うこと | 実際の意味 |
|--------|------------|----------|
| ReWOO | 「観測なしの推論」 | プランニング後、並行して証拠をフェッチし、その後解く。プランニングプロンプトに観測がない |
| Plan-and-Execute | 「LangChainのプラン・エグゼキュートパターン」 | 実行後のオプションの再プランナーノード付きReWOO |
| Plan-and-Act | 「スケーリングされたプラン・エグゼキュート」 | 長期的タスク用の合成プランニングトレーニングデータ付きの明示的なプランナー/実行器分割 |
| Evidence reference | 「#E1、#E2、...」 | 先行するワーカー出力でディスパッチ時に置き換えられるプランノードプレースホルダー |
| Planner distillation | 「小さいプランナー、大きい実行器」 | 大きい教師からのプランナーの痕跡で小さいモデルをファインチューニング |
| Token efficiency | 「より少ないラウンドトリップ」 | 論文でReActと比較してHotpotQAで約5倍のトークン削減 |
| DAG executor | 「位相的ディスパッチャー」 | 依存順序でプランノードを実行；各レベルで並行 |
