# Eval 駆動型エージェント開発

> Anthropic のガイダンス: 「単純なプロンプトで開始し、包括的な評価でそれらを最適化し、必要な場合のみマルチステップ エージェント システムを追加します」。評価は最後のステップではありません。これは Phase 14 の他のすべての選択を駆動する外側のループです。

**タイプ:** Learn + Build
**言語:** Python (stdlib)
**前提条件:** Phase 14 全体。
**所要時間:** 約 60 分

## 学習目標

- 3 つの評価レイヤー - 静的ベンチマーク、カスタム オフライン、オンライン本番環境 - とそれぞれが何のためかに名前を付けることができる。
- 評価器オプティマイザー タイト ループを説明できる。
- 2026 年のベスト プラクティス: eval はコードの隣に存在し、CI で実行され、PR をゲートすることを説明できる。
- すべての Phase 14 レッスンを生成する eval ケースに接続できる。

## 問題

エージェントはデモを成功させます。本番環境ではデモが予測できない方法で失敗します。ベンチマークは「このモデルは広く能力がありますか?」という質問に答えます。「このエージェントは私の製品に適切なパッチを出荷していますか?」という質問ではありません。答え: 3 つのレイヤーで評価し、継続的に実行し、すべてのガードレールと学習ルールが eval ケースにマップされます。

## コンセプト

### 3 つの評価レイヤー

1. **静的ベンチマーク** — SWE-bench Verified (Lesson 19) コード用、WebArena/OSWorld (Lesson 20) ブラウジング / デスクトップ用、GAIA (Lesson 19) ジェネラリスト用、BFCL V4 (Lesson 06) ツール使用用。クロスモデル比較と回帰ゲート用に使用します。汚染は実際: SWE-bench+ は 32.67% ソリューション漏洩を発見。常に Verified / + 監査スコアをレポート。

2. **カスタム オフライン eval** — あなたの製品の形:
   - LLM as judge (Langfuse、Phoenix、Opik - Lesson 24)。
   - 実行ベース (パッチを実行、テストを確認)。
   - 軌跡ベース (アクション シーケンスを金基準と比較; OSWorld-Human は最上位エージェントが金基準の 1.4-2.7x を示す)。

3. **オンライン eval** — 本番環境:
   - セッション リプレイ (Langfuse)。
   - ガードレール トリガー アラート (Lesson 16、21)。
   - ステップごとのコスト / レイテンシー トラッキング (Lesson 23 OTel スパン)。

### 評価器オプティマイザー (Anthropic)

タイト ループ:

1. 提案者が出力を生成。
2. 評価器が判定。
3. 評価器が成功するか最大ラウンドに達するまで改善。

これは Self-Refine (Lesson 05) を一般化したものです。あなたが気にするあらゆるエージェント フローは信頼性のため評価器オプティマイザーでラップできます。

### 2026 年ベスト プラクティス

- Eval はコードの隣に存在。
- すべての PR で CI で実行。
- eval スコアでマージをゲート (例 「メインに対して 5% を超える回帰なし」)。
- すべてのガードレールは eval ケースにマップ。
- すべての学習ルール (Reflexion、pro-workflow learn-rule) は障害ケースにマップ。

### Phase 14 を結ぶ

Phase 14 の各レッスンは eval ケースを生成します:

| レッスン | 生成する Eval ケース |

| 02 ReWOO | プランナー はツール失敗時に計画を立て直す |
| 03 Reflexion | 学習した反省は再試行で適用 |

| 13 LangGraph | 再開は状態を正確に再現 |
| 14 AutoGen Actors | DLQ は落ちたハンドラーをキャッチ |
| 16 OpenAI Agents SDK | ガードレール は正しい入力でトリップ |
| 17 Claude Agent SDK | サブエージェント結果はオーケストレーターに返される |
| 19-20 Benchmarks | SWE-bench Verified スコア、WebArena 成功率、OSWorld 効率 |

あなたの eval スイートにそれぞれのケースがある場合、Phase 14 をカバーしています。

### Eval 駆動型開発が失敗する場所

- **ベースラインなし。** ベースラインなしの eval は読めません。ベースラインを保存します。
- **根拠なし LLM 判定。** 判定も幻覚します。CRITIC パターン (Lesson 05) - 判定は外部ツールで根拠。
- **Eval への過学習。** eval に最適化することは本番環境の有用性から乖離します。ケースをローテーション。
- **不安定な eval。** 非決定論的ケースは誤検知を引き起こします。シードをピン、状態をスナップショット。

## Build It

`code/main.py` は stdlib eval ハーネスです:

- カテゴリー (ベンチマーク、カスタム、オンライン) のケース レジストリ。
- テスト下のスクリプト化されたエージェント。
- 評価器オプティマイザー ループ: 提案、判定、成功またはマックスラウンドまで改善。
- CI ゲート: 集計成功率 + ベースラインに対する回帰。

実行:

```
python3 code/main.py
```

出力: ケースごとの成功/失敗、回帰フラグ、CI ゲート判定。

## Use It

- Eval ケースを エージェント コード と同じリポジトリで記述。
- CI 経由ですべての PR で実行。
- 回帰時にビルドを失敗。
- 時間経過でパス率を追跡。
- すべての本番環境障害を新しいケースに結ぶ。

## Ship It

`outputs/skill-eval-suite.md` は CI ゲートと回帰トラッキング を持つエージェント製品用の 3 レイヤー eval スイートを構築します。

## 演習

1. 本番環境障害の 1 つを取得。それを再現する eval ケースを記述。あなたのエージェントは今それを成功しますか?
2. ドメイン用の LLM 判定ルーブリックを 3 つの次元 (事実的、トーン、スコープ) で構築。50 セッションをスコア。
3. eval スイートを CI に接続。 >= 5% 回帰時にビルドを失敗。
4. 軌跡効率メトリックを追加: エージェントは金基準軌跡に対してどのくらいのステップを取りましたか?
5. すべての Phase 14 レッスンを スイト内の eval ケースにマップ。欠落がありますか? それはクローズするギャップです。

## 主要用語

| 用語 | 人々が言うこと | 実際の意味 |
|------|----------------|----------|
| 静的ベンチマーク | 「既製 eval」 | SWE-bench、GAIA、AgentBench、WebArena、OSWorld |
| カスタム オフライン eval | 「ドメイン eval」 | LLM as judge / exec / 軌跡 (あなたの製品形) |
| オンライン eval | 「本番環境 eval」 | セッション リプレイ、ガードレール アラート、コスト/レイテンシー トラッキング |
| 評価器オプティマイザー | 「提案判定改善」 | 判定が成功するまで反復 |
| CI ゲート | 「マージ ブロッカー」 | eval 回帰時にビルドを失敗 |
| ベースライン | 「最後に既知の良好」 | 回帰を検出する参照スコア |
| 軌跡効率 | 「金基準以上のステップ」 | エージェント ステップ カウント / 人間専門家最小値 |
