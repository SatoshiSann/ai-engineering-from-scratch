# CrewAI：ロールベースのクルーとフロー

> CrewAIは2026年のロールベースのマルチエージェントフレームワーク。4つのプリミティブ：Agent、Task、Crew、Process。2つの最上位形状：Crews（自律、ロールベース協力）とFlows（イベント駆動、決定的）。ドキュメントは率直だ：「本番対応応用のため、フローで始める。」

**タイプ:** 学習 + ビルド
**言語:** Python（stdlib）
**前提条件:** Phase 14 · 12（ワークフローパターン）、Phase 14 · 14（アクターモデル）
**所要時間:** 約75分

## 学習目標

- CrewAIの4つのプリミティブを名前付けできる（Agent、Task、Crew、Process）と各が所有するもの。
- Sequential、Hierarchical、計画したConsensusプロセスを区別；ワークロードごとに1つを選ぶ。
- Crews（自律ロールベース）とFlows（イベント駆動決定的）を区別し、ドキュメントの本番推奨を説明できる。
- `@tool`デコレータと`BaseTool`サブクラスでツールをワイヤー；構造化出力対フリーテキストを推論。
- 4つのCrewAIメモリタイプを名前付けし、各が何のために支払うかを説明。
- stdlib 3エージェントクルー（研究者、作家、編集者）を実装、ブリーフを生成。
- 3つのCrewAI失敗モード：プロンプトブロート、マネージャーLLM税、脆いハンドオフをスポット。

## 問題

マルチエージェントフレームワークを採用するチームは同じ壁にぶつかる。「自律協力」はデモで聞こえる。その後顧客がバグを報告し、決定的なリプレイが必要。または財務は、実行ごとのLLMルートクルーのコストを聞く。またはオンコールは午前3時に、どのエージェントが止まったかを知る必要がある。

フリーフォームLLMルートクルーはそれらをクリーンに答えない。純粋DAGはすべてを答えるがブレーンストーミングエージェントが必要とする探索形を失う。

CrewAIの分割は取引について率直。自律、ロールベース、探索的作業のためのCrews。イベント駆動、コード所有、監査可能本番のためのFlows。同じフレームワーク、2形状、表面ごとに選ぶ。

## コンセプト

### 4つのプリミティブ

CrewAIの表面は小さい。これをメモ化すると、残りは設定だ。

- **Agent。** `role + goal + backstory + tools + (optional) llm`。backstoryはロード入っている。トーン、判断、エージェントが停止するとき形成。ツールはエージェントが呼べる関数（以下詳細）。
- **Task。** `description + expected_output + agent + (optional) context + (optional) output_pydantic`。再利用可能な仕事ユニット。`expected_output`は契約。`context`は上流タスクをリスト、その出力が渡される。`output_pydantic`は構造化形状を強制。
- **Crew。** コンテナ。`agents`リスト、`tasks`リスト、`process`、オプション`memory` + `verbose` + `manager_llm`設定を所有。
- **Process。** 実行戦略。Sequential、Hierarchical、Consensus（計画）。実行の形を選ぶ。

エージェントは他を直接見ない。タスクはエージェントをリファレンス。Crewはタスクをシーケンス。プロセスは次のタスク選びを決定。これが全体のメンタルモデル。

> **検証済みに対して** CrewAI 0.86（2026-05）。新しいバージョンはプロセスタイプをリネーム、マージしてもよい；特定形状に頼る前に[CrewAI Processes docs](https://docs.crewai.com/concepts/processes)をチェック。

### Sequential 対 Hierarchical 対 Consensus

- **Sequential。** タスク宣言順で実行。タスクNの出力はタスクN+1への`context`として利用可能。最下コスト。最予測可能。順序が固定のとき使用。
- **Hierarchical。** マネージャー Agent（別のLLM呼び出し）が専門家間でルート。CrewAIは`manager_llm`設定またはデフォルトからマネージャーをスポーン。マネージャーは各ラウンドで次のタスクを選び、拒否またはリルートできる。順序が本当に前の出力に依存するとき、4人以上の専門家がいるとき使用。
- **Consensus。** 計画済み、現在パブリックAPIで実装されていない。ドキュメントは投票ベースプロセスのための将来のために名前を予約。今日はそれに頼らない。

階層的はすべての専門家呼び出しの上にラウンドごとのLLM呼び出し（マネージャー）を追加。トークンコストは5ステップ実行で3倍にできる。ルーティングが出力に依存するときのみそれのために支払う。

### Crews 対 Flows

これはドキュメントが2026年で先頭をリードするフレーミング。

- **Crew。** LLM駆動自律性。フレームワークは実行時に形を選ぶ。良い：研究、ブレーンストーミング、初稿、パスが答えの部分のような場所。リプレイ難し。テスト難し。プロトタイプは安価。
- **Flow。** イベント駆動グラフ所有。`@start`はエントリをマーク。`@listen(topic)`は別ステップが話題を出すとき発火するステップをマーク。各ステップはプレーンPython（内部Crew.kickoff()呼び出しできる）。良い：本番。観測可能。テスト可能。決定的。

2026年ドキュメントの本番推奨：フローで始める。Flowステップ内からCrew.kickoff()呼び出しとして、自律がそのコストを獲得するときにCrewsを折り込む。フローは監査追跡を与える、Crewは探索を与える。合成し、選ばない。

### ツール統合

エージェントにツール与える3方法。最もシンプルなを選べ。

1. **`@tool`デコレータ。** 純関数はツールになる。シグネチャがスキーマ；docstringはLLMが見る説明。1回限りのヘルパーに最良。

   ```python
   from crewai.tools import tool

   @tool("Search the web")
   def search(query: str) -> str:
       """Return top results for the query."""
       return run_search(query)
   ```

2. **`BaseTool`サブクラス。** クラスベーストール、明示的argsスキーマ、非同期サポート、再試行。ツールが状態を持つ（クライアント、キャッシュ）またはネストされたargsが必要のとき使用。

   ```python
   from crewai.tools import BaseTool
   from pydantic import BaseModel

   class SearchArgs(BaseModel):
       query: str
       limit: int = 10

   class SearchTool(BaseTool):
       name = "web_search"
       description = "Search the web and return top results."
       args_schema = SearchArgs

       def _run(self, query: str, limit: int = 10) -> str:
           return self.client.search(query, limit=limit)
   ```

3. **組み込みツールキット。** CrewAIは1次アダプタを出荷：`SerperDevTool`、`FileReadTool`、`DirectoryReadTool`、`CodeInterpreterTool`、`RagTool`、`WebsiteSearchTool`。1インポートで接続。

構造化出力はPydanticを使用。Task上で`output_pydantic=MyModel`をパス。CrewAIはモデルに対してLLM応答を検証し、強制またはリトライのいずれか。厳密な`expected_output`文字列とペア。フリーテキスト出力は初稿に良い；構造化出力は下流Flowsが消費できるもの。

### メモリフック

CrewAIは既製の4メモリタイプを出荷。それらは合成：Crewは一度にすべての4つを有効化できる。

> **検証済みに対して** CrewAI 0.86（2026-05）。最近のリリースは統合`Memory`システムを通じてすべてをルート、これら4つのストアをラップ。下の概念モデルはまだ保持するが、パブリッククラス表面は新しいバージョンで単一`Memory`エントリポイントに崩壊してもよい；[CrewAI memory docs](https://docs.crewai.com/concepts/memory)で現在のAPIをチェック。

- **短期。** 単一実行内の会話バッファ。終了時にワイプ。
- **長期。** 実行間で永続化。ベクトルDB（デフォルトChroma、交換可能）に格納。現在のタスクへの相似度で検索。
- **エンティティ。** エンティティごとの事実。「顧客Xはエンタープライズプラン上だ。」エンティティでキーイング、相似度ではなく。実行間で生き残る。
- **コンテキスト。** アセンブリ時検索。エージェントがそれを必要のとき、メモリの関連を引く、プリロードされない。

Crew上で`memory=True`またはタイプごと設定で有効化。構成するレジでバックアップ（デフォルトOpenAI、ローカルに交換可能）。メモリはCrewAIがより薄いフレームワークに対して自分を獲得する場所の1つ；純LangGraphは各を自分でワイヤーする必要がある。

### CrewAIはいつフィット

- 3から6エージェント、名前ロールと協力ワークフロー。初稿、レビュー、計画、ブレーンストーミング。
- ルーティング、LLMの判断があるステップについてが価値の部分（Hierarchical）。
- どこでチーム内のグラフ定義を読むより`role + goal + backstory`を読む方が幸せ。

### CrewAIはいつフィットしない

- 厳密な順序付けを持つ決定的DAG。LangGraph（レッスン13）を使用。グラフ形状は正しい抽象化；CrewAIのロールフレーミングは摩擦。
- サブ秒レイテンシー予算。階層的はラウンドトリップを追加。Sequential時でもバックストーリーと前の出力が含まれるプロンプトをシリアル化。
- シングルエージェントループ。フレームワークをスキップ；エージェントループ（レッスン1）とツールレジストリは短い。

レッスン17（エージェントフレームワーク取引）はマトリックスでこれをレイアウト。短いバージョン：CrewAIは「協力ロールベース」コーナーに座る。

### 依存関係形状

LangChainから独立。Python 3.10から3.13。`uv`を使用。スターカウント：[crewAIInc/crewAI](https://github.com/crewAIInc/crewAI)を見よ（2026-05のスナップショット）。AWS Bedrock統合は文書化；ベンダーベンチマークはQAワークロード上でLangGraphに対して実質的加速をレポート、だがメソドロジー（データセット、ハードウェア、評価メトリック）は公開されない、フレームワークベンダー数を指向的のみとして扱う。

### このパターンがうまくいかない場合

- **バックストーリーからのプロンプトブロート。** エージェントごと2000ワードバックストーリーと5エージェント会員は最初のツール呼び出し前にコンテキスト予算を焼く。バックストーリーを200ワード下に保つ。エージェント全体で語句を再利用；ハウススタイルを5回繰り返さない。
- **マネージャーLLM トークン税。** 階層的プロセスは専門家呼び出しの前にマネージャーLLM呼び出しを追加。5タスク会員でこれは6 LLM呼び出しの代わり5、マネージャー呼び出しは完全なタスクリスト＋前の出力を運ぶ。出力に依存するルーティングが必要なとき以外Sequentialに交換。
- **脆いハンドオフ。** タスクNの`expected_output`は「アウトライン」。タスクN+1はそれを`context`として読み、3セクションをパース試みる。LLMが4を作成。下流エージェント即興。Task Nで`output_pydantic`で修正しタスクN+1はフリーテキストではなく型オブジェクトを読む。
- **Crew-as-prod。** フローラッパーなしで本番に出荷されたフリーフォームCrew。出力可変性は高い；リプレイは不可能；オンコールは悪い実行を良いものに対して差分できない。フローで何かをラップ。

## ビルドする

`code/main.py`は両方の形状のstdlibバージョンと3エージェント会員を実装。

形状:

- `Agent`、`Task`データクラスがCrewAIの表面を合致。
- `SequentialCrew.kickoff(inputs)`は宣言順でタスクを実行、出力を`context`としてスレッド。
- `HierarchicalCrew.kickoff(topic)`はマネージャーAgent選ぶそれぞれラウンドで次の専門家を追加、「done」で停止。
- `Flow`で`@start`と`@listen(topic)`デコレータ、小さなイベントループ、トレース。
- `tool(name)`デコレータがCrewAIの`@tool`形状をミラーリング。
- `Memory`で`short_term`、`long_term`、`entity`ストア；モック相似度はnumpy使用。
- モックLLM応答はロール＋入力プレフィックスでキーイングされた硬コード文字列。ネットワークなし。決定的。

具体的デモ：研究者、作家、編集者会員が「2026年エージェント工学」についてのブリーフを生成。研究者が（モック）ソースを引く。作家が初稿。編集者が締める。同じ会員は決定的な形を示すためにフローを通じて実行。

実行する:

```bash
python3 code/main.py
```

トレースは：sequential会員が`context`通じて出力をスレッド、階層的会員はマネージャーが選ぶ（研究者、作家、編集者、次に「done」）、フローは明示的話題で同じ3ステップを実行（`researched`、`drafted`、`edited`）、ツール呼び出しが`@tool`を通じてルート、長期メモリが2回のkickoff間で生き残る。

Crewトレースは液体；マネージャーは原則的にリオーダーできる。Flowトレースは固定。その選択がレッスン。

## 使う

- **CrewAI Flow**本番のため。フローが1つのステップ`Crew.kickoff()`を呼ぶときでも。フローは監査境界を与える。
- **CrewAI Crew（Sequential）** クリアな順序付け協力作業のため、特に初稿とレビューループ。
- **CrewAI Crew（Hierarchical）** ルーティングが出力に依存し4人以上の専門家がいるとき。
- **LangGraph**（レッスン13）明示的状態機械、永続再開、厳密な順序付けのため。
- **AutoGen v0.4**（レッスン14）アクターモデル並行性と障害分離のため。
- **OpenAI Agents SDK**（レッスン16）OpenAI1次製品にハンドオフとガードレール付き。
- **Claude Agent SDK**（レッスン17）Claude1次製品にサブエージェントとセッションストア付き。

## 出荷

`outputs/skill-crew-or-flow.md`はタスクのためCrewとFlowを選び、最小実装をスキャフォールド。Crew-without-backstory、Flow-without-explicit-topics、3人未満の専門家のHierarchicalな厳密拒否。

## 落とし穴

- **バックストーリーとしてのフレーバー。** 出力を形成。エージェントごと3変種をテスト；可変性は本物。1つを選び、凍らせ。
- **`expected_output`をスキップ。** タスクごとの契約なし、下流タスクはLLMが生じたものを選ぶ。会員は実行；監査は失敗。
- **メモリ常時オン。** 長期はすべての実行を書き込む。ベクトルDBは成長。検索はノイズ多くなる。事実が永続的なタスクへの書き込みをスコープ。
- **マネージャープロンプトドリフト。** 階層的のマネージャープロンプトは暗黙。ルーティングが奇妙になったら、冗長モードで吐き出し、読む。
- **Crewsのツール副作用。** Crewは予期されたより多くの回数ツール呼び出しできる。POST、DELETE、支払いはフロースステップに属し、決してCrewツール。

## 演習

1. Sequential会員をフローに変換。可変性が落ちる接触地点をカウント。読み易さが落ちたどこかをノート。
2. エンティティメモリをクルーに追加：顧客についての事実はkickoff間で永続化。検索が正しいエンティティを引く検証。
3. マネージャーが編集者へのルートを拒否するまでHierarchical プロセスを実装、作家の出力は最低3段落を持つ。リトライをトレース。
4. `BaseTool`サブクラスをワイヤー（モック）ウェブ検索のため。`@tool`デコレータバージョンに対してトレース形状を比較。
5. エディタータスク`output_pydantic=Brief`を追加、`Brief`が`title`、`summary`、`sections`を持つ。作家タスク出力がmalformed JSONを1度；CrewAIの再試行振る舞いをトレースで検証。
6. CrewAIのドキュメント入門を読む。おもちゃを本当の`crewai` APIにポート。stdlibバージョンはどの保証をスキップしたか？
7. AgentOpsまたはLangfuse（レッスン24）をリアル実行にワイヤー。stdlibバージョンでいくつのトレースをミスしたか？

## キー用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| エージェント | 「ペルソナ」 | ロール＋ゴール＋バックストーリー＋ツール |
| タスク | 「仕事ユニット」 | 説明＋予期出力＋アサイニー＋オプション構造化出力 |
| 会員 | 「エージェントチーム」 | エージェント＋タスク＋プロセスのコンテナ |
| プロセス | 「実行戦略」 | Sequential / Hierarchical / Consensus（計画） |
| フロー | 「決定的ワークフロー」 | イベント駆動、コード所有、テスト可能 |
| バックストーリー | 「ペルソナプロンプト」 | エージェントのためのトーンと判断形状 |
| `@tool` | 「関数ツール」 | デコレータ、エージェントが呼べるツールに関数を回す |
| `BaseTool` | 「クラスツール」 | args スキーマ、再試行、非同期サポート付きクラスベース |
| エンティティメモリ | 「エンティティごとの事実」 | 顧客 / アカウント / 問題にスコープされたメモリ |
| 長期メモリ | 「クロスラン メモリ」 | ベクトル・バックアップメモリが kickoff間で生き残る |
| コンテキストメモリ | 「ジャストインタイム検索」 | エージェントがそれを必要のとき引かれたメモリ |
| マネージャーLLM | 「ルータエージェント」 | 次タスク選ぶHierarchicalプロセスの追加LLM |
| `expected_output` | 「タスク契約」 | エージェント（と監査）に返す形を伝える文字列 |
