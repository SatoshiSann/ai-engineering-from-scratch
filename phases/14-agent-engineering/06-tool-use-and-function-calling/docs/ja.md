# ツール使用と関数呼び出し

> Toolformer (Schick et al., 2023)は自己教師付きツール注釈を開始しました。Berkeley Function Calling Leaderboard V4 (Patil et al., 2025)は2026年のバー：40%エージェンティック、30%マルチターン、10%ライブ、10%非ライブ、10%幻覚を設定します。シングルターンは解決しました。メモリ、動的意思決定、長期的ツールチェーンはそうではありません。

**タイプ:** ビルド
**言語:** Python (stdlib)
**前提条件:** Phase 14 · 01 (Agent Loop)、Phase 13 · 01 (Function Calling Deep Dive)
**所要時間:** 約60分

## 学習目標

- Toolformerの自己教師付き訓練信号を説明：ツール注釈は次トークン損失を削減する場合のみ保つ。
- BFCL V4の5つの評価カテゴリを名前付けし、各々が測定するものを名前付けする。
- スキーマ検証、引数強制、実行サンドボックス付きで標準ライブラリツールレジストリを実装する。
- 3つの2026年オープンな問題を診断：長期的ツールチェーン、動的意思決定、メモリ。

## 問題

初期ツール使用は質問しました：モデルは正しい関数呼び出しを予測できますか？モダンツール使用は質問します：モデルは40ステップ、メモリ、部分可観測性、ツール失敗からの回復、存在しないツール幻覚なしにわたるツールをチェーンできますか？

Toolformerはベースラインを確立しました：モデルは自己教師付きでツールをいつ呼び出すかを学べます。BFCL V4は2026年評価ターゲットを定義します。それらの間のギャップは本番エージェントが住んでいるスペースです。

## コンセプト

### Toolformer (Schick et al., NeurIPS 2023)

アイデア：候補API呼び出しで独自の事前訓練コーパスにモデル注釈を付けさせます。各候補について、実行します。ツール結果を含めることが次のトークンの損失を減らす場合のみ注釈を保ちます。フィルタされたコーパスでファインチューニングします。

対象ツール：計算機、QAシステム、検索エンジン、翻訳、カレンダー。自己教師付き信号は純粋にツールがテキストを予測するのを助けるかどうかについてです — 人間のラベルなし。

スケール結果：ツール使用はスケールで出現します。より小さいモデルはツール注釈で傷つき、より大きいモデルは利得を得ます。これは2026年フロンティアモデルが組み込まれたツール使用を強く持っている理由で、ほとんどの7Bモデルは信頼できるにはツール使用ファインチューニングの明示的な必要があります。

### Berkeley Function Calling Leaderboard V4 (Patil et al., ICML 2025)

BFCLは2026年の事実上の評価です。V4構成：

- **エージェンティック(40%)** — 完全なエージェントトラジェクトリ：メモリ、マルチターン、動的決定。
- **マルチターン(30%)** — ツールチェーンでのインタラクティブな会話。
- **ライブ(10%)** — ユーザー提出のリアルプロンプト(より難しい分布)。
- **非ライブ(10%)** — 合成テストケース。
- **幻覚(10%)** — ツール呼び出しがない場合を検出。

V3はステート・ベース評価を導入しました：ツールシーケンスの後、APIの実際のステート(例えば「ファイルは作成されたのか？」)をツール呼び出しのASTと比較する代わりにチェックします。V4はウェブサーチ、メモリ、フォーマット感度カテゴリを追加しました。

重大な2026年の発見：シングルターン関数呼び出しは近解決です。失敗はメモリ(ターン横切りにコンテキストを運ぶこと)、動的意思決定(先行結果に基づくツール選択)、長期的チェーン(20+ステップ後のドリフト)、幻覚検出(ツールが合わせる場合に拒否呼び出し)に集中します。

### ツールスキーマ

すべてのプロバイダはスキーマを持っています。詳細では異なりますが、同じ形を共有します：

```
name: string
description: string (what it does, when to use it)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropicは`input_schema`を直接使用します。OpenAIは`function.parameters`を使用します。両方ともJSON Schemaを受け入れます。説明はロード・ベアリングです — モデルは正しいツールを選択するために読みます。悪いツール説明は間違ったツール・ピック失敗の＃1根本原因です。

### 引数検証

ツール呼び出しに信頼なし。検証：

1. **型強制。** モデルはスキーマがintを言う場所に文字列「5」を返すかもしれません。曖昧でない場合は強制；そうでない場合は拒否。
2. **列挙検証。** スキーマが`status in {"open", "closed"}`を言い、モデルが`"in_progress"`を発行する場合、説明的なエラーで拒否。
3. **必須フィールド。** 必須フィールド欠落 -> クラッシュ、直接エラー観測をモデルに戻す。
4. **フォーマット検証。** 日付、メール、URL — コンクリートパーサー、regex以外でで検証。

すべての検証失敗は、モデルが正しい形で再試行できるように構造化された観測を返すべきです。

### 並行ツール呼び出し

モダンプロバイダは1つのアシスタントターンで並行ツール呼び出しをサポートします。ループ：

1. モデルは異なる`tool_use_id`で3つのツール呼び出しを発行します。
2. ランタイムは(独立に並行できる)実行します。
3. 各結果は`tool_use_id`で相関してきた`tool_result`ブロックとして戻ります。

エンジニアリングルール：相関IDをロード・ベアリングとして扱う。それらをスワップ間違ったツールから間違った結果へのルーティングを得ます。

### サンドボックス

ツール実行はサンドボックス境界です。レッスン09で詳細を参照してください。短いバージョン：すべてのツールは読み書き表面、ネットワーク・アクセス、タイムアウト、メモリ・キャップを指定すべきです。汎用`run_shell(cmd)`は赤フラグです；特定`git_status()`はより安全です。

## ビルド

`code/main.py`は本番形ツールレジストリを実装します：

- JSON Schemaサブセット検証器(標準ライブラリのみ)。
- 説明、入力スキーマ、タイムアウト、実行器でのツール登録。
- 引数強制と列挙検証。
- 相関IDで並行ツール・ディスパッチ。
- 構造化された文字列としてのエラー観測。

実行：

```
python3 code/main.py
```

トレースは1つのターンで3つのツールを呼び出すミニエージェント、モデルが行動できる説明的なエラーで拒否される1つの意図的に悪形式の呼び出しを示しています。

## 使う

すべてのプロバイダは独自のツールスキーマを持っています — Anthropic、OpenAI、Gemini、Bedrock。マルチプロバイダが必要な場合は翻訳層を使用します(OpenAI Agents SDK、Vercel AI SDK、LangChainツール・アダプター)。BFCLは参照ベンチマークです — ツール使用が製品に対して中心である場合、出荷前に対してエージェント上で実行します。

## 出荷

`outputs/skill-tool-registry.md`は与えられたタスク・ドメインのツール・カタログ、スキーマ、レジストリを生成します。説明品質チェック(各ツール説明はモデルにいつそれを使用するかを伝えます？)を含みます。

## 演習

1. モデルがしてもいいときに明示的に他のツールを使用することを拒否させる「ノー・オップ」ツールを追加します。BFCL様の幻覚テストで測定します。
2. int-as-stringとfloat-as-stringのための引数強制を実装します。強制はいつ実の虫を隠し始めますか？
3. ツールごとのタイムアウトと回路ブレーカー(3つの連続失敗後60秒間ツールを拒否)を追加します。これはモデルが回復する方法について何を変えますか？
4. BFCL V4説明を読みます。1つのカテゴリ(例えば「マルチターン」)を選んで、10例プロンプトをエージェント経由で実行します。パスレートを報告します。
5. 標準ライブラリ検証器をPydanticまたはZodにポートします。PydanticまたはZodは何をおもちゃが見逃したのを捕捉しましたか？
