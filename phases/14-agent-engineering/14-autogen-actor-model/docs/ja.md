# AutoGen v0.4：アクターモデルとエージェントフレームワーク

> AutoGen v0.4（マイクロソフト研究、2025年1月）はアクターモデルの周辺でエージェントオーケストレーションを再設計。非同期メッセージ交換、イベント駆動エージェント、障害分離、自然な並行性。フレームワークは現在保守モード中、一方マイクロソフト エージェント フレームワーク（パブリックプレビュー2025年10月）は後継品になる。

**タイプ:** 学習 + ビルド
**言語:** Python（stdlib）
**前提条件:** Phase 14 · 01（エージェントループ）、Phase 14 · 12（ワークフローパターン）
**所要時間:** 約75分

## 学習目標

- アクターモデルを説明できる：エージェントとしてのアクター、IPC唯一の方法としてのメッセージ、アクターごとの障害分離。
- AutoGen v0.4の3つのAPIレイヤーを名前付けできる—Core、AgentChat、Extensions—と各の用途。
- メッセージ配信処理から分離することが障害分離と自然な並行性を与える理由を説明できる。
- stdlibでアクターランタイムをPythonで実装し、2エージェントコードレビューフローにポートできる。

## 問題

ほとんどのエージェントフレームワークは同期的：1つのエージェントが生産、1つが消費、呼び出しスタック内。失敗はスタックをクラッシュさせる。並行性はボルトオン。分散は書き直しが必要。

AutoGen v0.4の答え：アクターモデル。各エージェントはプライベートインボックス付きのアクター。メッセージが唯一の相互作用。ランタイムは配信から処理を分離。障害は1つのアクターに分離。並行性はネイティブ。分散は異なるトランスポートのみ。

## コンセプト

### アクター

アクターは以下を持つ:

- プライベート状態（外から決して直接触らない）。
- インボックス（メッセージキュー）。
- ハンドラ：`receive(message) -> effects`、効果は「返信」「別のアクターに送信」「新しいアクターをスポーン」「状態を更新」「自分を停止」。

2つのアクターはメモリを共有できない。メッセージのみを送信できる。

### AutoGen v0.4の3つのAPIレイヤー

1. **Core。** 低レベルアクターフレームワーク。`AgentRuntime`、`Agent`、`Message`、`Topic`。非同期メッセージ交換、イベント駆動。
2. **AgentChat。** タスク駆動の高レベルAPI（v0.2の`ConversableAgent`の置換）。`AssistantAgent`、`UserProxyAgent`、`RoundRobinGroupChat`、`SelectorGroupChat`。
3. **Extensions。** 統合—OpenAI、Anthropic、Azure、ツール、メモリ。

### 分離がなぜ重要か

v0.2モデルで、`agent_a.chat(agent_b)`呼び出しは同期的にagent_bが復帰するまでagent_aをブロック。v0.4で、`send(agent_b, msg)`はメッセージをagent_bのインボックスに置きすぐ復帰。ランタイムは後から配信。3つの結果：

- **障害分離。** エージェントBがクラッシュするとエージェントAはクラッシュしない—ランタイムはBのハンドラで障害をキャッチし、何をするか決定（ログ、再試行、デッドレター）。
- **自然な並行性。** 多くのメッセージが飛行中；アクターは並行性を通じてインボックスを処理。
- **分散準備。** インボックス＋トランスポートは同じ抽象化かプロセス内アクターまたは別のホストかもかかわらず。

### 位相

- **RoundRobinGroupChat。** エージェントは固定ローテーションで順番に。
- **SelectorGroupChat。** セレクタエージェントは会話コンテキストに基づいて誰が次かを選ぶ。
- **Magentic-One。** ウェブブラウジング、コード実行、ファイルハンドリング向けの参照マルチエージェントチーム。AgentChatの上に構築。

### 観測可能性

OpenTelemetryサポートは組み込み。すべてのメッセージはスパンを出力；ツール呼び出しは2026年OTel GenAIセマンティック慣例（レッスン23）ごとに`gen_ai.*`属性を運ぶ。

### ステータス：保守モード

早期2026：AutoGen v0.7.xは研究とプロトタイピング向けに安定。マイクロソフトはマイクロソフト エージェント フレームワークへのアクティブ開発をシフト（パブリックプレビュー2025年10月1日；GA Q1 2026末を目指す）。AutoGenパターンはクリーンにポートフォーワード—アクターモデルは永続的アイデア。

## ビルドする

`code/main.py`はstdlibアクターランタイムを実装：

- `Message` — `sender`、`recipient`、`topic`、`body`を持つ型付けペイロード。
- `Actor` — `receive(message, runtime)`を持つ抽象。
- `Runtime` — 共有キュー、配信、障害分離を持つイベントループ。
- 2エージェントデモ：`ReviewerAgent`がコードをレビュー、`ChecklistAgent`がチェックリストを実行；合意に収束するまでメッセージを交換。

実行する:

```
python3 code/main.py
```

トレースはメッセージ配信、1つのアクター内の変調失敗が別のアクターをクラッシュしない、共有評決への収束を示す。

## 使う

- **AutoGen v0.4/v0.7**（保守）— 研究、プロトタイピング、マルチエージェントパターン向けに安定。
- **マイクロソフト エージェント フレームワーク**（パブリックプレビュー）— 前進パス；リフレッシュされたAPIでのアクターモデルのアイデア同じ。
- **LangGraph swarm位相**（レッスン13）— 共有ツールハンドオフ経由の同様パターン。
- **カスタムアクターランタイム** — 特定トランスポート（NATS、RabbitMQ、gRPC）が必要なとき。

## 出荷

`outputs/skill-actor-runtime.md`は与えられたマルチエージェントタスクのための最小アクターランタイム＋チームテンプレート（RoundRobinまたはSelector）を生成。

## 演習

1. デッドレターキュー追加：ハンドラが上げるとき、人間の検査のための失敗メッセージをパーク。おもちゃではDLQはどのくらい頻繁にヒットされるか？
2. `SelectorGroupChat`を実装：セレクタエージェントは会話状態に基づいて次を処理するメッセージを選ぶ。
3. 分散トランスポート追加：イン・プロセスキューをJSON超HTTPサーバーに交換し、アクターは別プロセスで走実できる。
4. OTelスパン対メッセージ（またはノーオプスタンドイン）をワイヤー。`gen_ai.agent.name`、`gen_ai.operation.name`をレッスン23あたりで出力。
5. AutoGen v0.4のアーキテクチャ投稿を読む。おもちゃを本当の`autogen_core` APIにポート。本番で重要なものをスキップしたか？

## キー用語

| 用語 | 人々が言うこと | 実際に意味すること |
|------|----------------|------------------------|
| アクター | 「エージェント」 | プライベート状態＋インボックス＋ハンドラ；共有メモリなし |
| メッセージ | 「イベント」 | 型付きペイロード；アクターが相互作用する唯一の方法 |
| インボックス | 「メールボックス」 | 保留中メッセージのエージェントごとキュー |
| ランタイム | 「エージェントホスト」 | メッセージをルート化し障害を分離するイベントループ |
| トピック | 「チャネル」 | アクター間の名前付きパブリッシュ-サブスクライブルート |
| 障害分離 | 「それをクラッシュさせる」 | 1つのアクター失敗は他をクラッシュさせない |
| RoundRobinGroupChat | 「固定ローテーションチーム」 | エージェント順序で順番に取る |
| SelectorGroupChat | 「コンテキストルートチーム」 | セレクタが誰が次かを選ぶ |
| Magentic-One | 「参照チーム」 | ウェブ＋コード＋ファイル向けのマルチエージェント分隊 |
