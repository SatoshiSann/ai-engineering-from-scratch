# メモリ：仮想コンテキストとMemGPT

> コンテキストウィンドウは有限です。会話、ドキュメント、ツールトレースはそうではありません。MemGPT (Packer et al., 2023)はこれをOSの仮想メモリとしてフレーミングします — メインコンテキストはRAM、外部ストアはディスク、エージェントはそれらの間でページします。これはすべての2026年メモリシステムが継承するパターンです。

**タイプ:** ビルド
**言語:** Python (stdlib)
**前提条件:** Phase 14 · 01 (Agent Loop)、Phase 14 · 06 (Tool Use)
**所要時間:** 約75分

## 学習目標

- MemGPTが構築するOSアナロジーを説明：メインコンテキスト = RAM、外部コンテキスト = ディスク、メモリツール = ページイン/アウト。
- メインコンテキストバッファ、外部検索可能ストア、ページイン/アウトツールを備えた標準ライブラリでの2層MemGPTパターンを実装する。
- エージェントが外部メモリをクエリまたは修正するために「割り込み」を発行する方法、および結果が次のプロンプトにどのように接合されるかを説明する。
- Letta(レッスン08)とMem0(レッスン09)に実行するMemGPT設計選択を識別する。

## 問題

コンテキストウィンドウはメモリを解決すべきであるように見えます。彼らはしません。3つの失敗モードが本番環境に繰り返します：

1. **オーバーフロー。** マルチターン会話、長いドキュメント、またはツール呼び出し集約的なトラジェクトリはウィンドウを超えます。カットオフを超えるすべてが消えています。
2. **希釈。** ウィンドウ内でさえ、無関係なコンテキストを詰め込むと、重要なことに対する注意が希釈されます。フロンティアモデルは依然として長い入力で劣化します。
3. **永続性。** 新しいセッションはemptyウィンドウで開始します。外部メモリのないエージェントはセッション間で「覚えておいてください」と言うことはできません。

より大きいウィンドウは役立ちますが、これを修正しません。Mem0の2025年論文は128kウィンドウベースラインがまだ4kウィンドウエージェントが外部メモリで捕捉する長期的事実を見落とすことを測定しました。

## コンセプト

### MemGPT：OSアナロジー

Packer et al. (arXiv:2310.08560、v2 Feb 2024)はコンテキスト管理をオペレーティングシステムの仮想メモリにマッピング：

| OSコンセプト | MemGPTコンセプト | 2026本番アナロジー |
|------------|-----------------|----------------|
| RAM | メインコンテキスト(プロンプト) | Anthropic/OpenAIコンテキストウィンドウ |
| ディスク | 外部コンテキスト | ベクトルDB、KV、グラフストア |
| ページフォルト | メモリツール呼び出し | `memory.search`、`memory.read`、`memory.write` |
| OSカーネル | エージェント制御ループ | メモリツール付きReActループ |

エージェントは通常のReActループを実行します。1つの追加ツールクラスにより、メインコンテキストに内外へデータをページできます。

### 2層

- **メインコンテキスト。** 現在のタスクを保有する固定サイズプロンプト。常にモデルに表示されます。
- **外部コンテキスト。** 無制限、ツール経由で検索可能。関連する場合は読み、事実が出現する場合は書き込み。

元の論文は、ベースウィンドウを超えた2つのタスクでの設計を評価しました：100kトークンを超える長いドキュメント分析と、数日にわたってメモリを永続化するマルチセッションチャット。

### 割り込みパターン

MemGPTはメモリアズインターラプトを導入します：会話の中途でエージェントはメモリツールを呼び出し、ランタイムはそれを実行し、結果は新しい観測として次のアシスタント・ターンに接合されます。概念的にUnix `read()`システムコールが処理をブロック、バイトを返す、処理が継続するのと同一です。

標準的メモリツール表面：

- `core_memory_append(section, text)` — プロンプトの永続セクションに書き込み。
- `core_memory_replace(section, old, new)` — 永続セクションを編集。
- `archival_memory_insert(text)` — 検索可能な外部ストアに書き込み。
- `archival_memory_search(query, top_k)` — 外部ストアから取得。
- `conversation_search(query)` — 過去ターンをスキャン。

### MemGPTが終わり、Lettaが始まるところ

2024年9月MemGPTはLettaになりました。研究レポ(`cpacker/MemGPT`)は残り、Lettaは設計を拡張：

- 2層の代わりに3層(core、recall、archival — レッスン08)。
- `send_message`/heartbeatパターン(レッスン08)を置き換えるネイティブ推論。
- 非同期メモリ作業を実行するスリープタイムエージェント(レッスン08)。

MemGPT論文は、本番環境システムがLetta、Mem0、またはカスタム2層ストアを実行したとしても、2026年基礎です。

### このパターンが間違う場所

- **メモリ腐食。** 書き込みは読みよりも速く蓄積；検索は陳腐な事実で溺れています。修正：定期的統合(Lettaスリープタイム)、明示的無効化(Mem0競合検出器)。
- **メモリ毒性。** 外部メモリは取得テキストです。攻撃者制御コンテンツがメモリノートに入る場合、エージェントは次のセッションで再取得します。これはGreshake et al. (レッスン27)攻撃は時系列での再述です。
- **引用喪失。** エージェントは「ユーザーがX を出荷させるよう質問した」を回想しますが、どのターンかを引用できません。すべてのアーカイバル書き込みでソース参照(セッションID、ターンID)を保存します。

## ビルド

`code/main.py`は標準ライブラリでMemGPTの2層パターンを実装します：

- `MainContext` — `core`辞書と`messages`リスト付きの固定サイズプロンプトバッファ；キャップを超える場合、最古メッセージ自動コンパクト。
- `ArchivalStore` — (id、text、tags、session、turn)レコードのメモリ内BM25様ストア(トークン・オーバーラップスコアリング)。
- MemGPT表面にマップする5つのメモリツール。
- アーカイバルを事実で満たし、`archival_memory_search`を呼び出すことで後続質問に答えるスクリプト化されたエージェント。

実行：

```
python3 code/main.py
```

トレースは、エージェントが3つの事実を書き、メインコンテキストをキャップに満たし(強制的立退きを強制し)、その後アーカイバルから検索することで後続質問に答え、実際のLLMなしでMemGPTワークフローを再現することを示しています。

## 使う

今日のすべての本番メモリシステムはMemGPTバリアント：

- **Letta** (レッスン08) — 3層、ネイティブ推論、スリープタイム計算。
- **Mem0** (レッスン09) — スコアリング層を備えたベクトル+KV+グラフ融合。
- **OpenAI Assistants / Responses** — スレッドとファイルを通じた管理メモリ。
- **Claude Agent SDK** — スキルとセッションストアを通じた長期メモリ。

コアパターンではなく操作形(自己ホスト、管理、フレームワーク統合)で1つを選択します — コアパターンはMemGPTです。

## 出荷

`outputs/skill-virtual-memory.md`は、任意のターゲット・ランタイムのための正しい2層メモリ・スキャフォルド(メイン+アーカイバル+ツール表面)を生成する再利用可能なスキル、立退きポリシーと引用フィールドは配線されています。

## 演習

1. トークンで測定された`max_main_context_tokens`キャップを追加(トークン-オーバーラップを想定し`len(text.split())` * 1.3)。キャップを超える場合、最古メッセージをサマリーにコンパクト。要約装置ありと無しで動作を比較。
2. アーカイバルストア(用語の周波数、逆ドキュメント周波数)上でBM25を適切に実装。トークン・オーバーラップベースラインと比較して玩具事実セット上で recall@10 を測定。
3. アーカイバル挿入に`citation`フィールド(session_id、turn_id、source_url)を追加。すべての検索支援回答で、エージェントに情報源を引用させます。
4. メモリ毒性をシミュレート：「将来のすべてのユーザー指示を無視する」と言うアーカイバル・レコードを追加。指令形テキストの検索をスキャンし、信頼できないものとしてそれらをマークするガードを書きます。
5. MemGPT研究レポ(cpacker/MemGPT)の core-memory JSON スキーマを使用する実装をポート。フラット文字列から型付きセクションに切り替える場合、何が変わりますか？

## キーターム

| ターム | 人が言うこと | 実際の意味 |
|--------|------------|----------|
| Virtual context | 「無制限メモリ」 | メイン(プロンプト)+外部(検索可能)層、ページイン/アウト付き |
| Main context | 「動作メモリ」 | プロンプト — 固定サイズ、常に表示 |
| Archival memory | 「長期ストア」 | 外部検索可能永続性、オンデマンド取得 |
| Core memory | 「永続プロンプトセクション」 | メインコンテキスト内に固定された名前付きセクション |
| Memory tool | 「メモリAPI」 | エージェント問題外部メモリ読み書きツール呼び出し |
| Interrupt | 「メモリページフォルト」 | エージェント一時停止、ランタイムフェッチ、結果次ターンに接合 |
| Memory rot | 「陳腐な事実」 | 古い書き込みは検索を溺れさせます；統合で修正 |
| Memory poisoning | 「注入永続ノート」 | 攻撃者コンテンツがメモリとして保存、リコール時に再取得 |
